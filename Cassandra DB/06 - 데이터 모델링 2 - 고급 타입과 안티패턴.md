[[05 - 데이터 모델링 1 - Query First]]에서 우리는 "쿼리가 테이블을 정의한다"는 원칙과, 파티션 키(partition key)·클러스터링 키(clustering key)가 데이터의 물리적 배치를 결정한다는 사실을 배웠다. 거기까지는 `text`, `int`, `uuid`, `timestamp` 같은 스칼라(scalar) 타입만으로 모델링했다. 하지만 CQL은 훨씬 풍부한 무기고를 제공한다. 컬렉션(list/set/map), 사용자 정의 타입(UDT), 튜플(tuple), 카운터(counter), 정적 컬럼(static column), 그리고 인덱싱 계열인 보조 인덱스(secondary index)·SAI·Materialized View까지.

문제는 이 무기들이 전부 양날의 검이라는 데 있다. RDB에서 옮겨온 엔지니어는 이들을 "익숙한 기능의 Cassandra판"으로 착각하기 쉽다. JSON 컬럼처럼 컬렉션을 쓰고, 트리거처럼 Materialized View를 쓰고, `SELECT COUNT`처럼 카운터를 쓴다. 그러다 운영 6개월 차에 5초짜리 읽기, 죽지 않는 톰스톤(tombstone), 한 노드만 불타는 핫스팟(hotspot)을 마주한다.

이 장이 답할 핵심 질문은 다음과 같다.

- 컬렉션은 디스크에서 **어떤 모양**으로 저장되는가? 왜 "전체를 읽어야" 하고, 왜 list는 동시성에 취약한가?
- `frozen`은 정확히 무엇을 얼리는가? non-frozen과 무엇이 다른가?
- UDT·tuple·static column은 각각 어떤 모델링 빈자리를 메우는가?
- 분산 카운터는 왜 **비멱등(non-idempotent)**이고, 타임아웃 재시도가 왜 위험한가?
- 보조 인덱스는 왜 "숨은 테이블"이며, 왜 멀티 노드에서 안티패턴인가?
- SASI는 왜 죽고 SAI가 그 자리를 차지했는가? SAI도 못 고치는 한계는?
- Materialized View의 자동 비정규화는 어디까지 믿을 수 있는가?
- 그리고 실전에서 반드시 피해야 할 안티패턴들의 **증상-원인-대안**은?

## 컬렉션 타입: list / set / map은 "셀의 묶음"이다

CQL에는 세 가지 컬렉션이 있다.

```cql
CREATE TABLE user (
    user_id     uuid PRIMARY KEY,
    emails      set<text>,            -- 중복 없음, 정렬됨
    login_log   list<timestamp>,      -- 순서 있음, 중복 허용
    attributes  map<text, text>       -- 키-값
);
```

겉보기에는 "한 컬럼에 여러 값을 담는 편의 기능"이지만 진짜 이해는 **이것들이 디스크에서 어떻게 표현되는가**에서 시작한다. 결론부터 말하면, **non-frozen 컬렉션의 각 원소는 하나의 셀(cell)로 저장된다.** 셀은 Cassandra 스토리지 엔진의 최소 단위로, `(이름, 값, 타임스탬프, [TTL])`의 묶음이다([[08 - Storage Engine 내부]]에서 상세). 스칼라 컬럼 `name text`는 행(row)당 셀 하나지만 `emails set<text>`는 원소 개수만큼 셀이 생긴다.

### 내부 표현: 컬렉션은 행 안의 "미니 클러스터링"이다

트릭은 단순하다. **컬렉션의 원소를 셀 이름(cell name) 안에 인코딩한다.** 행 안에 또 하나의 정렬된 키-값 구조(사실상 클러스터링 컬럼과 같은 메커니즘)를 만드는 셈이다.

```text
세 가지 컬렉션의 셀 표현 (논리적 모식)

set<text> emails = {'a@x.com', 'b@x.com'}
  ┌──────────────────────────────────────────────┐
  │ cell name = (emails, 'a@x.com')   value = ∅   │  ← 원소가 "이름"에, 값은 빈 바이트
  │ cell name = (emails, 'b@x.com')   value = ∅   │
  └──────────────────────────────────────────────┘

map<text,text> attributes = {'tier':'gold', 'city':'seoul'}
  ┌──────────────────────────────────────────────┐
  │ cell name = (attributes, 'city')  value='seoul'│  ← 맵 키가 "이름"에, 맵 값이 "값"에
  │ cell name = (attributes, 'tier')  value='gold' │
  └──────────────────────────────────────────────┘

list<text> login_log = ['x', 'y']
  ┌──────────────────────────────────────────────┐
  │ cell name = (login_log, <timeuuid_1>) value='x'│ ← 순서 보존용 timeuuid가 "이름"에
  │ cell name = (login_log, <timeuuid_2>) value='y'│
  └──────────────────────────────────────────────┘
```

여기서 세 컬렉션의 설계 의도가 그대로 드러난다.

- **set**: 원소 자체를 셀 이름에 넣으므로 자동으로 **중복이 제거**되고(같은 이름의 셀은 하나), 셀 이름의 바이트 정렬 순서로 **정렬**된다. 값은 비어 있다(원소가 이름에 다 있으니까).
- **map**: 맵의 키를 셀 이름에, 맵의 값을 셀 값에 넣는다. 그래서 같은 키는 덮어쓰여지고, 키 정렬 순서로 저장된다.
- **list**: 순서가 의미 있으므로 셀 이름에 **timeuuid**(시간 기반 UUID)를 부여한다. 그래서 같은 값('x')을 두 번 넣어도 서로 다른 셀 이름을 가져 **중복이 허용**되고, 삽입 순서가 보존된다. 바로 이 timeuuid 메커니즘이 뒤에서 다룰 list의 동시성 함정의 근원이다.

> **RDB였다면 vs Cassandra**: RDB의 배열/JSON 컬럼은 하나의 값으로 통째로 저장되어 한 원소를 바꾸려면 컬럼 전체를 다시 써야 한다. Cassandra의 **non-frozen** 컬렉션은 정반대다. 원소마다 독립된 셀이라 `attributes['city'] = 'busan'` 한 줄로 그 셀 하나만 갱신한다. 다른 원소는 건드리지 않는다. 이게 non-frozen의 존재 이유다.

> **숨은 비용 — 컬렉션 통째 재대입은 톰스톤을 만든다**: 부분 갱신(`+`/`-`)과 달리, non-frozen 컬렉션을 통째로 다시 대입하면(`SET tags = {'a','b'}`, 또는 `INSERT`로 컬렉션 컬럼을 지정) Cassandra는 새 셀을 쓰기 전에 먼저 **그 컬렉션 범위 전체를 덮는 톰스톤(range tombstone)**을 기록한다. "예전 원소를 확실히 비운다"는 의미상 불가피한 동작이지만 컬렉션을 자주 통째로 덮어쓰는 패턴은 톰스톤을 양산해 읽기·컴팩션을 무겁게 만든다. 부분 변경이 목적이라면 재대입이 아니라 반드시 `+`/`-` 연산을 써라. (톰스톤이 왜 비싼지는 이 장 뒤의 큐 패턴, 그리고 [[08 - Storage Engine 내부]]·[[09 - Compaction 전략]]에서 다룬다.)

### 한계 1: 읽을 때는 "전부" 읽는다

원소가 셀로 흩어져 있다는 것은 갱신엔 유리하지만 읽기엔 불리하다. **컬렉션 안에서 페이징은 없다.** `SELECT emails FROM user WHERE user_id = ?`를 하면 그 행의 emails에 속한 **모든 셀**이 한꺼번에 메모리로 올라와 직렬화되어 클라이언트로 간다. "emails의 처음 10개만" 같은 서버 측 슬라이싱은 불가능하다(클러스터링 컬럼이라면 `LIMIT`·범위 슬라이스가 되지만, 컬렉션 원소는 그 대상이 아니다). 예외가 하나 있다. non-frozen **map**은 키가 셀 이름에 정렬돼 있어 `SELECT attributes['city']`처럼 단일 키를 콕 집어 읽을 수 있고, frozen 컬렉션은 통째로 한 셀이라 단일 역직렬화로 끝난다. 그래도 "임의의 N개를 페이징"하는 것은 어느 경우든 불가능하다.

그래서 컬렉션이 수천~수만 개로 커지면, 그 행을 읽을 때마다 그 큰 덩어리가 통째로 직렬화된다. 읽기 지연이 튀고, 코디네이터(coordinator) 힙(heap)에 부담이 가며, 네트워크 페이로드가 비대해진다.

### 한계 2: 권장 원소 상한과 가드레일(guardrail)

역사적으로 native protocol v2 시절 컬렉션은 **원소 65,535개**라는 하드 리밋이 있었다(원소 개수를 16비트로 인코딩했기 때문). 이후 프로토콜에서 그 한계는 사라졌지만 **그게 "많이 담아도 된다"는 뜻은 아니다.** 실무 가이드는 보수적이다.

- 원소 수: 수백 개 이내가 안전, 길어도 **1,000개 미만**을 권장.
- 직렬화 크기: 한 컬렉션이 수십 KB를 넘기 시작하면 위험 신호.

Cassandra **4.1**부터 들어온 통합 **가드레일(guardrails) 프레임워크**가 이를 운영에서 강제할 수 있다(5.0에도 그대로 존재). `cassandra.yaml`에서 설정한다(키는 보통 최상위에 평탄화돼 있다).

```text
items_per_collection_warn_threshold   # 컬렉션 원소 수 경고 임계
items_per_collection_fail_threshold   # 컬렉션 원소 수 거부 임계
collection_size_warn_threshold        # 직렬화된 컬렉션 바이트 크기 경고
```

> **함정**: 이 가드레일들은 배포판에 따라 **기본값이 비활성(미설정)**인 경우가 많다. 클러스터가 알아서 막아주리라 기대하지 말고 직접 임계를 설정하는 것이 운영 안전장치다. (확인되지 않은 클러스터라면 `nodetool` / `cassandra.yaml`에서 실제 설정을 점검하라.)

### 대안: 진짜 큰 컬렉션은 "클러스터링 행"으로 풀어라

원소가 많아질 가능성이 있으면 컬렉션 대신 클러스터링 컬럼으로 모델링한다. 컬렉션은 "행 안의 미니 클러스터링"이고 진짜 클러스터링은 페이징·범위 쿼리·`LIMIT`을 지원하므로 확장에 유리하다.

```cql
-- 안티패턴: 로그인 기록을 컬렉션에 무한히 담기
CREATE TABLE user_bad (
    user_id uuid PRIMARY KEY,
    logins  list<timestamp>   -- 계속 append → 무한 성장, 읽으면 전체 직렬화
);

-- 대안: 클러스터링 행으로 풀기 (페이징·LIMIT·범위 슬라이스 가능)
CREATE TABLE user_logins (
    user_id   uuid,
    login_at  timestamp,
    ip        inet,
    PRIMARY KEY ((user_id), login_at)
) WITH CLUSTERING ORDER BY (login_at DESC);
-- SELECT * FROM user_logins WHERE user_id=? LIMIT 20;  ← 최근 20개만
```

컬렉션은 "값의 작은 부속물"일 때만 쓴다. 한 사용자의 전화번호 두세 개, 상품의 태그 몇 개. 무한히 늘 수 있는 것은 절대 컬렉션에 담지 않는다.

## frozen vs non-frozen: 한 덩어리냐, 셀의 모음이냐

`frozen` 키워드는 컬렉션·UDT의 저장 방식을 근본적으로 바꾼다.

```cql
CREATE TABLE t (
    id     uuid PRIMARY KEY,
    tags   set<text>,           -- non-frozen: 원소마다 셀
    coords frozen<set<int>>     -- frozen:     전체가 단일 블롭(blob) 1개 셀
);
```

- **non-frozen** (기본): 앞 절에서 본 대로 원소마다 독립 셀. 부분 갱신(`tags = tags + {'new'}`) 가능.
- **frozen**: 컬렉션 전체를 하나의 바이너리 블롭으로 직렬화해 **셀 한 개**에 넣는다. 부분 갱신 불가 — 바꾸려면 전체를 통째로 덮어써야 한다.

```text
non-frozen set<text> {'a','b','c'}        frozen<set<text>> {'a','b','c'}
  ┌─────────────────────┐                  ┌─────────────────────────────┐
  │ cell (tags,'a') = ∅ │                  │ cell (coords) = [a|b|c] blob │  ← 셀 1개
  │ cell (tags,'b') = ∅ │                  └─────────────────────────────┘
  │ cell (tags,'c') = ∅ │                   원소 하나 바꾸려면 blob 전체 재기록
  └─────────────────────┘
   원소 하나만 add/remove 가능
```

### frozen이 사는 자리

frozen의 결정적 효용은 **컬렉션·UDT를 기본 키(primary key)의 일부로 쓸 수 있다**는 데 있다. non-frozen은 "여러 셀의 집합"이라 단일 비교 가능한 값이 아니어서 키가 될 수 없다. frozen은 하나의 블롭, 즉 단일 비교 가능 값이라 파티션 키나 클러스터링 키에 넣을 수 있다.

```cql
CREATE TABLE route (
    -- 출발-도착 좌표쌍 전체를 하나의 키로
    endpoints frozen<tuple<text,text>>,
    departed_at timestamp,
    PRIMARY KEY (endpoints, departed_at)
);
```

frozen은 "이 컬렉션은 항상 통째로 다룬다(부분 갱신 안 한다)"는 의도를 타입으로 못 박는다. 통째로 다룰 거면 frozen이 셀 개수를 줄여 스토리지·읽기에 더 효율적이다.

| 구분 | non-frozen | frozen |
|------|-----------|--------|
| 저장 | 원소마다 셀 | 전체가 단일 블롭 1셀 |
| 부분 갱신 | O (원소 add/remove/set) | X (전체 덮어쓰기만) |
| PK 사용 | 불가 | 가능 |
| 셀 타임스탬프 | 원소마다 별도 | 컬렉션 1개 |
| 읽기 | 원소 셀 모두 머지 | 블롭 1개 역직렬화 |
| 적합한 경우 | 자주 부분 변경되는 작은 집합 | 불변/통째 교체, 키로 사용 |

> **튜플(tuple)은 언제나 frozen이다.** `tuple<...>`은 명시하지 않아도 항상 frozen으로 취급된다. 그래서 부분 갱신이 불가능하고 통째로만 교체된다.

## list의 함정: read-before-write와 동시성

list는 세 컬렉션 중 가장 위험하다. 순서를 보존하려고 timeuuid를 쓰는 구조가 두 가지 문제를 부른다.

### 함정 1: 인덱스 기반 연산은 read-before-write를 강제한다

list의 일부 연산은 **읽기 없이는 쓸 수 없다.** 셀 이름이 사용자가 지정한 인덱스(0,1,2…)가 아니라 내부 timeuuid이기 때문이다. "두 번째 원소"가 어떤 timeuuid를 가진 셀인지 알려면, 코디네이터가 먼저 list 전체를 읽어 위치를 계산한 다음 그 셀을 갱신/삭제해야 한다.

```cql
-- read-before-write가 필요한 연산 (느리고 race에 취약)
UPDATE t SET login_log[2] = ?      WHERE id = ?;   -- 인덱스로 set
DELETE login_log[2]                FROM t WHERE id = ?; -- 인덱스로 delete
UPDATE t SET login_log = login_log - [?] WHERE id = ?; -- 값으로 remove

-- read-before-write가 불필요한 연산 (빠름, 권장)
UPDATE t SET login_log = login_log + [?] WHERE id = ?; -- append
UPDATE t SET login_log = [?] + login_log WHERE id = ?; -- prepend
```

read-before-write는 단순히 느린 것을 넘어 **원자성이 깨진다.** 읽은 시점과 쓰는 시점 사이에 다른 클라이언트가 list를 바꾸면, 인덱스 2가 가리키던 원소가 이미 달라져 엉뚱한 원소를 덮어쓴다. Cassandra는 이 구간에 락을 걸지 않는다(LWT가 아닌 한). **index 기반 list 연산은 동시성 환경에서 신뢰할 수 없다.**

### 함정 2: append조차 멱등하지 않다

append는 read-before-write가 없어 빠르지만 **재시도 시 중복**을 만든다. append는 매번 새 timeuuid 셀을 만든다. 클라이언트가 append를 보냈는데 코디네이터 응답이 타임아웃됐다고 하자. 서버엔 이미 셀이 박혔는데 클라이언트는 모른다. 재시도하면 또 다른 timeuuid로 셀이 하나 더 생긴다 → **같은 값이 두 개.**

```text
append 재시도가 중복을 만드는 시퀀스

client: append('x') ──┐
                       └─> server: 셀(timeuuid_1, 'x') 기록 완료
client: (timeout!  응답 못 받음, 재시도)
client: append('x') ──┐
                       └─> server: 셀(timeuuid_2, 'x') 또 기록
결과: login_log = ['x', 'x']   ← 비멱등
```

set이라면 같은 값이 같은 셀 이름이 되어 멱등(중복 흡수)이지만 list는 timeuuid 때문에 멱등하지 않다.

> **실무 규칙**: 순서가 꼭 필요하고 중복을 허용해야 하는 게 아니라면 **list 대신 set이나 클러스터링 행을 써라.** 순서가 필요하면 list보다 `clustering order by`가 안전하다. list는 "작고, 거의 안 바뀌고, append만 하며, 중복이 의미 있는" 좁은 경우에만.

## 사용자 정의 타입(UDT)과 튜플: 구조체를 한 컬럼에

UDT는 여러 필드를 묶은 구조체를 하나의 타입으로 정의한다. 정규화하면 별도 테이블이 될 것을, 조인이 없는 Cassandra에서 한 행에 끼워 넣는 도구다.

```cql
CREATE TYPE address (
    street text,
    city   text,
    zip    text
);

CREATE TABLE customer (
    id       uuid PRIMARY KEY,
    name     text,
    home     frozen<address>,          -- 통째로 다룰 주소
    shipping list<frozen<address>>     -- 주소 여러 개
);
```

### frozen UDT vs non-frozen UDT

UDT도 컬렉션처럼 frozen/non-frozen이 나뉜다.

- **frozen UDT**: 전체 구조체를 단일 블롭으로 직렬화. 필드 하나 바꾸려면 전체 교체. 키에 사용 가능. **컬렉션 안에 UDT를 넣으려면 frozen이어야 한다**(`list<frozen<address>>`).
- **non-frozen UDT** (Cassandra 3.6+): 각 필드를 독립 셀로 저장해 `UPDATE ... SET home.city = ?`처럼 **필드 단위 갱신**이 가능하다. 단 제약이 있다 — 컬렉션의 원소로는 쓸 수 없고, 키로도 못 쓴다.

```text
frozen<address>                    non-frozen address
  ┌───────────────────────┐         ┌──────────────────────┐
  │ cell(home)=[s|c|z] blob│         │ cell(home,street)=...│
  └───────────────────────┘         │ cell(home,city)  =...│
   home.city만 못 바꿈              │ cell(home,zip)   =...│
                                     └──────────────────────┘
                                      home.city 단독 갱신 가능
```

내부 메커니즘은 컬렉션과 동일하다. frozen은 셀 1개, non-frozen은 필드마다 셀. 그래서 트레이드오프도 같다. 자주 부분 갱신하면 non-frozen, 통째로 다루고 키로 쓰면 frozen이다.

### 튜플: 이름 없는 가벼운 구조체

튜플은 필드 이름 없는 고정 길이 묶음이다. 항상 frozen이며, 스키마 변경 없이 잠깐 묶고 싶을 때 쓴다.

```cql
coords tuple<double, double>   -- (위도, 경도)
```

UDT는 이름이 있어 가독성·진화에 유리하고(필드 추가 `ALTER TYPE ... ADD`), 튜플은 가볍지만 의미가 위치로만 결정되어 깨지기 쉽다. **장기 스키마는 UDT, 일회성 묶음은 튜플**이 보통의 선택이다.

> **주의**: UDT는 진화 가능하지만(필드 추가만 허용, 삭제·타입 변경 불가) frozen UDT를 키에 쓰면 직렬화 형식이 키 정렬에 묶이므로 사실상 바꾸기 어렵다. [[13 - 실전 Event Sourcing on Cassandra]]의 이벤트 스키마 호환성 논리와 같은 주의가 필요하다. 한 번 박힌 직렬화 형식은 영원히 따라온다.

## Static column: 파티션이 공유하는 컬럼

정적 컬럼(static column)은 **한 파티션의 모든 행이 공유하는 단 하나의 값**이다. 파티션 단위 메타데이터를 행마다 중복 저장하지 않게 한다.

```cql
CREATE TABLE order_items (
    order_id     uuid,
    item_seq     int,
    -- 아래 두 개는 주문(파티션) 전체에 하나씩만 존재
    customer_name text    STATIC,
    order_status  text    STATIC,
    -- 아래는 아이템(행)마다
    product_id   uuid,
    qty          int,
    PRIMARY KEY ((order_id), item_seq)
);
```

여기서 `customer_name`, `order_status`는 `order_id` 파티션 전체에 **값 하나**다. 아이템이 100개여도 `order_status`는 한 번만 저장되고 어느 아이템 행에서 읽어도 같은 값이 보인다. 갱신도 파티션 단위다. `UPDATE order_items SET order_status='shipped' WHERE order_id=?`처럼 클러스터링 키 없이 파티션 전체의 정적 값을 바꾼다.

```text
파티션(order_id=A) 내부

  static:  customer_name='Kim'   order_status='shipped'   ← 파티션당 1벌
  ─────────────────────────────────────────────────────
  row item_seq=1:  product=P1  qty=2
  row item_seq=2:  product=P2  qty=1
  row item_seq=3:  product=P3  qty=5
  → 어느 행을 읽어도 동일한 static 값이 함께 보인다
```

### 용도와 제약

- **용도**: 파티션 헤더성 정보(주문 상태, 소유자, 집계 캐시), 또는 LWT로 파티션 단위 조건부 갱신을 걸 앵커(anchor). `IF order_status='pending'` 같은 조건을 정적 컬럼에 걸어 파티션 전역 상태 머신을 만들 수 있다([[11 - LWT Batch Counter 내부]]).
- **제약**: 정적 컬럼은 **클러스터링 컬럼이 있는 테이블에서만** 의미가 있다(파티션에 행이 여럿이어야 "공유"가 성립). 클러스터링 키가 없는 테이블엔 정적 컬럼을 둘 수 없다. 정적 컬럼은 파티션 키만으로 읽고 쓸 수 있어 클러스터링 행 없이도 정적 값만 존재하는 "행 없는 파티션"이 생길 수 있다.

> **결제 시스템 관점**: 한 결제(payment) 파티션 아래 여러 시도(attempt) 행이 클러스터링되는 모델에서, 결제 전체 상태(`paid`/`failed`/`cancelled`)를 정적 컬럼으로 두면 시도 행마다 상태를 복제하지 않고도 파티션 상태를 한 곳에서 관리할 수 있다. 단, 정적 컬럼 갱신은 파티션 전체에 영향이므로 동시 갱신 시 LWT가 필요하다.

## Counter: 분산 카운터의 비멱등성이라는 본질적 한계

카운터는 "분산 환경에서 숫자를 증감"하는 특수 타입이다. 좋아요 수, 조회 수처럼 동시에 여러 곳에서 +1 하는 값에 쓴다. 그런데 카운터는 Cassandra에서 **가장 제약이 많고 가장 위험한** 타입이다.

### 제약 1: 카운터는 전용 테이블에만

카운터 컬럼이 있는 테이블은 **기본 키를 제외한 모든 컬럼이 카운터여야 한다.** 일반 컬럼과 섞을 수 없다.

```cql
-- OK: PK 외 전부 counter
CREATE TABLE page_views (
    page_id text PRIMARY KEY,
    views   counter,
    likes   counter
);

-- 불가: counter와 일반 컬럼 혼재
CREATE TABLE bad (
    id    text PRIMARY KEY,
    views counter,
    title text          -- ERROR: 카운터 테이블에 일반 컬럼 금지
);
```

그래서 "조회수와 게시글 본문"을 한 테이블에 못 둔다. 카운터는 별도 테이블로 격리하고 키로 연결해야 한다. 카운터는 `INSERT`가 없다. 오직 `UPDATE ... SET c = c + n`만 가능하다. 절대값 설정 불가, TTL 불가다.

```cql
UPDATE page_views SET views = views + 1 WHERE page_id = ?;   -- 유일한 쓰기 방식
```

### 제약 2: 비멱등성 — 타임아웃 재시도가 곧 오차

이게 카운터에서 가장 위험한 지점이다. **카운터 증감은 멱등하지 않다.** 일반 쓰기 `SET x = 5`는 몇 번을 재시도해도 결과가 5다(멱등). 하지만 `SET views = views + 1`은 적용된 횟수만큼 결과가 달라진다.

문제는 **타임아웃**이다. 코디네이터가 클라이언트에 `WriteTimeout`을 돌려줬다고 해서 쓰기가 실패했다는 뜻이 아니다. "정해진 일관성 수준만큼 ack를 제때 못 모았다"는 뜻일 뿐, **실제로는 적용됐을 수도 있다.**

```text
카운터 타임아웃 재시도의 딜레마

client: views += 1  ──> coordinator ──> replica 적용 성공
                         (ack 지연으로 WriteTimeout 반환)
client: (적용됐나? 안 됐나? 알 수 없음)
  ├─ 재시도 안 함 → 실제로 안 됐다면 -1 누락 (under-count)
  └─ 재시도 함   → 실제로 됐다면   +1 중복 (over-count)
어느 쪽도 안전하지 않다. 멱등 키로 보호할 수도 없다.
```

일반 쓰기라면 같은 데이터를 다시 보내 멱등하게 재시도하면 그만이다. 카운터는 그게 안 된다. 그래서 **카운터로는 "정확히 한 번"을 보장할 수 없다.** 좋아요 수가 가끔 1~2 틀려도 되는 곳엔 쓰지만 **돈·재고처럼 정확해야 하는 값에는 절대 쓰지 않는다.**

### 제약 3: 내부 구조가 read-before-write다

2.1에서 재작성된 카운터 구현은 증분을 적용할 때 리더 레플리카(leader replica)에서 **현재 값을 읽고 → 새 값을 계산해 → 기록**한다. 카운터 쓰기는 사실상 읽기-수정-쓰기(read-modify-write)이며 일반 쓰기보다 무겁고 지연이 크다. 내부적으로 카운터 값은 단일 정수가 아니라 **레플리카별 증분 샤드의 합(counter context)**으로 표현된다. 그래서 증분에는 클라이언트가 부여한 식별자가 없고 서버는 "재시도"와 "진짜 두 번째 증가"를 구분할 도리가 없다. 부분 적용된 증분을 사후에 분해·상쇄해 되돌릴 방법도 없다. 삭제한 카운터를 다시 증가시키는 것은 동작이 정의되지 않아 사실상 금지다(한 번 delete한 카운터 키는 재사용하지 말 것).

> **결제 시스템 관점**: "오늘 거래 건수", "정산 합계" 같은 값을 카운터로 만들고 싶은 유혹이 크다. 하지만 타임아웃 한 번에 합계가 틀어지고 그 틀어짐을 사후에 검증·교정할 방법이 없다. 정확한 집계는 카운터가 아니라 **원본 이벤트를 적재하고 배치/스트림으로 재계산**하거나, LWT 기반 멱등 처리로 푼다. 카운터의 정확성 한계와 내부 동작은 [[11 - LWT Batch Counter 내부]]에서 끝까지 파헤친다.

## 보조 인덱스(Secondary Index): 숨은 테이블과 scatter-gather

RDB에서 "조회 조건이 늘면 인덱스를 추가한다"는 본능은 Cassandra에서 가장 자주 사고를 낸다. Cassandra의 (전통적) 보조 인덱스는 RDB의 B-tree 인덱스와 **이름만 같고 메커니즘이 전혀 다르다.**

### 내부: 보조 인덱스는 "숨은 로컬 테이블"이다

`CREATE INDEX`를 만들면 Cassandra는 내부적으로 **숨은 테이블(hidden table)**을 만든다. 인덱싱된 컬럼 값을 파티션 키로, 원본 행의 기본 키를 클러스터링으로 갖는 역색인이다. 결정적으로 이 인덱스는 **로컬(local)**이다. 각 노드는 **자기 노드에 있는 데이터만** 색인한다. 전역 인덱스가 아니다.

```text
원본 테이블 (user) — 파티션 키 user_id 로 링에 분산
  node A: user_id=1 (city=seoul), user_id=4 (city=busan)
  node B: user_id=2 (city=seoul), user_id=5 (city=seoul)
  node C: user_id=3 (city=busan), ...

CREATE INDEX ON user(city) → 각 노드에 "숨은 로컬 인덱스"
  node A index:  seoul→{1}      busan→{4}
  node B index:  seoul→{2,5}
  node C index:  busan→{3}
  ↑ 어느 노드도 "seoul 전체"를 알지 못한다. 자기 몫만 안다.
```

### 왜 멀티 노드에서 안티패턴인가: scatter-gather

`SELECT * FROM user WHERE city='seoul'`을 파티션 키 없이 인덱스로만 날리면, 코디네이터는 "seoul이 어느 노드에 있는지" 모른다. 인덱스가 로컬이니까. 그래서 **링의 모든 노드(정확히는 토큰 범위를 커버하는 모든 레플리카 세트)에 질의를 흩뿌리고(scatter), 각자 로컬 인덱스를 뒤져 돌려준 결과를 모은다(gather).** 이게 scatter-gather, 또는 fan-out이다.

```text
파티션 키 없는 인덱스 쿼리 = 전 노드 fan-out

         ┌─────────► node A (로컬 인덱스 조회) ─┐
client → coordinator ─► node B (로컬 인덱스 조회) ─┼─► 결과 머지
         └─────────► node C (로컬 인덱스 조회) ─┘
  노드가 100대면 100대를 다 깨운다. 한 노드라도 느리면 전체가 느리다.
  클러스터가 클수록 더 나빠진다 (안티-스케일).
```

노드 수가 늘수록 더 많은 노드를 깨우므로, **클러스터가 커질수록 더 느려지는** 반(反)확장적 패턴이다. Cassandra의 모든 좋은 쿼리는 "파티션 키로 노드를 특정"하는데, 인덱스만으로 거는 쿼리는 그 정반대다.

### 카디널리티(cardinality)의 양극단이 모두 나쁘다

- **고카디널리티**(값이 거의 다 다름, 예: email): 인덱스 파티션 수가 폭발하고, 각 인덱스 파티션은 행 한두 개. fan-out으로 전 노드를 깨워도 대부분 빈손. 매우 비효율.
- **저카디널리티**(값이 몇 개뿐, 예: `is_active boolean`, `status`): 인덱스 파티션 수는 적지만 각 파티션이 **거대**해진다. `is_active=true` 인덱스 파티션 하나에 수백만 행이 매달려 거대 파티션(large partition) 문제가 그대로 재현된다.

> **그럼 언제 써도 되나**: 보조 인덱스는 **파티션 키와 함께** 쓸 때만 안전하다. `WHERE user_id=? AND city='seoul'`처럼 파티션 키로 노드를 특정한 뒤, 그 파티션 안에서 인덱스로 거르는 경우다. 이때는 fan-out이 없고 한 노드 안의 좁은 검색이 된다. "파티션 키 없는 인덱스 단독 쿼리"가 안티패턴이지, 인덱스 자체가 금지는 아니다.

전통적 보조 인덱스의 이 모든 한계가 SAI를 낳았다.

## SASI에서 SAI로: 5.0의 Storage Attached Index

전통적 보조 인덱스(2i)의 한계를 넘으려는 시도가 둘 있었다. SASI(SSTable Attached Secondary Index)와 SAI(Storage Attached Index)다.

### SASI: 좋은 아이디어, 끝내 미완

SASI는 3.4에서 실험적으로 도입됐다. 인덱스를 SSTable에 붙여(attached) 컴팩션과 함께 관리하고 `LIKE` 접두/접미 검색과 범위 질의를 열었다. 진보였지만 메모리 사용·디스크 오버헤드·동작 안정성 문제로 **프로덕션 권장에 이르지 못했고, 5.0에서 사실상 폐기(deprecated) 수순**이다. SASI는 "여기 길이 있다"를 보여줬을 뿐, 그 길을 완성한 건 SAI다.

### SAI: 5.0의 권장 인덱스

SAI(Storage Attached Index)는 Cassandra 5.0의 간판 기능 중 하나다(CEP-7). 핵심 개선은 다음과 같다.

```text
            전통적 2i            SASI(deprecated)        SAI (5.0 권장)
저장        숨은 테이블          SSTable attached        SSTable attached
범위 쿼리   사실상 불가          부분 지원               숫자 range 1급 지원
텍스트      = 만                 LIKE prefix/suffix      = · analyzer 토큰 매칭
멀티 컬럼   인덱스마다 독립      독립                    여러 인덱스 AND 결합·공유
디스크      행마다 큼            큼                      매우 낮은 오버헤드
컴팩션      별도 관리            함께                    함께(수명주기 일치)
```

- **숫자 범위(numeric range)**: `WHERE amount > 10000 AND amount < 50000` 같은 범위 질의를 1급으로 지원한다. 전통적 2i는 이게 사실상 불가능했다.
- **텍스트 매칭**: 동등(`=`)에 더해, 컬럼에 **analyzer**(예: standard, 비토크나이징, n-gram 등 Lucene 계열)를 지정하면 토큰 단위 매칭(대소문자·악센트 정규화 등)을 할 수 있다. 다만 SASI의 간판이던 `LIKE 'foo%'`·`'%foo%'` 같은 **임의 접두·접미 와일드카드 검색을 SASI처럼 1급 기능으로 제공하지는 않는다**(필요하면 n-gram analyzer로 우회). SAI는 정확 매칭·범위·토큰 매칭에 초점이 있다.
- **여러 컬럼 인덱스의 결합**: 한 테이블의 여러 컬럼에 SAI를 걸고, `WHERE a=? AND b>?`처럼 **여러 SAI 인덱스를 AND로 교집합**할 수 있다. 내부적으로 인덱스들이 공유 인프라 위에서 동작해 컬럼이 늘어도 디스크 비용이 선형적으로 폭증하지 않는다.
- **SSTable에 붙음(storage-attached)**: 인덱스가 SSTable과 한 몸이라, 컴팩션·플러시(flush)와 수명주기가 일치한다. 별도 동기화가 필요 없고, 오버헤드(디스크·쓰기)가 전통적 2i보다 훨씬 낮다.

```cql
CREATE INDEX ON payments(amount) USING 'sai';
CREATE INDEX ON payments(status) USING 'sai';
-- 숫자 범위 + 동등을 결합 (단, 아래의 한계 주의)
SELECT * FROM payments WHERE user_id=? AND status='paid' AND amount > 10000;
```

> 위 `USING 'sai'`는 축약 별칭이고 긴 형식은 `CREATE CUSTOM INDEX ... USING 'StorageAttachedIndex'`다. 둘은 같은 인덱스를 만든다.

### SAI도 못 고치는 한계

SAI는 2i보다 훨씬 낫지만 **마법이 아니다.** 여전히 **로컬 인덱스**다. 따라서:

- **파티션 키 없는 쿼리는 여전히 scatter-gather다.** SAI는 fan-out을 더 효율적으로 만들 뿐, fan-out 자체를 없애지 못한다. 전 노드를 깨우는 구조는 그대로다. 클러스터가 크면 여전히 비싸다.
- **저카디널리티/거대 파티션 문제를 해결하지 않는다.** 인덱스가 가리키는 원본이 거대 파티션이면 읽기는 여전히 무겁다.
- 조인·집계·OR 같은 관계형 연산을 주지 않는다. SAI는 "필터"이지 쿼리 엔진이 아니다.
- 일관성은 베이스 데이터와 같은 최종 일관성(eventual)을 따른다.

> **결론**: SAI는 "쿼리 테이블을 추가로 만들 정도는 아닌, **파티션 키로 좁혀진 안에서의 보조 필터**"에 이상적이다. "파티션 키 없이 전역을 인덱스로 뒤지는" 용도라면 SAI를 써도 안티패턴이다. 그런 쿼리는 [[05 - 데이터 모델링 1 - Query First]]의 원칙대로 **전용 쿼리 테이블로 비정규화**하는 것이 정답이다.

## Materialized View: 자동 비정규화의 달콤함과 함정

수동 비정규화는 귀찮다. 같은 데이터를 쿼리별 테이블에 직접 여러 벌 쓰고 정합성을 애플리케이션이 책임져야 한다. Materialized View(MV)는 이걸 자동화하겠다는 약속이다. 베이스 테이블(base table)에 쓰면 Cassandra가 알아서 뷰(view) 테이블에 비정규화 사본을 만든다.

```cql
CREATE TABLE payments (
    user_id   text,
    payment_id text,
    status    text,
    paid_at   timestamp,
    PRIMARY KEY ((user_id), payment_id)
);

-- status로 조회하는 뷰를 자동 유지
CREATE MATERIALIZED VIEW payments_by_status AS
    SELECT * FROM payments
    WHERE user_id IS NOT NULL AND status IS NOT NULL AND payment_id IS NOT NULL
    PRIMARY KEY ((user_id), status, payment_id);
```

이제 `payments`에 INSERT/UPDATE하면 `payments_by_status`가 자동 갱신된다. 매력적이다. 하지만 **5.0에서도 MV는 experimental이며 기본 비활성**이다.

### 왜 experimental인가: 내부 동작과 일관성 문제

MV는 베이스에 쓰기가 들어올 때, 그 쓰기가 뷰의 어떤 행을 바꿔야 하는지 계산하기 위해 **베이스를 먼저 읽는다(read-before-write).** 예를 들어 `status`가 `pending`→`paid`로 바뀌면, 뷰에서 `pending` 행을 지우고 `paid` 행을 새로 넣어야 한다. 그러려면 "이전 status가 무엇이었는지"를 알아야 하므로 베이스 로컬 읽기가 필요하다.

```text
MV 쓰기 경로 (개략)

UPDATE payments SET status='paid' WHERE user_id=U AND payment_id=P
        │
        ▼ (베이스 레플리카에서)
   1) 기존 행 읽기: status_old = 'pending'    ← read-before-write
   2) 뷰 업데이트 생성:
        - DELETE payments_by_status (U,'pending',P)
        - INSERT payments_by_status (U,'paid',P)
   3) 베이스 쓰기 + 뷰 쓰기를 비동기 전파
```

여기서 두 가지 비용·위험이 나온다.

- **성능**: 모든 베이스 쓰기에 로컬 읽기가 붙는다. 쓰기 처리량이 떨어진다.
- **일관성**: 베이스와 뷰의 갱신은 원자적이지 않다. 노드 장애·전파 실패 시 **뷰가 베이스와 어긋날 수 있다.** 더 나쁜 건, 어긋난 뷰를 손쉽게 검증·교정할 도구가 빈약하다는 점이다(불일치 시 뷰 재구축이 사실상의 복구 수단이고, 대형 테이블에서는 매우 비싸다). repair가 베이스/뷰 정합을 항상 보장한다고 단정하기 어렵다.

MV에는 모델링 제약이 있다. **뷰의 기본 키는 베이스의 모든 기본 키 컬럼을 포함해야** 하고 거기에 더할 수 있는 베이스의 비키 컬럼은 **최대 하나**다. 그래서 표현할 수 있는 뷰가 제한적이다.

활성화하려면 `cassandra.yaml`에서 명시적으로 켜야 한다.

```text
materialized_views_enabled: true   # 5.0, 기본 false (experimental)
```

### MV vs 수동 비정규화 트레이드오프

| 관점 | Materialized View | 수동 비정규화 테이블 |
|------|-------------------|----------------------|
| 작성 편의 | 자동, 코드 적음 | 앱이 직접 다중 테이블 기록 |
| 일관성 제어 | 엔진에 위임(불일치 시 교정 난해) | 앱이 통제(LOGGED BATCH로 묶기 등) |
| 쓰기 비용 | 베이스 read-before-write | 추가 테이블 쓰기(보통 읽기 불필요) |
| 모델 유연성 | 키 제약(비키 +1만) | 자유롭게 설계 |
| 운영 성숙도 | experimental, 기본 off | 검증된 표준 패턴 |
| 불일치 복구 | 뷰 재구축(고비용) | 앱 레벨 재기록/검증 가능 |

> **흔한 오해**: "MV는 Cassandra가 공식 제공하니 안전하겠지." 아니다. 공식 기능이지만 experimental이고 기본 비활성이다. 정합성이 중요한 핵심 경로(결제·정산)에서는 **MV 대신 수동 비정규화**가 사실상의 업계 표준이다. 같은 데이터를 두 테이블에 쓰되, 둘 다 같은 파티션 키 그룹이면 `LOGGED BATCH`로 원자성을 흉내 내고([[11 - LWT Batch Counter 내부]]), 불일치는 멱등 재기록·검증 잡으로 잡는다. MV는 "정확성보다 편의가 우선이고 불일치를 감내할 수 있는" 비핵심 경로에서 신중히.

## 안티패턴 카탈로그: 증상 · 원인 · 대안

지금까지 본 기능들의 오용을 포함해, 실전에서 반복되는 안티패턴을 한 표로 정리한다. 모든 안티패턴의 공통 뿌리는 하나다. **"파티션의 물리적 현실을 무시한 모델링"**이다.

| 안티패턴 | 증상 | 근본 원인 | 대안 |
|----------|------|-----------|------|
| **무한 파티션 성장**(unbounded partition) | 시간이 갈수록 특정 파티션 읽기 지연·GC 압박 증가 | 시간 경계 없는 파티션 키(예: `user_id` 하나로 모든 이벤트) | 시간 버킷팅(`(user_id, yyyymm)`), 복합 파티션 키로 상한 부여 |
| **핫스팟**(hotspot) | 한 노드만 CPU/IO 폭주, 나머지 한가 | 단조·저카디널리티 파티션 키(예: 오늘 날짜로 전부 몰림) | 버킷/해시 접두로 분산, 키 카디널리티 상향 |
| **큐 패턴**(tombstone hell) | 읽기 지연이 시간에 비례해 악화, `TombstoneOverwhelmingException` | insert 후 처리 완료 행 delete → 좁은 범위에 톰스톤 누적, 읽을 때 톰스톤을 스캔 | Cassandra를 큐로 쓰지 않기; 불가피하면 시간 버킷+TTL로 톰스톤 스캔 회피 |
| **거대 파티션**(large partition) | 특정 파티션 읽기 느림, repair·컴팩션 시 힙 압박, 복구 지연 | 한 파티션에 너무 많은 행/바이트(~100MB·수십만 행 초과) | 파티션 분할(버킷팅), 행 폭 줄이기, 컬렉션→클러스터링 |
| **저카디널리티 인덱스** | 인덱스 쿼리 느림, 인덱스 파티션 비대 | `boolean`/`status` 등에 보조 인덱스 → 한 인덱스 파티션에 수백만 행 | 전용 쿼리 테이블, 또는 파티션 키와 함께 SAI 사용 |
| **read-before-write** | 레이스로 값 유실/중복, 지연 증가 | list index 연산, "읽고 검사 후 쓰기"를 무락(無lock)으로 수행 | LWT로 보호하거나(비용 인지), 그런 연산이 없도록 재모델링 |
| **컬렉션 남용** | 행 읽기마다 거대 컬렉션 직렬화, 쓰기 비대 | 무한 성장하는 데이터를 list/set/map에 적재 | 클러스터링 행으로 풀기, 컬렉션은 작은 부속물로 한정 |

### 안티패턴 깊이 보기

**무한 파티션 성장과 거대 파티션은 한 몸이다.** 둘 다 "파티션 키에 상한이 없다"는 같은 병의 다른 증상이다. 처방은 **버킷팅(bucketing)**이다. 파티션 키에 시간·해시 차원을 더해 하나의 파티션 크기를 물리적으로 제한한다. [[05 - 데이터 모델링 1 - Query First]]의 시계열 모델링에서 본 `(sensor_id, day)` 패턴이 바로 이것이다.

```cql
-- 무한 성장 (나쁨): 한 device의 모든 이벤트가 한 파티션에 영원히 쌓임
PRIMARY KEY ((device_id), event_at)

-- 버킷팅 (좋음): 일 단위로 파티션을 잘라 크기 상한 부여
PRIMARY KEY ((device_id, event_day), event_at)
```

**큐 패턴(tombstone hell)**은 Cassandra 입문자가 가장 흔히 빠지는 함정이다. "할 일을 INSERT하고, 처리하면 DELETE"는 RDB에선 자연스럽지만 Cassandra에선 재앙이다. DELETE는 즉시 지우는 게 아니라 **톰스톤**이라는 "삭제 표식"을 남기고 그 톰스톤은 `gc_grace_seconds`(기본 864,000초=10일) 동안 SSTable에 남는다. 큐의 머리(가장 오래된 미처리 항목)를 읽으려 하면, 코디네이터는 이미 삭제된 수많은 항목의 **톰스톤을 전부 스캔하고 버린 뒤**에야 살아있는 행에 도달한다. 처리량이 늘수록 톰스톤이 쌓이고 읽기는 점점 느려지다가 `tombstone_failure_threshold`(기본 100,000)에 걸려 쿼리가 통째로 실패한다.

```text
큐 패턴이 죽는 과정 (한 파티션 안의 SSTable 스캔)

읽기: "가장 오래된 미처리 항목" → 파티션 앞에서부터 스캔
  [tomb][tomb][tomb][tomb]...[tomb][LIVE]
   └────────── 다 읽고 버림 ──────┘  ↑ 겨우 여기
  tombstone_warn_threshold=1000 초과 → 경고 로그
  tombstone_failure_threshold=100000 초과 → 쿼리 실패(예외)
```

처방은 명확하다. **Cassandra를 작업 큐로 쓰지 마라**(Kafka/SQS 등 전용 큐를 써라). 굳이 써야 한다면, 시간 버킷으로 처리 완료 영역을 통째로 버리거나(파티션 단위 TTL/드롭), 읽기 범위가 톰스톤 더미를 가로지르지 않도록 모델링한다. 톰스톤·`gc_grace_seconds`의 내부 메커니즘은 [[08 - Storage Engine 내부]]와 [[09 - Compaction 전략]]에서 다룬다.

**핫스팟**은 일관성 모델이 아니라 **부하 분산**의 문제다. Cassandra는 파티션 키를 해시(murmur3)해 토큰 링([[02 - 분산 아키텍처]])에 배치한다. 키가 잘 분산되면 부하가 고르게 퍼지지만 모든 쓰기가 "오늘 날짜" 한 파티션으로 몰리면 그 파티션을 소유한 노드 세트만 불탄다. 처방은 파티션 키에 분산 차원을 더하는 것이다. 예를 들어 `(date)` 대신 `(date, bucket)`으로 두고 `bucket = hash(id) % N`처럼 쪼개면 같은 날 쓰기가 N개 파티션으로 흩어진다.

> **모든 안티패턴을 관통하는 한 문장**: Cassandra에서 좋은 모델은 "읽기와 쓰기가 **소수의, 크기가 제한된, 고르게 분산된 파티션**에 떨어지게" 만드는 것이다. 무한 성장·핫스팟·거대 파티션·큐는 전부 이 원칙을 어긴 결과다.

## 핵심 요약

- **컬렉션의 본질은 셀**이다. non-frozen은 원소마다 셀(부분 갱신 가능, 키 불가), frozen은 전체가 단일 블롭 1셀(통째 교체, 키 가능). 튜플은 언제나 frozen.
- 컬렉션은 **읽을 때 전체가 직렬화**되고 페이징이 없다. 1,000개 미만으로 유지하고, 무한히 늘 수 있는 것은 컬렉션이 아니라 **클러스터링 행**으로 모델링하라. non-frozen 컬렉션을 **통째로 재대입하면 range tombstone**이 생기므로 부분 변경엔 `+`/`-` 연산을 쓴다. 가드레일(4.1+)은 기본 비활성일 수 있으니 직접 설정.
- **list는 timeuuid 기반**이라 index 연산은 read-before-write를 강제하고 append조차 재시도 시 중복을 만든다(비멱등). 순서·중복이 꼭 필요한 게 아니면 set/클러스터링을 써라.
- **UDT**는 frozen(통째·키 가능)과 non-frozen(필드 단위 갱신)으로 나뉘고, 컬렉션 원소로 쓰려면 frozen이어야 한다. **static column**은 파티션이 공유하는 단일 값으로, 클러스터링이 있는 테이블에서만 의미가 있다.
- **카운터는 비멱등**이다. 타임아웃 재시도가 곧 over/under-count이며 정확성을 보장할 수 없다. 전용 테이블 제약, 절대값 설정·TTL 불가, 내부적으로 read-before-write. 돈·재고엔 쓰지 마라.
- **전통적 보조 인덱스는 로컬 숨은 테이블**이라 파티션 키 없는 쿼리는 전 노드 scatter-gather가 된다. 고/저카디널리티 모두 나쁘다. **파티션 키와 함께** 쓸 때만 안전.
- **SAI(5.0)**는 SASI를 대체하는 권장 인덱스로 숫자 범위·analyzer 토큰 매칭·멀티 컬럼 결합·낮은 오버헤드·SSTable attached를 제공하지만(임의 LIKE 와일드카드는 1급 기능이 아님), **여전히 로컬 인덱스**라 파티션 키 없는 전역 쿼리의 scatter-gather는 남는다.
- **Materialized View는 5.0에서도 experimental·기본 off**다. 베이스 read-before-write 비용과 베이스/뷰 불일치 위험이 있어, 핵심 경로에서는 **수동 비정규화**가 표준이다.
- 안티패턴(무한 성장·핫스팟·큐/톰스톤·거대 파티션·저카디널리티 인덱스·read-before-write·컬렉션 남용)의 공통 처방은 **소수의, 크기 제한된, 고르게 분산된 파티션**을 향한 모델링이다.
