# Cassandra DB 학습 로드맵

이 폴더는 Apache Cassandra를 **밑바닥부터(아키텍처 → 내부 동작 → 운영 → 실전)** 깊게 학습하기 위한 단계별 노트 모음이다. 각 챕터는 "개념 → 내부 메커니즘 → 안티패턴/실전" 흐름으로 구성된다.

## 이 학습의 목표

단순히 "CQL을 쓸 줄 안다"가 아니라, 다음 질문에 스스로 답할 수 있게 되는 것:

- 왜 `WHERE`에 아무 컬럼이나 못 쓰는가? (→ 02, 05)
- `CONSISTENCY QUORUM`이 정확히 무엇을 보장하고 무엇을 보장하지 않는가? (→ 04)
- 데이터를 지웠는데 디스크가 안 줄어드는 이유는? (→ 08, 09)
- 특정 파티션만 느려지는 hotspot은 왜 생기고 어떻게 막는가? (→ 05, 12)
- 우리 결제 시스템이 Cassandra 위에서 event sourcing을 하는 이유는? (→ 13)

## 선수지식

- **분산 시스템 기본**: CAP 정리, 일관성 모델(strong vs eventual), 합의(consensus)의 존재 이유
- **자료구조**: 해시, B-Tree vs LSM-Tree의 차이(없어도 08에서 다룸)
- **SQL 경험**: RDB를 써봤다면 "Cassandra가 무엇을 *포기*했는지" 이해가 빠르다

## 추천 학습 순서

1. **첫 바퀴 (큰 그림)**: 01 → 02 → 03 → 04. 분산 DB로서의 정체성과 일관성 모델을 먼저 머리에 넣는다.
2. **두 번째 바퀴 (모델링)**: 05 → 06 → 07. 여기서부터 직접 키스페이스/테이블을 만들어 보며 실습.
3. **세 번째 바퀴 (내부)**: 08 → 09 → 10. 디스크에서 무슨 일이 일어나는지 추적. 성능/운영 직관이 여기서 생긴다.
4. **네 번째 바퀴 (심화·운영)**: 11 → 12. 트랜잭션의 한계와 실제 장애 대응.
5. **마무리 (실전 연결)**: 13. 우리 코드베이스의 event sourcing이 왜 이렇게 생겼는지 역으로 이해.

## 실습 환경 준비

```bash
# 단일 노드 Docker (학습용)
docker run --name cass -p 9042:9042 -d cassandra:5.0

# CQL 셸 접속
docker exec -it cass cqlsh

# 클러스터 상태 확인 (운영 감각용)
docker exec -it cass nodetool status
```

> 분산 동작(복제, repair, consistency)을 체감하려면 단일 노드로는 부족하다. 02~04를 학습할 땐 `ccm`(Cassandra Cluster Manager)으로 로컬에 3노드 클러스터를 띄워 실험하는 것을 강력히 권장한다.

## 핵심 멘탈 모델 (전체를 관통하는 3가지)

1. **Cassandra는 "분산 해시맵 + 정렬된 컬럼"이다.** Partition Key로 노드를 찾고, Clustering Key로 그 안을 정렬한다. RDB의 "테이블에 조건 걸어 스캔" 사고를 버려야 한다.
2. **쓰기는 항상 append-only다.** UPDATE/DELETE조차 새 데이터를 *덧붙이는* 것이다(LSM-Tree). 이것이 빠른 쓰기, tombstone, compaction의 근원이다.
3. **일관성은 켜고 끄는 다이얼이다.** 강한 일관성이 default가 아니라, 쿼리마다 비용을 지불하고 사는 것이다.
