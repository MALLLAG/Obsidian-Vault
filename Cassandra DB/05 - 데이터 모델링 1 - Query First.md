앞선 장들에서 우리는 Cassandra가 토큰 링(token ring) 위에 데이터를 흩뿌리고([[02 - 분산 아키텍처]]), 그것을 여러 노드에 복제하며([[03 - 복제 전략과 데이터센터]]), 일관성 수준을 정족수(quorum)로 조율한다([[04 - Tunable Consistency]])는 사실을 보았다. 이 모든 메커니즘은 결국 하나의 질문으로 수렴한다. **"내가 방금 던진 이 쿼리는, 정확히 몇 개의 노드를 건드리는가?"** 이 장은 그 질문의 답을 데이터 모델 설계 단계에서 *미리* 강제하는 방법론을 다룬다.

이 장이 답하려는 핵심 질문은 다음과 같다.

- 왜 Cassandra에서는 "테이블을 먼저 설계하지 말라"고 말하는가? 무엇을 먼저 설계하는가?
- `PRIMARY KEY (a, b, c)`라고 썼을 때, 디스크 위에서는 실제로 어떤 일이 벌어지는가? `a`와 `b`, `c`는 각각 무슨 역할을 하는가?
- 왜 `WHERE` 절은 그렇게 까다로운가? 왜 임의의 컬럼으로 조건을 걸 수 없는가? 이것은 Cassandra의 "한계"인가, 아니면 "정직함"인가?
- `ALLOW FILTERING`은 왜 마치 핵폐기물 경고 라벨처럼 취급되는가?
- 파티션(partition)은 얼마나 커도 되는가? 무한히 자라는 파티션을 만들면 어느 날 무슨 일이 일어나는가?

이 질문들에 답하는 과정에서 우리는 "데이터 모델링"이라는 단어가 RDB와 Cassandra에서 거의 정반대의 작업을 가리킨다는 사실을 받아들이게 된다.

## 1. Query-First: 정규화의 정반대편에서 시작하기

### RDB의 세계관: 데이터가 먼저, 쿼리는 나중

관계형 데이터베이스로 시스템을 설계해 본 사람이라면 몸에 밴 순서가 있다.

1. 도메인의 엔티티를 식별한다 (사용자, 주문, 상품, 결제…).
2. 엔티티 간의 관계를 정의한다 (1:N, N:M).
3. 정규화(normalization)를 적용한다. 같은 데이터가 두 군데 저장되지 않도록, 모든 사실(fact)은 단 한 곳에만 존재하도록 테이블을 쪼갠다.
4. 그렇게 만들어진 깔끔한 스키마 위에서, 필요할 때마다 `JOIN`으로 데이터를 다시 조립해 쿼리를 작성한다.

이 접근은 **"쿼리는 나중에 얼마든지 만들 수 있다"**고 가정한다. 정규화된 스키마는 일종의 *진실의 단일 원천(single source of truth)*이고, 옵티마이저(optimizer)가 인덱스를 골라 가며 `JOIN`을 효율적으로 처리해 준다. 그래서 RDB에서 "좋은 스키마"란 중복이 없고 갱신 이상(update anomaly)이 발생하지 않는 스키마다.

이 세계관이 성립하는 이유는 단 하나, **`JOIN`이 싸기 때문**이다. 단일 노드의 RDB에서는 두 테이블이 같은 디스크, 같은 메모리, 같은 버퍼 풀 위에 있다. 인덱스를 타고 한쪽 테이블의 행을 찾은 뒤, 그 키로 다른 테이블을 다시 찾는 비용은 밀리초 단위다.

### Cassandra의 세계관: `JOIN`이 없다, 고로 쿼리가 먼저다

Cassandra에는 `JOIN`이 없다. 이것은 누락된 기능이 아니라 **설계상의 거부**다. 분산 환경에서 `JOIN`을 구현하려면, 임의의 두 행이 서로 다른 노드(심지어 다른 데이터센터, 다른 대륙)에 흩어져 있을 때 그것들을 네트워크를 건너 모아야 한다. 100개 노드 클러스터에서 무심코 던진 `JOIN` 하나가 100개 노드를 모두 깨우고, 노드 간 네트워크를 데이터로 가득 채울 수 있다. Cassandra는 이런 "조용히 비싸지는" 연산을 애초에 금지함으로써, 모든 쿼리의 비용을 *예측 가능하게* 만든다.

`JOIN`이 없고, 서버 사이드 집계도 사실상 없으며, 임의 컬럼 필터링도 막혀 있다면, "쿼리는 나중에 만들면 된다"는 가정은 무너진다. 내가 미래에 던질 쿼리를 *지금* 알지 못하면, 그 쿼리를 받쳐 줄 테이블 구조를 만들 수 없다. 그래서 Cassandra 데이터 모델링은 **읽기 패턴(read pattern)에서 시작해서 테이블로 끝난다.**

> **흔한 오해:** "Cassandra는 스키마가 유연하니까 일단 데이터를 넣고 나중에 쿼리를 고민하면 된다." → 정반대다. Cassandra는 컬럼을 추가하는 데는 관대하지만, 쿼리 패턴을 사후에 바꾸는 데는 가장 가혹한 DB다. 잘못 잡은 파티션 키는 테이블을 통째로 다시 만들고 데이터를 마이그레이션해야만 고칠 수 있다.

### 네 단계 방법론: conceptual → workflow → logical → physical

DataStax가 정리하고 업계가 받아들인 Cassandra 데이터 모델링의 표준 흐름은 네 단계다. RDB의 흐름과 나란히 놓고 보자.

```text
  RDB (Data-First)                  Cassandra (Query-First)
  ─────────────────                 ────────────────────────
  ① 개념 모델 (엔티티/관계)   →     ① 개념 모델 (엔티티/관계)   ← 여기까지는 같다
  ② 정규화 → 논리 스키마             ② 애플리케이션 워크플로
  ③ 물리 스키마 (인덱스)              ──> "어떤 화면이 어떤 데이터를
  ④ 쿼리를 SQL로 작성                     어떤 순서로 읽는가?"
                                          = 필요한 쿼리(Q1..Qn) 도출
                                    ③ 논리 모델: 쿼리 1개 = 테이블 1개
                                    ④ 물리 모델: CQL DDL, 파티션 크기 검증
```

각 단계를 뜯어보자.

**① 개념 모델 (conceptual model)** — 이 단계만큼은 RDB와 동일하다. 도메인을 ER 다이어그램으로 그린다. "결제(Payment)는 여러 개의 환불(Refund)을 가진다", "사용자(User)는 여러 활동(Activity)을 발생시킨다" 같은 엔티티와 관계를 식별한다. 비즈니스 개념을 이해하는 단계이지 저장 구조를 정하는 단계가 아니다.

**② 애플리케이션 워크플로 (application workflow)** — 여기서 RDB와 갈라진다. 우리는 "이 시스템의 화면과 API가 데이터를 *어떻게 접근하는가*"를 그린다. 결제 콘솔의 상세 화면은 무엇을 보여주는가? 사용자가 "내 최근 결제 내역"을 누르면 어떤 순서로 무엇을 읽는가? 이 워크플로 다이어그램의 각 화살표가 곧 하나의 **접근 패턴(access pattern)**, 즉 하나의 쿼리가 된다. 이 단계의 산출물은 다음과 같은 쿼리 목록이다.

```text
Q1: userId로 특정 사용자의 최근 결제 목록을 최신순으로 조회
Q2: paymentId로 단일 결제의 상세 조회
Q3: userId로 특정 사용자의 활동 타임라인을 시간 역순으로 조회
Q4: sensorId + 날짜로 그 날의 센서 측정값을 시간순으로 조회
...
```

**③ 논리 모델 (logical model)** — 이것이 Query-First의 심장이다. **쿼리 하나당 테이블 하나를 만든다.** Q1을 위한 테이블, Q2를 위한 테이블, Q3을 위한 테이블… 같은 데이터가 여러 테이블에 중복되어 들어가더라도 개의치 않는다. 각 테이블의 파티션 키와 클러스터링 컬럼은 그 쿼리의 `WHERE`/`ORDER BY`가 그대로 통하도록 역산해서 정한다.

**④ 물리 모델 (physical model)** — 논리 모델을 실제 CQL 데이터 타입과 DDL로 구체화하고, 각 테이블의 파티션이 너무 커지지 않는지, 핫스팟(hotspot)이 생기지 않는지 수치로 검증한다. 파티션이 너무 커진다면 버킷팅(bucketing)으로 쪼개는 등의 조정이 이 단계에서 이뤄진다.

### "이건 정규화의 죄악이 아닌가?"라는 본능을 잠재우기

RDB 출신 엔지니어가 ③단계에서 거의 항상 느끼는 거부감이 있다. "같은 결제 정보를 세 개 테이블에 복붙한다고? 한쪽만 수정되면 데이터가 어긋나잖아!" 이 본능은 옳다 — RDB에서는. 하지만 Cassandra에서는 트레이드오프의 축이 반대로 기울어 있다.

| 축 | RDB (정규화) | Cassandra (비정규화) |
|---|---|---|
| 최적화 대상 | 저장 공간, 쓰기 일관성 | 읽기 지연시간, 수평 확장성 |
| 디스크 | 비싸다고 가정 | 싸다고 가정 (그냥 더 사면 됨) |
| 중복 | 죄악 (anomaly의 원천) | 미덕 (각 쿼리가 자기 데이터를 소유) |
| 일관성 유지 | DB가 `JOIN`/트랜잭션으로 보장 | **애플리케이션이 쓰기 시점에 책임** |
| 쓰기 비용 | 한 곳에만 쓰면 됨 | 여러 테이블에 동시에 써야 함 |

이유는 하나다. **Cassandra에서 쓰기는 극도로 싸다.** [[08 - Storage Engine 내부]]에서 자세히 다루겠지만, Cassandra의 쓰기는 메모리(memtable)에 추가하고 커밋 로그에 append 하는 것이 전부다. 디스크 위의 기존 데이터를 읽어서 갱신하는(read-modify-write) 과정이 없다. 그래서 같은 데이터를 5개 테이블에 쓰는 것은 1개 테이블에 쓰는 것보다 5배 느린 게 아니라, 여전히 충분히 빠르다. 반면 읽기에서 `JOIN`이나 필터링을 하는 비용은 통제 불가능하게 커진다. **Cassandra는 "쓰기를 더 시키고 읽기를 거저 만드는" 거래를 한다.**

> **결제 시스템 비유:** 결제 도메인으로 옮겨 보면, "결제 이벤트"는 한 번 일어나면 변하지 않는다(append-only). 그런데 그 이벤트를 보는 화면은 여럿이다 — 사용자별 목록, 결제 ID 단건 조회, 정산 대사용 일자별 집계. 이벤트가 불변이라면, 같은 이벤트를 여러 형태로 여러 테이블에 미리 펼쳐 두는 것은 위험하기는커녕 자연스럽다. 이것이 [[13 - 실전 Event Sourcing on Cassandra]]가 Cassandra와 궁합이 좋은 근본 이유다.

## 2. Primary Key 해부: 파티션 키와 클러스터링 컬럼

Query-First가 "철학"이라면, 그 철학을 디스크 위에서 강제하는 "물리적 메커니즘"이 바로 PRIMARY KEY다. CQL에서 PRIMARY KEY는 RDB의 그것과 글자만 같을 뿐, 완전히 다른 두 가지 책임을 한 몸에 지고 있다.

```cql
PRIMARY KEY ( (partition_key) , clustering_col_1, clustering_col_2, ... )
              └──────┬───────┘  └──────────────────┬─────────────────┘
            데이터가 "어느 노드"에 가는가     파티션 "내부에서 어떻게 정렬"되는가
```

이 한 줄을 정확히 읽는 법을 익히는 것이 Cassandra 데이터 모델링 실력의 8할이다.

### 파티션 키(partition key): "어느 노드"를 결정하는 좌표

파티션 키는 PRIMARY KEY 정의에서 **맨 앞에 오는 컬럼(들)**이다. 괄호로 감싸면 여러 컬럼을 묶어 하나의 파티션 키로 만들 수 있고(복합 파티션 키, 뒤에서 다룬다), 괄호가 없으면 첫 번째 컬럼 하나가 파티션 키다.

파티션 키의 운명은 [[02 - 분산 아키텍처]]에서 본 토큰 링과 직결된다. Cassandra는 파티션 키 값을 **Murmur3 해시 함수**(`Murmur3Partitioner`가 기본값)에 통과시켜 -2⁶³ ~ 2⁶³-1 범위의 토큰(token) 하나를 만든다. 그 토큰이 토큰 링의 어느 구간에 떨어지느냐가, 이 데이터의 "주인 노드"를 결정한다. 다만 Cassandra는 마스터가 없는(masterless) 구조이므로, 여기서 "주인"이란 그 토큰 구간을 소유한 첫 복제 노드(natural endpoint)일 뿐 읽기/쓰기를 독점하는 마스터가 아니다 — 복제본 노드들은 모두 대등하며 어느 복제본이든 코디네이터를 거쳐 요청을 처리할 수 있다([[03 - 복제 전략과 데이터센터]]).

```text
       partition_key = "user_abc"
              │
              ▼
        Murmur3 해시
              │
              ▼
        token = 3812...  (64비트 정수)
              │
              ▼
   ┌─────────── 토큰 링 ───────────┐
   │   NodeA        NodeB          │
   │  [min..t1]   [t1..t2]         │
   │                  ▲            │
   │     token이 여기 떨어짐 → NodeB가 주인
   │   NodeD        NodeC          │
   │  [t3..max]   [t2..t3]         │
   └───────────────────────────────┘
```

여기서 가장 중요한 두 가지 성질이 나온다.

1. **같은 파티션 키 값은 항상 같은 토큰, 즉 항상 같은 노드 집합으로 간다.** 해시는 결정적(deterministic)이므로, `user_abc`의 모든 데이터는 항상 같은 노드(들)에 모여 있다. 이것이 "단일 노드에서 빠르게 읽을 수 있다"는 보장의 근원이다.
2. **해시는 순서를 파괴한다.** `user_abc`와 `user_abd`는 알파벳으로는 이웃이지만, 해시 후 토큰은 링의 정반대편에 떨어질 수 있다. 그래서 **파티션 키로는 범위 조회(`WHERE pk > x`)가 의미를 갖지 못한다.** `>`로 비교해 봐야 토큰 순서일 뿐 값의 순서가 아니기 때문이다. 이것이 뒤에서 "파티션 키는 등치(`=`)만 가능하다"는 규칙의 물리적 이유다.

> **직관:** 파티션 키를 정한다는 것은 "이 데이터의 우편번호를 정하는 일"이다. 같은 우편번호는 같은 배송 트럭(노드)에 실린다. 그리고 우체국은 우편번호를 *해시*해서 트럭을 배정하므로, 1번지와 2번지가 전혀 다른 트럭에 실릴 수 있다.

### 클러스터링 컬럼(clustering column): 파티션 "내부"의 정렬 축

PRIMARY KEY에서 파티션 키 *뒤에* 오는 컬럼들이 클러스터링 컬럼이다. 이들은 노드를 고르는 데는 전혀 관여하지 않는다. 대신, **하나의 파티션 안에서 행(row)들이 어떤 순서로 디스크에 물리적으로 정렬되어 저장될지**를 결정한다.

이 "물리적으로 정렬되어 저장된다"는 표현이 핵심이다. RDB의 인덱스는 데이터와 별도로 존재하는 보조 자료구조다. 반면 Cassandra의 클러스터링 컬럼은 데이터 자체의 저장 순서다 — RDB로 치면 클러스터드 인덱스(clustered index)에 가깝다. [[08 - Storage Engine 내부]]에서 보겠지만, 하나의 파티션은 SSTable 안에서 클러스터링 컬럼 순서로 정렬된 연속된 행들의 블록으로 저장된다.

```text
  파티션 키 = user_42 인 단 하나의 파티션 (한 노드 위, 디스크에 연속 저장)
  ┌──────────────────────────────────────────────────────────────┐
  │ user_42                                                        │
  │   ├─ [2026-06-26 10:00] activity=login    ← 클러스터링 컬럼   │
  │   ├─ [2026-06-26 09:30] activity=pay       (created_at DESC) │
  │   ├─ [2026-06-25 22:10] activity=logout     로 미리 정렬됨    │
  │   └─ [2026-06-25 18:00] activity=view                         │
  └──────────────────────────────────────────────────────────────┘
       ▲ 같은 파티션 내부는 디스크에서 이미 이 순서로 누워 있다
```

이 구조 덕분에 "최근 활동 10개"를 읽는 것은 정렬도, 필터링도 아닌 **이미 정렬된 블록의 앞에서부터 10줄을 순차 읽는(sequential scan)" 연산**이 된다. 디스크 헤드가 한 번 자리를 잡고 쭉 읽으면 끝이다. 이것이 시계열·타임라인 데이터에서 Cassandra가 빛나는 이유다.

`WITH CLUSTERING ORDER BY (created_at DESC)`를 테이블 정의에 넣으면, 그 정렬 방향이 *디스크에 저장될 때부터* 적용된다. 즉 "최신순 조회"가 압도적으로 흔하다면, 처음부터 내림차순으로 저장해 두어 읽을 때 뒤집는 비용조차 없앤다.

### 파티션 키 + 클러스터링 컬럼 = 한 행을 가리키는 좌표

전체 PRIMARY KEY는 한 행을 유일하게 식별한다. 단, 그 식별의 의미가 두 층으로 나뉜다.

- **파티션 키**: "어느 파티션(어느 노드)인가" — 클러스터 전체에서 데이터를 *찾아가는* 역할
- **클러스터링 컬럼**: "그 파티션 안에서 정확히 어느 행인가" — 파티션 내부에서 *정렬·정렬 범위*를 만드는 역할

```cql
-- 예: 결제 이벤트 타임라인
CREATE TABLE payment_events_by_payment (
    payment_id   text,        -- 파티션 키: 어느 결제의 이벤트인가
    event_seq    bigint,      -- 클러스터링: 그 결제 안에서 몇 번째 이벤트인가
    event_type   text,
    payload      text,
    occurred_at  timestamp,
    PRIMARY KEY ( payment_id, event_seq )
) WITH CLUSTERING ORDER BY ( event_seq ASC );
```

이 테이블에서 `payment_id`는 "이 결제의 모든 이벤트를 한 노드에 모아라"라고 말하고, `event_seq`는 "그 안에서 발생 순서대로 누워 있어라"라고 말한다. 그 결과 "특정 결제의 전체 이벤트 히스토리를 순서대로" 읽는 것은 단일 파티션의 순차 읽기 한 번으로 끝난다. 이것이 이벤트 소싱 히스토리 조회의 이상적 형태다.

## 3. 복합 파티션 키와 복합 클러스터링: 좌표를 다차원으로

### 복합 파티션 키 `((a, b))`: 묶어서 하나의 우편번호

때로는 단일 컬럼으로 파티션을 나누면 파티션이 너무 커지거나 데이터가 한쪽으로 쏠린다. 이때 **여러 컬럼을 괄호로 묶어 하나의 파티션 키**로 만든다.

```cql
PRIMARY KEY ( (sensor_id, day) , reading_time )
              └────────┬──────┘
              이 둘을 합쳐서 해시 → 토큰 1개
```

여기서 결정적으로 중요한 점이 있다. `(sensor_id, day)`는 **두 값을 합쳐서 하나의 해시 입력**으로 쓴다. 따라서 파티션을 특정하려면 `sensor_id`와 `day`를 **둘 다** 등치로 지정해야 한다. 하나만 알아서는 토큰을 계산할 수 없고 토큰을 모르면 어느 노드로 가야 할지 모른다.

```cql
-- ✅ 가능: 두 컬럼 모두 등치 → 토큰 계산 가능 → 단일 파티션 타격
SELECT * FROM readings WHERE sensor_id = 's1' AND day = '2026-06-26';

-- ❌ 불가: day가 빠지면 토큰을 못 만든다 → 어느 노드로 갈지 모름
SELECT * FROM readings WHERE sensor_id = 's1';
```

복합 파티션 키는 단일 키로는 너무 거대해지는 파티션을 잘게 나누는 핵심 도구다(다음 절의 버킷팅이 바로 이 기법이다).

### 복합 클러스터링: 다층 정렬과 prefix 규칙

클러스터링 컬럼은 여러 개를 둘 수 있고 이들은 **선언된 순서대로 다층 정렬(sort)**을 만든다. SQL의 `ORDER BY col1, col2, col3`를 디스크 저장 순서로 굳혀 둔 것이라 생각하면 정확하다.

```cql
CREATE TABLE sales_by_region (
    region      text,
    category    text,
    sold_at     timestamp,
    item_id     uuid,
    amount      bigint,
    PRIMARY KEY ( region, category, sold_at, item_id )
) WITH CLUSTERING ORDER BY ( category ASC, sold_at DESC, item_id ASC );
```

이 파티션(`region` 하나) 내부의 디스크 정렬 순서는 다음과 같다.

```text
region = "KR" 파티션 내부:
  category="book"   ├─ sold_at=10:00 ├─ item_id=a...
                    │                 └─ item_id=b...
                    └─ sold_at=09:00 └─ item_id=c...
  category="food"   ├─ sold_at=11:00 └─ item_id=d...
                    └─ sold_at=08:00 └─ item_id=e...
   ▲ 1차 정렬       ▲ 2차 정렬        ▲ 3차 정렬
   (category)       (sold_at)         (item_id)
```

이 다층 정렬이 바로 다음 절에서 다룰 "prefix 규칙"의 물리적 근거다. 전화번호부가 "성 → 이름" 순으로 정렬되어 있어서 성을 모르면 이름만으로 찾을 수 없는 것과 똑같은 원리다.

## 4. WHERE 절의 법칙: 왜 임의 조건이 불가능한가

이제 이 장에서 가장 자주 사람들을 좌절시키는 주제, `WHERE` 절의 제약을 다룬다. 핵심은 이것을 "Cassandra가 못 하는 것"의 목록으로 외우는 게 아니라, **저장 구조에서 필연적으로 따라 나오는 결과**로 이해하는 것이다. 그러면 외울 필요가 없어진다.

### 법칙 1: 파티션 키는 반드시, 전부, 등치(`=`)로

쿼리는 거의 항상 파티션 키 *전체*를 `=`로 지정해야 한다. 이유는 2절에서 본 그대로다. 파티션 키를 모르면 토큰을 계산할 수 없고, 토큰이 없으면 코디네이터(coordinator) 노드는 "도대체 어느 노드에게 이 질문을 던져야 하는가"를 알 수 없다. 파티션 키는 데이터의 주소이고 주소 없이는 배달이 불가능하다.

범위 조회(`>`, `<`)가 안 되는 이유도 같다. 해시는 값의 순서를 보존하지 않으므로(2절), `WHERE pk > 'm'`은 토큰 공간에서 아무 의미 없는 구간이 된다.

> **단 하나의 예외 — `token()` 함수:** 파티션 키 *값*에 대한 범위는 불가능하지만, 그 값의 *토큰*에 대한 범위는 합법이다. `WHERE token(payment_id) > ? AND token(payment_id) <= ?`는 토큰 링을 구간 단위로 훑는 정당한 쿼리이며, 이것이 바로 전체 테이블 스캔이나 Spark 같은 분석 커넥터가 모든 파티션을 노드별로 나눠 순회하는 방식이다. 하지만 이것은 "특정 값 범위"가 아니라 "링의 한 조각"을 읽는 것이므로, `WHERE pk > 'm'` 같은 값 기준 범위 조회의 대체재가 결코 아니다.

```cql
-- ✅ 파티션 키 전체를 등치로
SELECT * FROM payment_events_by_payment WHERE payment_id = 'pay_123';

-- ❌ 파티션 키에 범위 조건
SELECT * FROM payment_events_by_payment WHERE payment_id > 'pay_100';
-- Error: Only EQ relations are supported on partition key
```

### 법칙 2: 클러스터링 컬럼은 선행 컬럼부터 순서대로(prefix), 마지막에 범위 1개

클러스터링 컬럼에 거는 조건은 두 가지 규칙을 동시에 만족해야 한다.

1. **prefix 규칙**: 선언된 순서대로, 앞에서부터 빈틈없이 조건을 채워야 한다. `(c1, c2, c3)`이 있을 때 `c2`에 조건을 걸려면 `c1`에도 조건이 있어야 하고, `c3`에 조건을 걸려면 `c1`, `c2` 모두에 조건이 있어야 한다.
2. **마지막 범위 규칙**: 등치(`=`)로 연속해서 채운 다음, *그 바로 다음 컬럼 하나*에만 범위(`>`, `<`, `>=`, `<=`)를 걸 수 있다. 범위를 건 컬럼 뒤의 컬럼에는 더 이상 조건을 걸 수 없다. (`IN`은 범위가 아니라 "여러 등치의 묶음"에 가깝다. 그래서 `c1 IN (...) AND c2 = ...`처럼 IN 뒤의 컬럼에도 조건을 이어 걸 수 있다는 점이 진짜 범위 연산과 다르다. 다만 IN은 지정한 값마다 별도의 슬라이스를 읽으므로 값 개수가 늘면 비용도 그만큼 곱으로 늘어, 큰 IN 목록은 피하는 게 좋다.)

왜 그런가? 3절의 다층 정렬 그림을 다시 보자. 파티션 내부는 `category → sold_at → item_id` 순으로 정렬되어 있다. 이것은 정확히 "정렬된 1차원 줄(sorted run)"이다. 디스크 위에서 우리가 효율적으로 할 수 있는 일은 단 하나 — **이 줄에서 연속된 구간(contiguous slice)을 잘라내는 것**이다.

```text
정렬된 줄에서 "연속 구간"을 자를 수 있는 조건:
  category="book" AND sold_at >= 09:00      → ✅ 한 덩어리로 연속
  ├──┬──┬──┐
  │  │  │  │  ← 여기부터 여기까지, 한 칼로 잘림

"category는 무시하고 sold_at >= 09:00"      → ❌ 줄 전체에 흩어짐
  book/10:00  food/11:00  book/09:00  food/08:00
   ✓           ✓           ✓           ✗
  ← 자르고 싶은 행들이 줄 곳곳에 흩어져 있어 "한 칼"로 못 자른다
```

`category`를 건너뛰고 `sold_at`에만 조건을 거는 것이 불가능한 이유가 여기 있다. `sold_at`은 *각 `category` 블록 안에서만* 정렬되어 있지, 파티션 전체에서 정렬되어 있지 않다. 마치 전화번호부에서 "성을 무시하고 이름이 '철수'인 사람"을 찾으려면 책 전체를 넘겨야 하는 것과 같다. 그건 슬라이스가 아니라 풀스캔이다.

범위 조건을 한 컬럼에만, 그것도 마지막 등치 컬럼 *바로 다음*에만 걸 수 있는 이유도 동일하다. 한 컬럼에 범위를 걸어 구간을 열어버리면, 그 안에서는 다음 컬럼의 정렬이 더 이상 "연속 구간"을 보장하지 못한다. 범위를 두 군데 걸면 더는 한 칼로 자를 수 있는 모양이 아니게 된다.

```cql
-- ✅ prefix를 채우고 마지막에 범위 1개
SELECT * FROM sales_by_region
 WHERE region = 'KR' AND category = 'book' AND sold_at >= '2026-06-01';

-- ❌ category(선행 컬럼)를 건너뜀
SELECT * FROM sales_by_region
 WHERE region = 'KR' AND sold_at >= '2026-06-01';
-- Error: PRIMARY KEY column "sold_at" cannot be restricted
--        as preceding column "category" is not restricted

-- ❌ 범위를 두 컬럼에
SELECT * FROM sales_by_region
 WHERE region = 'KR' AND category > 'a' AND sold_at >= '2026-06-01';
-- Error: Clustering column "sold_at" cannot be restricted (range on "category")
```

### 한 장의 표로 정리하는 WHERE 법칙

| 키 종류 | 허용되는 조건 | 이유(물리적 근거) |
|---|---|---|
| 파티션 키 | `=` (또는 `IN`), **전체 컬럼** 필수 | 해시→토큰으로 노드를 찾아야 함. 순서 정보 없음 |
| 클러스터링 컬럼 (선행) | `=` (prefix로 빈틈없이) | 정렬된 줄에서 연속 구간을 열어두기 위함 |
| 클러스터링 컬럼 (마지막 1개) | 범위(`>`,`<`,`>=`,`<=`) 허용 | 정렬된 줄의 연속 슬라이스를 자르는 연산 |
| 범위 뒤의 클러스터링 컬럼 | 조건 불가 | 범위가 열린 뒤엔 하위 정렬이 무의미 |
| 일반(non-key) 컬럼 | 기본 불가 (`ALLOW FILTERING` 필요) | 정렬·인덱스 없음 → 전수 검사 필요 |

> **흔한 오해:** "`WHERE`에 `=`만 잘 쓰면 어떤 컬럼이든 조건을 걸 수 있겠지." → 아니다. `=`든 `>`든, 그 컬럼이 *PRIMARY KEY의 적절한 위치*에 있지 않으면 거부된다. 조건 연산자의 문제가 아니라 **저장 정렬 구조 상의 위치**의 문제다.

### `ORDER BY`도 같은 법칙의 그림자다

CQL의 `ORDER BY`는 RDB처럼 임의 정렬을 하지 못한다. 클러스터링 컬럼의 정렬 순서를 그대로 따르거나, 정확히 뒤집는 것(전체 컬럼을 일괄 reverse)만 가능하다. 이유는 이제 자명하다 — 데이터가 이미 그 순서로 디스크에 누워 있고, Cassandra가 할 수 있는 건 "그대로 읽기" 또는 "거꾸로 읽기"뿐이기 때문이다. 새로운 정렬을 하려면 전체를 메모리에 올려 재정렬해야 하는데, 그건 Cassandra가 거부하는 종류의 "조용히 비싸지는" 연산이다.

## 5. ALLOW FILTERING: 왜 이것이 위험 신호인가

법칙 2의 표 마지막 줄에서 보았듯, 일반 컬럼이나 prefix를 어긴 조건으로 쿼리하면 Cassandra는 에러를 낸다. 그런데 그 에러 메시지 끝에는 종종 유혹적인 한 줄이 붙어 있다.

```text
... If you want to execute this query despite the performance
unpredictability, use ALLOW FILTERING
```

`ALLOW FILTERING`을 붙이면 그 쿼리는 실행된다. 그래서 초심자는 "아, 이 키워드만 붙이면 되는구나"라고 생각하고 코드 곳곳에 뿌린다. 이것은 production을 향한 시한폭탄이다.

### `ALLOW FILTERING`이 실제로 하는 일

`ALLOW FILTERING`은 "내가 효율적으로 자를 수 없는 조건이지만, 그래도 **데이터를 다 읽어 와서 메모리에서 하나씩 검사해라**"라는 명령이다. 문제는 그 "다 읽어 오는" 범위가 통제 불가능하다는 데 있다.

파티션 키를 지정하지 않은 채 `ALLOW FILTERING`을 쓰면, 코디네이터는 **클러스터의 모든 노드에게 "네 파티션을 전부 스캔해서 조건에 맞는 걸 보내라"**고 요청한다. 노드 수가 100개면 100개 노드가 디스크를 갈아엎는다.

```text
SELECT * FROM payments WHERE status = 'FAILED' ALLOW FILTERING;

  Coordinator
      │ "status='FAILED'인 행 다 찾아와"
      ├──────────────► Node1: [전체 파티션 스캔] ─┐
      ├──────────────► Node2: [전체 파티션 스캔] ─┤
      ├──────────────► Node3: [전체 파티션 스캔] ─┼─► 다 모아서
      ├──────────────► ...                        │   메모리에서 필터링
      └──────────────► NodeN: [전체 파티션 스캔] ─┘
   읽는 데이터: 수억 행   →   결과: 12행
   (12행을 위해 수억 행을 디스크에서 읽고 네트워크로 옮긴다)
```

### 왜 "비결정적"인가가 진짜 함정이다

`ALLOW FILTERING`의 가장 음험한 점은 느린 게 아니라 **느려지는 시점을 예측할 수 없다**는 것이다.

- 개발 환경: 테이블에 1만 행. `ALLOW FILTERING` 쿼리가 5ms에 돌아온다. "어, 빠른데?" 하고 배포한다.
- 6개월 후 production: 테이블에 5억 행. 같은 쿼리가 30초 걸리고, 그 사이 코디네이터의 힙을 가득 채워 GC를 유발하고, 다른 정상 쿼리들까지 타임아웃에 빠뜨린다.

쿼리 자체는 한 글자도 바뀌지 않았는데, 데이터가 자라면서 성능이 **절벽처럼** 무너진다. 더 나쁜 것은, 이것이 부하가 가장 높은 시간대(데이터도 많고 트래픽도 많을 때)에 터진다는 점이다. RDB의 풀스캔은 그래도 옵티마이저가 경고를 주거나 인덱스를 추가해 구제할 여지가 있지만, Cassandra에서 `ALLOW FILTERING`은 *설계가 틀렸다는 신호*다.

> **현장 규칙:** `ALLOW FILTERING`을 production 코드에 쓰고 싶어지면, 그건 "이 쿼리를 위한 테이블이 아직 없다"는 뜻이다. 답은 `ALLOW FILTERING`이 아니라 **새 테이블(또는 보조 인덱스 전략)을 추가하는 것**이다. 예외는 *이미 단일 파티션으로 좁혀진 뒤* 그 파티션 내부의 소량 데이터를 추가 필터링하는 경우 정도다. 이때는 `WHERE pk = ... AND some_col = ... ALLOW FILTERING`처럼 파티션이 고정되어 있어 비용이 한 파티션 크기로 제한되므로 상대적으로 안전하다.

[[06 - 데이터 모델링 2 - 고급 타입과 안티패턴]]에서 보겠지만, "그래도 가끔은 키가 아닌 컬럼으로 조회하고 싶다"는 요구는 보조 인덱스(secondary index), SASI, Storage-Attached Index(SAI, Cassandra 5.0의 핵심 신기능) 같은 도구로 푸는 것이지 `ALLOW FILTERING`으로 푸는 것이 아니다.

## 6. 파티션 크기 설계: 너무 큰 파티션이라는 시한폭탄

Query-First로 테이블을 잘 잡았다 해도, 한 가지 검증을 더 통과해야 한다. **"이 파티션은 얼마나 커질 수 있는가?"** 이것은 ④ 물리 모델 단계의 핵심 작업이며, 가장 많은 production 사고가 여기서 난다.

### 파티션은 쪼개지지 않는다 — 한 노드에 통째로 산다

먼저 물리적 사실을 못 박자. **하나의 파티션은 결코 노드 사이에 쪼개지지 않는다.** 파티션 키 `user_42`에 속한 모든 행은, 그 행이 1개든 1억 개든, 전부 같은 노드(들)에 산다. 복제본도 마찬가지로 각 복제 노드에 통째로 복사된다. 이것이 "단일 파티션 읽기가 빠르다"는 보장의 대가다 — 빠른 대신, **한 파티션의 크기가 한 노드의 능력에 종속된다.**

따라서 어떤 파티션이 무한정 자라면, 그 파티션을 가진 노드는 다음과 같은 고통을 겪는다.

- 읽기 지연 증가: 거대한 파티션의 일부만 읽어도 인덱스 탐색과 디스크 I/O가 커진다.
- 복구·스트리밍 지연: 노드 교체/추가 시 거대 파티션을 통째로 스트리밍해야 한다.
- 컴팩션 부담: [[09 - Compaction 전략]]에서 다룰 컴팩션이 거대 파티션을 다룰 때 메모리/CPU를 폭식한다.
- JVM 힙 압박과 GC: 특정 연산은 파티션의 상당 부분을 힙에 올린다. 거대 파티션은 GC 폭주와 노드 다운으로 이어질 수 있다.

### 권장 상한: "100MB / 10만 셀(cells)"이라는 경험칙

업계와 DataStax가 오래 공유해 온 경험칙(rule of thumb)은 다음과 같다. *절대 법칙이 아니라 설계 가드레일로* 받아들이자.

- **파티션 크기: 대략 100MB 이하** (안전하게는 10MB 안팎을 목표로, 100MB를 경고선으로)
- **파티션당 셀(cell) 수: 대략 10만(100,000) 이하**
- **파티션당 행(row) 수: 대략 10만 이하** — 위 셀 기준과 *별개의* 독립 가드레일이다. 셀 ≈ (행 수 × 일반 컬럼 수)이므로, 컬럼이 여러 개인 테이블이라면 행 수가 10만에 닿기 한참 전에 셀 한계가 먼저 걸린다. 세 기준 중 어느 하나라도 넘으면 경고로 받아들이는 것이 안전하다.

여기서 "셀(cell)"이라는 단위가 중요하다. Cassandra의 저장 단위는 행이 아니라 셀이다.

```text
파티션의 셀 수 ≈ (행 수) × (행당 일반 컬럼 수)

예) 한 행에 일반 컬럼이 10개인 테이블에서
    행이 10,000개라면  → 셀 ≈ 10,000 × 10 = 100,000 셀 (경고선 도달)
```

컬럼이 많은 와이드(wide)한 테이블은 행 수가 적어 보여도 셀 기준으로는 일찍 한계에 닿는다. 파티션 크기를 추정할 때는 행 수만 보지 말고 **행 수 × 컬럼 수 × 평균 셀 크기**로 바이트와 셀 수를 함께 추산해야 한다.

여기서 한 가지는 분명히 해 두자. 위 10만이라는 숫자는 어디까지나 *설계 가드레일*이지 시스템이 강제하는 한계가 아니다. 저장 포맷상의 구조적 하드 리밋은 파티션당 약 20억(2³¹) 셀이지만, 실무에서 그 근처는 의미가 없다 — 읽기 지연, GC 압박, 컴팩션 비용이 폭증해 노드가 그 한참 전에 먼저 무너지기 때문이다. 그래서 실제로 신경 써야 하는 선은 언제나 하드 리밋이 아니라 가드레일이다.

`nodetool tablehistograms <keyspace> <table>` 로 실제 파티션 크기 분포(특히 99퍼센타일과 max)를 운영 중에 점검할 수 있다([[12 - 운영과 트러블슈팅]]).

```bash
nodetool tablehistograms myks payment_events_by_payment
# Partition Size (bytes)와 Cell Count의 분포에서
# Max / 99% 값이 가드레일을 넘는지 확인
```

### Unbounded partition: 가장 흔한 설계 실수

이 모든 위험은 하나의 안티패턴으로 수렴한다. **시간이 갈수록 무한히 자라는(unbounded) 파티션.** 전형적인 예를 보자.

```cql
-- ⚠️ 위험: sensor_id 하나가 영원히 자란다
CREATE TABLE readings_bad (
    sensor_id    text,
    reading_time timestamp,
    value        double,
    PRIMARY KEY ( sensor_id, reading_time )
) WITH CLUSTERING ORDER BY ( reading_time DESC );
```

이 테이블에서 센서 `s1`은 1초에 한 번 측정값을 쌓는다고 하자. 하루 86,400행, 한 달 약 260만 행, 1년 약 3,150만 행. `sensor_id='s1'` 파티션 하나가 끝없이 부풀어, 몇 달 안에 가드레일을 한참 넘기고, 결국 그 센서가 배정된 노드를 무릎 꿇린다. 설계 당시에는 멀쩡해 보였다는 점이 이 함정의 본질이다.

### 해법: 시간 버킷팅(time bucketing)으로 파티션을 자르기

해법은 파티션 키에 **시간 버킷(bucket)을 추가**해 "무한히 자라는 한 파티션"을 "유한한 크기의 많은 파티션"으로 쪼개는 것이다. 3절의 복합 파티션 키가 바로 이 일을 한다.

```cql
-- ✅ 일(day) 단위로 버킷팅
CREATE TABLE readings_by_day (
    sensor_id    text,
    day          date,           -- 시간 버킷: 파티션 키의 일부
    reading_time timestamp,
    value        double,
    PRIMARY KEY ( (sensor_id, day), reading_time )
) WITH CLUSTERING ORDER BY ( reading_time DESC );
```

이제 `(sensor_id='s1', day='2026-06-26')`가 하나의 파티션이고, 그 안에는 하루치(86,400행)만 들어간다. 파티션은 매일 새로 생기므로 영원히 자라지 않는다. 버킷 크기(일/시간/주/월)는 **데이터 유입 속도**에 맞춰 고른다.

```text
버킷 단위 선택의 기준:
  유입이 빠름(초당 다건)  → 시간(hour) 버킷, 심하면 분 버킷
  유입이 보통(분당 다건)  → 일(day) 버킷
  유입이 느림(시간당 다건) → 월(month) 버킷

목표: 한 버킷 파티션이 ~10만 셀 / ~100MB 안에 들도록 역산
```

버킷팅의 대가는 "여러 날에 걸친 조회"가 한 번의 쿼리로 안 된다는 것이다. 6월 한 달을 조회하려면 30개의 `day` 버킷을 각각(또는 `IN` / 애플리케이션 병렬 fan-out으로) 읽어야 한다. 이것은 비용이 아니라 *정직함*이다. 30일치를 읽는 일은 곧 30개 파티션을 읽는 일이고, Cassandra는 그 비용을 숨기지 않을 뿐이다.

```cql
-- 6월 한 달 조회: 버킷을 명시적으로 fan-out
SELECT * FROM readings_by_day
 WHERE sensor_id = 's1' AND day IN ('2026-06-01','2026-06-02', ... ,'2026-06-30')
   AND reading_time >= '2026-06-01' AND reading_time < '2026-07-01';
-- (실무에서는 IN의 파티션 수를 적게 유지하고, 보통은 앱에서 날짜별로
--  병렬 쿼리를 던져 코디네이터 한 곳에 부하가 몰리는 것을 피한다)
```

## 7. Hotspot: 데이터가 한곳으로 쏠릴 때

파티션 *크기* 문제가 "한 파티션이 너무 크다"였다면, hotspot 문제는 "트래픽이 소수 파티션·소수 노드로 몰린다"이다. 둘은 다른 문제이고 둘 다 파티션 키 선택에서 비롯된다.

### 저카디널리티 파티션 키: 우편번호가 너무 적을 때

파티션 키의 카디널리티(cardinality, 서로 다른 값의 개수)가 낮으면, 토큰 링 위에 소수의 파티션만 생기고 데이터·트래픽이 그 소수 노드로 집중된다.

```cql
-- ⚠️ 안티패턴: country를 파티션 키로
PRIMARY KEY ( country, user_id )
```

전 세계 사용자 대부분이 `country='KR'`이라면, `KR` 파티션 하나가 거대해지고(크기 문제) 동시에 그 파티션을 가진 노드로 읽기·쓰기가 폭주한다(hotspot 문제). 100개 노드가 있어도 실질적으로는 한두 노드만 일한다. 수평 확장의 이점이 통째로 사라진다.

> **가이드:** 파티션 키는 **높은 카디널리티 + 고른 분포**를 가져야 한다. 이상적으로는 클러스터의 노드 수보다 훨씬 많은 서로 다른 파티션이 생겨, 데이터와 트래픽이 토큰 링 전체에 골고루 흩어져야 한다.

### 단조 증가 파티션 키: 항상 "링의 같은 끝"만 두드린다

더 미묘한 hotspot은 파티션 키가 **단조 증가(monotonically increasing)**할 때 생긴다. 타임스탬프나 순차 ID를 파티션 키로 쓰는 경우다.

```cql
-- ⚠️ 안티패턴: 시간 버킷을 "단독" 파티션 키로
PRIMARY KEY ( bucket_minute, event_id )   -- bucket_minute = 분 단위 시각
```

해시는 값을 흩뿌리지만 문제는 *지금 이 순간 쓰기가 모두 같은 파티션 키 값*을 향한다는 데 있다. 현재가 10:00이면 모든 신규 쓰기가 `bucket_minute='10:00'` 파티션으로 간다. 그 파티션은 단 하나의 노드(와 복제본)에 있으므로, 그 순간 클러스터의 쓰기 부하 전체가 한 노드로 쏠린다. 1분 뒤엔 다음 노드로 옮겨가며 hotspot이 "링을 따라 행진"한다. 시계열을 "시간만으로" 파티셔닝하면 항상 이 함정에 빠진다.

### 회피: 합성 키(salting)와 복합 분산

해법은 파티션 키에 **분산을 유발하는 차원을 섞는 것**이다.

1. **자연 키와 결합**: 시간만 쓰지 말고 `(sensor_id, day)`처럼 카디널리티 높은 자연 차원을 함께 묶는다. 동시간대 쓰기라도 `sensor_id`가 제각각이라 서로 다른 토큰으로 흩어진다. (6절의 버킷팅이 크기 문제와 hotspot 문제를 *동시에* 푸는 이유다.)

2. **샤딩/솔팅(sharding/salting)**: 자연 키조차 한쪽으로 쏠린다면, 인위적인 분산 컬럼(예: `bucket = hash(id) % N`)을 파티션 키에 추가한다.

```cql
-- 핫한 단일 키를 N개 샤드로 인위 분산
CREATE TABLE hot_counter_sharded (
    entity_id  text,
    shard      int,            -- 0..N-1, 쓰기 시 무작위/해시로 분배
    event_time timestamp,
    payload    text,
    PRIMARY KEY ( (entity_id, shard), event_time )
);
-- 읽을 때는 N개 샤드를 fan-out해서 합친다 (쓰기 분산의 대가)
```

샤딩은 쓰기를 N개 파티션으로 흩어 hotspot을 N분의 1로 줄이지만, 읽을 때 N개 파티션을 모아야 하는 비용을 지운다. *쓰기 hotspot을 읽기 fan-out과 맞바꾸는* 전형적인 트레이드오프다. 카운터/리더보드처럼 단일 키에 쓰기가 폭주하는 경우에만 신중히 쓴다.

| 파티션 키 안티패턴 | 증상 | 처방 |
|---|---|---|
| 저카디널리티 (`country`) | 소수 거대 파티션 + 노드 쏠림 | 고카디널리티 자연 키로 교체/결합 |
| 단조 증가 (`timestamp` 단독) | hotspot이 링을 행진 | 자연 차원과 결합, 또는 솔팅 |
| 무한 성장 (버킷 없는 자연 키) | unbounded partition | 시간 버킷팅 |
| 단일 핫 키 (인기 셀럽 ID) | 한 파티션에 쓰기 폭주 | 샤딩(salting) + 읽기 fan-out |

## 8. 실전 1: 사용자별 활동 타임라인

이제 방법론을 처음부터 끝까지 한 바퀴 돌려 보자. 요구사항은 흔하다. **"사용자가 자기 페이지에서 최근 활동을 시간 역순으로, 무한 스크롤로 본다."**

### ① 개념 / ② 워크플로 → 쿼리 도출

화면을 따라가며 접근 패턴을 적는다.

```text
Q1: 특정 user의 활동을 최신순으로 페이지네이션 조회
Q2: 특정 활동 1건의 상세 조회 (알림 클릭 → 딥링크)
Q3: 특정 user의 특정 종류(예: 'PAYMENT') 활동만 최신순 조회
```

### ③ 논리 / ④ 물리 → 쿼리당 테이블

Q1은 user별 시간 역순이 핵심이다. `user_id`를 파티션 키로, 시간을 클러스터링 컬럼으로 내림차순 저장한다. 단, 활동이 활발한 사용자의 파티션이 unbounded가 되지 않도록 월(month) 버킷을 더한다.

```cql
-- Q1: 사용자 활동 타임라인 (최신순, 월 버킷)
CREATE TABLE activity_by_user (
    user_id      uuid,
    bucket_month text,          -- 'YYYY-MM' 시간 버킷
    activity_id  timeuuid,      -- 시간 순서 + 유일성을 한 컬럼에
    activity_type text,
    target_id    text,
    payload      text,
    PRIMARY KEY ( (user_id, bucket_month), activity_id )
) WITH CLUSTERING ORDER BY ( activity_id DESC );
```

여기서 `timeuuid`(시간 기반 UUID)를 클러스터링 컬럼으로 쓴 것이 묘수다. 이 한 컬럼이 **시간 정렬과 행 유일성을 동시에** 해결한다. 같은 밀리초에 두 활동이 발생해도 timeuuid는 서로 달라 충돌하지 않으면서도 시간 순서를 보존한다. (`now()`/`maxTimeuuid()`/`minTimeuuid()`로 범위 조회도 가능하다.)

```cql
-- Q1 실행: 최신 페이지 (파티션 1개 순차 읽기)
SELECT * FROM activity_by_user
 WHERE user_id = 8b... AND bucket_month = '2026-06'
 LIMIT 20;

-- 다음 페이지: 마지막으로 본 activity_id 이후로 이어서
SELECT * FROM activity_by_user
 WHERE user_id = 8b... AND bucket_month = '2026-06'
   AND activity_id < 7f...      -- 커서 기반 페이지네이션
 LIMIT 20;
```

이 페이지네이션이 왜 효율적인지 보자. 파티션이 `activity_id DESC`로 디스크에 누워 있으므로, `activity_id < cursor`는 4절의 "정렬된 줄의 연속 슬라이스"를 정확히 한 칼로 자르는 연산이다. RDB의 `OFFSET` 기반 페이지네이션처럼 앞부분을 세어 건너뛰는 낭비가 없다. 깊은 페이지로 가도 비용이 일정하다.

Q2(활동 단건 상세)는 user를 거치지 않고 activity_id로 바로 찾아야 한다. 그래서 **별도 테이블**을 만든다 — 같은 데이터를 다른 파티션 키로 중복 저장하는, 1절에서 말한 그 비정규화다.

```cql
-- Q2: 활동 단건 조회 (activity_id가 파티션 키)
CREATE TABLE activity_by_id (
    activity_id   timeuuid,
    user_id       uuid,
    activity_type text,
    target_id     text,
    payload       text,
    PRIMARY KEY ( activity_id )
);
```

Q3(종류별 필터)은 `ALLOW FILTERING`의 유혹이 오는 지점이다. 단호히 거부하고 *또 하나의 테이블*을 만든다. `activity_type`을 파티션 키에 포함시켜 종류별로 미리 갈라 둔다.

```cql
-- Q3: 사용자의 특정 종류 활동 (type을 파티션 키에 포함)
CREATE TABLE activity_by_user_and_type (
    user_id       uuid,
    activity_type text,
    bucket_month  text,
    activity_id   timeuuid,
    target_id     text,
    payload       text,
    PRIMARY KEY ( (user_id, activity_type, bucket_month), activity_id )
) WITH CLUSTERING ORDER BY ( activity_id DESC );
```

세 테이블이 같은 활동을 세 벌 저장한다. RDB 감각으로는 끔찍하지만 Cassandra 감각으로는 정확하다. 핵심은 **쓰기 시점에 세 테이블을 함께 갱신**하는 책임이 애플리케이션(또는 `BATCH`)에 있다는 것이다. 활동이 불변(append-only)이라 갱신 이상의 위험이 거의 없다는 점이 이 설계를 안전하게 만든다. 같은 파티션을 공유한다면 원자적 `BATCH`로 묶을 수도 있다([[11 - LWT Batch Counter 내부]]에서 logged batch의 의미와 비용을 다룬다).

## 9. 실전 2: 시계열 센서 데이터

두 번째 예제는 IoT/모니터링의 전형, **시계열(time-series)**이다. 결제 시스템이라면 "초당 거래량 메트릭", "서비스별 응답시간 시계열" 같은 형태로 똑같이 나타난다.

요구사항:

```text
Q1: 특정 센서의 특정 기간(하루/몇 시간) 측정값을 시간순 조회
Q2: 특정 센서의 가장 최근 측정값 N개 조회 (대시보드 실시간 패널)
```

6·7절에서 쌓은 모든 원칙을 한 테이블에 녹인다.

```cql
CREATE TABLE sensor_readings (
    sensor_id    text,
    bucket_hour  text,          -- 'YYYY-MM-DD-HH', 시간 버킷
    reading_time timestamp,
    value        double,
    unit         text,
    PRIMARY KEY ( (sensor_id, bucket_hour), reading_time )
) WITH CLUSTERING ORDER BY ( reading_time DESC )
  AND default_time_to_live = 7776000;   -- 90일 후 자동 만료 (TTL)
```

설계 선택을 하나씩 정당화하자.

- **`(sensor_id, bucket_hour)` 복합 파티션 키**: `sensor_id`는 카디널리티를 제공해 hotspot을 막고(7절), `bucket_hour`는 파티션을 1시간 단위로 잘라 unbounded를 막는다(6절). 초당 1건이면 한 파티션은 3,600행 안팎으로, 가드레일에 한참 못 미치는 건강한 크기다.
- **`reading_time DESC` 클러스터링**: Q2의 "최근 N개"가 `LIMIT N` 한 번으로 끝나도록 최신순으로 디스크에 눕힌다.
- **`default_time_to_live`(TTL)**: 시계열은 오래된 데이터의 가치가 급락한다. TTL을 걸면 90일 지난 데이터가 자동으로 만료되어 파티션·디스크가 무한히 자라지 않는다. (TTL 만료는 tombstone을 만들고, 이는 [[09 - Compaction 전략]] — 특히 TWCS, Time Window Compaction Strategy — 와 직결된다. 시계열 + TTL은 TWCS와 함께 쓰는 것이 정석이다.)

```cql
-- Q1: 특정 시간 버킷 내 기간 조회 (단일 파티션 슬라이스)
SELECT reading_time, value FROM sensor_readings
 WHERE sensor_id = 'temp-01' AND bucket_hour = '2026-06-26-10'
   AND reading_time >= '2026-06-26 10:15:00'
   AND reading_time <  '2026-06-26 10:45:00';

-- Q2: 최근 5개 (DESC 정렬 덕에 그냥 앞 5줄)
SELECT reading_time, value FROM sensor_readings
 WHERE sensor_id = 'temp-01' AND bucket_hour = '2026-06-26-10'
 LIMIT 5;
```

Q1의 `reading_time` 범위 조건이 합법인 이유를 다시 확인하자. 파티션 키(`sensor_id`, `bucket_hour`)가 전부 `=`로 고정되었고(법칙 1), 그다음 단 하나의 클러스터링 컬럼 `reading_time`에만 범위가 걸렸다(법칙 2). 교과서적인 단일 파티션 슬라이스다.

여러 시간대를 가로지르는 조회는 8절 타임라인과 같은 트레이드오프를 만난다. "오전 10시~12시"를 보려면 `bucket_hour`를 `10`, `11`, `12` 세 개로 fan-out 해야 한다. 버킷 단위(시간/일)는 이 fan-out 빈도와 파티션 크기 사이의 균형으로 정한다. 대시보드가 보통 "한 시간 창"을 본다면 시간 버킷이, "하루 추이"를 본다면 일 버킷이 fan-out을 최소화한다. **버킷 단위는 데이터 유입 속도와 지배적 조회 창(query window) 둘 다를 보고 정한다** — 이것이 시계열 모델링의 가장 실전적인 판단이다.

## 핵심 요약

- **Query-First는 RDB 정규화의 거울상이다.** RDB는 데이터 구조(정규화) → 쿼리 순으로 가지만, Cassandra는 개념 모델 → 애플리케이션 워크플로(필요 쿼리 도출) → 논리 모델(쿼리당 테이블) → 물리 모델 순으로 간다. `JOIN`이 없기 때문에 쿼리를 먼저 알아야만 테이블을 만들 수 있다.
- **"한 테이블 = 한 access pattern"이 원칙이고, 데이터 중복(비정규화)은 미덕이다.** Cassandra의 쓰기는 싸고 읽기의 `JOIN`/필터링은 통제 불가능하게 비싸므로, 쓰기를 더 시켜 읽기를 거저 만드는 거래를 한다. 중복 데이터의 일관성은 쓰기 시점에 애플리케이션이 책임진다.
- **PRIMARY KEY = 파티션 키 + 클러스터링 컬럼.** 파티션 키는 Murmur3 해시→토큰으로 *어느 노드*인지를 정하고(등치만 가능, 순서 없음), 클러스터링 컬럼은 파티션 *내부의 디스크 정렬 순서*를 정한다(범위·정렬 가능). `(a,b)`로 묶으면 복합 파티션 키, 여러 클러스터링 컬럼은 다층 정렬을 만든다.
- **`WHERE` 법칙은 저장 정렬 구조의 그림자다.** 파티션 키는 전체를 `=`로, 클러스터링은 선행부터 빈틈없이(prefix) 채운 뒤 마지막 한 컬럼에만 범위. 임의 컬럼 조건이 불가능한 이유는 "정렬된 줄에서 연속 슬라이스만 자를 수 있기" 때문이다.
- **`ALLOW FILTERING`은 전 파티션 스캔을 부른다.** 위험의 본질은 느린 것이 아니라 *비결정적*이라는 것 — 데이터가 자라면 어느 날 절벽처럼 무너진다. 필요해지면 그건 "테이블이 없다"는 신호이지 키워드를 붙일 신호가 아니다.
- **파티션은 노드 간에 쪼개지지 않는다.** 권장 가드레일은 대략 100MB / 10만 셀(셀 = 행 × 일반 컬럼). 무한 성장(unbounded) 파티션은 시간 버킷팅으로 자른다.
- **Hotspot은 파티션 키 선택의 문제다.** 저카디널리티·단조 증가 키를 피하고, 고카디널리티 자연 차원과 결합하거나 샤딩(salting)으로 쓰기를 흩는다(읽기 fan-out과의 트레이드오프).
- **실전 패턴**: 시간 역순 타임라인은 `(엔티티, 버킷)` 파티션 + `timeuuid DESC` 클러스터링 + 커서 페이지네이션. 시계열은 `(sensor, bucket_hour)` 파티션 + `reading_time DESC` + TTL(TWCS와 함께).
