지금까지의 장들이 Cassandra가 "정상적으로 동작할 때 내부에서 무슨 일이 일어나는가"를 다뤘다면, 이 장은 정반대의 질문에서 출발한다. **"무언가 어긋났을 때, 우리는 그것을 어떻게 알아채고, 무엇을 보고, 어떻게 되돌리는가?"** 결제 시스템을 운영해 본 엔지니어라면 이 감각에 익숙할 것이다. 시스템이 잘 돌아갈 때는 아무도 데이터베이스를 들여다보지 않는다. 사람들이 `nodetool`을 치기 시작하는 건 p99 latency 그래프가 천장을 뚫었을 때, 디스크가 차오를 때, 한 노드가 죽었을 때다.

이 장이 답하려는 핵심 질문들은 다음과 같다.

- Cassandra는 hinted handoff와 read repair라는 "자가 치유" 장치가 있는데, **왜 굳이 repair를 주기적으로 돌려야 하는가?** 안 돌리면 정확히 무슨 일이 일어나는가?
- Merkle tree는 어떻게 수 GB짜리 데이터의 차이를 단 몇 KB의 해시 교환으로 찾아내는가?
- `gc_grace_seconds`라는 마법의 숫자는 왜 repair 주기와 묶여 있고, 이걸 어기면 왜 **삭제한 데이터가 되살아나는가?**
- large partition은 왜 "조용한 시한폭탄"인가? 어떻게 미리 탐지하고, 발견하면 무엇을 해야 하는가?
- 노드 하나가 완전히 죽었을 때, `decommission`과 `removenode`와 `replace_address`는 각각 언제 쓰는가?
- 새벽 3시에 호출기가 울렸을 때, 어떤 순서로 무엇을 봐야 하는가?

RDB 운영 경험이 있다면 한 가지를 먼저 버려야 한다. RDB에서는 보통 하나의 마스터가 "진실의 단일 원천"이다. 데이터가 일관적이지 않으면 그건 버그다. Cassandra에서는 **여러 replica가 각자 조금씩 다른 진실을 들고 있는 것이 정상 상태**이고, 그 차이를 시간을 들여 수렴시키는 것이 운영의 본질이다. 이 패러다임 전환을 받아들이면 이 장의 모든 도구가 같은 목적을 향한다는 게 보인다. **엔트로피(entropy, 무질서)를 측정하고 줄이는 것.**

## Anti-entropy Repair: 왜 "자가 치유"만으로는 부족한가

[[10 - Write Path와 Read Path]]에서 본 것처럼, Cassandra는 쓰기 일관성을 두 가지 백그라운드 장치로 보조한다. **hinted handoff**(쓰기 시점에 down된 replica의 변경분을 coordinator가 보관했다가 살아나면 전달)와 **read repair**(읽기 시점에 replica 간 불일치를 발견하면 최신값으로 동기화). 이 둘이 있는데 왜 repair가 또 필요한가? 답하려면 두 장치의 **구멍**을 정확히 봐야 한다.

### hinted handoff의 구멍

hinted handoff는 두 가지 상황에서 새어 나간다.

1. **hint window 초과**: coordinator는 down된 노드의 hint를 `max_hint_window_in_ms`(5.0 cassandra.yaml에서는 duration 표기 `max_hint_window`, 기본 **3시간**) 동안만 보관한다. 노드가 3시간을 넘겨 죽어 있으면, 그 이후에 도착한 쓰기에 대한 hint는 **아예 생성되지 않는다**. 부팅된 노드는 자기가 놓친 3시간 이후의 데이터를 영영 모른다.
2. **hint 자체의 유실**: hint는 coordinator의 로컬 디스크에 쌓인다. 그 coordinator가 hint를 전달하기 전에 죽거나 디스크가 손상되면 hint는 사라진다. coordinator가 과부하라 hint 적재를 throttle하거나 drop해도 마찬가지다.

> 흔한 오해: "노드가 잠깐 죽었다 살아나면 hinted handoff가 알아서 다 맞춰준다." → 3시간을 넘겼거나, hint를 들고 있던 coordinator가 그사이 죽었거나, 쓰기 시점에 그 replica가 아예 응답 불가였던 게 아니라 coordinator로 선택조차 안 됐다면 구멍이 생긴다.

### read repair의 구멍

read repair는 더 근본적인 한계가 있다. **읽히지 않는 데이터는 절대 고쳐지지 않는다.** read repair는 그 row를 누군가 `SELECT`할 때만 발동한다. 일주일에 한 번 읽히는 결제 원장 row, 혹은 90일째 아무도 조회하지 않은 오래된 트랜잭션이 어느 replica에서만 누락됐다면, read repair는 영원히 그것을 건드리지 않는다. `read_repair = 'NONE'`(4.0+에서 테이블 옵션으로 조정 가능, 기본값은 `BLOCKING`)이거나 consistency level이 낮으면 비교 자체가 일어나지 않는다.

### 그래서 남는 것: 영구 drift(permanent drift)

이 두 구멍을 통과한 불일치는 **스스로 사라지지 않는다.** 이것을 영구 drift라 부른다. 시간이 지날수록 replica들은 미세하게 어긋나고, 그 어긋남이 누적된다. anti-entropy repair는 이 영구 drift를 **능동적으로** 찾아 메우는 유일한 장치다. read repair가 "수동적·기회주의적(누가 읽을 때만)"이라면, repair는 "능동적·전수조사(모든 데이터를 빠짐없이)"다.

```text
   불일치를 메우는 3가지 장치의 커버리지

   쓰기 시점        읽기 시점          주기적(능동)
   ┌─────────┐    ┌─────────┐      ┌──────────────┐
   │ hinted  │    │  read   │      │ anti-entropy │
   │ handoff │    │ repair  │      │   repair     │
   └─────────┘    └─────────┘      └──────────────┘
   구멍:          구멍:            구멍:
   - 3h window    - 안 읽히면      - (없음, 전수)
   - hint 유실      안 고침        - 비용: I/O+네트워크
   - throttle     - CL/옵션 의존    - 운영 부담
        │              │                   │
        └──────────────┴───────────────────┘
                       ▼
            세 장치의 교집합이 "진짜 일관성"
            repair만이 전체를 빠짐없이 덮는다
```

### 가장 무서운 시나리오: 삭제의 부활(zombie data)

영구 drift가 단순히 "오래된 값이 남는" 문제라면 그나마 견딜 만하다. 그러나 **삭제와 결합하면 데이터가 되살아난다.** 이것이 repair를 "선택"이 아니라 "의무"로 만드는 결정적 이유다.

[[08 - Storage Engine 내부]]에서 본 tombstone을 떠올려 보자. Cassandra에서 `DELETE`는 데이터를 지우는 게 아니라 **tombstone(묘비)**이라는 특수 마커를 쓴다. tombstone은 "이 시점 이후로 이 데이터는 없는 것으로 친다"는 의미의, timestamp를 가진 또 하나의 write다. 분산 환경에서 삭제를 마커로 표현하는 이유는 명확하다. 만약 삭제가 그냥 "지우기"라면, 그때 마침 죽어 있던 replica는 그 row를 여전히 들고 있다가 살아나서 "어? 다른 노드엔 이 데이터가 없네? 내가 가진 걸 전파해야지" 하며 **삭제를 되돌려 버린다.** tombstone은 이 부활을 막는다. "없음"을 명시적인, timestamp를 가진 사실로 기록하기 때문이다.

문제는 tombstone이 영원히 남으면 디스크가 무한정 차오른다는 것이다. 그래서 Cassandra는 일정 시간이 지난 tombstone을 compaction 과정에서 **물리적으로 제거(purge)**한다. 그 일정 시간이 바로 `gc_grace_seconds`이고, 기본값은 **864000초 = 10일**이다.

이제 죽음의 시나리오가 보인다.

```text
   gc_grace 안에 repair를 안 돌렸을 때: 좀비 데이터의 부활

   t=0     DELETE row X  →  Replica A: tombstone(X) 기록
                            Replica B: (이때 down) → tombstone 못 받음
                            Replica B: 여전히 X의 원본 데이터 보유

   t=10일  gc_grace 경과 →  Replica A: compaction이 tombstone(X) purge
                            이제 A는 X를 "없는 데이터"로 인식 (마커도 없음)

   t=11일  Replica B 복귀 →  B는 여전히 X 보유
                            read repair/repair 시 A에는 X도 tombstone도 없음
                            → B의 X가 "새 데이터"로 보여 A로 전파
                            → 삭제했던 X가 부활 (ZOMBIE)
```

관건은 분명하다. **tombstone이 purge되기 전에, 모든 replica가 그 tombstone을 받았다는 보장이 있어야 한다.** 그 보장을 만드는 유일한 방법이 `gc_grace_seconds` 주기 안에 클러스터 전체를 한 번 이상 repair하는 것이다. 그래서 Cassandra 운영의 제1계명은 이것이다.

> **모든 노드는 `gc_grace_seconds`(기본 10일) 안에 적어도 한 번 repair되어야 한다.** 이걸 어기면 삭제한 데이터가 되살아날 수 있다. repair는 디스크 청소나 성능 최적화가 아니라, **데이터 정합성의 안전장치**다.

| | RDB였다면 | Cassandra |
|---|---|---|
| 삭제 | row를 즉시 물리 삭제 | tombstone 마커 기록, gc_grace 후 purge |
| 일관성 | 단일 마스터가 진실 | 여러 replica가 수렴해야 진실 |
| 불일치 복구 | 거의 발생 안 함(트랜잭션) | repair로 능동적으로 메워야 함 |
| 삭제 부활 | 불가능 | repair 안 하면 가능(zombie) |

## Merkle Tree: 어떻게 수 GB의 차이를 KB로 찾아내는가

repair가 "모든 데이터를 빠짐없이 비교한다"고 했는데, 그러면 replica A와 B가 가진 수십 GB의 데이터를 통째로 네트워크로 주고받아 한 row씩 대조하나? 그건 불가능하다. 여기서 Cassandra는 **Merkle tree**(머클 트리, 해시 트리)라는 우아한 자료구조를 쓴다.

### 직관: 책 두 권의 차이를 찾기

두꺼운 책 두 권이 거의 같은데 어딘가 한 글자가 다르다고 하자. 전체를 한 글자씩 비교하면 너무 느리다. 대신 이렇게 한다.

1. 책 전체의 해시(체크섬)를 구해 비교한다. 같으면 끝. 다르면 어딘가 다르다.
2. 책을 절반으로 나눠 각 절반의 해시를 비교한다. 같은 절반은 건너뛰고, 다른 절반만 파고든다.
3. 다른 절반을 또 절반으로 나눈다. 이것을 반복하면, **log 시간에** 다른 페이지(혹은 단락)에 도달한다.

Merkle tree가 정확히 이 일을 한다. **차이가 있는 영역만 위에서 아래로 좁혀 들어가므로, 차이가 적으면 거의 데이터를 안 옮기고 끝난다.**

### 내부 메커니즘

Cassandra에서 repair가 시작되면, repair 대상이 된 token range를 책임지는 각 replica가 자기 데이터로부터 Merkle tree를 만든다.

```text
   Merkle Tree (단순화: 하나의 token range를 8개 leaf로 분할)

                       Root = H(L|R)
                      /              \
              H(LL|LR)                H(RL|RR)
             /        \              /        \
        H(a|b)      H(c|d)      H(e|f)      H(g|h)
        /   \        /   \       /   \       /   \
       a     b      c     d     e     f     g     h    ← leaf
       └─ 각 leaf = 하나의 sub-range에 속한 row들의 해시 ─┘

   - leaf: token range를 잘게 쪼갠 sub-range. 그 안의 모든 row를
           해시해서 하나의 값으로 압축.
   - 부모 노드: 자식 두 해시를 합쳐 다시 해시.
   - root: 전체 range를 대표하는 단 하나의 해시.
```

비교 절차는 이렇다.

1. coordinator(repair를 시작한 노드, validation 주관)가 각 replica에게 "이 range의 Merkle tree를 만들어 달라"고 요청한다. 이 트리 생성 과정이 **validation compaction**이다. 디스크의 SSTable들을 전부 읽어 해시를 계산하므로 **I/O가 무겁다**(이게 repair가 부하를 주는 주된 이유다).
2. 각 replica가 만든 트리를 coordinator로 보낸다. **트리는 데이터가 아니라 해시 집합이라 작다.** 수 GB 데이터라도 트리는 수 KB~수 MB 수준.
3. coordinator가 root부터 비교한다. root 해시가 같으면 → 두 replica는 이 range에서 완전히 동일 → **스트리밍 없음, 끝.** root가 다르면 자식으로 내려가며 다른 가지만 추적한다.
4. 차이가 확정된 leaf(sub-range)들을 모아, 그 범위의 **실제 데이터만** replica 간에 스트리밍(streaming)으로 교환한다. 각자 더 최신(timestamp 큰) 버전으로 수렴한다.

```text
   tree A:  Root=X1     tree B:  Root=X2   (X1≠X2 → 차이 있음)
            /    \                /    \
          ok     DIFF           ok     DIFF
                 /  \                   /  \
               ok   LEAF#5            ok   LEAF#5  ← 여기만 다름

   → leaf #5에 해당하는 sub-range의 데이터만 스트리밍
   → 나머지 99.9%는 해시만 비교하고 건너뜀
```

### Merkle tree의 해상도 한계 — "왜 멀쩡한 데이터까지 옮겨지나"

여기엔 실무에서 자주 부딪히는 함정이 하나 있다. **Merkle tree의 leaf 개수(해상도)는 유한하다.** Cassandra는 한 트리(repair 대상이 된 token range 하나)의 leaf 수에 상한을 둔다(역사적으로 약 32,768개 = 깊이 15 수준이었고, 지금은 `repair_session_max_tree_depth`로 조정 가능). leaf는 partition 단위가 아니라 token range를 잘게 쪼갠 sub-range 단위라는 점에 유의하라. 한 leaf가 거대한 token range를 대표하게 되면, 그 range 안에서 **단 한 row만 달라도 leaf 해시가 달라지고**, 그러면 그 leaf가 커버하는 **전체 range의 데이터를 통째로 스트리밍**한다.

> 함정: full repair에서 "겨우 몇 row 어긋났는데 수백 MB가 스트리밍되더라"는 현상은 버그가 아니라 Merkle tree 해상도의 자연스러운 결과다. **subrange repair**(아래에서 설명)로 한 번에 다루는 range를 좁히면 leaf의 해상도가 올라가 over-streaming을 줄일 수 있다. Cassandra Reaper가 자동으로 하는 일이 바로 이 잘게 쪼개기다.

> 또 하나의 함정: validation compaction은 데이터를 **실제로 다 읽는다.** repair는 "공짜 헬스 체크"가 아니라 클러스터에 상당한 read I/O와 네트워크 부하를 준다. 그래서 트래픽 낮은 시간대에, 한 번에 너무 많은 range를 돌리지 않도록 스케줄링하는 것이 핵심이다.

## Repair의 종류: full, incremental, primary range, subrange

repair는 한 종류가 아니다. 무엇을, 얼마나, 어떻게 비교할지에 따라 비용과 효과가 크게 달라진다. 이 선택을 잘못하면 repair가 클러스터를 마비시킬 수도, 데이터를 못 메울 수도 있다.

### full repair vs incremental repair

가장 중요한 구분이다. 이 구분의 뿌리는 [[09 - Compaction 전략]]에서 본 SSTable 개념에 있다.

**full repair**는 매번 **모든** SSTable(이미 예전에 repair된 데이터 포함)을 읽어 Merkle tree를 만든다. 정확하지만, 같은 오래된 데이터를 매번 다시 검증하므로 낭비가 크다. 1년 전 데이터는 한 번 맞춘 뒤로 안 변했는데도 매주 다시 해시한다.

**incremental repair**(Cassandra 2.1+, 4.0에서 안정화)는 이 낭비를 없앤다. 핵심 아이디어는 SSTable에 **"이미 repair됨(repaired)"** 플래그를 다는 것이다.

```text
   incremental repair의 핵심: repaired / unrepaired 분리

   디스크의 SSTable들
   ┌──────────────────────────────────────────┐
   │  repaired set        │   unrepaired set    │
   │  (이미 검증·동기화)   │   (새로 쓰여 미검증) │
   │  ┌────┐ ┌────┐       │   ┌────┐ ┌────┐     │
   │  │SST1│ │SST2│  ...  │   │SST9│ │SST10│ ...│
   │  └────┘ └────┘       │   └────┘ └────┘     │
   └──────────────────────────────────────────┘
          ▲                       ▲
          │                       │
      다음 repair 때          repair는 여기만
      건너뜀 (이미 OK)        검증 → 끝나면
                              repaired로 이동(anti-compaction)
```

incremental repair를 돌리면 **unrepaired set만** 검증·스트리밍한다. 끝나면 그 SSTable들을 repaired로 표시한다. 새 flush로 만들어진 SSTable은 항상 unrepaired에서 시작한다. 그래서 매 repair가 "지난번 이후 새로 쓰인 것만" 보면 되어 **반복 비용이 극적으로 줄어든다.**

이때 따라오는 중요한 부수 효과가 **anti-compaction**이다. 하나의 SSTable에 repaired 데이터와 unrepaired 데이터가 섞여 있으면, repair가 끝나는 시점에 그 SSTable을 둘로 쪼개야 한다(repaired 조각 / unrepaired 조각). compaction도 이 둘을 **절대 섞지 않는다.** repaired와 unrepaired는 영원히 다른 compaction 풀에서 산다.

| | full repair | incremental repair |
|---|---|---|
| 검증 대상 | 모든 SSTable | unrepaired SSTable만 |
| 반복 비용 | 매번 전체(비쌈) | 신규분만(저렴) |
| anti-compaction | 없음 | repaired/unrepaired 분리 |
| 정확성 | 가장 높음 | 높음(4.0+ 신뢰성 개선) |
| 권장 용도 | 주기적 검증, 의심스러울 때 | 일상적·잦은 repair |

> 역사적 주의: incremental repair는 3.x 시절 repaired set이 오염되는 버그들로 평판이 나빴다(over-streaming 누적, repaired/unrepaired 불일치). **Cassandra 4.0에서 대대적으로 재설계**되어 신뢰성이 크게 올라갔고, 5.0은 그 위에 서 있다. 그래도 운영 정석은 "incremental을 주력으로 자주 돌리되, **가끔(예: 분기별) full repair로 전체를 한 번 훑는** 하이브리드"다. 확신 없는 부분: 조직마다 full repair 주기 권장은 다르며, 어떤 팀은 incremental을 아예 안 쓰고 Reaper 기반 subrange full만 돌린다.

### primary range repair (`-pr`)

RF=3인 클러스터에서, 한 token range는 3개 노드에 복제된다. 만약 모든 노드에서 그냥 `nodetool repair`를 돌리면, **같은 range가 3번 검증된다**(자기가 책임지는 모든 range를 보므로). 끔찍한 중복이다.

`nodetool repair -pr`(primary range only)은 각 노드가 **자기가 "primary(첫 번째 replica)"인 range만** repair하게 한다. 모든 노드가 `-pr`로 한 바퀴 돌면, 클러스터의 모든 range가 **정확히 한 번씩** 커버된다. 중복 제거의 정석이다.

```text
   -pr 없이: 같은 range를 여러 노드가 중복 검증
   node1 repair → range R 검증
   node2 repair → range R 또 검증   ← 낭비
   node3 repair → range R 또 검증   ← 낭비

   -pr 사용: 각 노드는 자기 primary range만
   node1 repair -pr → R1만
   node2 repair -pr → R2만
   node3 repair -pr → R3만
   → 모든 노드가 한 바퀴 = 전체를 정확히 1회 커버
```

> 주의: `-pr`은 **모든 노드에서 빠짐없이 실행해야** 전체가 커버된다. 한 노드라도 빼먹으면 그 노드의 primary range는 검증되지 않는다. `-pr`은 incremental(`-inc` 또는 옵션)과 조합 시 주의가 필요하며, 다중 데이터센터에서는 DC별로 신경 써야 한다.

### subrange repair

하나의 token range를 더 잘게 쪼개(예: `-st`/`-et`로 시작·끝 토큰 지정) 그 조각만 repair한다. 위에서 본 Merkle tree 해상도 문제를 해결하고, 한 번에 거는 부하를 잘게 나눠 **점진적·연속적 repair**를 가능하게 한다. 사람이 손으로 토큰 범위를 계산해 쪼개는 건 비현실적이라 실무에서는 이것을 **자동화 도구(Reaper)**에 맡긴다.

### 주요 `nodetool repair` 옵션

```bash
# 전체 클러스터의 모든 keyspace를 full repair (가장 무거움, 보통 권장 안 함)
nodetool repair

# 특정 keyspace/table만
nodetool repair my_keyspace my_table

# primary range만 (모든 노드에서 한 바퀴 = 전체 1회 커버)
nodetool repair -pr

# full repair 강제 (incremental이 기본인 환경에서)
nodetool repair -full

# 로컬 데이터센터만 (DC 간 네트워크 비용 회피)
nodetool repair -local        # 또는 --in-local-dc

# 특정 토큰 subrange만
nodetool repair -st <start_token> -et <end_token>

# 병렬성 제어 (sequential / parallel / dc-parallel)
nodetool repair -seq          # 한 번에 한 replica씩 (부하↓, 느림)

# 진행 상황은 별도 터미널에서
nodetool netstats             # 스트리밍 진행 확인
```

> 버전 주의: repair 기본 동작은 버전마다 달랐다. Cassandra 2.2~3.x에서는 incremental이 기본이었다가, 4.0 이후 운영 권장이 바뀌었다. **5.0에서도 `nodetool repair`의 정확한 기본(full vs incremental)은 배포/설정에 따라 다를 수 있으니, 프로덕션에서는 항상 명시적으로 `-full` 또는 incremental 옵션을 지정**하는 것이 안전하다. 추측에 의존하지 말 것.

### Cassandra Reaper: repair를 사람이 손으로 돌리지 않는 이유

지금까지의 옵션을 보면, "gc_grace 안에 모든 노드를 빠짐없이, 부하 안 주게, subrange로 쪼개서, 실패하면 재시도하며 repair한다"는 게 사람이 cron으로 관리하기엔 너무 복잡하다는 걸 알 수 있다. 그래서 사실상 표준이 된 도구가 **Cassandra Reaper**(원래 Spotify, 현재 커뮤니티/여러 벤더가 유지)다.

Reaper가 해주는 일:

- token range를 자동으로 잘게 쪼개(subrange) 한 번에 거는 부하를 제한.
- **동시 실행 segment 수를 제어**해 클러스터가 repair로 죽지 않게 함(intensity 조절).
- 실패한 segment만 재시도, 진행률 추적, 웹 UI 대시보드.
- 스케줄링: 각 keyspace/table을 `gc_grace`보다 짧은 주기로 자동 반복.

운영 철학은 명확하다. **repair는 "이벤트"가 아니라 "항상 백그라운드에서 천천히 도는 연속 프로세스"여야 한다.** Reaper는 그것을 구현하는 표준 방식이다.

## Large Partition: 조용한 시한폭탄

[[05 - 데이터 모델링 1 - Query First]]와 [[06 - 데이터 모델링 2 - 고급 타입과 안티패턴]]에서 partition key 설계가 데이터 모델링의 전부라고 강조했다. 그 설계 실수가 운영 단계에서 터지는 가장 흔하고 치명적인 형태가 **large partition**(거대 파티션)이다.

### 왜 large partition이 위험한가 — 내부 메커니즘

Cassandra의 거의 모든 핵심 연산은 **partition 단위**로 일어난다. 한 partition은 항상 하나의 노드 집합(replica들)에 통째로 모여 있고, 절대 쪼개져 여러 노드에 분산되지 않는다. 이 설계는 빠른 조회를 위한 것이지만, partition이 너무 커지면 다음이 무너진다.

1. **읽기 시 메모리 압박과 GC**: 넓은 범위를 읽거나 partition 전체를 스캔하면, 그 데이터를 힙(heap)에 올려야 한다. partition이 수백 MB~GB면 한 번의 쿼리가 거대한 객체를 만들어 **young/old gen을 압박하고 GC pause를 유발**한다. [[10 - Write Path와 Read Path]]의 read path에서 본 row cache, partition index도 거대 partition 앞에서 무력해진다.
2. **partition index의 비효율**: SSTable 안에서 partition 내 특정 row를 찾으려면 partition index를 탄다. partition이 거대하면 index entry 자체가 커지고, 탐색이 느려진다. (5.0의 BIG/BTI 포맷 개선이 이를 일부 완화하지만 근본 해결은 아니다.)
3. **compaction 실패**: compaction은 여러 SSTable의 같은 partition을 병합한다. 거대 partition을 병합하려면 그만큼의 자원이 필요하고, 최악의 경우 **OOM으로 compaction이 실패**해 SSTable이 쌓이고 읽기가 더 느려지는 악순환에 빠진다.
4. **repair 실패와 hotspot**: 거대 partition은 한 노드에 부하를 집중시켜(hotspot) 그 노드만 느려지거나 죽게 만든다. 클러스터는 멀쩡한데 특정 노드만 p99가 튄다면 large partition을 의심한다.

> 하드 리밋: 운영 가이드(대략 100MB·row 수십만)와 별개로, 하나의 partition은 물리적으로 **최대 약 20억 개(2^31)의 cell**(대략 row 수 × 컬럼 수)까지만 담을 수 있다. 무한 성장 모델은 그 전에 GC·compaction으로 노드를 먼저 망가뜨리지만, 이론적으로 이 벽에도 부딪힌다. "권장 100MB"는 성능 가이드, "20억 cell"은 절대 천장이라고 기억하면 된다.

> RDB 직관과의 차이: RDB에서 "한 row가 크다"는 건 보통 BLOB 칼럼 정도의 국소적 문제다. Cassandra의 large partition은 **하나의 논리적 키 아래 수백만 row가 무한정 누적되는** 구조적 문제다. 예: `partition key = device_id`인데 한 디바이스가 1초에 100개씩 센서 값을 영원히 쓴다면, 그 partition은 끝없이 자란다.

### 탐지: large partition을 어떻게 찾는가

large partition의 무서운 점은 **조용하다**는 것이다. 평소엔 멀쩡하다가, 그 partition을 읽거나 compaction 대상이 되는 순간 터진다. 그래서 **증상이 나타나기 전에 능동적으로 탐지**해야 한다.

**1) compaction 경고 로그** — 가장 직접적인 신호. `compaction_large_partition_warning_threshold`(5.0 기본 **100MiB**)를 넘는 partition을 compaction하면 `system.log`에 경고가 찍힌다.

```text
WARN  [CompactionExecutor:5] BigTableWriter.java
  Writing large partition my_keyspace/sensor_data:device_42
  (256.000MiB) to sstable ...
```

이 한 줄이 데이터 모델이 잘못됐다는 가장 명확한 증거다. `device_42`라는 partition key가 256MiB를 차지한다는 뜻.

**2) `nodetool tablehistograms`** — 특정 테이블의 partition 크기, cell 수, 읽기/쓰기 latency를 백분위로 보여준다.

```bash
nodetool tablehistograms my_keyspace sensor_data
```

```text
Percentile  SSTables  Write Latency  Read Latency  Partition Size  Cell Count
                          (micros)      (micros)        (bytes)
50%             1.00         35.43        182.79          1597            12
95%             3.00         88.15       2759.43        943127          5000
99%             4.00        152.32      24601.18      268435456       1500000  ← !!
Max             5.00        315.85     943127.00      536870912       3000000
```

여기서 99% partition size가 256MB(268435456 bytes), Max가 512MB라면 명백한 large partition 문제다. p50은 1.6KB로 정상인데 p99/Max가 폭발하는 **꼬리(tail)가 핵심**이다.

**3) `nodetool tablestats`** — 테이블 단위 요약. `Compacted partition maximum bytes`가 거대 partition의 직접 지표다.

```bash
nodetool tablestats my_keyspace.sensor_data
```

```text
        Compacted partition minimum bytes: 87
        Compacted partition maximum bytes: 536870912   ← 512MB, 위험
        Compacted partition mean bytes: 4096
        Maximum live cells per slice (last five minutes): 1500000  ← 한 쿼리가 150만 cell
        Average tombstones per slice (last five minutes): ...
```

**4) `system.size_estimates`** — 각 노드가 자기 테이블의 partition 수와 크기를 추정해 기록하는 시스템 테이블. 클러스터 전반의 분포를 프로그램으로 훑을 때 유용(단, 추정치라 정밀하지 않음).

```cql
SELECT keyspace_name, table_name, range_start, range_end,
       mean_partition_size, partitions_count
FROM system.size_estimates
WHERE keyspace_name = 'my_keyspace' AND table_name = 'sensor_data';
```

### 해결: 버킷팅(bucketing)으로 재모델링

large partition은 설정으로 못 고친다. **데이터 모델을 바꿔야** 한다. 기본 기법은 partition key에 **버킷(bucket)** 차원을 추가해 무한 성장을 잘게 끊는 것이다.

```text
   문제: partition이 무한 성장
   PRIMARY KEY (device_id, ts)
   device_42 → [row, row, row, ... 수백만 개 ...] ← 한 파티션에 다 쌓임

   해결: 시간 버킷으로 분할
   PRIMARY KEY ((device_id, day_bucket), ts)
   (device_42, 2026-06-26) → 하루치만 (수만 개, 관리 가능)
   (device_42, 2026-06-27) → 다음 날은 새 파티션
   ...
```

```cql
-- before: 무한 성장 파티션
CREATE TABLE sensor_data (
    device_id  text,
    ts         timestamp,
    value      double,
    PRIMARY KEY (device_id, ts)
);

-- after: day 단위 버킷으로 파티션 크기 제한
CREATE TABLE sensor_data (
    device_id  text,
    day_bucket date,        -- 버킷 차원 추가
    ts         timestamp,
    value      double,
    PRIMARY KEY ((device_id, day_bucket), ts)
) WITH CLUSTERING ORDER BY (ts DESC);
```

버킷 크기 선택은 트레이드오프다. 버킷이 너무 크면 여전히 large partition, 너무 작으면 한 조회가 여러 partition을 가로질러야 해(scatter) 비효율적이다. **목표는 partition을 대략 100MB 이하, row 수 십만 이하**로 유지하는 것. 쓰기 속도(초당 row 수)와 보존 기간을 곱해 미리 계산해야 한다.

> 결제 시스템 관점: `partition key = user_id`로 모든 트랜잭션을 한 partition에 쌓으면, 특정 인기 사용자(헤비 유저)의 partition이 폭발한다. `((user_id, yyyymm), tx_time)` 같은 월 버킷이 정석. [[13 - 실전 Event Sourcing on Cassandra]]에서 이벤트 스트림 파티셔닝을 더 깊게 다룬다.

## Tombstone 문제 진단: 큐 안티패턴의 그림자

large partition과 쌍을 이루는 운영 골칫거리가 **tombstone 누적**이다. 앞서 tombstone이 삭제 부활을 막는 안전장치라고 했는데, 이게 과하게 쌓이면 **읽기 성능을 파괴**한다.

### 왜 tombstone이 읽기를 죽이는가

원리는 단순하다. **Cassandra는 읽을 때 tombstone도 읽어야 한다.** "이 데이터는 삭제됨"을 알려면 그 tombstone을 메모리로 올려 살아있는 데이터와 대조해야 하기 때문이다. 그래서 `SELECT` 한 번이 100개의 살아있는 row를 반환하기 위해 **10만 개의 tombstone을 스캔**하는 일이 벌어진다. 이때 Cassandra는 경고하거나 아예 쿼리를 중단한다.

```text
WARN  Read 100 live rows and 100000 tombstone cells for query
  SELECT * FROM queue WHERE ... (see tombstone_warn_threshold)

ERROR Scanned over 100001 tombstones during query ...;
  query aborted (see tombstone_failure_threshold)
```

관련 임계값(5.0 기본):
- `tombstone_warn_threshold`: **1000** — 단일 쿼리가 이만큼 tombstone을 스캔하면 경고 로그.
- `tombstone_failure_threshold`: **100000** — 이만큼 넘으면 **쿼리 자체를 실패(abort)**시켜 노드를 보호.

### 큐 안티패턴(queue anti-pattern) — 가장 악명 높은 사례

Cassandra로 작업 큐(queue)를 구현하는 것은 대표적인 안티패턴이고, tombstone 문제의 교과서적 원인이다. 시나리오:

```text
   큐 안티패턴: 쓰고 → 처리하고 → 삭제, 무한 반복

   파티션 queue_partition 안에서:
   t1  INSERT job1, job2, job3 ...
   t2  처리 후 DELETE job1     → tombstone(job1)
   t3  처리 후 DELETE job2     → tombstone(job2)
   ...
   t9999  파티션 앞부분은 전부 tombstone, 뒤에 살아있는 job 몇 개

   "맨 앞 미처리 job 조회" SELECT
   → 앞쪽 수만 개의 tombstone을 전부 스캔하고 나서야
     살아있는 job에 도달 → 점점 느려지다 결국 abort
```

clustering order로 정렬된 partition에서, **앞쪽을 계속 삭제하고 뒤에서 읽는 패턴**이 최악이다. 매 읽기가 누적된 tombstone 무덤을 매번 다시 파헤친다. gc_grace(10일)가 지나기 전엔 purge도 안 되므로, 그동안 tombstone은 계속 쌓인다.

### tombstone 진단 명령

```bash
# 테이블의 tombstone/슬라이스 통계
nodetool tablestats my_keyspace.queue
#   → Average tombstones per slice (last five minutes): ...
#   → Maximum tombstones per slice (last five minutes): ...

# partition 단위 tombstone 분포
nodetool tablehistograms my_keyspace queue
```

```text
WARN  log 패턴으로 grep:
  grep -i "tombstone" /var/log/cassandra/system.log
  → "Read N live rows and M tombstone cells" 가 자주 보이면 적신호
```

### 해결 방향

- **삭제를 설계에서 줄인다.** TTL을 쓰더라도 같은 partition에서 대량 만료가 동시에 일어나면 tombstone 폭증은 똑같다. (TTL 만료도 tombstone을 만든다.)
- **partition을 시간 버킷으로 나누고 통째로 버린다.** 큐가 정말 필요하면 "삭제" 대신 "오래된 버킷 partition 전체를 drop". 개별 row 삭제보다 partition 삭제가 tombstone 비용이 훨씬 적다.
- **Cassandra를 큐로 쓰지 않는다.** 큐는 Kafka/SQS 같은 전용 시스템에 맡기는 게 정답. Cassandra는 큐를 위해 만들어지지 않았다.
- 큐가 불가피하면 `LeveledCompactionStrategy`로 tombstone purge를 촉진하거나 `gc_grace_seconds`를 (정합성 위험을 감수하고) 조정하는 우회책이 있으나, 근본은 설계 변경이다.

> 함정: `gc_grace_seconds`를 0으로 낮춰 tombstone을 빨리 없애려는 유혹은 위험하다. 앞서 본 좀비 데이터 부활 위험이 바로 살아난다. 단일 노드(RF=1)이거나, 즉시 모든 replica에 전파됨이 보장되는 특수한 경우가 아니면 함부로 건드리지 말 것.

## nodetool: 운영자의 청진기

`nodetool`은 Cassandra 노드의 JMX에 붙어 상태를 조회하고 명령을 내리는 핵심 CLI다. 결제 시스템 엔지니어에게 RDB의 `SHOW PROCESSLIST`나 `EXPLAIN`이 청진기였다면, Cassandra에서는 `nodetool`이 그 역할을 한다. 다만 명령이 노드 단위라는 점을 늘 의식해야 한다(`-h`로 원격 지정 가능하지만, 많은 정보가 **그 노드 로컬 관점**이다).

| 명령 | 무엇을 보여주나 / 하나 | 언제 쓰나 |
|---|---|---|
| `nodetool status` | 클러스터 전체 노드의 UP/DOWN(`UN`/`DN`), 소유 비율(Owns), load(디스크), DC/rack | 가장 먼저 치는 명령. 누가 죽었나, 데이터가 고르게 분산됐나 |
| `nodetool info` | 이 노드의 heap/off-heap 사용량, uptime, cache 히트율, hint 수, gossip/native 상태 | 단일 노드의 건강 상태 종합 |
| `nodetool tablestats` | 테이블별 read/write count·latency, SSTable 수, partition min/max/mean bytes, bloom filter, tombstone | 테이블 성능·large partition·tombstone 진단 |
| `nodetool tablehistograms` | 테이블의 latency/partition size/cell count 백분위(p50~Max) | 꼬리 지연(p99)과 large partition 정밀 진단 |
| `nodetool compactionstats` | 진행 중인 compaction, 남은 바이트, pending task 수 | compaction이 밀리는지(읽기 성능 직결) 확인 |
| `nodetool netstats` | 스트리밍(repair/bootstrap) 진행, read repair 통계, 노드 간 전송 상태 | repair/bootstrap 진행률, 스트리밍 hang 확인 |
| `nodetool tpstats` | thread pool별 active/pending/completed/**blocked**, **dropped message(MUTATION/READ 등)** | 과부하 진단. dropped mutation은 쓰기 유실 신호 |
| `nodetool flush` | memtable을 강제로 SSTable로 디스크에 내림 | 백업 전, 또는 memtable 비우기 |
| `nodetool drain` | 쓰기를 멈추고 모든 memtable flush, commitlog 정리. 노드를 안전 정지 상태로 | **노드 재시작/종료 직전 필수.** 재시작 시 commitlog replay 회피 |
| `nodetool cleanup` | 이 노드가 더 이상 책임지지 않는(다른 노드로 넘어간) 데이터를 물리 삭제 | **노드 추가 후** 기존 노드들에서 실행. 안 하면 디스크 안 줄어듦 |
| `nodetool repair` | anti-entropy repair 실행(앞 섹션 참고) | 정합성 유지(gc_grace 내 필수) |
| `nodetool rebuild` | 다른 DC에서 데이터를 통째로 스트리밍해 채움 | **새 데이터센터 추가** 시 초기 적재 |
| `nodetool snapshot` | SSTable의 하드링크 스냅샷 생성(백업) | 백업, 위험한 작업 전 안전장치 |
| `nodetool decommission` | 이 노드를 클러스터에서 정상 제거(데이터를 다른 노드로 스트리밍) | 살아있는 노드를 줄일 때 |
| `nodetool removenode` | **죽은** 노드를 강제 제거(다른 replica가 데이터 보충) | 노드가 복구 불가로 죽었을 때 |
| `nodetool assassinate` | gossip에서 노드를 강제 추방(데이터 이동 없음, 최후의 수단) | removenode/decommission이 먹히지 않는 좀비 노드 |

### 자주 읽는 출력 해부

**`nodetool status`** — 운영의 첫 화면.

```text
Datacenter: dc1
===============
Status=Up/Down  |/ State=Normal/Leaving/Joining/Moving
--  Address      Load       Tokens  Owns(effective)  Host ID    Rack
UN  10.0.1.11    245.3 GiB  256     33.4%            a1b2...     rack1
UN  10.0.1.12    251.7 GiB  256     33.1%            c3d4...     rack2
DN  10.0.1.13    248.9 GiB  256     33.5%            e5f6...     rack3   ← Down!
```

- 첫 두 글자: `U`=Up/`D`=Down, `N`=Normal/`L`=Leaving/`J`=Joining/`M`=Moving. `DN`이 보이면 노드가 죽은 것.
- `Load`: 디스크 사용량. 한 노드만 크게 튀면 데이터 불균형(hotspot) 의심.
- `Owns`: token range 소유 비율. vnode를 쓰면 보통 고르게 분산된다. 단, 위 예시의 `Tokens 256`은 **구버전 기본값**이고, **Cassandra 4.0+/5.0의 기본 `num_tokens`는 16**으로 줄었다(+`allocate_tokens_for_local_replication_factor`로 균형 배치). 토큰이 많을수록 분산은 고르지만 repair·스트리밍·gossip 부담이 커지는 트레이드오프 때문이다.

**`nodetool tpstats`** — 과부하의 진실.

```text
Pool Name                    Active  Pending  Completed  Blocked  All time blocked
MutationStage                     0        0   45231234        0                0
ReadStage                         8       42    9823412        0                0   ← pending 쌓임
CompactionExecutor                2        5      82344        0                0

Message type           Dropped
MUTATION                  1823   ← 쓰기 1823건이 timeout으로 버려짐!
READ                       102
```

`Dropped MUTATION`이 0이 아니면 **그 노드가 쓰기를 처리하지 못해 버리고 있다는 뜻**이다. coordinator는 consistency level을 만족하면 성공으로 응답했을 수 있지만, 이 노드의 복제분은 유실됐다. 바로 이 유실을 나중에 메우는 게 repair다(영구 drift의 원천 중 하나). pending이 계속 쌓이면 그 stage가 병목.

## 노드 라이프사이클: 추가, 교체, 제거

노드는 죽고, 추가되고, 교체된다. 각 상황에 정확한 명령을 써야 한다. 잘못 쓰면 데이터 유실이나 토큰 충돌이 생긴다. 판단 기준은 **"노드가 살아있나 죽었나"** 와 **"데이터를 옮길 것인가 보충할 것인가"** 두 축이다.

```text
                       데이터 처리 방식
                  이동(streaming out)   보충(다른 replica로부터)
   살아있는 노드 │  decommission        │  (해당 없음)
   ─────────────┼──────────────────────┼─────────────────────
   죽은 노드     │  (불가, 못 옮김)     │  removenode / replace
```

### bootstrap: 새 노드 추가

새 노드를 클러스터에 합류시키는 과정이 **bootstrap**이다. 설정 파일에 seed 노드와 cluster name을 맞추고 시작하면, 새 노드는 gossip으로 클러스터에 합류해 token range를 할당받고, 자기가 책임지게 된 데이터를 기존 replica들로부터 **스트리밍으로 받아온다**(`status`에서 `UJ` = Up/Joining).

```text
   bootstrap 흐름
   1. 새 노드 시작 (seeds, cluster_name 일치)
   2. gossip 합류 → token range 할당 (auto_bootstrap=true)
   3. 책임지게 된 range의 데이터를 기존 노드들에서 streaming
      → status: UJ (Joining)
   4. 완료 → UN (Normal). 이제 읽기/쓰기 참여
   5. ★ 기존 노드들에서 nodetool cleanup 실행
      → 더 이상 자기 것이 아닌 데이터 물리 삭제 (디스크 회수)
```

> 함정: bootstrap 후 `nodetool cleanup`을 안 하면, 새 노드로 소유권이 넘어간 데이터가 **기존 노드에 그대로 남아** 디스크가 줄지 않는다. cleanup은 추가가 끝난 뒤 기존 노드들에서 순차적으로 돌려야 한다. 또한 한 번에 여러 노드를 동시에 bootstrap하지 말 것(token range 계산이 꼬일 수 있음 — 보통 2분 간격 또는 순차).

### replace_address_first_boot: 죽은 노드 교체

하드웨어가 죽어 노드가 영영 못 돌아올 때, **같은 token을 물려받는 새 노드**로 교체하는 게 가장 깔끔하다. `cassandra-env.sh` 또는 jvm 옵션에 죽은 노드의 IP를 지정한다.

```bash
# jvm.options 또는 시작 옵션에 추가 (5.0)
-Dcassandra.replace_address_first_boot=10.0.1.13
```

이렇게 시작한 새 노드는 죽은 노드의 token을 그대로 이어받아, 그 range의 데이터를 살아있는 다른 replica들로부터 스트리밍해 채운다. `_first_boot` 접미사가 붙은 버전은 **최초 부팅 시 한 번만** 적용되어 실수로 재시작 때마다 교체 모드로 도는 것을 막는다(과거 `replace_address`의 안전 개선판).

> 교체가 removenode보다 나은 이유: removenode는 죽은 노드의 데이터를 **남은 replica들에 재분배**하므로 클러스터 전체의 token 분포가 바뀌고 스트리밍이 광범위하다. replace는 **동일 token을 물려받아** 그 자리만 채우므로 영향 범위가 좁고 빠르다. 단, replace 중에는 RF가 일시적으로 줄어든 셈이라 consistency 여유가 줄어든다(QUORUM 계산 주의).

### decommission: 살아있는 노드의 정상 제거

클러스터를 축소하거나 노드를 빼낼 때, **살아있는** 노드에서 `nodetool decommission`을 실행한다. 이 노드는 자기가 책임지던 데이터를 다른 replica들로 **스트리밍해 넘긴 뒤** 클러스터를 떠난다(`status`에서 `UL` = Up/Leaving).

```bash
# 빼낼 노드 자신에서 실행 (살아있어야 함)
nodetool decommission
# 진행: nodetool netstats 로 스트리밍 확인
```

데이터를 안전하게 넘기고 나가므로 **데이터 유실이 없다.** 가장 깔끔한 제거 방식. 단, 스트리밍이 끝날 때까지(데이터 양에 따라 수십 분~수 시간) 기다려야 한다.

### removenode: 죽은 노드의 강제 제거

노드가 이미 죽어서(`DN`) decommission을 실행할 수 없을 때 쓴다. **살아있는 다른 노드에서** 죽은 노드의 Host ID를 지정해 실행한다.

```bash
nodetool status                       # 죽은 노드의 Host ID 확인 (DN)
nodetool removenode <host-id>         # 다른 노드에서 실행
nodetool removenode status            # 진행 확인
nodetool removenode force             # 멈췄을 때 강제 진행
```

죽은 노드는 데이터를 내보낼 수 없으므로, **남은 replica들이 가진 복제본으로 부족분을 서로 채운다**(RF를 다시 충족시키기 위한 재복제). 그래서 decommission보다 무겁고, 그 사이 일시적으로 일관성 여유가 줄어든다.

### assassinate: 최후의 수단

removenode조차 먹히지 않는 "좀비 노드"(gossip에 유령처럼 남아 있는 항목)를 강제로 추방한다.

```bash
nodetool assassinate 10.0.1.13
```

> 경고: `assassinate`는 **데이터 재분배를 하지 않고** gossip 상태에서 노드를 그냥 지워버린다. 그 노드가 들고 있던 데이터의 복제본 수를 회복시키지 않으므로, 이후 반드시 repair로 정합성을 맞춰야 한다. 정말 다른 방법이 없을 때만 쓰는 핵폭탄.

| 명령 | 노드 상태 | 데이터 처리 | 안전도 | 용도 |
|---|---|---|---|---|
| bootstrap | 신규 | 받아옴(streaming in) | 안전 | 노드 추가 |
| replace_address_first_boot | 죽음 | 동일 token 채움 | 안전·빠름 | 하드웨어 교체 |
| decommission | 살아있음 | 내보냄(streaming out) | 가장 안전 | 정상 축소 |
| removenode | 죽음 | 남은 replica가 재복제 | 보통 | 복구불가 노드 제거 |
| assassinate | 좀비 | 없음(강제 추방) | 위험 | 최후의 수단 |

## 모니터링 지표와 임계 가이드

운영의 핵심은 "터지기 전에 안다"는 것이다. nodetool은 그 순간의 스냅샷이지만, 진짜 운영은 지표를 **시계열로 수집**(Prometheus + Grafana, Cassandra의 JMX exporter 또는 metrics 라이브러리)해 추세와 임계를 본다. 결제 시스템에서 SLO를 보듯, Cassandra도 핵심 지표에 알람을 걸어야 한다.

| 지표 | 정상/주의 가이드 | 의미·왜 중요한가 |
|---|---|---|
| **Read/Write p99 latency** | write p99 < 10ms, read p99 < 수십 ms(워크로드 의존). p99가 평소의 수배로 튀면 경보 | 꼬리 지연이 사용자 경험을 좌우. 평균은 거짓말한다. p99/p999를 본다 |
| **Pending compactions** | 평상시 한 자릿수. 수십~수백으로 지속 상승하면 위험 | compaction이 쓰기 속도를 못 따라감 → SSTable 누적 → 읽기 latency 폭증. `compactionstats`로 확인 |
| **Dropped mutations** | **0이어야 정상.** 0이 아니면 즉시 조사 | 쓰기 유실. 노드 과부하·GC·디스크 병목. 영구 drift의 원천 |
| **Hints (stored/in progress)** | 평상시 0에 가까움. 지속 증가는 어떤 노드가 down/과부하 | hint가 쌓인다 = 어떤 replica가 못 받고 있다. 누가 아픈지 추적 |
| **GC pause (특히 old gen/STW)** | G1 기준 단일 pause < 수백 ms. 초 단위 pause면 심각 | 긴 STW pause 동안 노드는 사실상 멈춤 → timeout → dropped → 연쇄 장애 |
| **Heap usage** | old gen이 GC 후에도 70~80% 이상 지속되면 위험 | 거대 partition, 과한 cache, tombstone 스캔이 heap을 먹음. OOM 전조 |
| **SSTables per read** | 적을수록 좋음(LCS는 보통 1~3). 두 자릿수면 read amplification 문제 | 한 읽기가 너무 많은 SSTable을 뒤짐 → 느림. compaction 전략 점검 |
| **Partition size (max)** | < 100MB 목표. 그 이상은 large partition 경보 | 위 large partition 섹션 전체 |
| **Disk usage** | < 50~70%. compaction은 일시적으로 추가 공간(최악 2배)이 필요 | 디스크가 차면 compaction 불가 → 악순환. STCS는 특히 여유 필요 |

> 평균(mean) latency만 보는 것은 결제 시스템에서 가장 흔한 모니터링 실수다. 평균이 5ms여도 p99가 2초면, 100건 중 1건의 결제가 2초씩 걸린다는 뜻. **항상 백분위(p99, p999)로 본다.** `tablehistograms`가 이걸 노드 단위로, 시계열 대시보드가 클러스터 단위로 보여준다.

> dropped mutation이 보이는데 "쓰기는 성공했다"고 안심하지 말 것. consistency level QUORUM(RF=3 → 2개)에서 한 replica가 drop해도 나머지 2개가 성공하면 클라이언트엔 성공으로 응답된다. 그 drop된 1개의 복제분은 **silent하게 빠진 상태**이고, read repair나 repair가 메우기 전까지 그 노드는 오래된 값을 들고 있다.

## JVM과 GC: Cassandra가 멈추는 가장 흔한 이유

Cassandra는 JVM 위에서 돈다. 그래서 운영 장애의 상당수가 **데이터베이스가 아니라 가비지 컬렉션(garbage collection) 문제**다. 긴 GC pause(stop-the-world, STW) 동안 노드는 gossip ping에 응답을 못 해 다른 노드들이 그 노드를 `DN`으로 판단하고, 그동안 쌓인 요청이 timeout되어 dropped mutation이 생기고, 부하가 다른 노드로 쏠려 연쇄적으로 무너진다.

### heap sizing 가이드

직관과 반대로, **heap을 무작정 크게 잡으면 더 나빠진다.** heap이 클수록 GC가 한 번에 훑어야 할 영역이 커져 STW pause가 길어지기 때문이다. Cassandra는 의도적으로 많은 데이터 구조를 **off-heap(힙 바깥, GC 대상이 아닌 native 메모리)** 에 둔다.

```text
   메모리 배치 (개념도)
   ┌─────────────────── 머신 RAM (예: 64GB) ──────────────────┐
   │ ┌── JVM Heap (8~16GB 권장 상한 부근) ──┐                  │
   │ │ memtable(일부), 쿼리 처리 객체,        │                  │
   │ │ 단기 객체... ← GC가 관리              │                  │
   │ └──────────────────────────────────────┘                  │
   │ ┌── Off-heap (GC 대상 아님) ──────────────────────────┐    │
   │ │ bloom filter, compression metadata, (구성에 따라)    │    │
   │ │ memtable, key/row cache ... ← native, GC pause 무관  │    │
   │ └──────────────────────────────────────────────────────┘    │
   │ ┌── OS page cache (남은 RAM 전부) ─────────────────────┐    │
   │ │ SSTable 파일을 캐싱 → 디스크 읽기를 메모리 속도로     │    │
   │ │ ★ Cassandra read 성능의 핵심. 이걸 위해 heap을 아낀다 │    │
   │ └──────────────────────────────────────────────────────┘    │
   └────────────────────────────────────────────────────────────┘
```

대략의 휴리스틱(워크로드·GC에 따라 다름):
- **G1GC 기준 heap 8GB~16GB, 보통 31GB를 넘기지 않는다**(31GB 부근에서 JVM의 compressed OOP가 비활성화되어 포인터가 커지는 비효율 발생).
- 머신 RAM의 1/4 정도를 heap으로, **나머지는 OS page cache에 양보**한다. page cache가 SSTable을 메모리에 들고 있어야 read가 빠르다.
- heap을 키워 GC 문제를 푸는 게 아니라, **off-heap 활용과 GC 튜닝, 그리고 데이터 모델(거대 partition 제거)로** 푼다.

### G1 vs (구) CMS, 그리고 5.0

- **CMS(Concurrent Mark Sweep)**: 과거 Cassandra의 기본. 정교하게 튜닝하면 작은 heap에서 낮은 pause를 냈지만, **튜닝이 극도로 까다롭고** 단편화로 인한 Full GC(긴 STW)에 취약했다. 사실상 deprecated.
- **G1GC(Garbage First)**: 현대 Cassandra의 권장 기본. heap을 region으로 나눠 "쓰레기가 가장 많은 region부터" 수집해 **목표 pause time(`-XX:MaxGCPauseMillis`)을 지향**한다. 큰 heap에서도 비교적 일정한 pause를 내고 튜닝이 단순해, 16GB 이상 heap에서 특히 유리. 대부분의 배포에서 G1이 정답.
- **Cassandra 5.0의 개선**: 5.0은 최신 JDK(특히 **JDK 17** 지원)를 정식 지원하며, 그 위에서 **ZGC/Shenandoah** 같은 저지연(low-pause) 컬렉터를 쓸 수 있는 길이 열렸다. 이 컬렉터들은 수십 GB heap에서도 pause를 한 자릿수 ms로 억제하는 것을 목표로 한다. 다만 어떤 컬렉터가 특정 워크로드에 최적인지는 벤치마킹이 필요하다. 확신 없는 부분: 5.0+JDK17 환경에서 ZGC가 G1보다 항상 낫다고 단정할 수는 없으며, 처리량/메모리 오버헤드 트레이드오프를 측정해야 한다.

> GC 로그를 켜는 것은 선택이 아니라 필수다. 장애 사후분석의 절반은 "그 시각에 긴 GC pause가 있었나"를 확인하는 일이다. `gc.log`에서 초 단위 pause가 보이면 거의 항상 그게 범인이다.

## 백업과 복구: snapshot과 incremental backup

[[08 - Storage Engine 내부]]에서 본 중요한 성질을 떠올리자. **SSTable은 immutable(불변)** 이다. 한 번 디스크에 쓰이면 절대 수정되지 않고, 새 데이터는 새 SSTable로 쌓이며 compaction이 합칠 뿐이다. 이 불변성이 Cassandra의 백업을 놀랍도록 단순하고 저렴하게 만든다.

### snapshot: 하드링크의 마법

`nodetool snapshot`은 SSTable 파일들을 복사하지 않는다. 대신 **하드링크(hard link)** 를 만든다.

```text
   snapshot = 하드링크 (데이터 복사 없음, 즉시·디스크 거의 0)

   data/my_ks/sensor-xxxx/
   ├── nb-1-big-Data.db ◄──┐ 같은 inode를 가리킴
   ├── nb-2-big-Data.db    │
   └── snapshots/
       └── 2026-06-26/
           ├── nb-1-big-Data.db ─┘ (하드링크)
           └── nb-2-big-Data.db
```

하드링크는 같은 물리적 데이터(inode)를 가리키는 두 번째 이름일 뿐이라 **생성이 즉각적이고 추가 디스크를 거의 쓰지 않는다.** SSTable이 immutable이기 때문에 가능하다. 원본이 절대 안 바뀌니, 링크가 가리키는 데이터도 그 시점 상태로 고정된다. compaction이 원본 SSTable을 지워도, 하드링크가 살아있는 한 데이터는 디스크에 남는다(링크 카운트가 0이 될 때까지).

```bash
# 전체 keyspace 스냅샷 (태그 지정 권장)
nodetool snapshot -t before_upgrade my_keyspace

# 특정 테이블만
nodetool snapshot -t fix_2026 -cf sensor_data my_keyspace

# 스냅샷 목록 / 정리
nodetool listsnapshots
nodetool clearsnapshot -t before_upgrade
```

> 실전 습관: 위험한 작업(스키마 변경, 대량 삭제, 업그레이드) **직전에 snapshot**을 떠라. 비용이 거의 0이고, 사고가 나면 생명줄이 된다. 단, snapshot은 **그 노드 로컬**에만 생긴다. 진짜 백업은 이 스냅샷 파일을 S3 등 외부 스토리지로 복사해 보관해야 노드 자체가 죽어도 살아남는다. snapshot이 쌓이면 (compaction이 지운 원본을 붙들고 있어) 디스크를 갉아먹으니 주기적으로 정리/외부 이관이 필요하다.

### incremental backup: 새 SSTable마다 자동 링크

`incremental_backups: true`(기본 false)를 켜면, flush로 **새 SSTable이 생길 때마다** 자동으로 `backups/` 디렉터리에 하드링크가 생성된다. snapshot이 "특정 시점의 전체 스냅샷"이라면, incremental backup은 "그 이후의 변경분을 빠짐없이 모으는" 것이다.

```text
   전체 백업 전략 = snapshot(기준점) + incremental(변경분)

   [snapshot 0] ──── incremental ──── incremental ──── ... 
   특정 시점          이후 새 SSTable    또 새 SSTable
   전체 스냅샷        하드링크 자동       하드링크 자동

   복구 = snapshot 복원 + 이후 incremental SSTable들 얹기
```

incremental backup은 자동 삭제되지 않으므로(운영자가 외부 이관 후 정리), 디스크 관리에 주의해야 한다.

### 복구 절차 개요

복구의 큰 그림은 "백업해 둔 SSTable 파일을 제자리에 놓고 노드에게 그것을 읽게 하는 것"이다.

```text
   복구 시나리오 A: 같은 노드에 데이터 되돌리기
   1. nodetool drain 후 Cassandra 정지 (또는 truncate로 비우기)
   2. snapshot(+incremental) SSTable 파일을 data 디렉터리에 복원
   3. Cassandra 시작 → SSTable 자동 인식, 또는
      nodetool refresh 로 새 파일을 라이브에 편입
   4. nodetool repair 로 그동안의 정합성 보정

   복구 시나리오 B: 죽은 노드 → 새 하드웨어
   → 백업 복구보다 replace_address_first_boot 가 보통 더 깔끔
     (살아있는 replica에서 정상 스트리밍 받음)

   복구 시나리오 C: 새 클러스터로 통째 복원 (DR)
   → sstableloader 로 백업 SSTable을 새 클러스터에 적재
     (token 분포가 달라도 재분배하며 적재)
```

> RF=3 클러스터에서 **노드 하나가 죽는 것은 보통 백업 복구 사건이 아니다.** 다른 두 replica가 데이터를 들고 있으니 replace/removenode + repair로 해결된다. snapshot 백업이 진짜 빛나는 순간은 **논리적 사고**(실수로 `TRUNCATE`, 잘못된 대량 `DELETE`, 애플리케이션 버그로 인한 데이터 오염)다. 이런 건 모든 replica에 똑같이 복제되므로 클러스터 자체로는 못 되돌린다. 그래서 백업은 "하드웨어 장애 대비"보다 "사람·코드 실수 대비"로 더 중요하다.

## 실전 트러블슈팅 플레이북

새벽 3시에 알람이 울렸다. 패닉하지 말고 표를 따라간다. 원칙은 **항상 `nodetool status`부터** 보는 것(누가 죽었나), 그다음 **증상에서 의심으로, 의심에서 진단 명령으로** 좁혀가는 것이다.

| 증상 | 의심 원인 | 진단 명령 | 1차 조치 |
|---|---|---|---|
| 특정 노드만 read p99 폭증 | large partition / hotspot / compaction 밀림 | `tablehistograms`(partition size 꼬리), `compactionstats`, `tablestats` | 데이터 모델 점검(버킷팅), compaction throughput 상향, 핫 키 식별 |
| 클러스터 전반 write latency 상승, 노드 `DN` 깜빡임 | GC pause / 노드 과부하 | `tpstats`(dropped MUTATION, pending), `gc.log`, `nodetool info`(heap) | 과부하 노드 트래픽 경감, heap/GC 점검, 원인 노드 격리 |
| `SELECT`가 느리다가 갑자기 실패(query aborted) | tombstone 과다(큐 안티패턴) | system.log "tombstone" grep, `tablestats`(tombstones per slice) | 쿼리 패턴/모델 변경, partition 단위 삭제로 전환, gc_grace 검토 |
| 디스크가 빠르게 참 | compaction 밀림 / snapshot 누적 / 거대 SSTable | `compactionstats`(pending), `listsnapshots`, `df`, `tablestats` | 오래된 snapshot 정리, compaction 가속, 노드 추가 검토 |
| 삭제했던 데이터가 다시 보임 | gc_grace 내 미repair → 좀비 부활 | repair 이력 확인, 노드 다운 이력, `tablestats` | 즉시 repair, gc_grace/repair 스케줄(Reaper) 정비 |
| 노드 추가 후 디스크가 안 줄어듦 | bootstrap 후 cleanup 누락 | `nodetool status`(Owns/Load), `tablestats` | 기존 노드들에서 순차 `nodetool cleanup` |
| repair가 끝없이 돌거나 hang | over-streaming / 거대 partition / 스트리밍 stuck | `netstats`(스트리밍 진행), `compactionstats`(validation) | subrange/Reaper로 전환, 거대 partition 해소, stuck 세션 재시작 |
| 어떤 노드의 hint가 계속 쌓임 | 특정 replica가 down/과부하 | `nodetool info`(hint 수), `status`(DN) | 아픈 노드 복구, 복구 후 hint 자동 전달 확인, 안 되면 repair |
| `pending compactions`가 단조 증가 | 쓰기 속도 > compaction 속도 | `compactionstats`, `tablestats`(SSTable 수) | `compaction_throughput` 상향, compaction 전략 재검토(STCS→LCS 등) |
| 한 노드 Load가 다른 노드의 2배 | 토큰 불균형 / 데이터 hotspot | `nodetool status`(Load/Owns), `tablehistograms` | 파티션 키 분포 점검, vnode 수 점검, 필요 시 재밸런싱 |

### 트러블슈팅의 사고 흐름

```text
   알람 발생
      │
      ▼
   nodetool status  ── DN 노드 있나? ──► 있음 ─► 노드 라이프사이클 결정
      │                                          (replace/removenode/repair)
      ▼ 전부 UN
   증상 분류
   ├─ 느림(latency)  ─► tpstats(GC/dropped) + compactionstats(밀림)
   │                    + tablehistograms(large partition?)
   ├─ 실패(error)    ─► system.log grep (tombstone / OOM / timeout)
   ├─ 디스크 압박    ─► compactionstats + listsnapshots + df
   └─ 정합성 이상    ─► repair 이력 + 다운 이력 → repair
      │
      ▼
   가설 1개로 좁힘 → 진단 명령으로 검증 → 최소 침습 조치 → 모니터링으로 확인
```

> 가장 흔한 실수: 증상을 보자마자 노드를 재시작하는 것. 재시작은 증거(메모리 상태, 진행 중 쿼리, tpstats)를 날려버리고, drain 없이 재시작하면 commitlog replay로 더 느려진다. **먼저 진단(status, tpstats, compactionstats, 로그)으로 가설을 세우고**, 그다음 최소 침습 조치를 한다. 재시작은 보통 마지막 카드다.

## 핵심 요약

- **repair는 의무다.** hinted handoff(3시간 window, hint 유실)와 read repair(안 읽히면 안 고침)가 못 메우는 영구 drift를, anti-entropy repair만이 전수조사로 메운다. **`gc_grace_seconds`(기본 10일) 안에 모든 노드를 repair하지 않으면 삭제한 데이터가 부활(zombie)**한다.
- **Merkle tree**로 replica 간 차이를 log 비용으로 탐지하고, 다른 leaf의 데이터만 스트리밍한다. tree 해상도 한계로 over-streaming이 생기며, subrange/Reaper로 완화한다. validation compaction은 데이터를 다 읽어 부하가 무겁다.
- repair 종류: **full**(전체, 정확·비쌈) vs **incremental**(unrepaired만, anti-compaction으로 repaired/unrepaired 분리, 4.0에서 재설계), **primary range(`-pr`)**로 중복 제거, **subrange**로 부하 분할. 실무는 **Cassandra Reaper**로 연속 스케줄링.
- **large partition**은 조용한 시한폭탄: GC 압박·느린 읽기·compaction 실패·hotspot. 탐지는 compaction 경고 로그·`tablehistograms`·`tablestats`(max partition bytes)·`system.size_estimates`. 해결은 **버킷팅 재모델링**(설정으로는 못 고침).
- **tombstone 과다**(특히 큐 안티패턴)는 읽기를 죽인다. `tombstone_failure_threshold`(10만) 넘으면 쿼리 abort. 해결은 partition 단위 삭제, 큐는 전용 시스템 사용. `gc_grace`를 함부로 낮추지 말 것.
- **nodetool**: `status`(첫 화면), `tpstats`(dropped mutation = 쓰기 유실), `tablehistograms`(p99·partition 꼬리), `compactionstats`(밀림), `netstats`(스트리밍). 노드 정지 전 `drain`, 추가 후 `cleanup` 필수.
- 노드 라이프사이클: **bootstrap**(추가)→cleanup, **replace_address_first_boot**(죽은 노드 교체, 가장 깔끔), **decommission**(살아있는 노드 정상 제거), **removenode**(죽은 노드 강제 제거), **assassinate**(좀비, 최후의 수단).
- 모니터링은 **p99로** 본다(평균은 거짓말). dropped mutation은 0이어야, GC pause는 짧아야, heap은 31GB 미만·RAM의 1/4·나머지는 OS page cache에 양보. **G1GC가 현대 기본**, 5.0+JDK17은 ZGC 등 저지연 컬렉터의 길을 연다.
- 백업은 **snapshot(하드링크, SSTable immutable이라 즉시·저비용) + incremental backup**. 진짜 가치는 하드웨어 장애가 아니라 **사람·코드의 논리적 실수**(TRUNCATE, 잘못된 DELETE) 복구. 스냅샷은 외부 스토리지로 이관해야 진짜 백업.
- 트러블슈팅은 **항상 `nodetool status`부터** → 증상 분류 → 의심 → 진단 명령 → 최소 침습 조치. 진단 전 재시작은 증거를 날리는 최악수.
