이 장이 답하는 질문:

- `dns:///`, `unix:`, `xds:`, `passthrough:` 같은 target URI scheme은 채널 내부에서 무엇을 고르는가? resolver는 정확히 무엇을 반환하는가?
- DNS resolver만 믿으면 왜 위험한가? A/AAAA 조회로는 무엇을 알 수 없고, TTL·캐시·갱신 지연이 어떤 장애를 만드는가?
- `pick_first`와 `round_robin`은 내부에서 subchannel을 어떻게 다르게 관리하는가? 왜 기본값이 `pick_first`인가?
- 왜 gRPC는 L4 프록시 대신 클라이언트 사이드 L7 로드밸런싱을 택했는가? HTTP/2 멀티플렉싱이 L4 LB에서 왜 불균형을 만드는가?
- gRPCLB → xDS로 이어지는 "외부 컨트롤 플레인이 정책을 내려주는" 모델은 어떻게 동작하며, proxyless 서비스 메시는 무엇을 약속하는가?

## 12.0 들어가며: "주소를 안다"와 "어디로 보낼지 안다"는 다른 문제다

지금까지 우리는 [[07 - 채널 스텁 커넥션 생명주기]]에서 채널(channel)이 어떻게 만들어지고, 그 아래에서 실제 TCP/TLS 연결을 담당하는 subchannel이 어떤 상태(state) 머신을 따라 움직이는지 보았다. 그리고 [[04 - HTTP2 깊이 보기 - 전송 계층]]에서는 하나의 TCP 연결 위에서 여러 스트림(stream)이 멀티플렉싱(multiplexing)되는 HTTP/2의 동작을 깊이 들여다봤다.

그런데 한 가지 질문을 의도적으로 미뤄두었다. 우리가 채널을 만들 때 적는 그 한 줄,

```go
conn, err := grpc.NewClient("dns:///orders.svc.cluster.local:50051", opts...)
```

이 문자열은 **정확히 무엇을 의미하는가?** `orders.svc.cluster.local`이라는 이름은 어떻게 실제 IP 주소로 바뀌는가? 그 서비스 뒤에 파드(pod)가 10개 떠 있다면, 내 요청은 그중 어디로 가는가? 한 파드가 죽으면 누가 그것을 알아차리고 빼주는가? 새 파드가 스케일아웃(scale-out)으로 추가되면 내 클라이언트는 그것을 언제 알게 되는가?

이 모든 질문의 답이 바로 이 장의 주제, **이름 해석(name resolution)** 과 **로드밸런싱(load balancing)** 이다.

핵심부터 못 박자. 전통적인 모놀리식 서버나 단순한 REST 클라이언트의 세계에서는 이 두 가지가 대개 인프라(DNS + L4 로드밸런서)의 영역으로 외주화되어 있었다. 클라이언트는 그냥 `https://api.example.com`을 때리고, 그 뒤에서 무슨 일이 일어나는지 몰라도 됐다. 그러나 **gRPC는 이 결정 권한의 상당 부분을 클라이언트 라이브러리 안으로 끌어들였다.** 왜 그랬는가? 그 "왜"를 이해하는 것이 이 장의 절반이다. 나머지 절반은 "그래서 내부에서 실제로 무슨 일이 벌어지는가"이다.

그림으로 큰 그림을 먼저 잡자. gRPC 채널 내부에는 세 개의 협력자가 있다.

```text
              gRPC Channel (논리적 연결)
  ┌─────────────────────────────────────────────────────────┐
  │                                                           │
  │   target URI: "dns:///orders.svc:50051"                   │
  │        │                                                  │
  │        │  scheme("dns") 로 플러그인 선택                  │
  │        ▼                                                  │
  │  ┌──────────────┐   resolved addresses + service config   │
  │  │ Name Resolver│ ─────────────────────────────┐          │
  │  │  (DNS)       │   [10.0.0.1:50051,            │          │
  │  └──────────────┘    10.0.0.2:50051,            │          │
  │        ▲             10.0.0.3:50051]            ▼          │
  │        │ 재해석(re-resolve)        ┌────────────────────┐  │
  │        │ (TTL 만료/연결 실패 시)   │   LB Policy        │  │
  │        └───────────────────────────│ (round_robin 등)   │  │
  │                                     └────────────────────┘  │
  │                                       │     │     │        │
  │                            subchannel 집합을 생성·관리      │
  │                                       ▼     ▼     ▼        │
  │                                  ┌──────┐┌──────┐┌──────┐  │
  │                                  │ SC#1 ││ SC#2 ││ SC#3 │  │
  │                                  │READY ││READY ││TRANS.│  │
  │                                  └──┬───┘└──┬───┘└──┬───┘  │
  └─────────────────────────────────────┼───────┼───────┼─────┘
                                         ▼       ▼       ▼
                                    10.0.0.1  10.0.0.2  10.0.0.3
                                     (백엔드 파드들)
```

- **Name Resolver(이름 해석기)**: target URI의 호스트 부분을 입력받아 **주소 목록(list of addresses)** 과 **service config(JSON)** 를 출력한다. "이 이름은 이 IP들이고, 이 채널은 이런 정책으로 동작하라"는 두 가지를 채널에 알려준다.
- **LB Policy(로드밸런싱 정책)**: resolver가 준 주소 목록을 받아 각 주소에 대한 **subchannel** 을 만들고 그 상태를 추적하며 매 RPC마다 "이번엔 어느 subchannel로 보낼까"를 결정하는 **picker**(피커)를 채널에 제공한다.
- **Subchannel(서브채널)**: 단일 백엔드 주소에 대한 실제 연결. 상태 머신은 [[07 - 채널 스텁 커넥션 생명주기]]에서 본 `IDLE → CONNECTING → READY → TRANSIENT_FAILURE` 그대로다.

이 세 협력자의 협업 구조를 머릿속에 박아두면, 이 장의 나머지 내용은 전부 "이 그림의 각 박스를 확대해서 들여다보는 것"에 불과하다.

## 12.1 target URI: 채널 생성의 진짜 문법

### 12.1.1 그건 호스트:포트가 아니라 URI다

많은 사람이 `grpc.NewClient("localhost:50051")`을 보고 "아 호스트랑 포트구나"라고 생각하고 넘어간다. 절반만 맞다. gRPC가 받는 그 문자열은 사실 **RFC 3986** 형식의 **URI**이며, 정식 형태는 다음과 같다.

```text
  scheme :// authority / endpoint_name
  ─────     ─────────   ─────────────
  어떤      (선택)       이름 해석기에
  resolver  authority    넘길 실제 "이름"
  를 쓸까
```

예를 들어 `dns:///orders.svc.cluster.local:50051`을 뜯어보면:

| 구성 요소 | 값 | 의미 |
|-----------|-----|------|
| scheme | `dns` | DNS resolver 플러그인을 쓰겠다 |
| authority | `` (비어 있음) | DNS 서버를 명시 안 함 → 시스템 기본 resolver |
| endpoint | `orders.svc.cluster.local:50051` | 해석 대상 이름과 포트 |

슬래시가 세 개(`///`)인 이유가 여기서 드러난다. `scheme://` 다음에 authority가 오고, 그게 비어 있으면 곧장 `/`로 endpoint가 시작된다. `dns://8.8.8.8/orders.svc:50051`처럼 적으면 authority 자리에 `8.8.8.8`이 들어가, "구글 DNS 서버를 써서 해석하라"는 뜻이 된다.

그런데 우리가 `grpc.NewClient("localhost:50051")`처럼 scheme 없이 적어도 동작하는 이유는 무엇인가? gRPC는 scheme이 파싱되지 않거나 등록된 resolver가 없을 때 **기본 scheme(default scheme)** 을 적용하는데, 대부분의 구현에서 이 기본값이 `dns`다(Go의 경우 `resolver.SetDefaultScheme`으로 바꿀 수 있다). `localhost:50051`은 사실상 `dns:///localhost:50051`로 취급된다.

> 한 가지 함정: Windows 경로나 일부 호스트 문자열이 우연히 `C:\...`처럼 보이면 `C`가 scheme으로 오인될 수 있다. 그래서 명시적으로 scheme을 붙이는 습관이 안전하다.

> 또 하나의 미묘한 함정(Go): "기본 scheme이 `dns`"는 권장 API인 `grpc.NewClient`의 동작이다. NewClient는 scheme이 없으면 `dns`를 적용한다(`grpc.WithDefaultScheme`로 변경 가능). 반면 **레거시 `grpc.Dial`(deprecated)은 전역 기본 scheme이 `passthrough`** 였다. 같은 `"localhost:50051"` 문자열이라도 어떤 API로 채널을 만드느냐에 따라 DNS 해석을 거치느냐(`NewClient`) 그대로 dial하느냐(`Dial`)가 달라졌다. 이 차이가 "왜 옛날 코드는 DNS 재해석이 안 일어나지?" 같은 혼란의 원인이 된다. 신규 코드는 `NewClient`를 쓰고, 의도가 분명하도록 scheme을 명시하자.

### 12.1.2 scheme → resolver 플러그인 매핑

scheme은 일종의 **플러그인 선택 키**다. gRPC 런타임은 내부에 resolver 레지스트리(registry)를 가지고 있고, 각 resolver는 자기가 처리할 scheme을 등록한다. 대표적인 내장 scheme을 정리하면:

| scheme | resolver | 동작 | 대표 용도 |
|--------|----------|------|-----------|
| `dns:///` | DNS resolver | A/AAAA (+ 선택적 SRV/TXT) 조회로 주소 목록 생성 | Kubernetes headless service, 일반 도메인 |
| `passthrough:///` | passthrough resolver | 이름을 해석하지 않고 **문자열 그대로** 단일 주소로 사용 | 이미 IP를 아는 경우, 프록시 뒤, 사이드카 |
| `unix:` / `unix-abstract:` | unix resolver | 유닉스 도메인 소켓(UDS) 경로 | 같은 호스트 내 IPC, 사이드카 통신 |
| `ipv4:` | ipv4 resolver | 콤마로 구분된 IPv4 주소 목록을 그대로 사용 | 정적 주소 집합, 테스트 |
| `ipv6:` | ipv6 resolver | IPv6 주소 목록을 그대로 사용 | 정적 IPv6 집합 |
| `xds:///` | xDS resolver | xDS 컨트롤 플레인(예: Istiod)에서 동적으로 받아옴 | proxyless 서비스 메시 |

각각의 정확한 표기를 손에 익히자.

```text
dns:///my-service.default.svc.cluster.local:50051
   └─ K8s 클러스터 내부 DNS. headless service면 모든 파드 IP 반환.

dns://10.96.0.10/my-service:50051
   └─ authority에 특정 DNS 서버 IP를 지정.

unix:/var/run/app.sock
   └─ 절대 경로. 슬래시 하나 주의(unix:///path 도 허용되는 구현 있음).

unix-abstract:my-socket
   └─ 리눅스 추상 네임스페이스 소켓(파일시스템에 안 보임). 앞에 NUL 바이트.

ipv4:10.0.0.1:50051,10.0.0.2:50051,10.0.0.3:50051
   └─ 세 개의 정적 주소. round_robin과 조합하면 즉석 분산 테스트.

ipv6:[::1]:50051,[2001:db8::1]:50051
   └─ IPv6는 대괄호로 감싼다.

passthrough:///already.resolved.host:50051
   └─ DNS 안 거치고 "already.resolved.host"를 그대로 dial.

xds:///orders.svc.cluster.local
   └─ scheme이 xds면 컨트롤 플레인이 라우팅·정책·엔드포인트를 내려줌.
```

`passthrough`가 왜 따로 있는지가 흥미롭다. 만약 당신이 이미 IP를 알고 있거나, **앞단에 프록시/사이드카가 있어서 gRPC 클라이언트가 해석을 하지 말아야** 하는 상황이라면, DNS resolver의 "여러 IP를 반환하고 round_robin을 돌리는" 동작이 오히려 방해가 된다. 이때 `passthrough`는 "이 문자열은 이미 최종 목적지다. 손대지 말고 그냥 단일 subchannel로 연결해라"라고 말한다. 실제로 Envoy 사이드카 뒤에서 동작하는 전통적 서비스 메시는 클라이언트가 `passthrough`로 로컬 사이드카에 연결하고, 분산은 사이드카가 처리하는 구조를 쓴다.

### 12.1.3 authority의 두 가지 얼굴

authority는 두 군데서 의미를 갖는다.

1. **resolver 입장에서의 authority**: 위에서 본 것처럼 "어떤 DNS 서버를 쓸까" 같은 resolver 동작 파라미터.
2. **HTTP/2 `:authority` 의사 헤더(pseudo-header)로서의 authority**: [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 본 `HEADERS` 프레임의 `:authority`. TLS의 SNI/인증서 검증, 그리고 서버 측 가상 호스팅·라우팅에 쓰인다.

이 둘이 헷갈리기 쉽다. 기본적으로 gRPC는 target의 endpoint 호스트 부분을 `:authority`로 쓰지만, `grpc.WithAuthority("override.example.com")`(Go) 같은 옵션으로 명시적으로 덮어쓸 수 있다. 이는 TLS 인증서 검증 호스트를 바꾸거나, 같은 IP에 여러 논리 서비스가 사는 환경에서 라우팅 키를 바꿀 때 쓴다. 보안 측면의 디테일은 [[11 - 보안 - TLS mTLS 인증]]에서 다룬다.

## 12.2 Name Resolver: resolver가 정확히 무엇을 반환하는가

resolver의 출력은 단순한 IP 목록이 아니다. **두 가지**다. 이걸 명확히 구분하는 게 중요하다.

```text
   Resolver.Resolve(target)  ──▶  {
       Addresses:     []Address{ {Addr:"10.0.0.1:50051", ...}, ... },
       ServiceConfig:  <파싱된 JSON 객체 또는 에러>,
       Attributes:     <LB 정책에 전달할 부가 메타데이터>,
   }
```

### 12.2.1 출력 1: 주소 목록(addresses)

각 주소(Address)는 단순 문자열이 아니라 부가 속성을 달 수 있는 구조체다. 특히 중요한 게 **localities(지역성)** 나 **weight(가중치)** 같은 attribute다. round_robin은 이를 무시하지만, weighted round robin이나 xDS 기반 LB는 이 attribute를 읽어 동작을 바꾼다.

또한 gRPC의 추상화에서 주소는 **endpoint** 단위로 묶일 수 있다. 하나의 논리적 엔드포인트가 여러 물리 주소(예: IPv4/IPv6 듀얼스택)를 가질 때, LB는 "이 엔드포인트에 연결한다(둘 중 되는 쪽으로)"는 의미로 처리한다. 이건 비교적 최근에 정교해진 부분이라 구현별 차이가 있으니, 개념만 잡아두자: **주소 목록은 "flat한 IP 리스트"가 아니라 "엔드포인트들의 집합이고 각 엔드포인트는 하나 이상의 주소를 가질 수 있는" 구조**라는 것.

### 12.2.2 출력 2: service config (JSON)

이게 진짜 강력하고 자주 간과되는 부분이다. resolver는 주소뿐 아니라 **이 채널이 어떻게 동작해야 하는지를 기술한 JSON 설정**을 함께 내려줄 수 있다. 이를 **service config**라고 부른다.

service config가 담을 수 있는 대표 항목:

```json
{
  "loadBalancingConfig": [
    { "round_robin": {} }
  ],
  "methodConfig": [
    {
      "name": [
        { "service": "orders.v1.OrderService", "method": "GetOrder" }
      ],
      "timeout": "2s",
      "retryPolicy": {
        "maxAttempts": 4,
        "initialBackoff": "0.1s",
        "maxBackoff": "1s",
        "backoffMultiplier": 2.0,
        "retryableStatusCodes": ["UNAVAILABLE"]
      }
    }
  ],
  "healthCheckConfig": {
    "serviceName": "orders.v1.OrderService"
  }
}
```

이 한 덩어리에 **이 장과 직접 관련된 두 가지가 들어 있다.**

1. `loadBalancingConfig`: "이 채널은 round_robin을 써라" — **LB 정책을 resolver가 정한다.** 클라이언트 코드를 안 고치고도 서버 쪽 설정만으로 정책을 바꿀 수 있다.
2. `healthCheckConfig`: "subchannel 헬스체크를 켜고, 이 서비스 이름으로 watch 하라" — 헬스 인지 LB의 스위치.

나머지(`methodConfig`의 timeout, retryPolicy)는 각각 [[10 - Deadline 취소 타임아웃]]과 [[13 - 안정성 - Retry Health Check Keepalive]]의 주제이므로 여기서는 깊이 들어가지 않고 "resolver가 이것들도 같이 내려준다"는 사실만 기억하자.

service config의 출처는 두 가지다.

- **resolver가 내려주는 것**: DNS resolver의 경우 `_grpc_config.<host>`라는 이름의 `TXT` 레코드에 JSON을 박아둘 수 있고(레코드 값은 `grpc_config=` 접두사 뒤에 service config JSON 배열이 오는 형식이다), xDS resolver는 컨트롤 플레인에서 받아 변환한다. 이 방식의 장점은 **중앙 관리**다. 운영자가 DNS/컨트롤 플레인만 바꾸면 모든 클라이언트가 자동으로 새 정책을 받는다.
- **클라이언트가 코드로 박는 default service config**: `grpc.WithDefaultServiceConfig(jsonString)`. resolver가 service config를 안 주거나, 보안상 DNS TXT를 신뢰하지 않을 때 쓴다. 실무에서는 이 방식이 더 흔하다.

```go
// 클라이언트 코드에서 직접 service config를 박는 방법 (Go)
const svcCfg = `{
  "loadBalancingConfig": [ { "round_robin": {} } ],
  "methodConfig": [{
    "name": [{ "service": "orders.v1.OrderService" }],
    "timeout": "3s"
  }]
}`

conn, err := grpc.NewClient(
    "dns:///orders.svc.cluster.local:50051",
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithDefaultServiceConfig(svcCfg),
)
```

> 보안 주의: DNS TXT로 service config를 받는 기능은 임의의 DNS 응답이 클라이언트 동작을 바꿀 수 있다는 뜻이라, 많은 환경에서 기본 비활성화이거나 명시적 opt-in이다. 신뢰 경계를 넘는 DNS를 함부로 믿지 말 것 — "외부 라이브러리·외부 입력 신뢰 최소화" 원칙과도 통한다.

### 12.2.3 resolver의 생명주기: 한 번 해석하고 끝이 아니다

resolver의 결정적 특징: **이것은 일회성 함수가 아니라 watch(감시) 모델이다.** resolver는 한 번 주소를 내려준 뒤에도 살아서, 다음 조건에서 **재해석(re-resolution)** 을 수행하고 채널에 갱신된 목록을 푸시한다.

```text
재해석을 트리거하는 이벤트들
  ┌───────────────────────────────────────────────────────┐
  │ 1. LB policy가 "연결 실패가 났다, 주소를 다시 받고      │
  │    싶다"고 resolver에게 ResolveNow()를 호출            │
  │ 2. DNS resolver의 경우, 내부 타이머(최소 갱신 주기)가  │
  │    돌면서 주기적으로 재조회 (예: 기본 30초 최소 간격)  │
  │ 3. xDS resolver의 경우, 컨트롤 플레인이 새 설정을      │
  │    스트리밍으로 푸시 → 즉시 갱신                       │
  └───────────────────────────────────────────────────────┘
            │
            ▼
   새 주소 목록을 LB policy에 전달
            │
            ▼
   LB policy가 diff 계산:
     - 사라진 주소 → 해당 subchannel 종료(graceful)
     - 새로 생긴 주소 → subchannel 생성·연결
     - 그대로인 주소 → 유지
```

여기서 **DNS resolver의 갱신 주기**가 운영상 매우 중요하다. gRPC DNS resolver는 매 RPC마다 DNS를 조회하지 않는다(그러면 DNS 서버가 죽는다). 대신 **최소 재해석 간격(minimum re-resolution interval)** 을 두고, 그 사이에는 캐시된 결과를 쓴다. 이 값이 크면 "백엔드가 바뀌었는데 클라이언트가 한참 모르는" 갱신 지연(staleness)이 생긴다. 이게 12.8에서 다룰 운영 함정의 뿌리다.

## 12.3 DNS Resolver: 가장 흔하고, 가장 함정이 많다

DNS resolver는 가장 많이 쓰이지만 동시에 가장 오해받는다. 그 동작과 **구조적 한계**를 정확히 알아야 한다.

### 12.3.1 무엇을 조회하는가

DNS resolver가 `orders.svc.cluster.local:50051`을 받으면 내부적으로:

```text
1. A 레코드 조회   → IPv4 주소들  (10.0.0.1, 10.0.0.2, 10.0.0.3 ...)
2. AAAA 레코드 조회 → IPv6 주소들  (2001:db8::1 ...)
3. (선택) TXT 조회  → _grpc_config.orders.svc... 에서 service config JSON
4. (선택/구현별) SRV 조회 → grpclb 백엔드 발견용 (레거시)
```

이렇게 모은 IP들에 **포트(50051)를 붙여** 주소 목록을 만든다. 핵심은 **DNS는 "이 이름에 해당하는 IP가 여럿이면 여럿 다 준다"** 는 점이다. Kubernetes의 **headless service**(`clusterIP: None`)가 바로 이 동작을 활용한다 — headless service의 A 레코드 조회는 모든 백엔드 파드의 IP를 반환한다. 그래서 `dns:///my-svc:50051` + headless service + round_robin 조합이 K8s에서 가장 표준적인 클라이언트 사이드 LB 셋업이 된다.

반대로 일반(ClusterIP) service는 A 레코드가 **단 하나의 가상 IP(VIP)** 만 반환하고, 분산은 kube-proxy의 L4(iptables/IPVS)가 처리한다. 이러면 12.4에서 설명할 L4 LB의 쏠림 문제에 그대로 빠진다. **그래서 gRPC를 K8s에서 제대로 분산하려면 headless service를 써야 한다**는 격언이 나온다.

### 12.3.2 DNS resolver의 세 가지 구조적 한계

이게 이 절의 핵심이다. DNS는 1980년대에 설계된 프로토콜이고, 마이크로서비스 로드밸런싱을 위해 만들어지지 않았다. 단독으로 쓰면 다음 세 가지가 발목을 잡는다.

**한계 1: 헬스 정보가 없다.**

DNS A 레코드는 "이 이름의 IP는 이것들"이라고만 말한다. 그중 어떤 파드가 살아 있고 어떤 게 죽어가는지 **DNS는 모른다.** 죽은 파드의 IP가 TTL 만료 전까지 응답에 남아 있을 수 있다. 그래서 DNS만으로는 "건강한 백엔드만 고르는" 게 불가능하다. 이 공백을 메우는 게 **subchannel 자체의 연결 상태**(연결이 끊기면 TRANSIENT_FAILURE)와 **gRPC 헬스체크 프로토콜**([[13 - 안정성 - Retry Health Check Keepalive]])이다. 즉, DNS는 "후보 명단"만 주고, "실제로 살아 있는지"는 LB policy + subchannel + 헬스체크가 판단하는 분업 구조다.

**한계 2: 갱신이 느리고 캐시가 끼어든다.**

DNS는 계층적 캐시 시스템이다. 클라이언트 OS resolver, 사이드카, 클러스터 DNS(CoreDNS), 그 위의 권위 서버까지 각 단계에 TTL 기반 캐시가 있다. 게다가 gRPC DNS resolver 자체도 최소 재해석 간격을 둔다. 결과적으로 백엔드 토폴로지가 바뀐 순간과 클라이언트가 그걸 아는 순간 사이에 **수 초~수십 초의 지연**이 생긴다. 트래픽이 급변하는 오토스케일링 환경에서 이 지연은 "새로 뜬 파드에 트래픽이 한참 안 가는" 또는 "죽은 파드로 계속 시도하는" 증상을 만든다.

**한계 3: 가중치·우선순위·메타데이터 표현력이 빈약하다.**

DNS A 레코드는 그냥 IP 리스트다. "이 파드는 트래픽 70%, 저 파드는 30%" 같은 가중치를, 또는 "이 리전을 우선하고 장애 시 다른 리전으로" 같은 지역성 라우팅을 A 레코드로 표현할 방법이 사실상 없다(SRV의 priority/weight가 있지만 gRPC LB에서 일반적으로 활용되지 않고, 표현력도 제한적이다). 이런 정교한 정책이 필요하면 DNS의 한계를 넘어서 **xDS** 같은 풍부한 컨트롤 플레인이 필요해진다(12.6).

이 세 한계를 한 장의 표로:

| 요구사항 | DNS resolver 단독 | 보완책 |
|----------|-------------------|--------|
| 건강한 백엔드만 선택 | 불가 (헬스 모름) | subchannel 상태 + gRPC 헬스체크 ([[13 - 안정성 - Retry Health Check Keepalive]]) |
| 빠른 토폴로지 반영 | 느림 (TTL/캐시/최소간격) | MAX_CONNECTION_AGE 강제 재분배, xDS push |
| 가중치/지역성 라우팅 | 사실상 불가 | xDS (LDS/RDS/CDS/EDS) |
| 중앙 정책 배포 | TXT로 가능하나 빈약·보안우려 | xDS, 또는 코드의 default service config |

기억하라: **DNS resolver는 "충분히 좋은 기본값"이지 "완전한 솔루션"이 아니다.** 단순한 환경에선 충분하고, 정교한 트래픽 관리가 필요해지는 순간 한계에 부딪힌다. 그 한계가 바로 gRPCLB와 xDS가 태어난 이유다.

## 12.4 클라이언트 사이드 L7 LB: gRPC가 L4 프록시를 버린 이유

이 절은 이 장 전체에서 **가장 중요한 "왜"** 다. 천천히 가자.

### 12.4.1 L4 로드밸런서와 HTTP/2의 불행한 만남

전통적인 로드밸런싱은 **L4(전송 계층) 프록시**로 한다. 클라이언트는 LB의 VIP로 TCP 연결을 맺고, LB는 그 연결을 백엔드 하나로 포워딩한다. **연결 단위로** 백엔드를 고른다는 게 핵심이다.

HTTP/1.1에서는 이게 잘 작동했다. 왜냐하면 HTTP/1.1은 한 연결에서 한 번에 하나의 요청만 처리하고(파이프라이닝은 사실상 안 씀), 클라이언트가 부하를 위해 여러 연결을 열기 때문이다. 연결이 많으니 L4가 연결을 분산하면 요청도 자연히 분산됐다.

그런데 **HTTP/2는 정반대로 설계됐다.** [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 봤듯, HTTP/2의 핵심 미덕은 **단 하나의 롱리브드(long-lived) TCP 연결에서 수많은 스트림을 멀티플렉싱**하는 것이다. 연결 핸드셰이크 비용을 한 번만 내고, HOL(head-of-line) 블로킹 없이 동시에 여러 RPC를 처리한다. gRPC는 이 위에서 동작하므로, **gRPC 클라이언트는 한 백엔드당 보통 연결을 딱 하나만 유지한다.**

이제 L4 LB 뒤에 gRPC를 놓으면 무슨 일이 벌어지는지 보자.

```text
   [L4 LB 뒤의 gRPC: 연결 쏠림(connection pinning) 문제]

   Client A ──(1개의 영속 TCP 연결)──┐
                                      │  L4 LB가 연결을
   Client B ──(1개의 영속 TCP 연결)──┤  맺을 때 백엔드 고정
                                      ▼
                              ┌───────────────┐
                              │     L4 LB      │
                              └───┬───────┬────┘
                  연결 고정! ─────┘       └──── 연결 고정!
                                  ▼               ▼
                            ┌─────────┐     ┌─────────┐     ┌─────────┐
                            │Backend 1│     │Backend 2│     │Backend 3│
                            │ A,B의   │     │  (놀고   │     │ (아예    │
                            │ 모든    │     │  있음)   │     │  연결    │
                            │ RPC!    │     │          │     │  없음)   │
                            └─────────┘     └─────────┘     └─────────┘
```

문제가 보이는가? L4 LB는 **연결을 맺는 순간 한 번** 백엔드를 고른다. 그 연결이 한 시간 동안 살아 있으면서 그 위로 RPC가 10만 개가 흐르든, **그 10만 개 전부가 같은 백엔드로 간다.** L4는 연결 안에서 흐르는 개별 요청(스트림)을 보지 못하기 때문이다(그게 L7의 일이다). 클라이언트 수가 백엔드 수보다 적거나, 클라이언트들의 연결이 우연히 같은 백엔드로 몰리면, **일부 백엔드는 과부하, 일부는 놀고, 일부는 아예 연결조차 못 받는** 극심한 불균형이 생긴다.

게다가 스케일아웃을 해도 소용없다. 새 백엔드(Backend 4)를 추가해도, **이미 맺어진 영속 연결들은 그대로 기존 백엔드에 붙어 있다.** 새 백엔드는 새 연결이 생길 때까지 놀게 된다. 롱리브드 연결의 미덕이 분산의 독이 된다.

### 12.4.2 두 가지 해법: L7 프록시 vs 클라이언트 사이드 LB

이 문제를 푸는 방법은 둘이다.

**해법 A — L7 프록시 LB**: LB가 HTTP/2를 이해하게 만든다. Envoy나 nginx 같은 L7 프록시는 연결을 종단(terminate)하고, **각 스트림(요청)을 개별적으로** 백엔드에 분배한다. 한 클라이언트 연결 위의 100개 RPC가 100개 백엔드로 흩어질 수 있다. 정확하고 강력하다. 대신 **모든 요청이 프록시를 한 번 더 거친다.** 추가 홉(hop)에 따른 지연(latency), 프록시 자체의 운영·확장 부담, 그리고 프록시가 또 하나의 장애점·병목이 된다는 비용을 치른다.

**해법 B — 클라이언트 사이드 LB**: 아예 프록시를 없애고, **클라이언트가 직접 모든 백엔드에 연결을 맺고, 요청마다 어느 백엔드로 보낼지 스스로 고른다.** 이게 gRPC가 채택한 기본 모델이다.

```text
   [클라이언트 사이드 L7 LB: 요청 단위 분산]

   Client (round_robin LB policy)
     │
     ├──(연결1)──▶ Backend 1   ┐
     ├──(연결2)──▶ Backend 2   ├─ 모든 백엔드에 직접 연결
     └──(연결3)──▶ Backend 3   ┘
     │
     매 RPC마다 picker가 "이번엔 1, 다음엔 2, 다음엔 3..." 순환
     → 요청 단위(L7) 분산이 클라이언트 안에서 일어남
     → 추가 네트워크 홉 없음 (프록시 없음)
```

클라이언트 사이드 LB의 장점:

- **요청 단위 분산**: round_robin이면 RPC마다 다음 백엔드를 골라 진짜로 고르게 분산한다. L4의 연결 쏠림이 원천 봉쇄된다.
- **추가 홉 없음**: 클라이언트가 백엔드에 직접 연결하므로 프록시 지연·병목이 없다. 마이크로서비스 간 east-west 트래픽처럼 양이 많은 구간에서 특히 유리하다.
- **백엔드 상태 직접 인지**: 클라이언트가 각 subchannel의 연결 상태를 직접 보므로, 죽은 백엔드를 즉시 picker에서 뺄 수 있다.

대가도 있다:

- **클라이언트가 똑똑해야 한다**: 모든 클라이언트의 gRPC 라이브러리가 resolver/LB policy를 구현해야 한다. 그래서 gRPC가 이 로직을 라이브러리 안에 내장했다. (다국어 SDK가 다 이걸 구현해야 한다는 부담 — 이게 [[16 - gRPC-Web과 REST 게이트웨이]]에서 브라우저 환경이 클라이언트 LB를 못 하는 이유와 연결된다.)
- **연결 수 폭증 가능성**: 클라이언트 N개 × 백엔드 M개 = N×M개의 연결. 백엔드가 수천 개면 클라이언트마다 수천 연결을 들고 있어야 한다. round_robin은 "모든 백엔드에 연결"하므로 이 문제가 실재한다. 그래서 대규모에서는 "일부 백엔드에만 연결하는" subsetting이나, look-aside LB로 후보를 줄이는 전략이 나온다.
- **멤버십 최신화는 여전히 resolver 책임**: 클라이언트가 아무리 똑똑해도, 백엔드 목록이 갱신 안 되면 새 백엔드를 모른다(12.3.2의 갱신 지연). 이건 12.8에서.

### 12.4.3 그래서 gRPC의 기본 철학

gRPC의 설계 결정은 이렇게 요약된다:

> "HTTP/2의 롱리브드 멀티플렉싱 연결은 성능엔 좋지만 L4 분산엔 독이다. 그러니 분산을 **연결 계층(L4)이 아니라 요청 계층(L7)에서**, 그것도 **프록시가 아니라 클라이언트 안에서** 하자. 그 대신 클라이언트 라이브러리가 resolver와 LB policy를 내장해 똑똑하게 만들자."

이 한 문장이 이 장 전체의 뼈대다. resolver(어디 있나), LB policy(어디로 보내나), subchannel(실제 연결)이라는 3분할 구조가 전부 이 철학의 구현체다.

## 12.5 LB Policy 깊이 보기: pick_first와 round_robin의 내부

이제 LB policy 박스를 확대하자. LB policy의 일은 두 가지다.

1. resolver가 준 주소 목록으로 **subchannel을 만들고 관리**한다(연결 시작, 상태 추적, 사라진 주소의 subchannel 정리).
2. 각 RPC가 보낼 때 호출되는 **picker(피커)** 를 채널에 제공한다. picker는 "이 RPC를 어느 subchannel로 보낼지"를 동기적으로 결정한다.

이 picker 추상화가 핵심이다. 채널의 RPC 발신 경로는 "지금 picker에게 subchannel 하나 달라고 한다 → 받은 subchannel로 보낸다"가 전부다. LB policy를 바꾼다는 건 **picker의 선택 알고리즘을 바꾸는 것**이다.

### 12.5.1 pick_first: 기본값, 그리고 그 이유

`pick_first`는 gRPC의 **기본 LB policy**다. 이름 그대로 동작한다.

```text
   pick_first 동작
   ────────────────
   resolver가 [A, B, C] 주소를 줌
        │
        ▼
   주소들을 순서대로 시도하며 "처음으로 연결되는 하나"를 찾음
   (A 시도 → 실패하면 B 시도 → ... 최근엔 happy-eyeballs식 병렬 시도도)
        │
        ▼
   연결된 그 하나의 subchannel만 READY로 유지
        │
        ▼
   picker는 항상 "그 하나"를 반환
   → 모든 RPC가 단일 백엔드로 감
        │
        ▼
   그 연결이 끊기면? 다음 주소로 넘어가 다시 "처음 되는 하나"를 찾음
   (필요시 resolver에 재해석 요청)
```

pick_first는 **백엔드 하나에만 연결**한다. 분산을 안 한다. 그런데 왜 이게 기본값인가? 세 가지 이유가 있다.

1. **단일 엔드포인트가 가장 흔한 케이스**: 많은 target은 사실 단일 서버이거나, 앞단에 이미 L7 프록시/L4 LB가 있어서 클라이언트는 그냥 "거기 하나"로 보내면 되는 경우다. 이럴 때 모든 백엔드에 연결을 까는 round_robin은 과하다.
2. **연결 수 최소화**: pick_first는 연결을 딱 하나만 쓴다. 12.4.2에서 본 N×M 연결 폭증 문제가 없다. 리소스 친화적이다.
3. **프록시·메시 친화적**: 앞에 사이드카가 있으면 분산은 사이드카가 하고, 클라이언트는 사이드카로 연결 하나만 맺으면 된다. pick_first가 딱 맞는다.

pick_first의 기본값 선택은 "안전하고 가볍고, 대부분의 단순 케이스에 맞으며, 분산이 필요하면 명시적으로 round_robin으로 바꿔라"는 보수적 설계다.

> 참고: pick_first도 resolver가 여러 주소를 주면 그것들을 **후보**로 받아 "되는 것 하나"를 고른다. 그래서 DNS가 여러 IP를 줘도 pick_first면 그중 하나로만 트래픽이 간다 — 이걸 "분산되겠지" 하고 오해하면 안 된다. 분산을 원하면 round_robin을 명시해야 한다.

### 12.5.2 round_robin: 모두에 연결하고 순환한다

`round_robin`은 분산을 원할 때 쓰는 대표 정책이다.

```text
   round_robin 동작
   ─────────────────
   resolver가 [A, B, C] 주소를 줌
        │
        ▼
   "모든" 주소에 대해 subchannel 생성 + 연결 시작
        │
        ▼
   각 subchannel의 상태를 추적
   READY인 subchannel들만 모아 "ready 리스트" 구성
        │  예: A=READY, B=READY, C=TRANSIENT_FAILURE
        │      → ready 리스트 = [A, B]   (C는 제외!)
        ▼
   picker: 호출될 때마다 ready 리스트를 원형으로 한 칸씩 진행
        RPC1 → A
        RPC2 → B
        RPC3 → A   (C는 빠졌으니 A,B만 순환)
        RPC4 → B
        ...
   C가 복구되어 READY가 되면 ready 리스트에 다시 합류 → [A,B,C] 순환
```

round_robin의 두 가지 핵심 포인트:

- **모든 백엔드에 연결을 유지**한다. 그래서 어느 백엔드든 즉시 보낼 수 있다(연결 예열 비용 없음). 대신 연결 수가 백엔드 수만큼 든다.
- **READY인 것만 순환 대상**이다. 이게 헬스 인지(health-aware)의 1차 방어선이다. subchannel이 TRANSIENT_FAILURE면 picker가 자동으로 제외한다. 즉 **죽은 백엔드로는 안 보낸다.** 별도 헬스체크 없이도, 연결이 끊긴 백엔드는 자동으로 빠진다. (애플리케이션 레벨 헬스 — "TCP는 살아 있지만 앱이 not-serving" — 까지 잡으려면 [[13 - 안정성 - Retry Health Check Keepalive]]의 gRPC 헬스체크가 필요하다.)

round_robin을 켜는 법은 앞서 본 service config의 `loadBalancingConfig`다.

```go
const svcCfg = `{ "loadBalancingConfig": [ { "round_robin": {} } ] }`

conn, err := grpc.NewClient(
    "dns:///orders.svc.cluster.local:50051",
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithDefaultServiceConfig(svcCfg),
)
```

이 조합 — `dns:///` (headless service) + `round_robin` — 이 K8s에서 gRPC 클라이언트 사이드 LB의 가장 기본적이고 표준적인 셋업이다. DNS가 모든 파드 IP를 주고, round_robin이 그 모두에 연결해 RPC를 고르게 돌린다.

### 12.5.3 더 똑똑한 정책들: weighted, least_request, ring_hash

round_robin은 단순해서 좋지만, **모든 백엔드의 처리 능력이 같다고 가정**한다. 현실은 그렇지 않다. 그래서 더 정교한 정책들이 있다.

| 정책 | 핵심 아이디어 | 언제 쓰나 |
|------|---------------|-----------|
| `round_robin` | 무조건 차례대로 | 균질한 백엔드, 단순함 우선 |
| `weighted_round_robin` | 각 백엔드의 가중치(처리 능력, 또는 서버가 보고하는 부하)에 비례해 분배 | 백엔드 스펙이 다르거나, 부하 기반 동적 가중 (서버가 부하 리포트) |
| `least_request` | 진행 중(in-flight)인 요청이 가장 적은 백엔드로 | 요청 처리 시간 편차가 클 때, 느린 백엔드 회피 |
| `ring_hash` / `maglev` | 요청의 어떤 키(예: 세션 ID)를 해시해 **같은 키는 같은 백엔드로** (consistent hashing) | 세션 어피니티(sticky), 캐시 지역성이 중요할 때 |
| `pick_first` | 되는 것 하나만 | 단일 엔드포인트, 프록시 뒤 |

`weighted_round_robin`이 특히 흥미롭다. 정적 가중치뿐 아니라, **서버가 자신의 실시간 부하(CPU, 큐 길이 등)를 응답에 실어 보내면(ORCA: Open Request Cost Aggregation 같은 메커니즘)** 클라이언트가 그걸 받아 가중치를 동적으로 조정할 수 있다. "지금 한가한 서버로 더 보내는" 적응형 분산이 가능하다. 이건 round_robin의 "맹목적 순환"보다 훨씬 똑똑하지만, 그만큼 클라이언트·서버 양쪽 구현 복잡도가 올라간다.

`ring_hash`는 분산과는 결이 다른, **어피니티(affinity)** 를 위한 정책이다. 예를 들어 사용자 ID를 해시해서 같은 사용자의 요청을 항상 같은 백엔드로 보내면, 그 백엔드의 로컬 캐시 적중률이 올라간다. consistent hashing을 쓰므로 백엔드가 추가·제거돼도 **재배치되는 키가 최소화**된다(전체가 다 흩어지지 않는다). 세션이 있는 워크로드나 캐시 지역성이 중요한 곳에서 쓴다.

### 12.5.4 LB policy는 계층적으로 합성된다

여기서 한 단계 더 들어가자. 실제 gRPC의 LB policy는 **단일 평면이 아니라 트리(tree)로 합성**된다. 예를 들어 xDS 환경에서는:

```text
   xds_cluster_manager        (어느 클러스터로?)
        └─ priority            (리전 우선순위: 1순위 죽으면 2순위로)
             └─ weighted_target (지역성 가중치: us-east 70%, us-west 30%)
                  └─ round_robin (그 지역 안에서 백엔드 순환)
                       └─ (각 백엔드 subchannel)
```

각 계층이 자식 policy에게 "너는 이 부분집합을 맡아라"라고 위임하고, 자식이 다시 picker를 만들어 부모에게 올린다. picker들이 중첩되어 최종적으로 하나의 "이 RPC는 이 subchannel로"라는 결정이 나온다. 이 합성 구조 덕분에 "리전 페일오버 + 지역성 가중 + 그 안에서 라운드로빈"같은 복합 정책을 레고처럼 조립할 수 있다. 이걸 직접 손으로 구성하는 건 끔찍하지만, **xDS 컨트롤 플레인이 이 트리를 자동으로 만들어 내려준다**(12.6).

## 12.6 Look-aside LB와 xDS: 외부 컨트롤 플레인의 등장

DNS resolver의 한계(12.3.2)와 클라이언트 사이드 LB의 한계(연결 폭증, 멤버십 지연)를 동시에 극복하려는 흐름이 **look-aside LB**, 그리고 그 후계자 **xDS**다.

### 12.6.1 in-band vs look-aside

지금까지 본 DNS + round_robin은 **in-band(인밴드)** 모델이다 — 클라이언트가 직접 백엔드들에 연결하고, 그 데이터 연결 자체로 상태를 파악한다. resolver는 "주소 목록"만 주고 빠진다.

**look-aside(룩어사이드)** 모델은 다르다. 별도의 **로드밸런서 서버(컨트롤 플레인)** 가 있어서, 클라이언트는 데이터 트래픽과 **별개의 채널**로 그 LB 서버에게 묻는다: "지금 어느 백엔드로 보내면 돼? 가중치는? 헬스는?" LB 서버는 전체 백엔드를 모니터링하다가 "현재 살아 있고 가중치는 이렇다"는 목록을 **스트리밍으로** 내려준다. 클라이언트는 실제 데이터는 백엔드에 직접 보내되(여전히 클라이언트 사이드 LB), **누구에게 보낼지/가중치는 외부 두뇌가 정해준다.**

```text
   look-aside LB 모델
   ───────────────────
                  ┌──────────────────────┐
                  │  LB 서버 (컨트롤플레인)│  ← 백엔드 헬스/부하를 모니터
                  └─────────┬────────────┘
       "이 백엔드들 써,      │  스트리밍으로
        가중치는 이렇게"      ▼  백엔드 목록+정책 push
                  ┌──────────────────────┐
                  │       Client          │
                  └───┬─────────┬────────┘
        실제 RPC는    │         │   (데이터는 백엔드로 직접)
        백엔드로 직접 ▼         ▼
                  Backend 1   Backend 2 ...
```

이 모델의 첫 구현이 **gRPCLB**였다. SRV 레코드로 LB 서버를 발견하고, 그 LB 서버와 `LoadBalancer` gRPC 서비스로 통신해 백엔드 목록을 받는 방식이었다. gRPCLB는 "DNS로는 못 하는 헬스·가중치 인지 분산"을 가능케 했지만, **gRPC 전용**이라는 한계가 있었다. HTTP, 다른 RPC 프레임워크는 못 썼다. 그래서 **gRPCLB는 현재 deprecated(폐기 예정)** 이고, 그 자리를 더 일반적이고 강력한 **xDS**가 차지했다.

### 12.6.2 xDS: Envoy가 만든 보편 표준

**xDS**는 원래 **Envoy 프록시**가 자신의 설정(라우팅, 클러스터, 엔드포인트)을 동적으로 받기 위해 만든 **API 군(group)** 이다. "x Discovery Service"의 약자로, x 자리에 여러 종류가 들어간다. gRPC는 이 xDS API를 **직접** 말하도록 구현해서, Envoy와 **같은 컨트롤 플레인(예: Istio의 Istiod)** 을 공유할 수 있게 됐다.

xDS의 주요 디스커버리 서비스 네 가지를 알아야 한다.

| 약자 | 풀네임 | 무엇을 내려주나 | 비유 |
|------|--------|----------------|------|
| **LDS** | Listener Discovery Service | 리스너(어떤 트래픽을 어떻게 받을지) | "현관문 설정" |
| **RDS** | Route Discovery Service | 라우팅 규칙(호스트/경로/헤더 → 어느 클러스터로) | "이 손님은 이 방으로" |
| **CDS** | Cluster Discovery Service | 클러스터(논리적 백엔드 그룹)와 LB 정책 | "이 방엔 이런 그룹의 서버들" |
| **EDS** | Endpoint Discovery Service | 엔드포인트(클러스터에 속한 실제 IP들 + 가중치 + 지역성 + 헬스) | "그 그룹에 실제로 이 IP들이 있고 상태는 이렇다" |

이 네 개가 **계단식으로 연결**된다. 클라이언트가 `xds:///orders.svc`로 채널을 만들면:

```text
   xds:///orders.svc 채널 생성
        │
        ▼
   [LDS] "orders.svc 리스너 설정 줘" → 라우트 설정(RDS)을 가리킴
        │
        ▼
   [RDS] "이 호스트의 라우팅 규칙 줘"
         → "/orders.v1.OrderService/* 는 cluster=orders-prod 로,
            가중치 카나리: orders-prod 95% / orders-canary 5%"
        │
        ▼
   [CDS] "cluster=orders-prod 의 설정 줘"
         → "LB정책=round_robin, 헬스체크=켬, 엔드포인트는 EDS로"
        │
        ▼
   [EDS] "orders-prod 의 엔드포인트 줘"
         → [ {10.0.0.1:50051, weight=1, locality=us-east, health=HEALTHY},
              {10.0.0.2:50051, weight=1, locality=us-east, health=HEALTHY},
              {10.0.1.7:50051, weight=1, locality=us-west, health=HEALTHY} ]
        │
        ▼
   이 모든 정보가 gRPC 내부에서
   resolver(주소+config) + LB policy 트리(12.5.4)로 변환됨
```

위 그림은 네 디스커버리 서비스를 별개의 단계로 그렸지만, 실제 gRPC의 xDS 클라이언트는 이들을 **하나의 양방향 스트림으로 다중화**해서 받는다. 이를 **ADS(Aggregated Discovery Service)** 라 한다. 왜 굳이 한 스트림으로 묶는가? LDS/RDS/CDS/EDS를 제각각 별도 스트림으로 받으면 **갱신 순서가 어긋나** 일시적으로 라우팅이 깨질 수 있기 때문이다 — 예컨대 새 CDS(클러스터)가 먼저 도착했는데 그 클러스터의 EDS(엔드포인트)가 아직 안 와서 "가리키는 곳은 있는데 보낼 IP가 없는" 중간 상태가 생긴다. 단일 스트림으로 묶으면 컨트롤 플레인이 갱신을 **일관된 순서로**(CDS→EDS, LDS→RDS) 내려보낼 수 있어 이 틈을 없앤다. gRPC의 xDS 구현은 이 ADS 변형을 기본으로 쓴다. 프로토콜 차원에서도 클라이언트가 받은 설정을 잘 적용했는지 컨트롤 플레인에 되돌려 알리는 ACK/NACK 흐름(State-of-the-World 또는 증분 xDS)을 갖고 있어, 잘못된 설정이 내려오면 NACK으로 거부하고 직전 정상 설정을 유지한다.

핵심은 이거다. **xDS는 DNS resolver가 못 하던 모든 것을 한다.**
- 헬스 인지: EDS가 health 상태를 직접 내려준다.
- 가중치: EDS의 weight, RDS의 트래픽 분할(카나리/A-B 테스트).
- 지역성(locality) 라우팅: locality 정보로 "같은 리전 우선, 장애 시 다른 리전" 페일오버.
- 즉시 갱신: 스트리밍 push라 TTL 지연이 없다. 백엔드가 죽으면 컨트롤 플레인이 바로 빼서 내려준다.
- 중앙 정책: 모든 트래픽 규칙이 컨트롤 플레인 한 곳에서 관리된다.

### 12.6.3 proxyless 서비스 메시: 사이드카 없이 메시의 두뇌만

전통적 서비스 메시(Istio + Envoy)는 **모든 파드 옆에 Envoy 사이드카**를 붙인다. 애플리케이션은 평범하게 통신하고, 사이드카가 트래픽을 가로채 라우팅·mTLS·관측을 처리한다. 강력하지만, **사이드카마다 프록시 한 홉씩 지연이 추가**되고, 메모리·CPU를 잡아먹는다.

**proxyless(프록시리스) 모드**는 영리한 절충이다. **사이드카 프록시를 없애고, 대신 gRPC 라이브러리가 직접 xDS를 말한다.** gRPC 클라이언트가 컨트롤 플레인(Istiod)에 직접 연결해 LDS/RDS/CDS/EDS를 받아, **자기 안의 resolver와 LB policy로 변환**한다. 데이터 트래픽은 사이드카를 거치지 않고 백엔드로 직접 간다.

```text
   사이드카 메시 vs proxyless 메시
   ────────────────────────────────

   [사이드카]                        [proxyless]
   App → Envoy(사이드카)             App(gRPC, xDS 내장)
          │  xDS                          │  xDS
          ▼                               ▼
       Istiod ◀──────────────────────  Istiod
          ▲                               
          │ (반대편도 사이드카)            데이터는 직접:
   Envoy ← App                        App ──직접──▶ Backend
                                      (프록시 홉 없음!)
   데이터: App→Envoy→네트워크→Envoy→App
   (홉 2개 추가)
```

- **장점**: 프록시 홉이 없어 지연↓, 리소스↓. 그러면서도 메시의 두뇌(중앙 정책, 가중치 라우팅, 지역성)는 그대로 누린다.
- **대가**: 메시의 기능이 gRPC 라이브러리에 구현된 만큼만 동작한다. 사이드카가 주던 일부 기능(L7 정책 일부, 비-gRPC 프로토콜 처리, 일부 보안 기능)은 못 쓸 수 있다. 또 "내 클라이언트가 컨트롤 플레인에 직접 의존"하므로 부트스트랩 설정(어디가 컨트롤 플레인인지)이 필요하다.

proxyless gRPC는 보통 **부트스트랩 파일**(`GRPC_XDS_BOOTSTRAP` 환경변수가 가리키는 JSON)로 "xDS 서버는 여기, 노드 ID는 이것"을 알려주고 시작한다. 그 뒤로는 `xds:///` target만 쓰면 모든 동적 설정이 컨트롤 플레인에서 흘러든다.

이 영역의 mTLS·인증서 자동 발급(SDS: Secret Discovery Service)은 [[11 - 보안 - TLS mTLS 인증]]과 맞닿는다 — xDS는 사실 보안 설정(인증서)도 SDS로 내려줄 수 있다.

## 12.7 헬스 인지 로드밸런싱: 죽은 백엔드를 빼는 메커니즘

"건강한 백엔드만 고른다"는 목표는 여러 계층의 협업으로 달성된다. 한 가지 마법이 아니라 **방어선의 중첩**이다.

```text
   헬스 인지의 3중 방어선
   ──────────────────────
   1차) subchannel 연결 상태
        TCP/HTTP2 연결이 끊기면 그 subchannel은 TRANSIENT_FAILURE.
        round_robin picker가 ready 리스트에서 자동 제외.
        → "TCP가 죽은" 백엔드를 잡는다. 추가 설정 불필요.

   2차) gRPC 애플리케이션 헬스체크 (grpc.health.v1.Health)
        TCP는 살아 있어도 앱이 "NOT_SERVING"일 수 있다(예: DB 끊김).
        service config의 healthCheckConfig로 활성화하면,
        LB가 각 subchannel에 대해 Health/Watch 스트림을 열어
        SERVING/NOT_SERVING을 구독.
        NOT_SERVING이면 그 subchannel을 ready에서 제외.
        → "TCP는 살아 있으나 앱이 아픈" 백엔드를 잡는다.
        (상세는 [[13 - 안정성 - Retry Health Check Keepalive]])

   3차) outlier detection (xDS/Envoy 계열, 통계적 추방)
        실시간 에러율/지연을 보고 "통계적으로 이상한" 백엔드를
        일정 시간 추방(eject)했다가 복귀시킴.
        → 헬스체크는 통과하지만 실제로 에러를 뱉는 백엔드를 잡는다.
```

핵심 직관: **DNS는 후보 명단만 주고, "실제로 보낼지"는 이 3중 방어선이 거른다.** 그래서 12.3.2에서 "DNS는 헬스 정보가 없다"는 한계가 치명적이지 않다 — 그 공백을 subchannel 상태와 헬스체크가 메운다. 다만 1차 방어선(subchannel 상태)만으로는 "앱은 아픈데 TCP는 멀쩡한" 경우를 못 잡으므로, 신뢰성이 중요하면 2차(gRPC 헬스체크)를 켜야 한다.

service config에서 헬스체크를 켜는 모습:

```json
{
  "loadBalancingConfig": [ { "round_robin": {} } ],
  "healthCheckConfig": {
    "serviceName": "orders.v1.OrderService"
  }
}
```

`serviceName`은 서버의 헬스 서비스에 `Check`/`Watch`할 때 넘기는 이름이다. 빈 문자열이면 "서버 전체"의 헬스를, 특정 서비스 이름이면 그 서비스만의 헬스를 본다. 자세한 프로토콜과 서버 측 구현은 [[13 - 안정성 - Retry Health Check Keepalive]]에서.

## 12.8 운영 함정: DNS만 믿다가 겪는 일들

이론을 알았으니, 실전에서 사람들이 실제로 부딪히는 함정들을 보자. 전부 "이름 해석과 LB의 상호작용을 오해해서" 생긴다.

### 12.8.1 함정 1: 스케일아웃해도 기존 연결이 새 파드를 모른다

가장 흔하고 가장 아픈 함정이다. 시나리오:

```text
   T0: 백엔드 파드 3개. 클라이언트가 round_robin으로 셋에 연결. 잘 분산됨.
   T1: 트래픽 급증 → 오토스케일러가 파드를 3 → 10으로 늘림.
   T2: ... 그런데 클라이언트는 여전히 기존 3개로만 보낸다!
       새 파드 7개는 놀고 있다. 기존 3개는 과부하.
```

왜? **클라이언트가 새 파드의 존재를 모르기 때문이다.** 클라이언트가 새 파드를 알려면 **resolver가 재해석해서 갱신된 주소 목록을 줘야** 하는데(12.2.3), DNS resolver는 최소 재해석 간격 + TTL 캐시 때문에 한참 뒤에야 갱신한다. 게다가 — 더 교묘하게 — **기존 연결이 멀쩡히 살아 있으면 재해석을 trigger할 이유조차 없다.** round_robin은 연결이 다 READY면 "굳이 resolver를 다시 부를" 동기가 없다.

해결책은 여러 겹이다.

**(a) 서버가 연결을 주기적으로 끊게 한다 — `MAX_CONNECTION_AGE`.**

이게 정석 해법이다. 서버 측에 "한 연결은 최대 N분/N초만 유지하고, 그 후엔 우아하게 종료(GOAWAY)하라"는 설정을 건다.

```go
// 서버 측 (Go): 연결 강제 재분배
import "google.golang.org/grpc/keepalive"

srv := grpc.NewServer(
    grpc.KeepaliveParams(keepalive.ServerParameters{
        MaxConnectionAge:      30 * time.Minute, // 연결 최대 수명
        MaxConnectionAgeGrace: 5 * time.Minute,  // 진행 중 RPC 마무리 유예
    }),
)
```

연결이 `MaxConnectionAge`에 도달하면 서버가 `GOAWAY` 프레임([[04 - HTTP2 깊이 보기 - 전송 계층]])을 보내 "이 연결 그만 쓰자"고 알린다. 클라이언트는 진행 중인 RPC를 마치고 연결을 닫은 뒤, **새로 연결을 맺으려 resolver를 다시 부른다 → 그 사이 갱신된 DNS 결과를 받아 새 파드를 발견한다.** 즉 MAX_CONNECTION_AGE는 "연결을 일부러 주기적으로 갈아엎어서 멤버십을 강제로 최신화"하는 트릭이다. 이게 K8s + gRPC 운영의 거의 필수 설정으로 통한다.

```text
   MAX_CONNECTION_AGE의 효과
   ──────────────────────────
   연결 수명 ──────────┐ MaxConnectionAge 도달
   ─────────────────────▶ 서버가 GOAWAY 전송
                          │
                          ▼ 클라이언트: 진행 RPC 마치고 연결 닫음
                          │
                          ▼ 새 연결 위해 resolver 재호출
                          │
                          ▼ DNS 재조회 → [기존3 + 새7] 받음!
                          │
                          ▼ round_robin이 10개 모두에 연결 → 분산 복구
```

**(b) DNS 최소 재해석 간격을 짧게**(구현이 허용하면)하거나, **(c) xDS로 갈아탄다**(컨트롤 플레인이 즉시 push하므로 이 문제가 원천 소멸).

### 12.8.2 함정 2: pick_first인 줄 모르고 분산을 기대

"DNS가 IP 10개를 주니까 알아서 분산되겠지" — 틀렸다. **기본 LB policy는 pick_first**라 10개 중 **하나로만** 간다(12.5.1). 분산을 원하면 service config로 `round_robin`을 **명시**해야 한다. 이 한 줄을 빼먹어서 "왜 한 파드만 터지지?"를 디버깅하는 경우가 정말 많다.

### 12.8.3 함정 3: ClusterIP service로 gRPC 분산을 기대

K8s의 일반 ClusterIP service는 DNS A 레코드로 **VIP 하나**만 준다(12.3.1). 클라이언트는 그 VIP로 연결 하나를 맺고, 분산은 kube-proxy(L4)에 맡겨진다 → 12.4.1의 **연결 쏠림** 그대로다. gRPC를 제대로 분산하려면 **headless service(`clusterIP: None`)** 를 써서 모든 파드 IP가 노출되게 하고 round_robin을 돌려야 한다.

```yaml
# gRPC 클라이언트 사이드 LB를 위한 headless service
apiVersion: v1
kind: Service
metadata:
  name: orders
spec:
  clusterIP: None          # ← 핵심! headless. A 레코드가 모든 파드 IP 반환
  selector:
    app: orders
  ports:
    - port: 50051
      targetPort: 50051
```

### 12.8.4 함정 4: 단일 연결 쏠림 — 멤버십 최신화 지연의 일반형

함정 1의 일반화다. **클라이언트 사이드 LB는 "resolver가 준 멤버십"이 최신일 때만 잘 분산한다.** 멤버십이 stale하면(파드 교체, 롤링 배포, 노드 드레인 등으로 IP가 바뀌었는데 클라이언트가 모름), 트래픽이 사라진 IP로 가서 에러가 나거나, 살아남은 소수에 쏠린다. 롤링 배포 중에 특히 두드러진다. 해법은 함정 1과 같다: MAX_CONNECTION_AGE로 강제 갱신, 또는 push 기반 xDS.

### 12.8.5 함정 5: 연결 폭증

round_robin은 **모든 백엔드에 연결**한다(12.5.2). 백엔드가 수백~수천 개면, 클라이언트마다 그만큼의 연결을 들고 있어야 한다. 클라이언트도 많으면 서버 측 연결 수가 폭발한다. 이럴 땐 (a) subsetting(각 클라이언트가 백엔드의 부분집합에만 연결), (b) look-aside/xDS로 후보 축소, (c) 애초에 pick_first + L7 프록시 같은 다른 토폴로지를 고려한다. "round_robin은 공짜가 아니다 — 연결 비용을 낸다"를 기억하자.

함정들을 한 표로 정리하면:

| 함정 | 증상 | 근본 원인 | 처방 |
|------|------|-----------|------|
| 스케일아웃 무시 | 새 파드 노는데 기존 과부하 | resolver 갱신 안 됨 + 연결 살아있어 재해석 안 함 | MAX_CONNECTION_AGE, xDS |
| pick_first 오해 | 한 파드만 받음 | 기본이 pick_first | round_robin 명시 |
| ClusterIP 분산 기대 | VIP 하나로 L4 쏠림 | A 레코드가 VIP만 | headless service |
| 멤버십 지연 | 롤링 중 에러/쏠림 | stale resolver 결과 | 강제 재해석, push 기반 |
| 연결 폭증 | 연결 수 폭발 | round_robin이 전체 연결 | subsetting, 후보 축소 |

## 12.9 직접 만들어보기: 커스텀 resolver와 LB policy 등록

내장 플러그인으로 안 되는 요구(특수한 서비스 디스커버리, 회사 자체 메타데이터 기반 분산 등)가 있으면 **직접 등록**할 수 있다. 전체 구현은 방대하므로, "어떤 인터페이스를 채우는가"의 골격만 보자. 개념을 잡는 게 목적이다.

### 12.9.1 커스텀 resolver의 골격 (Go)

resolver는 두 인터페이스를 구현한다: `Builder`(scheme을 등록하고 인스턴스를 만든다)와 `Resolver`(재해석을 처리한다).

```go
package myresolver

import "google.golang.org/grpc/resolver"

// 1) Builder: scheme 등록 + resolver 인스턴스 생성
type myBuilder struct{}

func (b *myBuilder) Scheme() string { return "my" } // my:/// 를 처리

func (b *myBuilder) Build(
    target resolver.Target,
    cc resolver.ClientConn,        // 결과를 채널에 보고하는 통로
    opts resolver.BuildOptions,
) (resolver.Resolver, error) {
    r := &myResolver{target: target, cc: cc}
    r.start() // 최초 해석 + watch 시작
    return r, nil
}

// 2) Resolver: 재해석 처리
type myResolver struct {
    target resolver.Target
    cc     resolver.ClientConn
}

func (r *myResolver) start() {
    // 어딘가(컨피그서버/etcd/커스텀 디스커버리)에서 주소를 구해 온다
    addrs := []resolver.Address{
        {Addr: "10.0.0.1:50051"},
        {Addr: "10.0.0.2:50051"},
    }

    // service config(JSON)를 파싱해서 함께 보고 (round_robin 강제 등)
    sc := r.cc.ParseServiceConfig(`{"loadBalancingConfig":[{"round_robin":{}}]}`)

    // 핵심: 채널에 "이게 새 상태다"라고 보고. 이게 LB policy로 흘러간다.
    r.cc.UpdateState(resolver.State{
        Addresses:     addrs,
        ServiceConfig: sc,
    })
}

// LB policy가 "주소 다시 받고 싶다"고 할 때 호출됨
func (r *myResolver) ResolveNow(resolver.ResolveNowOptions) {
    r.start() // 재해석. 보통은 watch로 비동기 갱신을 권장.
}

func (r *myResolver) Close() { /* watch 정리 */ }

// 3) 프로세스 시작 시 전역 등록
func init() {
    resolver.Register(&myBuilder{})
}
```

이제 클라이언트가 `grpc.NewClient("my:///some-name", ...)`을 호출하면, scheme `my`가 이 builder를 골라 우리 resolver가 동작한다. 핵심 한 줄은 `r.cc.UpdateState(...)` — 이게 12.2에서 말한 "resolver의 출력(주소 + service config)"을 채널에 밀어 넣는 지점이다.

### 12.9.2 커스텀 LB policy의 골격 (개념)

LB policy도 비슷하게 `Builder`로 등록하고, 핵심은 두 콜백이다.

```text
   LB policy가 구현해야 하는 두 가지
   ─────────────────────────────────
   1. UpdateClientConnState(state):
        resolver가 준 주소 목록을 받아
        - 새 주소 → cc.NewSubConn(addr)로 subchannel 생성·연결
        - 사라진 주소 → 해당 subchannel 종료
        를 수행

   2. UpdateSubConnState(subConn, state):
        각 subchannel의 상태(IDLE/CONNECTING/READY/TRANSIENT_FAILURE)
        변화를 받아
        - "지금 보낼 수 있는 subchannel 집합"을 갱신하고
        - 새 picker를 만들어 cc.UpdateState(balancer.State{Picker: ...})로 보고

   그리고 Picker:
        Pick(info) (PickResult, error)
        → 매 RPC마다 호출. 어느 subConn으로 보낼지 반환.
          round_robin이면 여기서 원형 카운터를 한 칸 돌린다.
```

예를 들어 "round_robin인데, 짝수 시각엔 us-east만, 홀수 시각엔 전체"처럼 회사 고유 정책이 필요하면, 이 Picker의 `Pick` 로직과 subchannel 집합 관리만 커스터마이즈하면 된다. 실무에서 커스텀 LB를 직접 짜는 일은 드물지만, **"LB policy = subchannel 집합 관리 + picker 제공"이라는 본질**을 이해하면 내장 정책들의 동작이 전부 자명해진다.

### 12.9.3 다른 언어에서의 등가물

- **Java**: `NameResolverProvider`/`NameResolver`와 `LoadBalancerProvider`/`LoadBalancer`를 구현하고 SPI(`META-INF/services`)나 `NameResolverRegistry`로 등록. `ManagedChannelBuilder.defaultLoadBalancingPolicy("round_robin")` 또는 `defaultServiceConfig(...)`로 정책 지정.
- **Python**: 채널 옵션 `grpc.lb_policy_name` 또는 service config JSON(`grpc.service_config`)으로 지정. 커스텀 resolver는 C-core 플러그인 수준이라 Go/Java만큼 친화적이진 않다.
- **C++/core**: 모든 언어의 토대인 gRPC Core에 resolver/LB 레지스트리가 있고, 위 언어들은 대개 이를 래핑한다.

언어마다 표면 API는 달라도 **모델은 동일**하다: scheme→resolver, resolver→(주소+service config), LB policy→(subchannel 집합+picker). 이 공통 모델이 [[01 - gRPC란 무엇인가]]에서 강조한 "언어 중립적 설계"의 또 다른 증거다.

## 12.10 종합 예제: 처음부터 끝까지 따라가기

지금까지의 조각을 하나의 흐름으로 꿰어보자. K8s에서 `orders` 서비스를 round_robin으로 호출하는 클라이언트의 전 생애를 따라간다.

```proto
// orders.proto
syntax = "proto3";
package orders.v1;

service OrderService {
  rpc GetOrder(GetOrderRequest) returns (Order);
}

message GetOrderRequest { string order_id = 1; }
message Order {
  string order_id = 1;
  string status   = 2;
  int64  amount   = 3;
}
```

```go
// client.go
package main

import (
    "context"
    "log"
    "time"

    "google.golang.org/grpc"
    "google.golang.org/grpc/credentials/insecure"
    orderspb "example.com/orders/v1"
)

const serviceConfig = `{
  "loadBalancingConfig": [ { "round_robin": {} } ],
  "methodConfig": [{
    "name": [{ "service": "orders.v1.OrderService" }],
    "timeout": "2s",
    "retryPolicy": {
      "maxAttempts": 3,
      "initialBackoff": "0.1s",
      "maxBackoff": "1s",
      "backoffMultiplier": 2.0,
      "retryableStatusCodes": ["UNAVAILABLE"]
    }
  }],
  "healthCheckConfig": { "serviceName": "orders.v1.OrderService" }
}`

func main() {
    // dns:/// + headless service "orders" → 모든 파드 IP를 받는다
    conn, err := grpc.NewClient(
        "dns:///orders.default.svc.cluster.local:50051",
        grpc.WithTransportCredentials(insecure.NewCredentials()),
        grpc.WithDefaultServiceConfig(serviceConfig),
    )
    if err != nil {
        log.Fatalf("dial: %v", err)
    }
    defer conn.Close()

    client := orderspb.NewOrderServiceClient(conn)

    // 여러 번 호출하면 round_robin으로 파드들에 고르게 분산된다
    for i := 0; i < 6; i++ {
        ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
        resp, err := client.GetOrder(ctx, &orderspb.GetOrderRequest{OrderId: "o-123"})
        cancel()
        if err != nil {
            log.Printf("call %d failed: %v", i, err)
            continue
        }
        log.Printf("call %d → status=%s (served by some pod)", i, resp.Status)
    }
}
```

이 코드가 실행될 때 내부에서 벌어지는 일의 타임라인:

```text
   t=0   grpc.NewClient("dns:///orders...:50051", ...)
         - target 파싱: scheme="dns", endpoint="orders...:50051"
         - dns resolver builder 선택 → Build() 호출
         - default service config 적용: round_robin + 헬스체크 + 재시도

   t=1   dns resolver가 A/AAAA 조회
         → [10.244.1.5:50051, 10.244.2.7:50051, 10.244.3.9:50051]
         - cc.UpdateState({addrs, serviceConfig})

   t=2   round_robin LB policy가 상태 수신
         - 세 주소 모두에 NewSubConn → 연결 시작 (CONNECTING)
         - healthCheckConfig 켜져 있으니 각 subchannel에 Health/Watch 스트림 염

   t=3   subchannel들이 하나씩 READY로
         - 헬스체크도 SERVING 확인
         - ready 리스트 = [SC1, SC2, SC3]
         - round_robin picker 생성 → cc.UpdateState(picker)
         - 채널 상태 IDLE→READY (이제 RPC 가능)

   t=4+  GetOrder 호출 6번
         - call 0 → picker → SC1 (10.244.1.5)
         - call 1 → picker → SC2 (10.244.2.7)
         - call 2 → picker → SC3 (10.244.3.9)
         - call 3 → picker → SC1
         - call 4 → picker → SC2
         - call 5 → picker → SC3
         → 6번 호출이 3개 파드에 2번씩 고르게!

   (만약 t=4 시점에 SC2의 파드가 죽으면)
         - SC2: READY → TRANSIENT_FAILURE
         - round_robin이 ready 리스트에서 SC2 제외 → [SC1, SC3]
         - 호출은 SC1, SC3로만 순환 (에러 없이 자동 우회)
         - LB가 resolver에 ResolveNow → DNS 재조회로 교체 파드 발견 시도
```

이 한 장의 타임라인에 이 장의 모든 개념이 들어 있다: target 파싱(12.1) → DNS resolver의 주소+config 출력(12.2, 12.3) → round_robin의 subchannel 관리와 picker(12.5) → 헬스 인지 제외(12.7) → 재해석(12.2.3, 12.8). 이걸 머릿속에서 한 번 재생할 수 있으면 이 장을 소화한 셈이다.

## 12.11 디버깅: 지금 내 채널이 어디에 연결돼 있나

이름 해석/LB 문제는 "보이지 않아서" 어렵다. 가시화 도구가 필수다. 깊은 내용은 [[14 - 관찰성과 디버깅 - Reflection grpcurl]]에 있지만, 이 장 맥락에서 유용한 것만.

- **channelz**: gRPC 내장 디버그 서비스. 현재 채널이 어떤 subchannel을 가지고 있고, 각각의 상태(READY/TRANSIENT_FAILURE)와 대상 주소, 누적 RPC 수/에러 수를 들여다본다. "round_robin이라는데 왜 한 곳만?"을 진단할 때 channelz로 subchannel 목록을 보면 즉시 안다(혹시 ready가 1개뿐인가?).
- **환경변수 트레이싱**(Go): `GRPC_GO_LOG_SEVERITY_LEVEL=info`, `GRPC_GO_LOG_VERBOSITY_LEVEL=2`로 resolver/balancer 로그를 켜면 "DNS가 무슨 주소를 줬는지, LB가 무슨 subchannel을 만들었는지"가 찍힌다. C-core 계열은 `GRPC_TRACE=resolver,round_robin,pick_first` + `GRPC_VERBOSITY=DEBUG`.
- **dig / nslookup**: 의심스러우면 DNS부터 의심하라. `dig +short orders.default.svc.cluster.local`로 "DNS가 실제로 몇 개의 IP를 주는지" 직접 확인한다. 한 개만 나오면 headless service가 아니거나 ClusterIP다(함정 3).

```bash
# DNS가 정말 여러 파드 IP를 주는지 확인 (클러스터 내부에서)
dig +short orders.default.svc.cluster.local
# 기대: 여러 줄의 파드 IP. 한 줄(VIP)만 나오면 headless가 아님!

# resolver/LB 내부 동작 로깅 (Go 클라이언트)
GRPC_GO_LOG_SEVERITY_LEVEL=info \
GRPC_GO_LOG_VERBOSITY_LEVEL=2 \
  ./client
```

[[14 - 관찰성과 디버깅 - Reflection grpcurl]]에서 본 `grpcurl` 자체는 LB를 거의 안 쓰지만(보통 단일 타깃), 서버가 정말 응답하는지 백엔드별로 직접 찔러보는 데 유용하다.

## 12.12 한 발 물러서서: 이 장을 관통하는 설계 철학

마지막으로 큰 그림을 다시 묶자. 이 장의 모든 디테일은 사실 **하나의 긴장 관계**에서 파생된다.

```text
   HTTP/2의 미덕              ↔        분산의 요구
   ─────────────                       ──────────
   적은 수의                            많은 백엔드에
   롱리브드 멀티플렉싱 연결    vs       고르게 퍼뜨리기
   (연결 비용↓, HOL 회피)              (한 곳 쏠림 방지)
```

이 긴장을 해소하는 방식의 스펙트럼이 곧 이 장의 목차였다.

| 접근 | 어디서 분산? | 똑똑함의 위치 | 트레이드오프 |
|------|-------------|--------------|--------------|
| L4 프록시 | 연결 계층 | 인프라 | 단순하지만 HTTP/2에서 쏠림 |
| L7 프록시(Envoy) | 요청 계층, 프록시에서 | 프록시 | 정확하지만 추가 홉·운영비 |
| 클라 사이드 + DNS | 요청 계층, 클라에서 | 클라이언트 라이브러리 | 빠르지만 멤버십 갱신·헬스 표현 한계 |
| 클라 사이드 + xDS(proxyless) | 요청 계층, 클라에서 | 클라 + 중앙 컨트롤 플레인 | 강력하지만 컨트롤 플레인 의존·복잡 |

gRPC의 **기본**은 "클라이언트 사이드 + DNS + pick_first(또는 round_robin)"다. 이건 단순한 환경에 더없이 적합한, 의도적으로 보수적인 출발점이다. 그리고 요구가 커지면(헬스 인지, 가중치, 지역성, 즉각 갱신, 카나리) **xDS로 점진적으로 진화**할 수 있는 길을 열어뒀다. resolver/LB policy를 플러그인으로 추상화한 이유가 바로 이것이다 — **같은 채널 API를 유지한 채, 그 아래의 디스커버리·분산 메커니즘만 갈아끼울 수 있게.**

이 "추상화를 통한 진화 가능성"은 [[01 - gRPC란 무엇인가]]에서 본 gRPC의 일관된 철학이자, [[17 - 실전 - proto 관리와 하위호환성]]에서 볼 "인터페이스를 안정적으로 유지하며 구현을 진화시킨다"는 원칙의 인프라 버전이다.

## 핵심 요약

- **target URI는 호스트:포트가 아니라 RFC 3986 URI다.** `scheme://authority/endpoint` 구조이며 scheme(`dns`, `unix`, `ipv4`, `xds`, `passthrough`, ...)이 **name resolver 플러그인을 고른다.** scheme 생략 시 기본값은 보통 `dns`.
- **resolver의 출력은 두 가지다**: (1) 주소 목록, (2) service config(JSON). 그리고 resolver는 일회성이 아니라 **watch 모델**이라 TTL 만료·연결 실패·컨트롤 플레인 push 때 **재해석**한다.
- **service config가 LB 정책을 정한다.** `loadBalancingConfig`로 round_robin 같은 정책을, `healthCheckConfig`로 헬스체크를, `methodConfig`로 타임아웃([[10 - Deadline 취소 타임아웃]])·재시도([[13 - 안정성 - Retry Health Check Keepalive]])를 클라이언트 코드 수정 없이 내려줄 수 있다.
- **DNS resolver는 충분히 좋은 기본값이지만 세 가지 구조적 한계**가 있다: 헬스 정보 없음, 갱신 지연(TTL/캐시/최소간격), 가중치·지역성 표현력 빈약. 그래서 단독으로는 부족하다.
- **gRPC의 기본은 클라이언트 사이드 L7 LB다.** HTTP/2의 롱리브드 멀티플렉싱 연결은 L4 프록시에서 "연결 쏠림(connection pinning)"으로 불균형을 만들기 때문에, 분산을 **요청 단위로, 클라이언트 안에서** 한다. 추가 홉이 없고 요청 단위로 고르게 퍼진다.
- **pick_first(기본)는 되는 백엔드 하나에만 연결**한다 — 가볍고 프록시 친화적. **분산을 원하면 round_robin을 명시**해야 하며, round_robin은 **모든 백엔드에 연결하고 READY인 것만 순환**한다(죽은 백엔드 자동 제외).
- **헬스 인지는 3중 방어선**이다: subchannel 연결 상태(1차) + gRPC 애플리케이션 헬스체크(2차) + outlier detection(3차). DNS의 "헬스 모름" 공백을 이들이 메운다.
- **gRPCLB(look-aside, deprecated) → xDS**로 이어지는 외부 컨트롤 플레인 모델은 LDS/RDS/CDS/EDS로 라우팅·클러스터·엔드포인트·가중치·지역성을 동적으로 push한다. gRPC가 사이드카 없이 xDS를 직접 말하는 게 **proxyless 서비스 메시**다.
- **운영 최대 함정**: 스케일아웃해도 기존 롱리브드 연결이 새 파드를 모른다. **`MAX_CONNECTION_AGE`로 연결을 주기적으로 강제 종료(GOAWAY)** 시켜 재해석을 유발해 멤버십을 최신화하라. K8s에선 **headless service + round_robin**이 표준.
