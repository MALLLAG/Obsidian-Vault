**이 장이 답하는 질문**

- RPC는 1980년대부터 있었는데, 왜 그 많은 시도(Sun RPC, CORBA, RMI, DCOM, SOAP, Thrift)는 주류가 되지 못했고 gRPC는 됐는가?
- 구글은 이미 잘 돌아가던 내부 RPC(Stubby)를 두고 왜 굳이 2015년에 gRPC를 새로 써서 오픈소스로 풀었나?
- gRPC를 떠받치는 "네 기둥"은 정확히 무엇이며, 각각이 이 책의 어느 챕터로 이어지는가?
- RPC(동사·프로시저 중심)와 REST(명사·리소스 중심)는 철학적으로 무엇이 다르고, 그 트레이드오프는 어디서 갈리는가?
- 클라이언트가 메서드 하나를 호출했을 때, 바이트 한 줄까지 내려가면 와이어 위에서 실제로 무슨 일이 벌어지는가?

## 1. 들어가며: "함수를 호출하듯 네트워크 너머를 부른다"는 오래된 꿈

프로그래밍을 처음 배울 때 우리는 함수 호출을 너무 당연하게 여긴다.

```go
total := order.CalculateTotal(items)
```

이 한 줄에 우리는 엄청난 것을 가정한다. `CalculateTotal`은 **반드시** 실행된다. 호출은 **즉시** 반환된다(나노초 단위). 인자 `items`는 **손실 없이** 전달된다. 호출하는 쪽과 호출당하는 쪽은 **같은 메모리·같은 언어·같은 타입 시스템**을 공유한다. 실패한다면 그것은 명확한 예외(panic, exception)로 드러난다. 이 다섯 가지 가정은 한 프로세스 안에서는 거의 공짜로 주어진다.

분산 시스템의 역사는 이 편안한 함수 호출의 추상을 **네트워크 너머로 연장하려는 시도와, 그 시도가 반드시 부딪히는 물리 법칙 사이의 긴 줄다리기**다. RPC(Remote Procedure Call, 원격 프로시저 호출)라는 아이디어의 본질은 단순하고 매혹적이다.

> "원격 서버의 함수를, 로컬 함수를 부르듯이 부르자. 네트워크가 끼어 있다는 사실은 프레임워크가 숨겨주자."

이 문장 안에 RPC의 천재성과 함정이 동시에 들어 있다. 천재성은 **개발자 경험**이다. 개발자는 소켓을 열고, 바이트를 직렬화하고, 패킷 경계를 다루고, 재조립하는 저수준 잡일에서 해방되어 도메인 로직(`주문 합계 계산`, `재고 차감`)에만 집중한다. 함정은 **숨김 그 자체**다. 네트워크는 로컬 메모리가 아니다. 패킷은 유실되고, 지연은 마이크로초가 아니라 밀리초·초 단위이며, 서버는 호출 도중 죽고, 같은 요청이 두 번 도착하기도 한다. RPC가 네트워크를 "숨겨" 줄수록, 개발자는 이 잔혹한 현실을 잊고 코드를 짠다.

gRPC를 제대로 이해한다는 것은, 이 **"숨김의 유혹"과 "현실의 폭로" 사이에서 gRPC가 어디에 선을 그었는가**를 이해하는 일이다. 결론부터 말하면 gRPC는 함수 호출의 *문법적 편안함*은 빌려오되, *분산의 잔혹한 현실*(타임아웃, 취소, 부분 실패, 백프레셔)은 절대 숨기지 않고 오히려 API의 1급 시민으로 끌어올렸다. 이 균형 감각이 어디서 왔는지를 알려면, 먼저 gRPC가 밟고 올라선 거인들의 어깨 — RPC의 40년 계보 — 를 봐야 한다.

```text
        "로컬 함수처럼 편하게"  ←──────────────→  "분산의 현실을 인정하라"
              (추상의 유혹)                          (물리 법칙)
                  │                                      │
   Sun RPC ─ CORBA ─ RMI ─ DCOM ─ SOAP ─ Thrift ─────────┤
   (대체로 '편함'에 베팅, 현실에 데임)                    │
                                                          │
   gRPC: 문법은 함수처럼, 의미론은 분산을 정직하게 ────────┘
```

## 2. RPC의 역사적 계보 — 40년의 시행착오

gRPC는 진공에서 태어나지 않았다. 적어도 다섯 세대에 걸친 RPC 실험의 **누적된 교훈**이다. 각 세대가 "무엇을 풀었고, 무엇에 실패했는가"를 따라가면, gRPC의 설계 결정 하나하나가 "선배들의 실패에 대한 의도적 응답"이라는 사실이 보인다.

### 2.1 1980년대 초: Sun RPC / ONC RPC와 XDR

현대적 의미의 RPC가 산업에 대규모로 자리 잡은 첫 사례는 **Sun Microsystems의 Sun RPC**(나중에 ONC RPC, Open Network Computing RPC로 표준화, RFC 1057/1831/5531)다. 1980년대 중반, Sun은 네트워크 파일 시스템 **NFS**를 만들면서 그 토대로 RPC가 필요했다. 여기서 두 가지 결정적 아이디어가 등장한다.

1. **IDL(Interface Definition Language)의 분리**: `.x` 파일에 인터페이스(프로시저 시그니처와 데이터 구조)를 *언어 중립적으로* 기술하고, `rpcgen` 컴파일러가 클라이언트 **스텁(stub)**과 서버 **스켈레톤(skeleton)**을 자동 생성한다. 손으로 직렬화 코드를 짜지 않는다는 이 아이디어는 이후 거의 모든 RPC의 뼈대가 된다. gRPC의 `.proto` + `protoc`는 이 패턴의 직계 후손이다.
2. **XDR(External Data Representation)**: 서로 다른 머신(빅엔디안/리틀엔디안, 워드 크기가 다른 CPU)이 데이터를 손실 없이 교환하려면 **표준 바이트 표현**이 필요하다. XDR은 "와이어 위에서는 모두가 합의한 한 가지 형식"이라는, 오늘날 Protocol Buffers 와이어 포맷이 푸는 것과 똑같은 문제를 1980년대에 이미 풀었다.

Sun RPC가 풀어낸 것: **언어/플랫폼 중립의 인터페이스 계약과 코드 생성.** 한계: 절차적이고(객체 개념 없음), C 중심이며, 전송이 단순하고(주로 UDP/TCP 위 자체 포맷), 인증·스트리밍·버전 관리 같은 현대적 요구가 빈약했다.

### 2.2 Apollo NCS와 DCE RPC: 표준화의 첫 큰 시도

비슷한 시기 Apollo Computer는 **NCS(Network Computing System)**와 그 IDL인 **NIDL**을 내놨다. NCS의 유산은 곧 OSF(Open Software Foundation)의 **DCE(Distributed Computing Environment) RPC**로 흡수된다. DCE RPC(1990년대 초)는 오래 살아남을 개념들을 정립했다.

- **UUID(Universally Unique Identifier)**로 인터페이스와 객체를 식별 — 오늘날 우리가 쓰는 그 UUID의 표준화된 뿌리다.
- 인터페이스 **버전 번호**를 1급 개념으로 다룸 — 하위호환성 문제를 일찍 의식했다(→ [[17 - 실전 - proto 관리와 하위호환성]]에서 gRPC가 이 문제를 어떻게 다루는지 본다).
- **Microsoft RPC(MSRPC)**가 DCE RPC를 기반으로 만들어졌고, 이는 곧 Windows의 DCOM으로 이어진다.

DCE RPC가 풀어낸 것: **엔터프라이즈급 식별·버전·보안 모델.** 한계: 무거웠고, 구현이 복잡했으며, 특정 벤더 생태계에 묶이는 경향이 있었다.

### 2.3 CORBA와 IIOP: 분산 "객체"의 황금기와 몰락

1990년대는 객체지향(OOP)의 시대였다. 자연스럽게 "원격 *프로시저*"를 넘어 "원격 *객체*"를 부르고 싶어졌다. OMG(Object Management Group)가 1991년 발표한 **CORBA(Common Object Request Broker Architecture)**가 이 야망의 정점이다.

CORBA의 핵심 구성은 다음과 같다.

```text
   클라이언트 객체                         서버 객체(Servant)
        │                                      │
     [Stub]                                 [Skeleton]
        │                                      │
        └────────► ORB ◄── IIOP ──► ORB ◄──────┘
              (Object Request Broker)
        - IDL로 인터페이스 정의
        - ORB가 원격 객체 참조(IOR)를 들고 메시지 라우팅
        - IIOP(Internet Inter-ORB Protocol)로 ORB 간 통신
```

CORBA는 **언어 중립 IDL**, **ORB(Object Request Broker)**라는 미들웨어 버스, 그리고 서로 다른 벤더의 ORB가 상호운용하도록 하는 **IIOP**(CORBA 2.0, 1996)를 제시했다. 야망은 거대했다. C++ 객체가 자바 객체의 메서드를, 다시 코볼 객체의 메서드를 투명하게 부르는 세상.

그런데 CORBA는 무너졌다. 그 실패의 원인은 gRPC 설계자들에게 **반면교사 교과서**가 됐다.

| CORBA의 문제 | 결과 | gRPC가 취한 반대 선택 |
|---|---|---|
| 사양이 거대하고 모호 | 벤더마다 ORB 구현이 미묘하게 달라 상호운용 실패 | 명세를 작게, HTTP/2라는 기존 표준 위에 구축 |
| 분산 객체 모델 자체 | 원격 객체의 생명주기·GC·상태 공유가 지옥 | **객체가 아니라 무상태 메서드 호출**(서비스/메서드)로 회귀 |
| IIOP는 방화벽·프록시에 적대적 | 인터넷 환경에서 통과 곤란 | **HTTP/2** 위에 올려 기존 인프라(LB, 프록시)와 친화 |
| 가파른 학습 곡선 | 개발자 외면 | `.proto` 한 파일에서 시작하는 단순함 |

CORBA의 가장 깊은 교훈은 **"원격 객체는 환상"**이라는 것이다. 객체에는 *상태*와 *정체성(identity)*과 *생명주기*가 있다. 이걸 네트워크 너머로 투명하게 공유하려는 순간, 분산 가비지 컬렉션, 객체 마이그레이션, 상태 동기화 같은 풀기 힘든 문제들이 폭발한다. gRPC는 이 교훈을 받아들여 **"원격 객체"가 아니라 "원격 *메서드 호출*"로 의식적으로 후퇴**한다. gRPC의 서비스는 상태를 공유하는 객체가 아니라, 입력 메시지를 받아 출력 메시지를 돌려주는 *무상태 함수의 모음*에 가깝다(상태는 서버 내부가 알아서 관리하며, 와이어 모델에는 노출되지 않는다).

### 2.4 Java RMI와 Microsoft DCOM: 언어/플랫폼 감옥

CORBA가 "너무 보편적이려다" 무너졌다면, 같은 시기의 두 강자는 정반대 함정에 빠졌다 — **특정 언어/플랫폼 감옥.**

- **Java RMI(Remote Method Invocation)**(JDK 1.1, 1997): 자바 객체가 다른 JVM의 자바 객체 메서드를 부른다. `java.io.Serializable` 기반 객체 직렬화, 스텁/스켈레톤, 심지어 **분산 가비지 컬렉션**까지 제공했다. 아름답게 동작했지만 — **오직 자바끼리만.** 직렬화 포맷이 자바 전용이라 폴리글랏(다언어) 세상에 부적합했고, 자바 직렬화는 훗날 심각한 보안 취약점의 온상이 됐다.
- **Microsoft DCOM(Distributed COM)**(1996): MSRPC 위에 COM 객체를 분산. Windows 생태계에서는 강력했으나 본질적으로 **윈도우 종속**이었고, 인터넷 친화성이 낮았다.

두 기술의 공통 교훈: **RPC가 진짜 가치를 가지려면 언어·플랫폼 경계를 자유롭게 넘어야 한다.** 폴리글랏 마이크로서비스 세상(자바 서비스가 Go 서비스를, Python 데이터 파이프라인이 Kotlin 서비스를 호출)에서 "자바끼리만/윈도우끼리만"은 치명적 제약이다. gRPC는 처음부터 **언어 중립 IDL + 언어별 코드 생성**을 핵심으로 삼아 이 감옥을 부순다(→ [[06 - 코드 생성과 protoc 툴체인]]).

### 2.5 분산 객체의 환상과 "분산 컴퓨팅의 8가지 오류"

위의 모든 실패를 관통하는 한 가지 근본 오류가 있다. **"원격 호출을 로컬 호출과 똑같이 취급하려는 욕망."** 1994년 Sun의 L. Peter Deutsch(이후 James Gosling이 8번째 항목 추가)는 이 욕망이 부르는 재앙을 **"분산 컴퓨팅의 8가지 오류(8 Fallacies of Distributed Computing)"**로 정리했다. 신참 분산 시스템 엔지니어가 *암묵적으로 참이라고 가정하지만 실제로는 거짓인* 명제들이다.

| # | 오류 (틀린 가정) | 현실 | gRPC가 이를 다루는 챕터 |
|---|---|---|---|
| 1 | 네트워크는 신뢰할 수 있다 | 패킷은 유실되고 연결은 끊긴다 | [[13 - 안정성 - Retry Health Check Keepalive]] |
| 2 | 지연(latency)은 0이다 | RTT는 ms~수백 ms | [[10 - Deadline 취소 타임아웃]] |
| 3 | 대역폭은 무한하다 | 유한하며, 직렬화 크기가 중요 | [[03 - Protocol Buffers 2 - 인코딩과 와이어 포맷]] |
| 4 | 네트워크는 안전하다 | 도청·변조·위장 위협 상존 | [[11 - 보안 - TLS mTLS 인증]] |
| 5 | 토폴로지는 변하지 않는다 | 인스턴스는 뜨고 진다 | [[12 - 이름 해석과 로드밸런싱]] |
| 6 | 관리자는 한 명이다 | 여러 팀·조직이 운영 | [[14 - 관찰성과 디버깅 - Reflection grpcurl]] |
| 7 | 전송 비용은 0이다 | 직렬화/역직렬화·인프라 비용 존재 | [[15 - 성능과 Flow Control 백프레셔]] |
| 8 | 네트워크는 균질하다 | 다양한 OS·언어·하드웨어 | [[06 - 코드 생성과 protoc 툴체인]] |

이 표는 우연이 아니다. **gRPC의 기능 목록은 사실상 "8가지 오류에 대한 체계적 방어"로 읽을 수 있다.** 과거 RPC들은 이 오류들을 "추상으로 숨기려" 했고(그래서 개발자가 방심하게 만들었고), gRPC는 정반대로 **각 오류를 명시적 API로 노출**한다. 데드라인을 안 주면 호출이 영원히 매달릴 수 있다는 사실을, gRPC는 숨기지 않고 `context`/`Deadline`으로 들이민다. 이것이 1세대 RPC와 gRPC를 가르는 철학적 분수령이다.

### 2.6 2000년대 초: XML-RPC, SOAP, 그리고 WS-* 의 무게

2000년대로 넘어오며 바람이 바뀐다. 인터넷·방화벽·HTTP가 표준이 되자, "이진 프로토콜로 방화벽과 싸우지 말고, 어디서나 통과되는 HTTP와 사람이 읽는 XML 위에 RPC를 올리자"는 흐름이 생긴다.

- **XML-RPC**(1998, Dave Winer): 극도로 단순. HTTP POST 본문에 `<methodCall>` XML, 응답에 `<methodResponse>`. 매력은 단순함, 약점은 장황함과 빈약한 타입 시스템.
- **SOAP**(원래 Simple Object Access Protocol, ~2000): XML-RPC를 엔터프라이즈급으로 확장. **WSDL**(인터페이스 정의), 그리고 WS-Security, WS-Addressing, WS-ReliableMessaging 등 **WS-\* 스택**이 따라붙었다.

SOAP의 운명은 의미심장하다. SOAP은 "엄격한 계약(WSDL)"과 "코드 생성"이라는 — gRPC와 닮은 — 올바른 본능이 있었다. 그러나 **XML의 장황함**과 **WS-\* 표준의 폭발적 복잡성**이 발목을 잡았다. 단순한 호출 하나에 수 KB의 XML 봉투(envelope), 네임스페이스, 스키마 검증이 따라붙었다. 개발자들은 "이렇게까지 복잡할 일인가"라고 물었고, 그 질문이 다음 절의 REST 부상으로 이어진다.

SOAP의 교훈을 gRPC는 이렇게 흡수한다: **엄격한 계약과 코드 생성은 옳다. 그러나 그 직렬화는 사람이 읽는 장황한 텍스트(XML)가 아니라, 작고 빠른 이진 포맷(Protocol Buffers)이어야 하며, 표준은 거대한 위원회가 아니라 작고 실용적이어야 한다.**

### 2.7 Apache Thrift(2007): gRPC 직전의 가장 가까운 선배

gRPC를 이해할 때 절대 빼놓을 수 없는 직전 선배가 **Apache Thrift**다. 2007년 Facebook의 Mark Slee 등이 만들어 이후 Apache 재단에 기증했다. Thrift는 사실상 gRPC가 8년 뒤 다시 하게 될 것의 대부분을 이미 했다.

- 언어 중립 **IDL**(`.thrift` 파일)에 서비스·구조체 정의
- 수많은 언어로 **코드 생성**(C++, Java, Python, PHP, Ruby, Go ...)
- **직렬화 프로토콜 교체 가능**(Binary, Compact, JSON)
- **전송(transport) 교체 가능**(socket, framed, HTTP)

Thrift는 폴리글랏 사내 RPC로 훌륭히 동작했다. 그렇다면 왜 gRPC가 또 필요했나? 차이는 주로 **전송 계층의 야망**에 있다. Thrift가 등장하던 2007년에는 아직 **HTTP/2가 없었다**(SPDY가 2009년, HTTP/2 RFC 7540이 2015년). Thrift의 전송은 대체로 단순 TCP 스트림이라, **단일 연결 위 다중 스트림 멀티플렉싱, 헤더 압축, 서버 스트리밍, 흐름 제어** 같은 현대적 기능이 약했다. gRPC는 "Thrift가 옳게 한 IDL·코드 생성·다언어"는 계승하되, 마침 성숙한 **HTTP/2를 전송으로 채택**해 멀티플렉싱과 양방향 스트리밍을 1급으로 끌어올렸다(→ [[04 - HTTP2 깊이 보기 - 전송 계층]], [[05 - 통신의 4가지 방식 - Unary와 Streaming]]).

### 2.8 REST의 부상: RPC의 잠정적 패배와 그 빈틈

2000년 Roy Fielding의 박사 논문이 정의한 **REST(REpresentational State Transfer)**는 2000년대~2010년대 웹 API의 패권을 쥔다. REST의 승리는 RPC의 복잡성(SOAP/WS-\*)에 대한 반작용이자, HTTP 자체의 의미론을 영리하게 재사용한 결과다.

REST의 핵심은 **리소스(명사) 중심**이다. `/orders/42`라는 URL이 "주문 42번"이라는 *리소스*를 가리키고, HTTP 동사(GET/POST/PUT/DELETE)가 그 리소스에 대한 표준 행위를 표현한다. 여기에 JSON이라는 "사람이 읽고 어디서나 파싱되는" 텍스트 포맷이 결합하면서, REST는 **낮은 진입 장벽**과 **브라우저·캐시·프록시 등 기존 웹 인프라와의 완벽한 친화성**을 얻었다.

그러나 REST에도 빈틈이 있었고, 그 빈틈이 정확히 gRPC가 채우는 자리다.

- **계약의 느슨함**: REST는 본질적으로 규약(convention)이지 강제되는 계약이 아니다. "이 엔드포인트가 어떤 필드를 어떤 타입으로 받는가"는 문서·OpenAPI에 의존하며, 컴파일 타임 보장이 없다.
- **텍스트 직렬화 비용**: JSON은 파싱이 느리고 크기가 크다. 서비스 간 초당 수십만 호출이 오가는 내부망에서는 이 오버헤드가 누적된다(→ [[03 - Protocol Buffers 2 - 인코딩과 와이어 포맷]]).
- **스트리밍의 어색함**: REST/HTTP1.1 위에서 양방향 실시간 스트리밍은 부자연스럽다(SSE, 롱폴링, WebSocket 등 별도 기법 필요).
- **N+1·과다/과소 페칭**: 자원 중심이라 한 화면을 그리는 데 여러 번 왕복하기 쉽다.

요약하면, 2010년대 초 업계의 풍경은 이러했다: **외부 공개 API와 브라우저 대상으로는 REST가 잘 맞았지만, 폴리글랏 마이크로서비스가 내부망에서 저지연·고처리량으로 수다 떠는 영역에는 REST가 최적이 아니었다.** 바로 이 내부망의 빈틈을, 구글은 이미 사내에서 **Stubby**라는 자신들만의 무기로 메우고 있었다.

## 3. Stubby에서 gRPC로: 구글 내부의 이야기

구글은 2000년대 초부터 사내 모든 서비스를 잇는 **Stubby**라는 RPC 프레임워크를 운영해 왔다. Stubby는 구글의 데이터센터에서 **초당 약 100억 건(10^10) 규모**(gRPC 공개 당시 구글이 밝힌 수치)의 호출을 처리하는, 사실상 구글 인프라의 혈관이었다. Stubby는 직렬화로 **Protocol Buffers**를 썼고(프로토버프 역시 구글이 2001년경부터 내부적으로 만든 것이고 2008년 오픈소스화됐다), 로드밸런싱·인증·데드라인·추적 같은 운영 요구를 깊이 반영하고 있었다.

그렇다면 질문. **이미 세계 최대 규모로 검증된 Stubby가 있는데, 왜 굳이 2015년에 gRPC를 새로 썼나?** 이유는 기능 부족이 아니라 **개방성과 표준의 부재**였다.

1. **사내 인프라에 깊게 결합**: Stubby는 구글 내부의 네이밍, 인증, 부하 분산 인프라에 강하게 묶여 있어, 그대로는 바깥 세상에 내놓을 수 없었다. 표준 위에 서 있지 않았다.
2. **표준 전송 부재의 한계**: Stubby는 자체 프로토콜 위에 돌았다. 그사이 업계 표준으로 **HTTP/2**(SPDY의 후예, RFC 7540, 2015년 5월)가 성숙했다. HTTP/2는 단일 연결 멀티플렉싱·헤더 압축·스트리밍을 표준으로 제공한다 — Stubby가 사내에서 자체 구현하던 것을, 이제 *표준*으로 누구나 쓸 수 있게 됐다.
3. **오픈소스 생태계의 시대정신**: 마이크로서비스·클라우드 네이티브 흐름 속에서, "구글이 내부에서 쓰는 그 RPC"를 외부도 쓰고 싶어 했다.

그래서 구글은 **"Stubby의 개념과 운영 지혜를, HTTP/2라는 개방 표준 위에 다시 구현"**하기로 한다. 그것이 **gRPC**다. 2015년 2월 공개됐고, 2017년 3월 **CNCF(Cloud Native Computing Foundation)**에 기증되어(인큐베이팅을 거쳐 클라우드 네이티브 스택의 일원으로 성숙) Kubernetes·Envoy 등과 함께 표준 통신 계층으로 올라섰다.

핵심을 한 줄로: **gRPC = "Stubby의 검증된 RPC 모델" + "HTTP/2라는 개방 표준 전송" + "오픈소스·다언어 생태계".** Stubby가 닫힌 정원의 비밀 무기였다면, gRPC는 그 무기를 누구나 쓸 수 있도록 표준 부품으로 재조립했다.

```text
   2001~      구글 내부
   ┌──────────────────────────────┐
   │ Protocol Buffers (직렬화)     │  ← 2008년 오픈소스
   │ Stubby (RPC, 사내 전송)       │  ← 비공개, 인프라 결합
   └──────────────┬───────────────┘
                  │  "표준 위에 다시 짓자"
                  ▼
   2015  gRPC = Stubby의 개념 + HTTP/2 + 오픈소스
                  │
   2017  CNCF 기증 → 클라우드 네이티브 표준 통신 계층
```

## 4. 이름에 담긴 농담과 의미

**gRPC**라는 이름은 공식적으로 **재귀 약어(recursive acronym)**다. 풀면 **"gRPC Remote Procedure Calls"** — 약어 안에 자기 자신(g**RPC**)이 들어가 있다. GNU("GNU's Not Unix")와 같은 해커 전통의 농담이다.

그런데 더 재미있는 부분은 맨 앞 소문자 **g**다. gRPC 팀은 **마이너 릴리스마다 "g가 무엇을 뜻하는가"를 매번 다르게 정하는** 장난을 이어왔다. 저장소에는 버전별 "g의 뜻"을 모아둔 문서(`g_stands_for`)가 있다. 예를 들면:

| 버전 | g가 뜻하는 것 | 비고 |
|---|---|---|
| 1.0 | **g**RPC | 그냥 자기 자신 |
| 1.1 | **g**ood | |
| 1.2 | **g**reen | |
| ... | ... | 매 릴리스마다 새 단어 |

(정확한 전체 목록은 버전마다 바뀌며 공식 저장소가 정본이다. 여기서 중요한 건 정확한 단어가 아니라 **이 장난이 보여주는 문화** — gRPC가 거대 위원회 표준이 아니라, 빠르게 릴리스하며 유머를 잃지 않는 실용적 오픈소스 프로젝트라는 점이다.)

이름이 알려주는 두 가지 진실:

- **"Remote Procedure Calls"** — gRPC는 자신을 *프로시저(절차/함수) 호출*로 정의한다. *리소스*(REST)도 *객체*(CORBA)도 아니다. 이 단어 선택은 6장의 RPC vs REST 철학 대비로 직결된다.
- **재귀 약어와 g 장난** — 진지한 인프라이되, 권위적이지 않고 실용·유머 지향이라는 정체성.

## 5. gRPC의 네 가지 기둥

이제 gRPC를 떠받치는 **네 개의 기둥**을 명확히 세우자. 이 책의 나머지 16개 챕터는 사실상 이 네 기둥을 하나씩 확대해 들여다보는 작업이다. 먼저 그림으로 전체를 잡는다.

```text
        ┌───────────────────────────────────────────────────────┐
        │                      gRPC                              │
        ├───────────────┬───────────────┬───────────────────────┤
        │  기둥 1        │  기둥 2        │  기둥 3                │
        │  IDL+직렬화    │  전송          │  다언어 코드 생성       │
        │  Protocol      │  HTTP/2        │  protoc 툴체인          │
        │  Buffers       │  (RFC 7540/    │  (Go/Java/Py/C++/...)  │
        │                │   9113)        │                        │
        ├───────────────┴───────────────┴───────────────────────┤
        │  기둥 4: 교체 가능한(pluggable) 런타임                  │
        │  인증 · 이름해석 · 로드밸런싱 · 재시도 · 인터셉터 ...    │
        └───────────────────────────────────────────────────────┘
```

### 5.1 기둥 1 — IDL과 직렬화: Protocol Buffers

gRPC의 **계약(contract)**은 `.proto` 파일에 적힌다. 여기서 `.proto`는 두 가지 역할을 동시에 한다.

1. **IDL(인터페이스 정의 언어)**: 어떤 *서비스*가 어떤 *메서드*를 갖고, 각 메서드가 어떤 *요청/응답 메시지*를 주고받는지 — 언어 중립적으로 — 선언한다.
2. **직렬화 스키마**: 그 메시지가 와이어 위에서 어떤 이진 바이트로 인코딩되는지를 규정한다.

가장 작은 예:

```proto
syntax = "proto3";

package order.v1;

// 주문 서비스: 무엇을 할 수 있는지(동사)를 선언한다
service OrderService {
  // 주문 하나를 조회하는 단순(unary) 호출
  rpc GetOrder(GetOrderRequest) returns (Order);
}

// 메시지: 무엇을 주고받는지(데이터 모양)를 선언한다
message GetOrderRequest {
  string order_id = 1;   // 필드 번호 1
}

message Order {
  string order_id = 1;
  string item      = 2;
  int32  quantity  = 3;
  int64  total     = 4;  // 통화 최소단위(예: 원). 부동소수 금지!
}
```

여기서 두 가지가 결정적이다.

- **강한 타입 계약**: `GetOrder`는 *반드시* `GetOrderRequest`를 받아 `Order`를 돌려준다. 클라이언트와 서버는 이 계약을 *컴파일 타임에* 공유한다. REST에서 "그 엔드포인트가 무슨 필드를 받지?"를 문서로 확인하던 불확실성이 사라진다.
- **필드 번호(tag)**: `order_id = 1`의 `1`은 변수 이름이 아니라 **와이어 위에서 쓰이는 식별자**다. 이 번호가 직렬화의 핵심이며, *이름은 바뀌어도 번호만 유지하면 하위호환이 유지*되는 마법의 원천이다.

문법(메시지·열거형·oneof·map·중첩 등)은 [[02 - Protocol Buffers 1 - 문법과 타입 시스템]]에서, 그 바이트가 어떻게 인코딩되는지(varint, zigzag, 와이어 타입)는 [[03 - Protocol Buffers 2 - 인코딩과 와이어 포맷]]에서 깊이 판다. 계약을 바꿀 때 무엇이 안전하고 무엇이 재앙인지는 [[17 - 실전 - proto 관리와 하위호환성]]에서 다룬다.

**왜 텍스트(JSON)가 아니라 이진(Protocol Buffers)인가?** 세 가지 이유다. (1) **크기** — 필드 이름을 매번 보내지 않고 작은 정수 태그만 보내므로 페이로드가 훨씬 작다. (2) **속도** — 파싱이 문자열 스캔이 아니라 바이트 시프트라 빠르다. (3) **스키마 진화** — 필드 번호 기반이라 안전한 진화 규칙이 명확하다. 대가는 **사람이 눈으로 못 읽는다**는 것인데, 이는 [[14 - 관찰성과 디버깅 - Reflection grpcurl]]의 리플렉션/grpcurl로 보완한다.

### 5.2 기둥 2 — 전송: HTTP/2

gRPC는 **HTTP/2**(RFC 7540, 개정판 RFC 9113) 위에서 달린다. 이것은 단순한 구현 선택이 아니라 gRPC를 과거 RPC들과 가르는 핵심이다. HTTP/2가 gRPC에 주는 선물은 다음과 같다.

| HTTP/2 기능 | gRPC에 주는 의미 |
|---|---|
| **스트림 멀티플렉싱** | 하나의 TCP 연결 위에서 수많은 RPC가 동시에 흐른다. 호출마다 새 소켓을 열 필요가 없다(HTTP/1.1의 head-of-line blocking 회피). |
| **양방향 스트리밍** | 클라이언트·서버가 같은 스트림에서 메시지를 *계속* 주고받을 수 있다 → 4가지 통신 방식의 토대(→ [[05 - 통신의 4가지 방식 - Unary와 Streaming]]). |
| **헤더 압축(HPACK)** | 메타데이터/헤더를 압축해 오버헤드를 줄인다(→ [[08 - 메타데이터와 인터셉터]]). |
| **흐름 제어(flow control)** | 스트림·연결 단위 윈도로 백프레셔를 표준 제공(→ [[15 - 성능과 Flow Control 백프레셔]]). |
| **바이너리 프레이밍** | 모든 통신이 정형화된 *프레임* 단위라 파싱이 명확하고 효율적이다. |

> **정밀한 단서**: HTTP/2의 멀티플렉싱은 HTTP/1.1 *애플리케이션 계층*의 head-of-line blocking을 없앤다. 그러나 모든 스트림이 단 하나의 TCP 연결을 공유하므로, TCP 세그먼트 한 개가 유실되면 그 연결의 *모든* 스트림이 재전송을 기다리며 함께 멈추는 **TCP 계층의 head-of-line blocking**은 여전히 남는다. 이 마지막 한 겹까지 걷어내려면 전송을 UDP 기반 QUIC으로 바꾼 HTTP/3가 필요하다. gRPC는 HTTP/2를 1차 전송으로 삼되 이 한계와 HTTP/3 대안은 [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 다룬다.

gRPC 호출 하나는 HTTP/2의 **하나의 스트림(stream)**에 대응한다. 요청은 `:method: POST`, `:path: /order.v1.OrderService/GetOrder` 같은 **HEADERS 프레임**으로 시작하고, 직렬화된 메시지는 **DATA 프레임**으로 흐르며, 호출 결과 상태(`grpc-status`)는 마지막 **트레일러(trailing HEADERS) 프레임**으로 전달된다. 이 구조는 9장에서 바이트 단위로 다시 본다. HTTP/2 프레임 타입·우선순위·SETTINGS·윈도우 업데이트 등 전송 계층의 속살은 [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 전적으로 다룬다.

> 참고: HTTP/2를 골랐기 때문에 생기는 **대가**도 있다. 브라우저는 자바스크립트에서 HTTP/2 프레임을 raw로 제어할 수 없어, *브라우저가 gRPC 서버를 직접 호출하지 못한다*. 이 제약과 우회책(gRPC-Web, 프록시)은 [[16 - gRPC-Web과 REST 게이트웨이]]에서 다룬다.

### 5.3 기둥 3 — 다언어 코드 생성

`.proto` 계약 하나로부터, **`protoc`** 컴파일러와 언어별 플러그인이 각 언어의 **타입 안전한 스텁/메시지 코드**를 자동 생성한다. 같은 `order.proto`에서 Go 서버, Java 클라이언트, Python 분석 스크립트가 *모두 동일한 계약*을 공유한다.

```text
                 order.proto  (단일 진실의 원천)
                      │
            ┌─────────┴──────────┐    protoc + 언어 플러그인
            ▼         ▼          ▼
        Go 코드   Java 코드   Python 코드   ... (C++, C#, Ruby, Node, Kotlin, Dart)
        (서버)    (클라)      (배치잡)
```

이것이 2.4절에서 본 "언어 감옥"을 부수는 열쇠다. RMI는 자바끼리만, DCOM은 윈도우끼리만 가능했지만, gRPC는 한 계약에서 **N개 언어로 코드를 뿜어낸다.** 폴리글랏 마이크로서비스 환경의 결정적 이점이다. `protoc` 툴체인, 플러그인, 빌드 통합, 버전 관리는 [[06 - 코드 생성과 protoc 툴체인]]에서 깊이 다룬다.

### 5.4 기둥 4 — 교체 가능한(pluggable) 런타임

gRPC의 가장 과소평가된 기둥이다. gRPC는 단지 "직렬화 + 전송"이 아니라, **분산 시스템 운영에 필요한 횡단 관심사를 교체 가능한 컴포넌트로 추상화한 런타임**이다. 1세대 RPC가 "함수처럼 편하게"에 베팅하다 8가지 오류에 데였다면, gRPC는 그 오류 각각에 대한 *방어 장치*를 끼울 수 있는 슬롯으로 만들었다.

- **인증·보안(pluggable credentials)**: TLS, mTLS, 토큰 기반 인증을 자격증명(credentials) 추상으로 끼운다(→ [[11 - 보안 - TLS mTLS 인증]]).
- **이름 해석(name resolver)**: `dns:///`, `xds:///` 등 스킴별 resolver로 "이 서비스가 어느 주소에 있나"를 해결한다(→ [[12 - 이름 해석과 로드밸런싱]]).
- **로드밸런싱 정책**: round_robin, pick_first 등 정책을 끼운다(→ [[12 - 이름 해석과 로드밸런싱]]).
- **재시도·헬스체크·keepalive**: 일시적 장애를 흡수하는 안정성 장치(→ [[13 - 안정성 - Retry Health Check Keepalive]]).
- **인터셉터(interceptor)**: 모든 호출을 가로채 로깅·인증·메트릭·트레이싱을 주입하는 미들웨어(→ [[08 - 메타데이터와 인터셉터]]).
- **데드라인·취소**: 모든 호출에 시간 상한을 부여하고 전파한다(→ [[10 - Deadline 취소 타임아웃]]).
- **에러 모델**: 상태 코드와 Rich Error(상세 에러)로 실패를 구조화한다(→ [[09 - 에러 모델 - 상태 코드와 Rich Error]]).

이 네 번째 기둥 덕분에 gRPC는 "RPC 라이브러리"를 넘어 **마이크로서비스 통신을 위한 미들웨어 플랫폼**이 된다. 개발자는 도메인 로직만 짜고, 재시도·인증·관찰성 같은 운영 관심사는 런타임의 슬롯에 끼우면 된다.

## 6. RPC 패러다임 vs REST 패러다임: 철학의 대비

gRPC를 처음 만나는 REST 베테랑이 가장 자주 헷갈리는 지점이 이것이다. 둘은 단지 "프로토콜이 다른" 것이 아니라 **세계를 모델링하는 방식 자체가 다르다.**

- **REST: 명사(리소스) 중심.** 세계를 *리소스*들의 집합으로 본다. `/orders/42`는 "주문 42번"이라는 명사다. 행위는 HTTP 동사라는 *고정된 작은 집합*(GET/POST/PUT/PATCH/DELETE)으로만 표현한다. "주문을 환불 처리한다"를 표현하려면 `POST /orders/42/refunds` 같은 *새 리소스*를 만들어 끼워 맞춘다.
- **RPC(gRPC): 동사(프로시저) 중심.** 세계를 *행위(메서드)*의 집합으로 본다. `OrderService.RefundOrder(...)`처럼 도메인 동사가 그대로 API가 된다. "주문을 환불한다"가 곧 하나의 메서드다. 리소스로 우회할 필요가 없다.

같은 "주문 환불"을 두 패러다임이 어떻게 표현하는지 보자.

```text
REST (리소스/명사 중심)              gRPC (프로시저/동사 중심)
─────────────────────────           ──────────────────────────
POST /orders/42/refunds             rpc RefundOrder(RefundOrderRequest)
Content-Type: application/json          returns (RefundOrderResponse);
{
  "amount": 5000,                   message RefundOrderRequest {
  "reason": "고객 변심"               string order_id = 1;
}                                      int64  amount   = 2;
                                       string reason   = 3;
→ 200 / 201 / 4xx (의미는 약속)      }
→ 본문은 JSON (스키마는 문서 의존)    → grpc-status + RefundOrderResponse
                                        (계약이 컴파일 타임 강제)
```

핵심 트레이드오프를 표로 정리한다.

| 축 | REST | gRPC(RPC) |
|---|---|---|
| 모델 단위 | 리소스(명사) | 프로시저(동사) |
| 계약 강제 | 느슨함(규약/OpenAPI, 런타임 의존) | 강함(`.proto`, 컴파일 타임 강제) |
| 직렬화 | 텍스트(주로 JSON) | 이진(Protocol Buffers) |
| 사람이 읽기 | 쉬움(`curl`로 즉시) | 어려움(도구 필요, → grpcurl) |
| 페이로드 크기 | 큼 | 작음 |
| 파싱 속도 | 느림 | 빠름 |
| 스트리밍 | 어색(SSE/WebSocket 등 별도) | 1급 지원(4가지 방식) |
| 멀티플렉싱 | HTTP/1.1에선 제약 | HTTP/2로 기본 제공 |
| 브라우저 직접 호출 | 자연스러움 | 불가(gRPC-Web 필요) |
| 캐시(HTTP 캐시) | 친화적 | 부적합 |
| 진입 장벽 | 낮음 | 다소 높음(툴체인) |
| 코드 생성 | 선택(OpenAPI codegen) | 핵심(필수에 가까움) |
| 적합 영역 | 공개 API, 브라우저, 단순 CRUD | 내부 마이크로서비스, 저지연·고처리량, 폴리글랏, 스트리밍 |

여기서 흔한 오해 하나를 깨자. **"gRPC가 REST보다 우월하다"는 것은 틀린 프레임이다.** 둘은 *다른 문제*를 푼다. REST는 "전 세계 누구나, 브라우저로, 사람이 읽으며, HTTP 캐시·프록시 인프라를 그대로 활용해" 자원을 다루는 데 최적이다. gRPC는 "신뢰하는 내부망에서, 폴리글랏 서비스들이, 강한 계약으로, 작고 빠른 이진으로, 때로는 스트리밍으로" 수다 떠는 데 최적이다. **계약의 강함과 텍스트 가독성은 교환 관계**이며, 어느 쪽을 택할지는 *누가 이 API를 소비하는가*에 달려 있다. 사람이(브라우저·외부 개발자가) 소비하면 REST 쪽으로, 기계가(내부 서비스가) 소비하면 gRPC 쪽으로 추가 기운다.

> 덧붙임: GraphQL은 또 다른 축(클라이언트 주도 질의)으로, 이 책의 범위를 벗어난다. 다만 "리소스(REST)·동사(RPC)·질의(GraphQL)"가 서로 다른 모델링 철학이라는 큰 그림만 기억해 두자.

## 7. 언제 gRPC를 쓰고, 언제 피하나

도구는 적재적소가 전부다. gRPC의 강점이 빛나는 곳과, 오히려 독이 되는 곳을 분명히 하자.

### 7.1 gRPC가 빛나는 곳

1. **내부 마이크로서비스 간 통신**: 신뢰 경계 안에서 서비스들이 저지연·고처리량으로 호출을 주고받는다. 이진 직렬화의 작은 크기와 HTTP/2 멀티플렉싱이 직접적인 성능 이득이 된다. 강한 계약은 팀 간 통합 버그를 컴파일 타임으로 끌어올린다.
2. **폴리글랏 환경**: Go·Java·Python·Kotlin·C++ 서비스가 섞여 있고, 모두가 *하나의 `.proto` 계약*을 공유해야 할 때. 코드 생성이 언어 경계를 매끄럽게 잇는다.
3. **스트리밍이 본질인 도메인**: 실시간 위치 업데이트, 채팅, 로그/메트릭 스트림, 대용량 파일 청크 전송처럼 *연속적 데이터 흐름*이 필요할 때. 양방향 스트리밍이 1급이다(→ [[05 - 통신의 4가지 방식 - Unary와 Streaming]]).
4. **엄격한 스키마와 진화가 중요한 시스템**: 여러 팀·여러 버전의 클라이언트가 공존하며, 하위호환을 유지하며 진화해야 할 때. 필드 번호 기반 진화 규칙이 강력하다(→ [[17 - 실전 - proto 관리와 하위호환성]]).
5. **모바일 ↔ 백엔드(대역폭·배터리 민감)**: 작은 페이로드와 효율적 파싱이 모바일에서 의미 있는 이득을 준다.

### 7.2 gRPC를 피하거나 신중해야 할 곳

1. **브라우저에서 서버로 직접 호출**: 브라우저 JS는 HTTP/2 프레임을 raw 제어하지 못해 gRPC를 *직접* 호출할 수 없다. **gRPC-Web**과 프록시(Envoy 등)가 있지만 추가 계층이 생기고 양방향 스트리밍 등에 제약이 있다(→ [[16 - gRPC-Web과 REST 게이트웨이]]). 브라우저가 1차 소비자라면 REST/GraphQL이 더 단순한 선택일 때가 많다.
2. **공개 외부 API**: 외부 개발자들에게 공개하는 API라면, 진입 장벽이 낮고 어디서나 `curl`로 두드려볼 수 있는 REST/JSON가 채택률이 훨씬 높다. gRPC는 클라이언트에 툴체인·코드 생성을 강요한다.
3. **사람이 손으로 디버깅하는 단순 CRUD**: 이진 페이로드는 사람이 눈으로 못 읽는다. grpcurl/리플렉션으로 보완하지만, 정말 단순한 CRUD에 굳이 그 무게를 짊어질 이유가 없을 수 있다.
4. **HTTP 캐시·CDN을 적극 활용하는 읽기 트래픽**: GET 캐싱 등 HTTP 캐시 의미론에 기대는 워크로드는 REST가 자연스럽다.
5. **인프라가 HTTP/2를 온전히 지원하지 않는 경우**: 일부 구형 프록시·로드밸런서·게이트웨이가 HTTP/2 트레일러(gRPC가 상태 코드를 싣는 곳)나 장수명 스트림을 제대로 다루지 못하면 운영 함정이 된다.

**실전 패턴**은 흔히 *둘 다 쓰는 것*이다. 외부 경계(브라우저·외부 개발자)에는 REST/GraphQL을 노출하고, 그 게이트웨이 뒤의 **내부 서비스 메시는 gRPC**로 잇는다. gRPC ↔ REST 변환은 게이트웨이로 자동화할 수 있다(→ [[16 - gRPC-Web과 REST 게이트웨이]]).

## 8. 생태계 지형도: 구현체가 둘로 갈리는 이유

gRPC를 실무에서 다루다 보면 곧 마주치는 사실이 있다. **gRPC 구현체는 크게 두 갈래**다. 이 분기를 모르면 "왜 Go의 동작과 Python의 동작이 미묘하게 다른가" 같은 미스터리에 빠진다.

```text
                 gRPC 구현체의 두 갈래
   ┌───────────────────────────────┬───────────────────────────────┐
   │ (A) 순수 네이티브 구현          │ (B) C-core 위의 래퍼            │
   ├───────────────────────────────┼───────────────────────────────┤
   │ gRPC-Go  : 순수 Go             │ C++  : C-core 직접/근접         │
   │ gRPC-Java: 순수 Java(Netty)    │ Python, Ruby, C#, PHP,          │
   │ gRPC-Dart: 순수 Dart           │ Node.js(과거), Objective-C ...  │
   │                                │  → 모두 동일한 C 코어를 공유     │
   │                                │    하고 언어별 얇은 바인딩       │
   └───────────────────────────────┴───────────────────────────────┘
```

- **(A) 순수 네이티브 구현**: **gRPC-Go**(순수 Go), **gRPC-Java**(JVM, 내부적으로 Netty 등 사용), **gRPC-Dart** 등은 해당 언어로 *처음부터 끝까지* 구현됐다. C 라이브러리에 의존하지 않는다.
- **(B) C-core 위의 래퍼**: **C++, Python, Ruby, C#, PHP, Objective-C** 등 상당수는 **공통 C 코어(gRPC C-core)**를 공유하고, 그 위에 언어별 *얇은 바인딩*을 올린 형태다. 핵심 전송·HTTP/2·흐름 제어 로직은 C로 한 번 구현하고, 각 언어는 그것을 호출한다.

**왜 둘로 갈렸나?** 이것은 엔지니어링 트레이드오프의 결과다.

- *공통 C 코어 전략*의 장점은 **일관성과 유지보수 효율**이다. HTTP/2·흐름 제어처럼 복잡하고 까다로운 핵심 로직을 한 번만 정확히 구현하면, 수많은 언어가 그 검증된 코어를 공유한다. 버그 수정·성능 개선이 한곳에서 전 언어로 퍼진다.
- 그러나 *Go와 Java는 다른 길*을 갔다. 이유는 **런타임 모델의 충돌**이다. Go의 고루틴·스케줄러·GC, JVM의 스레드·GC는 C 라이브러리(자체 스레드 풀·메모리 관리·이벤트 루프)와 매끄럽게 섞이기 어렵다. C로 짠 이벤트 루프를 Go 런타임에 끼우면 고루틴 스케줄링과 충돌하고, FFI(cgo/JNI) 경계 비용도 크다. 그래서 이 두 거대 생태계는 **각자 언어의 동시성 모델에 자연스럽게 녹아드는 순수 구현**을 택했다.

**실무적 함의**는 분명하다.

- **언어마다 기능 성숙도·세부 동작이 미묘하게 다를 수 있다.** 재시도, keepalive, 특정 로드밸런싱 정책 같은 고급 기능의 지원 시점·정확한 의미가 구현체마다 다를 수 있으니, 다언어 시스템에서는 *각 언어 구현의 문서와 버전*을 확인해야 한다.
- **성능 특성도 다르다.** 순수 Go/Java 구현은 해당 런타임에 최적화돼 있고, C-core 래퍼 언어는 FFI 경계 비용을 진다.
- **디버깅 시 "어느 계층의 문제인가"를 의식하라.** C-core 래퍼 언어에서 이상 동작이 보이면, 그것이 언어 바인딩의 문제인지 C 코어의 문제인지 구분해야 할 수 있다.

이 분기점은 채널·연결 생명주기([[07 - 채널 스텁 커넥션 생명주기]])나 안정성 기능([[13 - 안정성 - Retry Health Check Keepalive]])을 다룰 때 다시 등장한다. "gRPC의 동작"이라고 뭉뚱그리지 말고, "이 *구현체*에서 일어나는 동작"으로 사고하는 습관을 들이자.

## 9. "Hello World" 한 사이클을 끝까지 따라가기

이제 모든 추상을 한 줄의 호출로 응축해, **클라이언트가 메서드 하나를 부르는 순간부터 응답이 돌아오기까지 와이어 위에서 실제로 무슨 일이 벌어지는지**를 끝까지 따라가 보자. 이 절의 그림이 이 책 전체의 **로드맵**이다 — 이후 챕터들은 이 그림의 각 부분을 확대경으로 들여다본다.

### 9.1 계약: `.proto`

```proto
syntax = "proto3";

package greet.v1;

option go_package = "example.com/greet/v1;greetv1";

service Greeter {
  // 단순(unary) RPC: 요청 1개 → 응답 1개
  rpc SayHello(HelloRequest) returns (HelloReply);
}

message HelloRequest {
  string name = 1;
}

message HelloReply {
  string message = 1;
}
```

### 9.2 코드 생성

```bash
# protoc로 Go용 메시지 + gRPC 스텁 생성
protoc \
  --go_out=. --go_opt=paths=source_relative \
  --go-grpc_out=. --go-grpc_opt=paths=source_relative \
  greet/v1/greet.proto
# → greet/v1/greet.pb.go      (메시지 타입)
# → greet/v1/greet_grpc.pb.go (Greeter 클라이언트/서버 스텁)
```

(`protoc`와 플러그인의 작동 원리는 [[06 - 코드 생성과 protoc 툴체인]].)

### 9.3 서버 (Go)

```go
package main

import (
	"context"
	"log"
	"net"

	"google.golang.org/grpc"
	greetv1 "example.com/greet/v1"
)

// 생성된 인터페이스(greetv1.GreeterServer)를 구현한다
type server struct {
	greetv1.UnimplementedGreeterServer
}

// 우리가 짜는 것은 '도메인 로직'뿐. 직렬화/전송은 런타임이 처리한다.
func (s *server) SayHello(ctx context.Context, req *greetv1.HelloRequest) (*greetv1.HelloReply, error) {
	return &greetv1.HelloReply{Message: "안녕하세요, " + req.GetName() + "님"}, nil
}

func main() {
	lis, err := net.Listen("tcp", ":50051")
	if err != nil {
		log.Fatal(err)
	}
	s := grpc.NewServer()
	greetv1.RegisterGreeterServer(s, &server{})
	log.Println("listening on :50051")
	if err := s.Serve(lis); err != nil { // HTTP/2 서버 시작
		log.Fatal(err)
	}
}
```

### 9.4 클라이언트 (Go)

```go
package main

import (
	"context"
	"log"
	"time"

	"google.golang.org/grpc"
	"google.golang.org/grpc/credentials/insecure"
	greetv1 "example.com/greet/v1"
)

func main() {
	// 1) 채널(연결 추상)을 만든다. 아직 도메인 호출은 없다.
	conn, err := grpc.NewClient(
		"localhost:50051",
		grpc.WithTransportCredentials(insecure.NewCredentials()), // 데모용 평문
	)
	if err != nil {
		log.Fatal(err)
	}
	defer conn.Close()

	// 2) 스텁: 마치 로컬 객체처럼 보이는 원격 프록시
	client := greetv1.NewGreeterClient(conn)

	// 3) 데드라인을 '반드시' 건다 — 분산의 현실을 숨기지 않는다(8가지 오류 #2)
	ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
	defer cancel()

	// 4) 호출: 문법은 로컬 함수와 똑같다
	reply, err := client.SayHello(ctx, &greetv1.HelloRequest{Name: "Shawn"})
	if err != nil {
		log.Fatalf("RPC 실패: %v", err) // 실패는 error로 정직하게 드러난다
	}
	log.Println(reply.GetMessage()) // → 안녕하세요, Shawn님
}
```

다른 언어 클라이언트도 *같은 계약*에서 생성된다. 예를 들어 Python:

```python
import grpc
import greet.v1.greet_pb2 as pb
import greet.v1.greet_pb2_grpc as rpc

with grpc.insecure_channel("localhost:50051") as channel:
    stub = rpc.GreeterStub(channel)
    reply = stub.SayHello(pb.HelloRequest(name="Shawn"), timeout=2.0)
    print(reply.message)  # → 안녕하세요, Shawn님
```

Go 서버 ↔ Python 클라이언트가 한 줄 수정 없이 맞물린다. 이것이 기둥 3(다언어 코드 생성)의 위력이다.

### 9.5 그 한 줄 호출의 내부에서 벌어지는 일 (전체 그림)

`client.SayHello(ctx, &HelloRequest{Name:"Shawn"})` 한 줄 뒤에서 실제로 일어나는 일을, 단계별로 펼친다.

```text
[클라이언트]                                            [서버]
   │
   │ ① 스텁이 호출을 가로챈다
   │    (스텁 = 원격 메서드의 로컬 프록시)
   │
   │ ② Protocol Buffers 직렬화
   │    HelloRequest{name:"Shawn"}  →  바이트 0a 05 53 68 61 77 6e
   │    (필드1 LEN, 길이5, "Shawn")     [[03 - Protocol Buffers 2 - 인코딩과 와이어 포맷]]
   │
   │ ③ gRPC 길이-접두 프레이밍
   │    [압축플래그 00][길이 00 00 00 07][payload 0a 05 53 68 61 77 6e]
   │
   │ ④ 채널/연결에서 HTTP/2 스트림 1개 확보
   │    [[07 - 채널 스텁 커넥션 생명주기]]
   │
   │ ⑤ HEADERS 프레임 전송 ───────────────────────────►
   │    :method: POST
   │    :scheme: http
   │    :path: /greet.v1.Greeter/SayHello
   │    :authority: localhost:50051
   │    content-type: application/grpc
   │    grpc-timeout: 2S         ← ctx의 데드라인 [[10 - Deadline 취소 타임아웃]]
   │    (+ 사용자 메타데이터)      [[08 - 메타데이터와 인터셉터]]
   │
   │ ⑥ DATA 프레임 전송(③의 바이트) ─────────────────►
   │    (END_STREAM 플래그로 "요청 끝" 표시)
   │                                                  ⑦ 서버가 프레임 수신
   │                                                     디프레이밍 → 역직렬화
   │                                                     → HelloRequest{name:"Shawn"}
   │                                                  ⑧ 핸들러(SayHello) 실행
   │                                                     도메인 로직 → HelloReply
   │                                                  ⑨ 응답 직렬화 + 프레이밍
   │ ◄──────────── HEADERS(응답 헤더) ─────────────────
   │ ◄──────────── DATA(응답 메시지) ──────────────────
   │ ◄──────────── HEADERS(트레일러) ──────────────────
   │                grpc-status: 0 (OK)  ← 결과 상태!
   │                grpc-message: ""      [[09 - 에러 모델 - 상태 코드와 Rich Error]]
   │
   │ ⑩ 클라이언트: 디프레이밍 → 역직렬화 → HelloReply
   │    grpc-status==0 이므로 정상 반환
   ▼
 reply.GetMessage() == "안녕하세요, Shawn님"
```

이 그림에서 주목할 설계 포인트 세 가지.

1. **요청과 결과 상태가 분리돼 있다.** HTTP 상태코드(200 등)는 *전송* 성공만 말한다. *도메인* 성공/실패는 **트레일러의 `grpc-status`**로 별도 전달된다. 그래서 "HTTP 200인데 grpc-status는 에러" 같은 일이 자연스럽다. 이 분리가 [[09 - 에러 모델 - 상태 코드와 Rich Error]]의 출발점이다. (응답 본문 없이 곧장 실패·종료하는 경우엔 초기 헤더와 트레일러를 단 한 번의 HEADERS 프레임으로 합쳐 보내는 *Trailers-Only* 형태가 쓰이기도 한다.)
2. **데드라인이 헤더(`grpc-timeout`)로 와이어를 타고 전파된다.** 클라이언트가 "2초"라고 정한 시간 상한이 서버까지 전달되고, 더 깊은 호출로 이어 전파될 수 있다. 8가지 오류 #2(지연은 0이 아니다)에 대한 구조적 응답이다(→ [[10 - Deadline 취소 타임아웃]]). (위 그림에서는 사람이 읽기 좋게 `2S`로 그렸지만, 실제 구현은 보통 더 작은 단위와 8자리 이내 정수로 — 게다가 *남은 시간*을 카운트다운하며 — 인코딩한다. `grpc-timeout`의 값은 절대 시각이 아니라 상대 잔여 시간이다.)
3. **인터셉터가 ①과 ⑧을 감쌀 수 있다.** 모든 호출의 진입/이탈을 가로채 인증·로깅·메트릭·트레이싱을 한곳에서 주입한다(→ [[08 - 메타데이터와 인터셉터]]).

### 9.6 ②~③의 바이트를 손으로 디코딩하기

추상이 아니라 진짜 바이트를 보자. `HelloRequest{ name: "Shawn" }`이 Protocol Buffers로 어떻게 인코딩되는가?

- 필드 `name`은 **필드 번호 1**, 타입 `string` → 와이어 타입 **2(LEN, 길이 지정)**.
- **태그 바이트** = `(field_number << 3) | wire_type` = `(1 << 3) | 2` = `0x0A`.
- 그다음 **길이** = `"Shawn"`은 5바이트 → `0x05`.
- 그다음 **UTF-8 바이트**: `S=0x53, h=0x68, a=0x61, w=0x77, n=0x6E`.

```text
프로토버프 페이로드 (7바이트):
  0A        05        53 68 61 77 6E
  └tag┘     └len┘     └─── "Shawn" ───┘
  field1    길이5      UTF-8 bytes
  LEN(=2)
```

이제 gRPC가 이 7바이트를 **길이-접두 메시지(Length-Prefixed-Message)**로 감싼다. 형식은 `[1바이트 압축 플래그][4바이트 빅엔디안 길이][payload]`.

```text
gRPC DATA 프레임 본문 (5 + 7 = 12바이트):
  00            00 00 00 07          0A 05 53 68 61 77 6E
  └압축플래그┘   └─ 길이=7(빅엔디안)┘  └──── 프로토버프 payload ────┘
  0=비압축
```

이 12바이트가 HTTP/2 **DATA 프레임**의 페이로드로 실려 나간다. (HTTP/2 프레임은 다시 `[길이3 | 타입1 | 플래그1 | 스트림ID4]`의 9바이트 헤더로 감싸지는데, 그 디테일은 [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 본다.) varint·zigzag·packed 같은 더 깊은 인코딩 규칙은 [[03 - Protocol Buffers 2 - 인코딩과 와이어 포맷]]에서 바이트 단위로 파고든다.

요점: **"로컬 함수처럼 부른다"는 한 줄의 편안함 뒤에는, 직렬화 → 길이 프레이밍 → HTTP/2 프레임 → 멀티플렉싱 → 전송 → 역과정이 정확히 짜여 돌아간다.** gRPC의 가치는 이 복잡함을 *숨기되*, 분산의 현실(데드라인·상태코드·취소)은 *숨기지 않는* 균형에 있다.

## 핵심 요약

- **RPC의 본질적 꿈**은 "원격 함수를 로컬 함수처럼 부른다"는 것이고, 분산 시스템의 역사는 이 꿈과 물리 법칙 사이의 줄다리기다. 과거 RPC들은 네트워크를 *숨기려다* "분산 컴퓨팅의 8가지 오류"에 데였다.
- **gRPC의 설계는 선배들의 실패에 대한 의도적 응답**이다. CORBA의 "분산 객체 환상"을 버리고 *무상태 메서드 호출*로 후퇴했고, RMI/DCOM의 "언어·플랫폼 감옥"을 다언어 코드 생성으로 부쉈으며, SOAP의 "장황한 텍스트·거대 표준"을 작은 이진 포맷(Protocol Buffers)과 실용적 오픈소스로 대체했다. Thrift가 옳게 한 IDL·코드생성·다언어는 계승하되, 마침 성숙한 **HTTP/2**를 전송으로 채택해 멀티플렉싱·스트리밍을 1급으로 끌어올렸다.
- **gRPC = Stubby의 검증된 RPC 모델 + HTTP/2 개방 표준 + 오픈소스·다언어 생태계.** 구글은 닫힌 정원의 무기(Stubby)를 표준 부품으로 재조립해 2015년 공개, 2017년 CNCF에 기증했다.
- **네 가지 기둥**: (1) IDL+직렬화 = Protocol Buffers, (2) 전송 = HTTP/2, (3) 다언어 코드 생성, (4) 교체 가능한 런타임(인증·이름해석·로드밸런싱·재시도·인터셉터). gRPC의 기능 목록은 사실상 "8가지 오류에 대한 체계적 방어"로 읽힌다.
- **RPC vs REST는 우열이 아니라 모델링 철학의 차이**다. 동사(프로시저) 중심 vs 명사(리소스) 중심, 강한 컴파일 타임 계약 vs 느슨한 규약, 이진 vs 텍스트. *기계가 소비하면 gRPC, 사람이 소비하면 REST* 쪽으로 기운다. 실전에선 외부 REST + 내부 gRPC의 조합이 흔하다.
- **gRPC가 빛나는 곳**: 내부 마이크로서비스, 폴리글랏, 저지연·고처리량, 스트리밍, 엄격한 스키마 진화. **피할 곳**: 브라우저 직접 호출, 공개 외부 API, 사람이 손으로 디버깅하는 단순 CRUD, HTTP 캐시 의존 트래픽.
- **구현체는 두 갈래** — 순수 네이티브(gRPC-Go, gRPC-Java)와 공통 C-core 위 래퍼(C++/Python/Ruby/C#/PHP 등). Go·Java가 갈라진 이유는 각자의 동시성·런타임 모델이 C 코어와 충돌하기 때문이며, 그래서 *언어별 기능 성숙도·세부 동작이 다를 수 있다*.
- **"로컬 함수처럼 부른다"는 한 줄의 편안함 뒤에는** 직렬화 → 길이 프레이밍 → HTTP/2 프레임 → 멀티플렉싱 → 전송 → 역과정이 정확히 돌아간다. gRPC의 핵심 미덕은 이 복잡함은 *숨기되*, 데드라인·상태코드·취소 같은 분산의 현실은 *숨기지 않는* 균형이다.
