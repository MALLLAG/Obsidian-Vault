이 장이 답하는 질문:

- 채널(channel)은 소켓(socket)과 무엇이 다른가? "가상 연결"이라는 말은 내부에서 정확히 무엇을 의미하는가?
- 채널은 어떤 상태(IDLE/CONNECTING/READY/TRANSIENT_FAILURE/SHUTDOWN)를 거치며, 각 전이는 무엇이 트리거하는가? `WaitForReady`는 그 상태 기계와 어떻게 맞물리는가?
- 연결이 끊겼을 때 gRPC는 어떻게 스스로 복구하는가? 지수 백오프와 지터(jitter)의 기본값과 의미는?
- 스텁은 왜 가볍고 채널은 왜 무거운가? "요청마다 채널을 새로 만든다"가 왜 재앙인가?
- 하나의 HTTP/2 연결에는 동시 스트림 한계(`MAX_CONCURRENT_STREAMS`)가 있는데, 그 한계가 고부하에서 어떻게 병목이 되고 채널 풀링은 왜 필요한가?

## 1. 들어가며: "연결을 연다"는 말의 함정

REST/HTTP1.1 시절의 직관에서 출발해 보자. 보통 우리는 "서버에 연결한다"고 하면 머릿속에 이런 그림을 그린다.

```text
클라이언트  ──── TCP 소켓 1개 ────▶  서버
              (요청 보내고 응답 받고 닫음)
```

소켓 하나가 곧 연결이고, 연결 하나가 곧 요청 하나(혹은 keep-alive로 몇 개)다. 이 직관은 단순해서 좋지만 gRPC를 이해하는 데는 오히려 방해가 된다. gRPC에서 우리가 코드로 만드는 `Channel`은 소켓이 **아니다**. 그것은 "어떤 목적지(target)로 RPC를 보내고 싶다"는 의도를 담은 **논리적 추상화**이며, 실제 TCP 소켓은 그 추상화 아래에서 gRPC 라이브러리가 알아서 만들고, 끊고, 다시 잇는다.

이 차이를 한 문장으로 요약하면 이렇다.

> **채널은 "정책"이고, 소켓은 "메커니즘"이다.** 당신은 "이 서비스에 말을 걸겠다"는 정책을 선언하고, gRPC 런타임은 그 정책을 만족시키기 위해 필요한 만큼의 소켓을 알아서 운영한다.

이 장은 그 "알아서 운영한다"의 내부를 한 꺼풀씩 벗긴다. 채널이라는 추상화의 층위, 그 아래의 서브채널, 서브채널이 들고 있는 실제 HTTP/2 연결, 그 연결이 거치는 상태 기계, 끊겼을 때의 자동 복구, 그리고 그 위에 얹히는 스텁까지. gRPC가 처음인 백엔드/네트워크 엔지니어라면 이 장을 읽고 나서 "아, 채널을 요청마다 만들면 안 되는 거였구나"가 뼛속까지 박히는 것이 목표다.

전송 계층의 메커니즘(프레임, 스트림, 멀티플렉싱) 자체는 [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 이미 다뤘다고 가정한다. 이 장은 그 위에 앉은 **클라이언트 측 연결 관리**에 집중한다.

## 2. 채널이라는 추상화: 가상 연결의 해부

### 2.1 정의부터 정확히

gRPC 공식 용어로 **채널(channel)** 은 다음과 같이 정의된다.

> A gRPC channel provides a connection to a gRPC server on a specified host and port. (gRPC 채널은 지정된 호스트와 포트의 gRPC 서버로 가는 연결을 제공한다.)

여기서 "a connection"이라는 단수 표현이 오해의 씨앗이다. 더 정확하게 말하면 채널은 **하나의 타겟(target)으로 향하는 가상 연결**이다. 그 타겟이 DNS 이름이면 이름 뒤에 여러 개의 백엔드 IP가 달릴 수 있고, 그러면 채널 하나가 내부적으로 **여러 개의 실제 연결**을 거느린다.

층위를 그림으로 보자.

```text
   당신의 코드가 만지는 것 ──────────────────────────────┐
                                                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  Channel  ("orderservice.internal:50051" 로 가는 가상 연결)    │
   │                                                                │
   │   ┌─ Name Resolver: 타겟 → [10.0.0.11, 10.0.0.12, 10.0.0.13] │
   │   │                                                            │
   │   ┌─ LB Policy: 어느 백엔드로 보낼지 고르는 정책 (round_robin) │
   │   │                                                            │
   │   ▼                                                            │
   │  ┌────────────┐  ┌────────────┐  ┌────────────┐                │
   │  │ Subchannel │  │ Subchannel │  │ Subchannel │   ← 백엔드별   │
   │  │ 10.0.0.11  │  │ 10.0.0.12  │  │ 10.0.0.13  │                │
   │  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘                │
   │        │               │               │                       │
   └────────┼───────────────┼───────────────┼───────────────────────┘
            ▼               ▼               ▼
      ┌──────────┐    ┌──────────┐    ┌──────────┐
      │ TCP +    │    │ TCP +    │    │ TCP +    │   ← 실제 소켓
      │ TLS +    │    │ TLS +    │    │ TLS +    │     (HTTP/2 conn)
      │ HTTP/2   │    │ HTTP/2   │    │ HTTP/2   │
      └──────────┘    └──────────┘    └──────────┘
```

즉, 머릿속 모델은 이렇게 3층으로 정리된다.

| 층 | 이름 | 정체 | 개수 | 당신이 직접 만지나? |
|----|------|------|------|---------------------|
| 1 | **Channel** | 타겟으로 가는 가상 연결(정책) | 보통 1 | O (코드로 생성) |
| 2 | **Subchannel** | 단일 백엔드 주소로 가는 연결 관리 단위 | 백엔드 수만큼 | X (런타임이 관리) |
| 3 | **Transport / Socket** | 실제 TCP+TLS+HTTP/2 연결 | 서브채널당 1개(보통) | X (런타임이 관리) |

채널과 소켓을 헷갈리지 않는 결정적 한마디: **소켓은 끊어질 수 있지만 채널은 끊어지지 않는다.** 백엔드 하나가 죽어서 그쪽 소켓이 깨져도 채널은 살아 있다. 채널은 그저 그 백엔드의 서브채널을 `TRANSIENT_FAILURE` 상태로 표시하고, 다른 살아 있는 서브채널로 트래픽을 보내거나 끊긴 서브채널을 백오프 후 다시 잇는다. 이것이 "가상"의 진짜 의미다. 물리적 연결의 생사와 무관하게 논리적으로 계속 존재하는 통로.

### 2.2 왜 이렇게 설계했는가 (why)

이 3층 구조가 단순한 과잉 추상화가 아니라는 점을 짚자. 만약 채널이 곧 소켓이었다면:

1. **로드밸런싱이 불가능하다.** [[12 - 이름 해석과 로드밸런싱]]에서 다루듯, gRPC는 L4 로드밸런서 없이 **클라이언트 측 로드밸런싱(client-side load balancing)** 을 1급 기능으로 지원한다. 채널이 여러 서브채널을 거느려야 round_robin 같은 정책으로 요청을 분산할 수 있다.
2. **장애 격리가 불가능하다.** 백엔드 3대 중 1대가 죽었을 때, 채널=소켓 모델이라면 채널 전체가 죽는다. 서브채널 분리 덕분에 죽은 1대만 격리하고 나머지로 서비스를 이어간다.
3. **연결 비용을 숨길 수 없다.** TCP 핸드셰이크 + TLS 핸드셰이크 + HTTP/2 셋업은 비싸다. 이 비용을 매번 노출하면 애플리케이션 코드가 연결 풀을 직접 관리해야 한다. 채널 추상화가 이 복잡성을 통째로 흡수한다.

채널은 **"연결 관리라는 분산 시스템의 골치 아픈 부분을 라이브러리 안으로 밀어 넣기 위한 경계선"** 이다.

## 3. 서브채널과 HTTP/2 멀티플렉싱: 연결 하나에 얼마나 욱여넣을 수 있나

### 3.1 한 연결로 수천 개의 요청 — 멀티플렉싱의 위력

HTTP/1.1에서는 한 TCP 연결로 한 번에 하나의 요청만 처리한다(파이프라이닝은 사실상 죽었다). 그래서 브라우저가 도메인당 6개씩 연결을 여는 식의 꼼수가 필요했다.

HTTP/2는 이 모델을 근본적으로 바꿨다. 하나의 TCP 연결 위에 **스트림(stream)** 이라는 논리적 채널을 여러 개 동시에 흘려보낸다. 이것이 **멀티플렉싱(multiplexing)** 이다. gRPC의 RPC 하나는 HTTP/2 스트림 하나에 대응한다.

```text
            하나의 TCP 연결 (HTTP/2)
   ┌───────────────────────────────────────────────┐
   │  stream 1  ▶ RPC: GetOrder(id=1)               │
   │  stream 3  ▶ RPC: GetOrder(id=2)   ← 동시 진행  │
   │  stream 5  ▶ RPC: ListOrders()                 │
   │  stream 7  ▶ RPC: CreateOrder(...)             │
   │   ...      각 스트림의 DATA/HEADERS 프레임이      │
   │            인터리빙되어 한 소켓으로 흐름           │
   └───────────────────────────────────────────────┘
```

그래서 gRPC는 **연결 하나로도 매우 많은 동시 RPC를 처리**할 수 있다. 이것이 "채널 하나를 만들어 앱 전체가 공유하라"는 권고의 기술적 근거다. 수백 개의 동시 요청이 있어도 연결 하나로 충분할 때가 많다.

참고로 HTTP/2 스트림 ID는 클라이언트가 시작하는 스트림이면 항상 홀수(1, 3, 5, 7, ...)다. 서버가 시작하는 스트림(gRPC에서는 거의 안 쓰임)은 짝수다. 그래서 위 그림에서 RPC들이 홀수 ID를 달고 있다.

### 3.2 그러나 한계가 있다: MAX_CONCURRENT_STREAMS

무한정 욱여넣을 수는 없다. HTTP/2의 SETTINGS 프레임에는 `SETTINGS_MAX_CONCURRENT_STREAMS`라는 파라미터가 있고 서버는 이것으로 "내 연결 하나당 동시에 열 수 있는 스트림은 N개까지"라고 선언한다. gRPC 서버들은 흔히 이 값을 **100** 근처로 잡는다(구현/설정에 따라 다름).

서버가 보내는 SETTINGS 프레임을 실제 바이트로 손 디코딩해 보자. HTTP/2 프레임 헤더는 9바이트 고정이다.

```text
HTTP/2 Frame Header (9 bytes)
+-----------------------------------------------+
|                 Length (24 bit)               |
+---------------+---------------+---------------+
|   Type (8)    |   Flags (8)   |
+-+-------------+---------------+-------------------------------+
|R|                 Stream Identifier (31 bit)                 |
+=+=============================================================+
|                   Frame Payload (Length bytes)              ...
+---------------------------------------------------------------+
```

서버가 "MAX_CONCURRENT_STREAMS = 100"을 알리는 SETTINGS 프레임의 바이트:

```text
바이트열:  00 00 06   04   00   00 00 00 00   00 03   00 00 00 64
           └──┬───┘   │    │    └────┬─────┘   └─┬─┘   └────┬────┘
           Length=6  Type Flags  StreamID=0   ID=0x3   Value=0x64
            (6바이트  =0x04 =0    (SETTINGS는   (=MAX_   (=100)
             payload) SETTINGS     stream 0     CONCURRENT
                                   고정)         _STREAMS)
```

- `00 00 06`: payload 길이 6바이트(설정 1개 = 식별자 2바이트 + 값 4바이트).
- `04`: 프레임 타입 0x4 = SETTINGS.
- `00`: 플래그 0 (ACK 비트가 꺼져 있으므로 "이건 새 설정값 선언"이라는 뜻. ACK이면 0x1).
- `00 00 00 00`: 스트림 ID 0. SETTINGS는 반드시 연결 전체에 적용되므로 스트림 0(연결 자체)에서만 보낸다.
- `00 03`: 설정 식별자 0x3 = `SETTINGS_MAX_CONCURRENT_STREAMS`.
- `00 00 00 64`: 값 0x64 = 100.

이제 핵심 질문. **101번째 동시 RPC를 보내면 어떻게 되는가?** 연결이 거부하는 게 아니라, gRPC 클라이언트가 그 RPC를 **큐에 넣고 기다리게** 한다. 앞선 스트림 중 하나가 끝나서 슬롯이 비면 그때 새 스트림을 연다. 즉 단일 연결의 동시 스트림 한계는 **암묵적인 동시성 상한**으로 작동한다.

### 3.3 그래서 고부하에서 단일 연결이 병목이 된다

이 지점이 실전에서 사람들을 놀라게 하는 부분이다. "채널 하나면 충분하다며? 그런데 왜 처리량이 안 나오지?"

답은 이렇다. **단일 서브채널 = 단일 TCP 연결**이고, 그 연결은 `MAX_CONCURRENT_STREAMS`(예: 100)에 묶여 있다. 만약 한 백엔드로만 트래픽이 몰리고(예: 백엔드가 1대뿐이거나, LB 정책이 `pick_first`라 연결을 하나만 쓰는 경우), 평균 RPC 지연이 10ms인데 초당 50,000건을 보내고 싶다면, 리틀의 법칙(Little's Law)으로 필요한 동시성을 계산할 수 있다.

```text
필요한 동시성 ≈ throughput × latency
            = 50,000 req/s × 0.010 s
            = 500 개의 동시 인-플라이트(in-flight) RPC

그런데 연결 하나의 한계 = 100 스트림
→ 500 / 100 = 최소 5개의 연결이 필요
→ 연결 1개로는 동시성 100에 묶여 throughput ≈ 10,000 req/s 에서 천장
```

이때 해법은 두 가지다.

1. **백엔드를 여러 대로 두고 `round_robin` 같은 LB 정책을 쓴다.** 그러면 채널이 백엔드마다 서브채널을 만들어 여러 연결로 분산한다. (→ [[12 - 이름 해석과 로드밸런싱]])
2. **백엔드가 한 대라면 채널을 여러 개 만들거나(채널 풀링), 라이브러리의 멀티 커넥션 옵션을 쓴다.** 단, 같은 타겟으로 채널을 여러 개 만들 때는 각 채널이 `subchannel`을 공유하지 않도록 일부러 다르게 만들어 줘야 하는 트릭이 필요하다(언어별로 로컬 서브채널 풀을 강제하거나 의미 없는 더미 인자를 붙여 채널을 구분).

이 백프레셔(backpressure)와 처리량 한계, flow control window의 상호작용은 [[15 - 성능과 Flow Control 백프레셔]]에서 깊게 다룬다. 여기서 기억할 한 줄: **멀티플렉싱은 강력하지만 무한하지 않다. `MAX_CONCURRENT_STREAMS`가 단일 연결의 동시성 천장이다.**

## 4. 연결 생명주기의 심장: 연결 상태 기계(Connectivity State Machine)

이제 이 장의 핵심이다. 채널(정확히는 채널 안의 각 서브채널, 그리고 이를 집계한 채널 전체)은 5개의 **연결 상태(connectivity state)** 사이를 오간다. gRPC의 모든 언어 구현이 이 상태 기계를 공유한다.

```text
                         ┌─────────────────────────────────────┐
                         │                                     │
   (생성 직후 / 유휴)      ▼                                     │
        ┌──────────────────────┐                               │
   ┌───▶│        IDLE          │                               │
   │    └──────────┬───────────┘                               │
   │               │  RPC 발생 OR Connect() 호출                 │
   │               │  OR 일정 유휴시간 후 재진입                  │
   │               ▼                                            │
   │    ┌──────────────────────┐                               │
   │    │     CONNECTING       │  TCP+TLS+HTTP/2 핸드셰이크 중   │
   │    └──────┬────────────┬──┘                               │
   │           │ 성공        │ 실패                              │
   │           ▼            ▼                                   │
   │  ┌────────────┐   ┌──────────────────────┐                │
   │  │   READY    │   │  TRANSIENT_FAILURE   │                │
   │  └─────┬──────┘   └──────────┬───────────┘                │
   │        │ 연결 끊김 / GOAWAY    │ 백오프 타이머 만료           │
   │        │ / keepalive 실패     │ → 재시도                    │
   │        ▼                     └────────────────────────────┘
   │  ┌──────────────────────┐         (다시 CONNECTING 으로)
   │  │  TRANSIENT_FAILURE   │
   │  └──────────────────────┘
   │
   │  유휴 타임아웃(idle timeout) 경과 시 READY → IDLE 로 복귀 가능
   └──────────────────────────────────────────────────────────┐
                                                               │
   어느 상태에서든 channel.shutdown() 호출 시 ──────────────────▶│
                                                  ┌────────────▼──────┐
                                                  │     SHUTDOWN      │ (종착)
                                                  └───────────────────┘
```

각 상태를 하나씩, "무엇이 트리거하고 내부에서 무슨 일이 일어나는가"를 중심으로 본다.

### 4.1 IDLE — 게으른 시작

채널을 생성하면 보통 **IDLE** 상태로 시작한다. 중요한 통찰: **채널을 만든다고 해서 즉시 TCP 연결이 열리는 게 아니다.** gRPC는 게으르다(lazy). "아직 보낼 RPC가 없는데 굳이 연결을 열어 둘 이유가 없다"는 철학이다. 연결은 자원(파일 디스크립터, 메모리, 서버 측 슬롯)이므로, 실제로 필요할 때까지 아낀다.

IDLE에서 벗어나는 트리거는 두 가지다.

1. **첫 RPC가 발생할 때.** 누군가 스텁으로 RPC를 호출하면 채널이 "아, 이제 연결이 필요하구나" 하고 CONNECTING으로 전이한다.
2. **명시적으로 `Connect()`(또는 언어별 동등 API)를 호출할 때.** "지금 미리 연결을 데워 놔라(warm up)"고 요청하는 것. 첫 RPC의 지연을 줄이고 싶을 때 쓴다.

또한 READY였던 채널도 **일정 시간 동안 RPC가 한 건도 없으면 다시 IDLE로 돌아온다.** 이를 유휴 타임아웃(idle timeout)이라 하며, 예컨대 grpc-go의 기본값은 30분이다. 이때 연결을 닫아 자원을 회수하고, 다음 RPC가 오면 다시 CONNECTING으로 깨어난다. 서버 입장에서도 놀고 있는 연결의 슬롯을 회수할 수 있어 양쪽에 이득이다.

### 4.2 CONNECTING — 핸드셰이크의 시간

CONNECTING은 "지금 실제 연결을 맺는 중"이다. 이 안에서 순차적으로 일어나는 일:

```text
CONNECTING 내부의 단계들 (TLS 채널 기준)
  1) 이름 해석 결과에서 백엔드 주소 하나를 고른다 (LB가 결정)
  2) TCP 3-way handshake         (SYN → SYN/ACK → ACK)
  3) TLS handshake               (ClientHello → ... → Finished)
        └ ALPN 으로 "h2" 협상      ([[11 - 보안 - TLS mTLS 인증]])
  4) HTTP/2 connection preface 교환
        클라이언트: "PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n" + SETTINGS
        서버:       SETTINGS (MAX_CONCURRENT_STREAMS 등)
  5) 양쪽 SETTINGS ACK 교환
  → 여기까지 성공하면 READY
```

이 과정이 왜 비싼지가 한눈에 보인다. RTT가 여러 번 왕복한다(TCP 1 RTT + TLS 1~2 RTT + HTTP/2 셋업). 거기에 TLS의 비대칭 암호 연산까지. **이 비용을 매 RPC마다 치르면 안 된다**는 것이 8절 안티패턴 논의의 핵심 근거다.

`MIN_CONNECT_TIMEOUT`이라는 상수도 여기서 작동한다. 연결 시도 1회에 최소한 이만큼은 기다려 본다는 하한(기본 20초)이다. 백오프 간격이 이보다 짧아도 연결 시도 자체에는 최소 20초의 기회를 준다.

CONNECTING에서 핸드셰이크가 실패하면(SYN에 응답이 없거나, RST를 받거나, TLS 검증 실패 등) → **TRANSIENT_FAILURE**로 간다. 성공하면 → **READY**.

### 4.3 READY — 일하는 상태

READY는 "연결이 살아 있고 RPC를 보낼 수 있다"는 상태다. 가장 행복한 상태. 이 상태에서 스트림이 열리고 DATA 프레임이 오가며 실제 일이 일어난다.

READY에서 벗어나는 트리거:

- **연결이 끊김**(상대가 FIN/RST, 네트워크 단절, keepalive ping 실패). → TRANSIENT_FAILURE.
- **서버가 GOAWAY 프레임을 보냄**(서버가 우아하게 연결을 닫겠다고 예고). → 진행 중 RPC는 마치고 새 연결을 준비. (11절 graceful shutdown에서 상술)
- **유휴 타임아웃**으로 한동안 RPC가 없음. → IDLE.

### 4.4 TRANSIENT_FAILURE — "지금은 안 되지만 곧 될 거야"

이름이 중요하다. **Transient(일시적)**. 영구 실패가 아니라 "지금 이 순간 연결이 안 되지만, 잠시 후 다시 시도하면 될 수도 있다"는 낙관적 상태다. 백엔드가 재시작 중이거나, 네트워크가 잠깐 끊겼거나, 일시적 과부하일 때 여기에 머문다.

이 상태의 가장 중요한 성질: **TRANSIENT_FAILURE는 가만히 있지 않는다.** gRPC는 자동으로 백오프 타이머를 걸고 타이머가 만료되면 스스로 CONNECTING으로 돌아가 재연결을 시도한다. 애플리케이션 코드가 재연결 로직을 짤 필요가 전혀 없다. 이 자동 복구가 gRPC 연결 관리의 백미다.

여기서 **RPC 입장의 분기**가 생긴다. 채널이 TRANSIENT_FAILURE인데 새 RPC가 들어오면?

- **Fail Fast (기본)**: RPC를 즉시 `UNAVAILABLE` 에러로 실패시킨다. "지금 연결이 안 되니 기다리지 말고 바로 알려줄게."
- **Wait for Ready (옵션)**: RPC를 실패시키지 않고 **채널이 READY가 될 때까지 큐에 넣고 기다린다.** 데드라인이 만료되면 그때 실패. (5절에서 상술)

### 4.5 SHUTDOWN — 종착역

`channel.shutdown()`(언어별 명칭 차이) 호출 시 진입하는 **종착 상태**. 한 번 SHUTDOWN이 되면 그 채널로는 새 RPC를 시작할 수 없고 다시 살아나지도 않는다. 새 채널을 만들어야 한다. graceful shutdown의 메커니즘은 11절에서 다룬다.

### 4.6 상태 전이 표로 정리

| From | To | 트리거 |
|------|-----|--------|
| (생성) | IDLE | 채널 생성(게으른 기본값) |
| IDLE | CONNECTING | 첫 RPC 발생 / `Connect()` 호출 |
| CONNECTING | READY | 핸드셰이크 성공 |
| CONNECTING | TRANSIENT_FAILURE | 핸드셰이크 실패 |
| READY | TRANSIENT_FAILURE | 연결 끊김 / keepalive 실패 |
| READY | IDLE | 유휴 타임아웃 |
| READY | CONNECTING/IDLE | 서버 GOAWAY (새 연결 준비) |
| TRANSIENT_FAILURE | CONNECTING | 백오프 타이머 만료 → 재시도 |
| (any) | SHUTDOWN | `channel.shutdown()` |

### 4.7 상태를 코드로 관찰하기

상태 기계는 추상 개념이 아니라 API로 노출된다. 채널의 현재 상태를 묻고, 상태 변화를 구독할 수 있다.

Go:

```go
package main

import (
	"context"
	"log"
	"time"

	"google.golang.org/grpc"
	"google.golang.org/grpc/connectivity"
	"google.golang.org/grpc/credentials/insecure"
)

func main() {
	// grpc.NewClient: 채널을 만들지만 즉시 연결하지 않는다(IDLE).
	conn, err := grpc.NewClient(
		"dns:///orderservice.internal:50051",
		grpc.WithTransportCredentials(insecure.NewCredentials()),
	)
	if err != nil {
		log.Fatal(err)
	}
	defer conn.Close()

	log.Printf("생성 직후 상태: %v", conn.GetState()) // IDLE

	// 명시적으로 연결을 데운다(IDLE → CONNECTING).
	conn.Connect()

	// 상태 변화를 구독하며 READY 까지 기다린다.
	for {
		s := conn.GetState()
		log.Printf("현재 상태: %v", s)
		if s == connectivity.Ready {
			break
		}
		// 상태가 s에서 바뀔 때까지 블록(또는 데드라인까지).
		ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
		changed := conn.WaitForStateChange(ctx, s)
		cancel()
		if !changed {
			log.Println("상태 변화 없이 타임아웃")
			break
		}
	}
}
```

Java:

```java
import io.grpc.ConnectivityState;
import io.grpc.ManagedChannel;
import io.grpc.ManagedChannelBuilder;

ManagedChannel channel = ManagedChannelBuilder
        .forTarget("dns:///orderservice.internal:50051")
        .usePlaintext()           // 데모용. 운영은 TLS.
        .build();

// requestConnection=true 이면 IDLE 일 때 연결을 트리거한다.
ConnectivityState state = channel.getState(/* requestConnection */ true);
System.out.println("현재 상태: " + state);

// 상태가 바뀌면 콜백을 받는다.
channel.notifyWhenStateChanged(state, () -> {
    ConnectivityState next = channel.getState(false);
    System.out.println("상태 전이: " + next);
});
```

Python에서도 동일한 개념이 있다.

```python
import grpc

channel = grpc.insecure_channel("orderservice.internal:50051")

def on_change(state: grpc.ChannelConnectivity):
    print("상태 전이:", state)

channel.subscribe(on_change, try_to_connect=True)
# state 는 grpc.ChannelConnectivity.IDLE / CONNECTING / READY /
#         TRANSIENT_FAILURE / SHUTDOWN 중 하나
```

이 API들은 운영에서 매우 유용하다. 예를 들어 헬스 대시보드에 "이 채널이 지금 READY인가 TRANSIENT_FAILURE인가"를 표시하거나, 채널이 READY가 될 때까지 앱 부팅을 막는 readiness 게이트를 만들 수 있다.

## 5. WaitForReady: 실패를 기다림으로 바꾸는 스위치

4.4절에서 잠깐 나온 `WaitForReady`를 제대로 파헤치자. 분산 시스템 설계에서 의외로 깊은 함의를 지닌 옵션이다.

### 5.1 두 가지 철학: Fail Fast vs Wait for Ready

```text
                      RPC 시작 시점에 채널이 READY 가 아니라면?

  ┌─────────────────────────────┐     ┌─────────────────────────────┐
  │   Fail Fast (기본값)         │     │   Wait for Ready (옵션)       │
  ├─────────────────────────────┤     ├─────────────────────────────┤
  │ 즉시 UNAVAILABLE 로 실패      │     │ READY 될 때까지 큐에서 대기     │
  │ "지금 안 되니 바로 알려줄게"    │     │ "될 때까지 기다릴게"           │
  │                             │     │ 단, 데드라인 만료되면 실패        │
  │ 빠른 피드백, 빠른 폴백 가능     │     │ 일시적 장애를 호출자에게 숨김     │
  └─────────────────────────────┘     └─────────────────────────────┘
```

기본은 **Fail Fast**다. 채널이 TRANSIENT_FAILURE면 RPC는 기다리지 않고 즉시 `UNAVAILABLE`로 떨어진다. 왜 이게 기본일까? "빠른 실패"가 분산 시스템에서 더 안전한 기본값이기 때문이다. 호출자는 즉시 실패를 알고 재시도하거나 폴백하거나 서킷 브레이커를 작동시킬 수 있다. 무한정 매달려 있다가 전체가 느려지는 것보다 낫다.

**Wait for Ready**는 이 동작을 뒤집는다. RPC가 시작될 때 채널이 아직 연결 중이거나 일시 장애 상태여도, 실패시키지 않고 채널이 READY가 될 때까지 RPC를 보류한다. 이게 유용한 전형적 상황:

- **앱 부팅 시 의존 서비스가 아직 안 떴을 때.** 부팅 순서를 강제하지 않고 "곧 뜰 테니 기다렸다가 보내라"고 할 수 있다.
- **일시적 네트워크 흔들림을 호출자에게 숨기고 싶을 때.** 단, 반드시 데드라인([[10 - Deadline 취소 타임아웃]])과 함께 써야 한다. 데드라인 없는 Wait for Ready는 영원히 매달릴 수 있는 위험한 조합이다.

주의: Wait for Ready가 보장하는 것은 "연결 단계의 일시 실패를 기다린다"는 것이지, 일단 RPC가 서버에 도달해 처리되다가 실패한 것까지 무한히 기다려 준다는 뜻은 아니다. 연결이 확립되고 RPC가 시작된 뒤의 재시도는 별개 메커니즘([[13 - 안정성 - Retry Health Check Keepalive]])이다.

### 5.2 코드

Go (per-call 옵션):

```go
import "google.golang.org/grpc"

// 반드시 데드라인과 함께!
ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

resp, err := client.GetOrder(ctx, &pb.GetOrderRequest{Id: "order-1"},
	grpc.WaitForReady(true), // 채널이 READY 될 때까지 기다림(데드라인 내에서)
)
```

Java (CallOptions를 통해 스텁 단위로):

```java
OrderServiceGrpc.OrderServiceBlockingStub stub =
        OrderServiceGrpc.newBlockingStub(channel)
            .withWaitForReady()                          // wait-for-ready 켬
            .withDeadlineAfter(3, TimeUnit.SECONDS);     // 반드시 데드라인
```

Python:

```python
resp = stub.GetOrder(
    order_pb2.GetOrderRequest(id="order-1"),
    wait_for_ready=True,
    timeout=3.0,   # 데드라인
)
```

### 5.3 "연결 실패가 곧 RPC 실패는 아니다"

이 절의 핵심 통찰을 한 문장으로. **채널이 TRANSIENT_FAILURE라는 사실 자체는 당신의 RPC가 실패한다는 뜻이 아니다.** Fail Fast면 실패하고, Wait for Ready면 기다린다. 즉 RPC의 운명은 채널 상태 × wait-for-ready 정책 × 데드라인의 함수다. 이 셋을 분리해서 생각하는 습관이 gRPC 운영의 성숙도다.

## 6. 재연결 백오프: 폭주를 막는 지수적 인내

TRANSIENT_FAILURE에서 자동 재시도한다고 했다. 그런데 **얼마나 자주** 재시도할까? 1ms마다? 그러면 죽은 서버에 클라이언트 수천 대가 초당 수백만 번 SYN을 퍼부어 부활하려던 서버를 다시 죽인다. 이것이 그 악명 높은 **thundering herd(쇄도) / retry storm(재시도 폭풍)** 문제다.

gRPC는 이를 **지수 백오프 + 지터(exponential backoff with jitter)** 로 막는다. 이 정책은 gRPC 공식 "Connection Backoff" 명세에 표준 상수와 함께 정의되어 있다.

### 6.1 표준 알고리즘과 기본 상수

```text
파라미터 (gRPC connection-backoff 명세 권장값)
  INITIAL_BACKOFF     = 1 초     (첫 실패 후 대기)
  MULTIPLIER          = 1.6      (실패할 때마다 곱하는 배수)
  JITTER              = 0.2      (±20% 무작위 흔들기)
  MAX_BACKOFF         = 120 초   (대기 상한)
  MIN_CONNECT_TIMEOUT = 20 초    (연결 시도 1회의 최소 허용 시간)
```

알고리즘(의사코드):

```text
current_backoff = INITIAL_BACKOFF              # 1s
첫 연결 시도 즉시 수행
loop:
    이번 연결 시도에 최소 MIN_CONNECT_TIMEOUT(20s) 기회를 준다
    실패하면:
        sleep_for = current_backoff
                    × random(1 - JITTER, 1 + JITTER)   # ±20%
        wait(sleep_for)
        current_backoff = min(current_backoff × MULTIPLIER, MAX_BACKOFF)
    성공하면:
        current_backoff = INITIAL_BACKOFF   # 다시 1s 로 리셋
        break
```

### 6.2 실제 대기 시간이 어떻게 커지는가

지터를 빼고 중앙값만 보면 대기 시간은 이렇게 늘어난다.

| 시도 | 백오프(지터 전) | 지터 적용 범위(±20%) |
|------|----------------|----------------------|
| 1차 실패 후 | 1.0s | 0.8s ~ 1.2s |
| 2차 | 1.6s | 1.28s ~ 1.92s |
| 3차 | 2.56s | 2.05s ~ 3.07s |
| 4차 | 4.10s | 3.28s ~ 4.92s |
| 5차 | 6.55s | 5.24s ~ 7.86s |
| 6차 | 10.49s | 8.39s ~ 12.58s |
| ... | (×1.6씩) | ... |
| n차 | min(1.6^(n-1), 120)s | ... |
| 충분히 큰 n | 120s (상한) | 96s ~ 144s |

```text
대기시간(초)
120 ┤                                      ●━━━━━━━━━●  ← MAX_BACKOFF 천장
    │                                 ●
    │                            ●
    │                       ●
 10 ┤                  ●
    │             ●
  5 ┤        ●
    │    ●
  1 ┤●
    └┬───┬───┬───┬───┬───┬───┬───┬───┬───┬──▶ 재시도 횟수
     1   2   3   4   5   6   7   8  ...
```

### 6.3 왜 지터가 결정적인가

지터가 없다고 상상해 보자. 데이터센터가 잠깐 끊겼다가 복구되는 순간, **클라이언트 1만 대가 정확히 동시에** 백오프 타이머를 시작했다면, 모두 정확히 1초 후, 2.56초 후, ... 같은 순간에 일제히 재연결을 시도한다. 복구된 서버는 동기화된 트래픽 파도에 다시 익사한다.

지터는 이 동기화를 깬다. ±20%의 무작위성이 1만 대의 재시도 시점을 시간 축에 흩뿌려서 파도를 잔물결로 바꾼다. **지터는 "예의 바른 분산 시스템 시민"의 필수 덕목이다.**

```text
지터 없음:           지터 있음:
  ████ (동시 폭주)      ▁▂▃▄▃▂▁▂▃▂▁  (시간에 분산)
   ↑                    완만하게 퍼짐
 서버 다시 죽음          서버 살아남음
```

### 6.4 백오프와 RPC의 관계

다시 강조: 백오프는 **연결 레벨**의 자동 복구이지, **RPC 레벨**의 재시도가 아니다. 둘은 다른 층이다.

- **연결 백오프**: 끊긴 서브채널을 다시 잇는 타이밍 제어. 항상 켜져 있고, 끌 수 없으며, 애플리케이션이 신경 쓸 필요 없다.
- **RPC 재시도(retry)**: 실패한 개별 RPC를 다시 보낼지에 대한 정책. 명시적으로 설정해야 하고, 멱등성(idempotency)을 고려해야 한다. 이건 [[13 - 안정성 - Retry Health Check Keepalive]]에서 다룬다.

두 메커니즘이 협력한다. 연결이 백오프로 복구되는 동안, wait-for-ready RPC는 데드라인까지 기다리고 retry 정책은 재시도 가능한 에러에 한해 재전송한다.

## 7. 스텁(Stub): 채널 위에 얹는 가벼운 손잡이

지금까지는 채널(무거운 것)을 봤다. 이제 스텁(가벼운 것)을 본다.

### 7.1 스텁이란 무엇인가

스텁(stub)은 [[06 - 코드 생성과 protoc 툴체인]]이 `.proto`로부터 생성해 주는 **타입 안전한 클라이언트 객체**다. "원격 메서드를 마치 로컬 함수처럼" 호출하게 해 주는 얇은 어댑터. 스텁 자체에는 연결도, 상태도, 자원도 거의 없다. 스텁은 그저 **채널 참조 + 메서드 시그니처 묶음**일 뿐이다.

```text
   채널 (무겁다: 소켓·상태기계·백오프·LB 보유)
        ▲
        │ 참조만 들고 있음
   ┌────┴────┬──────────┬──────────┐
 스텁A      스텁B       스텁C       스텁D    ← 전부 같은 채널 공유
(blocking) (async)   (future)   (다른서비스)
```

여기서 핵심: **스텁은 싸고, 채널은 비싸다.** 스텁은 마음껏 만들어도 된다. 요청마다 만들어도 좋고 메서드마다 다른 옵션을 붙인 스텁을 파생시켜도 된다. 하지만 채널은 절대 함부로 만들면 안 된다(8절).

### 7.2 스텁의 종류 — 호출 스타일별

같은 서비스라도 보통 여러 종류의 스텁이 생성된다. "RPC의 결과를 어떻게 받을 것인가"에 따라 나뉜다.

| 스텁 종류 | 호출 방식 | 반환 | 적합한 곳 |
|-----------|-----------|------|-----------|
| **Blocking / Sync** | 결과 올 때까지 스레드 블록 | 응답 객체 직접 | 단순 코드, 스크립트, 순차 로직 |
| **Async (callback/observer)** | 콜백/옵저버로 결과 받음 | void, 콜백 호출 | 논블로킹, 스트리밍, 고동시성 |
| **Future** | `Future<Response>` 반환 | 나중에 `.get()` | 여러 RPC 병렬 후 합류 |

Java는 이 세 가지를 명시적으로 분리해 생성한다.

```java
// 같은 채널 위에 세 가지 스타일의 스텁을 만든다 (모두 같은 채널 공유)
ManagedChannel channel = ManagedChannelBuilder
        .forTarget("dns:///orderservice.internal:50051")
        .useTransportSecurity()      // 운영은 TLS
        .build();

// 1) Blocking: 결과가 올 때까지 현재 스레드를 막는다.
OrderServiceGrpc.OrderServiceBlockingStub blocking =
        OrderServiceGrpc.newBlockingStub(channel);
OrderResponse r = blocking.getOrder(GetOrderRequest.newBuilder().setId("o1").build());

// 2) Future: ListenableFuture 로 받아 나중에 합류.
OrderServiceGrpc.OrderServiceFutureStub future =
        OrderServiceGrpc.newFutureStub(channel);
ListenableFuture<OrderResponse> f =
        future.getOrder(GetOrderRequest.newBuilder().setId("o2").build());

// 3) Async: StreamObserver 콜백으로 받는다. 스트리밍에 필수.
OrderServiceGrpc.OrderServiceStub async =
        OrderServiceGrpc.newStub(channel);
async.getOrder(GetOrderRequest.newBuilder().setId("o3").build(),
    new StreamObserver<OrderResponse>() {
        @Override public void onNext(OrderResponse value) { /* 응답 도착 */ }
        @Override public void onError(Throwable t) { /* 에러 */ }
        @Override public void onCompleted() { /* 종료 */ }
    });
```

스트리밍(서버/클라이언트/양방향)은 원래 비동기이므로 보통 async 스텁(또는 언어별 스트리밍 API)을 쓴다. 스트리밍의 4가지 통신 방식은 [[05 - 통신의 4가지 방식 - Unary와 Streaming]]을 보라.

### 7.3 Go의 단일 클라이언트 모델

Go는 위처럼 스텁을 세 종류로 나누지 않는다. 생성된 클라이언트는 하나이고 동기/비동기/타임아웃/취소 같은 제어는 모두 첫 번째 인자인 `context.Context`로 표현한다. "스타일"이 아니라 "컨텍스트"로 호출 정책을 주입하는 설계다.

```go
// 생성된 클라이언트는 하나. 채널(conn)을 감싸는 얇은 래퍼.
client := pb.NewOrderServiceClient(conn)

// 동기처럼 보이지만, ctx 로 데드라인/취소를 주입한다.
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()

resp, err := client.GetOrder(ctx, &pb.GetOrderRequest{Id: "o1"})
// 병렬이 필요하면 goroutine 으로 여러 번 호출하면 된다.
```

이 차이는 외워둘 가치가 있다. **Java/Python = 스텁 종류로 스타일 분기, Go = 단일 client + context로 제어.** 둘 다 "채널을 공유하는 가벼운 손잡이"라는 본질은 같다.

### 7.4 스텁에 옵션을 파생시키기 (불변 객체 패턴)

Java 스텁은 불변(immutable)이다. `withDeadlineAfter`, `withWaitForReady`, `withCompression` 같은 메서드는 **원본을 바꾸지 않고 옵션이 적용된 새 스텁을 반환**한다. 이 덕분에 한 채널 위에서 RPC마다 다른 정책을 안전하게 적용할 수 있다.

```java
OrderServiceGrpc.OrderServiceBlockingStub base =
        OrderServiceGrpc.newBlockingStub(channel);

// 빠르게 끊어야 하는 호출: 짧은 데드라인 + fail fast
OrderResponse fast = base
        .withDeadlineAfter(500, TimeUnit.MILLISECONDS)
        .getOrder(req);

// 배치처럼 느려도 되는 호출: 긴 데드라인 + wait-for-ready + 압축
OrderResponse slow = base
        .withDeadlineAfter(30, TimeUnit.SECONDS)
        .withWaitForReady()
        .withCompression("gzip")
        .listOrders(req2);
```

`base` 스텁은 그대로다. 파생 스텁들은 채널을 공유하면서 호출별 옵션만 다르다. 이것이 "스텁은 싸다"의 실용적 표현이다.

## 8. 황금률: 채널은 만들어서 공유하고, 절대 요청마다 만들지 마라

이 장 전체에서 단 하나만 가져가야 한다면 이것이다.

> **채널은 thread-safe하고 비싸다. 앱 생애주기 동안 타겟당 하나(또는 소수)를 만들어 모든 곳에서 공유하라. 요청마다 채널을 새로 만드는 것은 치명적인 안티패턴이다.**

### 8.1 채널은 thread-safe하다

gRPC 채널은 여러 스레드/고루틴이 동시에 사용해도 안전하도록 설계됐다. 멀티플렉싱 덕분에 한 채널에 동시에 수백 개의 RPC를 흘려보내도 내부에서 알아서 스트림을 나눠 처리한다. 그러니 채널을 스레드마다 만들 이유가 전혀 없다. 하나를 공유하라.

### 8.2 요청마다 채널을 만들면 무슨 일이 일어나는가

안티패턴 코드를 보자(절대 따라 하지 말 것).

```go
// 끔찍한 안티패턴: 요청마다 채널 생성/파괴 (절대 금지)
func handleRequest(orderID string) (*pb.OrderResponse, error) {
	conn, _ := grpc.NewClient("dns:///orderservice:50051",
		grpc.WithTransportCredentials(insecure.NewCredentials()))
	defer conn.Close() // 매번 닫는다 = 매번 연결을 버린다

	client := pb.NewOrderServiceClient(conn)
	ctx, cancel := context.WithTimeout(context.Background(), time.Second)
	defer cancel()
	return client.GetOrder(ctx, &pb.GetOrderRequest{Id: orderID})
}
```

이 코드가 요청마다 치르는 비용을 분해하면:

```text
요청 1건당 발생하는 낭비:
  1) DNS 조회                              (이름 해석 캐시 무력화)
  2) TCP 3-way handshake          1 RTT
  3) TLS handshake               1~2 RTT  (비대칭 암호 연산까지)
  4) HTTP/2 preface + SETTINGS 교환  1 RTT
  ───────────────────────────────────────
     => 핸드셰이크에만 3~4 RTT + CPU 암호연산
        그리고 곧바로 close 로 다 버림

  5) TCP Slow Start: 막 연 연결은 혼잡 윈도우(cwnd)가 작아
     처음엔 느리게 보낸다. 워밍업이 되기도 전에 닫음.
  6) 파일 디스크립터(FD) 낭비 + TIME_WAIT 소켓 누적
     → 고부하 시 "too many open files" / 포트 고갈
  7) 서버 측 연결 슬롯/메모리 폭증, 서버 GC 압박
```

핵심 손해를 표로:

| 항목 | 채널 공유 | 요청마다 생성 |
|------|-----------|----------------|
| 핸드셰이크 비용 | 앱 시작 시 1회 | 요청마다 매번 |
| TCP slow start | 한 번 데우면 끝 | 매번 콜드 스타트 |
| 멀티플렉싱 활용 | O (한 연결로 수백 RPC) | X (연결당 1 RPC) |
| FD/소켓 | 소수 유지 | 폭증, TIME_WAIT 누적 |
| 지연(latency) | 낮음(핸드셰이크 생략) | 매 요청 +수 RTT |
| 서버 부하 | 안정적 | 연결 폭주 |

특히 5번 **TCP Slow Start**가 은근히 치명적이다. 갓 연결된 TCP는 혼잡 제어상 처음엔 적은 데이터만 보낸다(혼잡 윈도우가 작음). 연결을 오래 유지하면 윈도우가 커지며 처리량이 오르는데, 매번 새로 열면 영원히 콜드 상태에 머문다. 채널 공유는 "연결을 데워서 계속 쓴다"는 점에서 그만큼 빠르다.

### 8.3 올바른 패턴

```go
// 앱 시작 시 한 번 만들어 전역(또는 DI 컨테이너)으로 공유 (올바른 패턴)
type App struct {
	orderClient pb.OrderServiceClient
	conn        *grpc.ClientConn
}

func NewApp() (*App, error) {
	conn, err := grpc.NewClient(
		"dns:///orderservice.internal:50051",
		grpc.WithTransportCredentials(insecure.NewCredentials()),
		// keepalive, 메시지 크기 한계 등은 여기서 한 번 설정
	)
	if err != nil {
		return nil, err
	}
	return &App{
		conn:        conn,
		orderClient: pb.NewOrderServiceClient(conn), // 스텁은 가볍다
	}, nil
}

// 요청 핸들러는 공유된 client(스텁)만 쓴다. 채널은 건드리지 않는다.
func (a *App) handleRequest(ctx context.Context, id string) (*pb.OrderResponse, error) {
	return a.orderClient.GetOrder(ctx, &pb.GetOrderRequest{Id: id})
}

// 앱 종료 시 한 번만 닫는다.
func (a *App) Shutdown() { a.conn.Close() }
```

```java
// Java: 채널은 싱글턴/빈으로 한 번, 스텁은 그 위에서 자유롭게 (올바른 패턴)
@Configuration
class GrpcConfig {
    @Bean(destroyMethod = "shutdownNow")   // 컨테이너 종료 시 닫힘
    ManagedChannel orderChannel() {
        return ManagedChannelBuilder
                .forTarget("dns:///orderservice.internal:50051")
                .useTransportSecurity()
                .keepAliveTime(30, TimeUnit.SECONDS)
                .build();
    }

    @Bean
    OrderServiceGrpc.OrderServiceBlockingStub orderStub(ManagedChannel ch) {
        return OrderServiceGrpc.newBlockingStub(ch);
    }
}
```

### 8.4 "소수"라고 한 이유

"하나"가 원칙이지만 "소수"인 경우도 있다. 3.3절에서 봤듯 단일 연결의 `MAX_CONCURRENT_STREAMS`에 묶여 처리량이 막히는 초고부하 상황에서는 같은 타겟에 채널을 2~4개 두는 **채널 풀링**이 정당화된다. 또한 지연 민감 트래픽과 배치 트래픽을 분리하려고 채널을 나누기도 한다. 핵심은 "요청마다"가 아니라 "용도/부하에 따라 정적인 소수"라는 것. 이 트레이드오프의 정량적 판단은 [[15 - 성능과 Flow Control 백프레셔]]에서.

## 9. 이름 해석·로드밸런싱과 채널의 관계

채널이 생성될 때 가장 먼저 하는 일은 **타겟(target) 문자열을 실제 주소 목록으로 해석**하는 것이다. 이 과정의 상세는 [[12 - 이름 해석과 로드밸런싱]]에 양보하고, 여기서는 **채널 생명주기 관점에서 무슨 일이 일어나는지**만 짚는다.

### 9.1 타겟 문자열의 구조

```text
   dns:///orderservice.internal:50051
   └─┬─┘   └──────────┬──────────┘└─┬─┘
   scheme          host            port
  (resolver를      (해석 대상)
   고름)

  예:
   dns:///foo.bar:50051        → DNS resolver 로 foo.bar 의 A/AAAA 레코드 조회
   passthrough:///10.0.0.5:50051 → 해석 없이 주소 그대로 사용
   unix:///var/run/svc.sock    → 유닉스 도메인 소켓
   xds:///foo.bar              → xDS(서비스 메시) 제어 평면에서 해석
```

scheme이 **resolver(이름 해석기)** 를 고르고, resolver가 host를 주소 목록으로 푼다.

### 9.2 채널 생성 → 해석 → 서브채널 → LB

```text
channel 생성
   │
   ▼
[Name Resolver]  dns:///orderservice.internal:50051
   │  → [10.0.0.11:50051, 10.0.0.12:50051, 10.0.0.13:50051]
   │  (그리고 백그라운드에서 주기적으로 재해석 → 주소 변동 반영)
   ▼
[LB Policy]  예: round_robin
   │  → 받은 주소마다 Subchannel 을 만들라고 지시
   ▼
Subchannel(.11)  Subchannel(.12)  Subchannel(.13)
   각자 4절의 상태기계(IDLE→CONNECTING→READY...)를 독립적으로 돈다
   │
   ▼
RPC 가 오면 LB Policy 가 READY 인 서브채널 중 하나를 골라(picker) 보낸다
```

여기서 채널 생명주기와 맞물리는 중요한 사실들:

- **채널의 전체 상태는 서브채널 상태들의 집계**다. 예컨대 round_robin에서 서브채널이 하나라도 READY면 채널은 READY로 보고 전부 TRANSIENT_FAILURE여야 채널이 TRANSIENT_FAILURE다.
- **resolver는 한 번 해석하고 끝이 아니다.** 주기적으로(또는 연결 실패 등 이벤트 트리거로) 재해석해서 백엔드 추가/제거를 반영한다. 새 주소가 생기면 새 서브채널이, 사라지면 해당 서브채널이 graceful하게 정리된다.
- **LB 정책에 따라 연결 개수가 달라진다.** `pick_first`(기본)는 주소 목록을 순서대로 시도해 **하나의 연결만** 쓴다. `round_robin`은 **모든 백엔드에 연결**을 만들어 분산한다. 이 선택이 3.3절의 처리량 한계와 직결된다.

## 10. 채널 옵션/인자: 생성 시점에 박아 두는 정책들

채널을 만들 때 넘기는 옵션(Go의 `DialOption`, Java의 빌더 메서드, C-core의 channel args)은 그 채널의 동작 전반을 결정한다. 대부분 다른 장에서 깊게 다루므로, 여기서는 **무엇을 어디서 설정하는지의 지도**만 제공한다.

| 옵션 범주 | 무엇 | 어느 장에서 깊이 | Go 예 | Java 예 |
|-----------|------|------------------|-------|---------|
| 보안/자격증명 | TLS/mTLS/plaintext | [[11 - 보안 - TLS mTLS 인증]] | `WithTransportCredentials` | `useTransportSecurity()` |
| 메시지 크기 한계 | 송수신 최대 바이트 | [[15 - 성능과 Flow Control 백프레셔]] | `WithDefaultCallOptions(MaxCallRecvMsgSize)` | `maxInboundMessageSize()` |
| Keepalive | ping 주기/타임아웃 | [[13 - 안정성 - Retry Health Check Keepalive]] | `WithKeepaliveParams` | `keepAliveTime/keepAliveTimeout` |
| 압축 | gzip 등 | [[15 - 성능과 Flow Control 백프레셔]] | `UseCompressor` (call opt) | `withCompression("gzip")` |
| 인터셉터 | 횡단 관심사 훅 | [[08 - 메타데이터와 인터셉터]] | `WithUnaryInterceptor` | `intercept(...)` |
| 이름 해석/LB | resolver, LB 정책 | [[12 - 이름 해석과 로드밸런싱]] | `WithResolvers`, `WithDefaultServiceConfig` | `defaultLoadBalancingPolicy()` |
| 재시도 | retry 정책(service config) | [[13 - 안정성 - Retry Health Check Keepalive]] | service config JSON | service config JSON |
| 유휴 타임아웃 | READY→IDLE 시간 | (이 장) | `WithIdleTimeout` | `idleTimeout()` |

메시지 크기 한계만 짧게 짚자. gRPC는 기본적으로 **수신 메시지 크기를 4MB로 제한**한다(보낼 때는 기본 무제한인 구현이 많다). 큰 메시지를 주고받아야 하면 이 한계를 올려야 한다. 단, 큰 메시지는 메모리·지연 측면에서 비싸므로 스트리밍으로 쪼개는 편이 보통 낫다([[05 - 통신의 4가지 방식 - Unary와 Streaming]]).

```go
conn, _ := grpc.NewClient(target,
	grpc.WithTransportCredentials(creds),
	grpc.WithDefaultCallOptions(
		grpc.MaxCallRecvMsgSize(16*1024*1024), // 수신 16MB 허용
	),
)
```

인터셉터 등록 지점이 채널 생성 시점이라는 것도 기억하자. 메타데이터 주입, 인증 토큰 부착, 로깅/트레이싱/메트릭 같은 횡단 관심사를 인터셉터로 한 번 등록하면 그 채널의 모든 RPC에 적용된다([[08 - 메타데이터와 인터셉터]]).

## 11. Graceful Shutdown: 끝맺음의 예의

채널과 연결의 생애 마지막, **종료**를 본다. 그냥 소켓을 `close()` 해버리면 진행 중이던 RPC들이 중간에 끊겨 에러가 된다. gRPC는 **우아한 종료(graceful shutdown)** 를 양쪽(클라이언트 채널, 서버)에서 지원한다.

### 11.1 클라이언트 측: shutdown vs shutdownNow

Java가 둘을 명확히 구분한다.

```java
ManagedChannel channel = /* ... */;

// 우아한 종료: 진행 중 RPC는 끝까지 처리, 새 RPC만 거부.
channel.shutdown();

// 종료 완료를 일정 시간 기다린다(블록).
if (!channel.awaitTermination(5, TimeUnit.SECONDS)) {
    // 5초 안에 못 끝내면 강제 종료: 진행 중 RPC도 취소(CANCELLED).
    channel.shutdownNow();
    channel.awaitTermination(2, TimeUnit.SECONDS);
}
```

```text
shutdown()      : "문을 닫되, 안에 있는 손님은 다 나갈 때까지 기다린다"
                  - 새 RPC 거부, 기존 RPC 완료 대기, 끝나면 연결 종료
shutdownNow()   : "지금 당장 닫는다. 안에 있는 손님도 내보낸다"
                  - 진행 중 RPC를 CANCELLED 로 취소하고 즉시 종료
awaitTermination: 종료가 실제로 끝날 때까지 블록(타임아웃 포함)
```

Go는 `conn.Close()` 하나다. 진행 중 RPC가 있으면 컨텍스트 취소로 정리되며 실무에서는 `Close()` 전에 워커들이 끝나길 기다리는 패턴을 직접 짠다.

```go
// 우아한 종료 패턴
func (a *App) Shutdown(ctx context.Context) error {
	// 1) 새 작업 받기를 멈추고, 진행 중 작업이 끝나길 기다린다(앱 레벨).
	a.stopAccepting()
	a.waitInflight(ctx)
	// 2) 채널을 닫는다.
	return a.conn.Close()
}
```

### 11.2 서버 측: GOAWAY 프레임

서버가 종료하거나(배포/스케일다운) 연결을 정리하고 싶을 때, 갑자기 끊지 않고 **GOAWAY** 프레임을 보낸다. 이건 HTTP/2 표준 메커니즘(RFC 7540 §6.8, RFC 9113로 갱신)이다. "나 이제 이 연결로는 새 스트림 안 받을 거야. 단, 마지막으로 처리한 스트림 ID까지는 책임지고 끝낼게"라는 예고.

GOAWAY 프레임을 바이트로 디코딩해 보자.

```text
바이트열:  00 00 08   07   00   00 00 00 00   00 00 00 09   00 00 00 00
           └──┬───┘   │    │    └────┬─────┘   └────┬─────┘   └────┬────┘
           Length=8  Type Flags  StreamID=0   LastStreamID    ErrorCode
            (8바이트  =0x07 =0    (GOAWAY도     =9 (이 ID 이하   =0 (NO_ERROR)
             payload) GOAWAY      stream 0)    스트림은 처리됨)
```

- `00 00 08`: payload 8바이트 (Last-Stream-ID 4 + Error Code 4. 디버그 데이터는 없음).
- `07`: 프레임 타입 0x7 = GOAWAY.
- `00`: 플래그 없음.
- `00 00 00 00`: 스트림 0(연결 전체).
- `00 00 00 09`: **Last-Stream-ID = 9.** "스트림 9까지는 내가 처리했거나 처리 중이다. 그보다 큰 번호의 스트림은 시작도 안 했으니 너희가 다른 연결에서 재시도해도 안전하다."
- `00 00 00 00`: 에러 코드 0 = NO_ERROR(정상적 종료).

GOAWAY를 받은 클라이언트 채널의 반응:

```text
1) Last-Stream-ID 이하의 진행 중 RPC → 계속 진행해 끝맺는다.
2) Last-Stream-ID 초과로 보내려던/막 보낸 RPC → 이 연결에선 처리 보장 안 됨.
   클라이언트는 이들을 새 연결에서 안전하게 재시도할 수 있다.
3) 이 연결로는 새 스트림을 더 열지 않는다.
4) 동시에 새 연결(새 서브채널 또는 같은 백엔드로의 새 연결)을 준비한다.
   → 채널 전체로 보면 무중단(connection draining).
```

이 덕분에 서버를 **무중단 배포(graceful rolling deploy)** 할 수 있다. 새 파드가 뜨고, 기존 파드가 GOAWAY로 연결을 비우고(drain), 진행 중 요청을 다 끝낸 뒤 종료한다. 클라이언트는 그 사이 다른 연결로 자연스럽게 옮겨 간다. GOAWAY와 keepalive, health check가 어떻게 협력해 무중단을 만드는지는 [[13 - 안정성 - Retry Health Check Keepalive]]에서 더 본다.

### 11.3 양방향 종료 시퀀스 한눈에

```text
   클라이언트 채널                         서버
        │                                  │
        │     RPC 들 진행 중 (stream 1,3,5)  │
        │◀────────── 데이터 ──────────────▶│
        │                                  │  (서버 종료 시작)
        │◀──────────── GOAWAY(last=5) ─────│  "5까지만 받음"
        │                                  │
        │  stream 1,3,5 는 끝까지 마무리      │
        │◀────────── 응답 ─────────────────│
        │                                  │
        │  새 RPC 는 새 연결로 ──────────────│ (다른 파드/연결)
        │                                  │
        │           (기존 연결 close)        │  서버 종료 완료
```

## 12. 언어별 차이 메모

같은 개념이지만 API 표면은 언어마다 다르다. 자주 헷갈리는 포인트만 모은다.

### 12.1 Go: Dial에서 NewClient로

grpc-go는 비교적 최근(v1.63 무렵)에 채널 생성 API를 크게 바꿨다. 핵심 변화를 알아둬야 옛 코드와 새 코드를 둘 다 읽을 수 있다.

| 구분 | 옛 방식 `grpc.Dial` / `DialContext` (deprecated) | 새 방식 `grpc.NewClient` (권장) |
|------|--------------------------------------------------|---------------------------------|
| 연결 시점 | 옵션에 따라 eager 연결 가능(`WithBlock`) | 항상 lazy. 생성 시 IDLE, 첫 RPC에 연결 |
| `WithBlock`/`WithReturnConnectionError` | 지원(생성이 연결까지 블록) | 없음(개념적으로 부적합해 제거) |
| 기본 resolver scheme | `passthrough`(주소 그대로) | `dns`(DNS 조회) |
| 권장 여부 | deprecated | 권장 |

```go
// 옛 방식 (deprecated): 연결될 때까지 블록하던 패턴
conn, err := grpc.DialContext(ctx, "orderservice:50051",
	grpc.WithTransportCredentials(insecure.NewCredentials()),
	grpc.WithBlock(),                    // 연결 완료까지 대기 (NewClient엔 없음)
)

// 새 방식 (권장): 게으른 생성. 연결 워밍업이 필요하면 Connect()를 명시 호출
conn, err := grpc.NewClient("dns:///orderservice:50051",
	grpc.WithTransportCredentials(insecure.NewCredentials()),
)
// conn.Connect()  // 원하면 미리 연결을 데운다
```

`WithBlock`이 사라진 것이 처음엔 불편하지만 철학적으로 옳다. "채널 생성"과 "연결 성공"은 별개 사건이고 채널은 본래 게으르고 자가 복구적이어야 하기 때문이다. 부팅 시 연결 보장이 필요하면 `Connect()` 후 `WaitForStateChange`로 READY를 기다리거나, wait-for-ready RPC로 첫 호출을 보낸다.

또 하나 함정: 기본 scheme이 `passthrough`에서 `dns`로 바뀌었기 때문에, `grpc.Dial("localhost:50051", ...)`처럼 scheme 없이 쓰던 코드를 `grpc.NewClient`로 옮기면 `localhost`를 DNS로 해석하려 든다. 명시적으로 `dns:///` 또는 `passthrough:///`를 붙이는 습관이 안전하다.

### 12.2 Java: ManagedChannel과 빌더

Java에서 채널 타입은 `ManagedChannel`이고 만들 때는 `ManagedChannelBuilder`를 쓴다. "Managed"는 "수명/연결을 라이브러리가 관리한다"는 의미.

```java
ManagedChannel channel = ManagedChannelBuilder
        .forTarget("dns:///orderservice.internal:50051")
        .useTransportSecurity()           // 또는 usePlaintext() (데모)
        .keepAliveTime(30, TimeUnit.SECONDS)
        .maxInboundMessageSize(16 * 1024 * 1024)
        .build();

// ... 스텁으로 RPC ...

// 종료(11.1절)
channel.shutdown();
channel.awaitTermination(5, TimeUnit.SECONDS);
```

종료 메서드 3종(`shutdown`/`shutdownNow`/`awaitTermination`)의 의미 차이는 11.1절에서 본 그대로다. 운영 코드에서는 보통 `shutdown()` → `awaitTermination(timeout)` → (안 끝나면) `shutdownNow()`의 단계적 패턴을 쓴다.

### 12.3 Python / 기타

- **Python**: `grpc.insecure_channel(target)` / `grpc.secure_channel(target, creds)`로 채널 생성. `channel.subscribe(callback, try_to_connect=True)`로 상태 구독. `with grpc.insecure_channel(...) as channel:` 컨텍스트 매니저로 자동 종료. 동기 스텁과 `grpc.aio`(asyncio) 비동기 스텁이 별도.
- **C++ / C-core 계열(Python, Ruby, PHP, C# 일부)**: 내부적으로 C-core를 공유하며 channel args(키-값)로 옵션을 준다. `GRPC_ARG_KEEPALIVE_TIME_MS` 같은 문자열 키.
- 어떤 언어든 **개념(채널=가상연결, 서브채널, 상태기계, 백오프, 스텁 공유)은 동일**하다. API 이름만 갈아 끼우면 된다.

## 13. 전체를 잇는 시나리오: 한 RPC의 일생

지금까지의 조각들을 하나의 타임라인으로 꿰어 보자. "앱이 부팅하고, 첫 주문 조회 RPC를 보내고, 백엔드가 잠깐 죽었다가 살아나고, 앱이 종료될 때까지."

```text
t0  앱 시작: grpc.NewClient(dns:///orderservice:50051)
    → Channel 생성. 상태 = IDLE. 아직 소켓 0개.

t1  첫 RPC: client.GetOrder("o-1")
    → IDLE → CONNECTING.
    → resolver: dns 조회 → [10.0.0.11, 10.0.0.12]
    → LB(round_robin): 두 백엔드에 Subchannel 생성
    → 각 서브채널 TCP+TLS+HTTP/2 핸드셰이크
    → 둘 다 성공: Subchannel READY → Channel READY
    → stream 1 열림, GetOrder 전송, 응답 수신. 끝.

t2  정상 트래픽: 수백 개 RPC가 두 연결로 멀티플렉싱.
    한 연결당 MAX_CONCURRENT_STREAMS(=100) 한도 내에서 인터리빙.

t3  백엔드 .11 파드가 죽음(배포):
    → 서버가 GOAWAY(last_stream_id) 전송
    → 진행 중 RPC는 마무리, 새 RPC는 .12 로
    → .11 소켓 끊김 → Subchannel(.11) READY → TRANSIENT_FAILURE
    → Channel 은 여전히 READY (.12 가 살아 있으므로)
    → .11 서브채널: 백오프 시작 (1s, 1.6s, 2.56s ... ±지터)

t4  새 .11 파드가 뜸, resolver 가 재해석으로 인지:
    → 백오프 타이머 만료 시 .11 Subchannel CONNECTING → READY
    → 다시 두 연결로 분산. (애플리케이션은 아무것도 안 했다!)

t5  앱 종료: channel.shutdown() / conn.Close()
    → 진행 중 RPC 마무리(graceful), 새 RPC 거부
    → 소켓 정리, Channel SHUTDOWN. 끝.
```

이 타임라인에서 애플리케이션 코드가 손댄 것은 t0(채널 생성), t1~t2(RPC 호출), t5(종료)뿐이다. t3~t4의 장애 감지·격리·백오프·재연결·재해석은 **전부 채널이 알아서** 했다. 이것이 채널 추상화가 우리에게 주는 선물이며 동시에 "채널을 요청마다 만들면 이 모든 자가 복구 메커니즘을 매번 버리는 것"이라는 8절 경고의 의미다.

## 14. 흔한 오해와 함정 (FAQ)

**Q. 채널을 만들었는데 연결이 안 열려요. 버그인가요?**
A. 아니다. `grpc.NewClient`(및 최신 권장 경로)는 게으르다. 채널은 IDLE로 시작하고 첫 RPC나 `Connect()` 때 연결한다. 부팅 시 연결을 확인하려면 명시적으로 `Connect()` + 상태 대기, 또는 헬스 체크 RPC를 한 번 던져라.

**Q. `GetState()`가 TRANSIENT_FAILURE인데 RPC는 성공해요. 왜죠?**
A. 채널 상태는 서브채널들의 집계다. 일부 서브채널이 실패해도 READY인 서브채널이 있으면 RPC는 그쪽으로 간다. 또한 wait-for-ready RPC라면 잠깐의 TRANSIENT_FAILURE를 기다렸다가 성공할 수 있다.

**Q. 채널을 스레드마다 하나씩 만들어야 동시성이 좋아지지 않나요?**
A. 아니다. 채널은 thread-safe하고 한 연결로 멀티플렉싱한다. 스레드마다 채널을 만들면 연결만 늘고 핸드셰이크/메모리만 낭비된다. 처리량이 정말 `MAX_CONCURRENT_STREAMS`에 막힐 때만(3.3절) 정적인 소수 채널 풀을 고려하라.

**Q. RPC가 `UNAVAILABLE`로 떨어져요.**
A. 채널이 연결을 못 맺었거나(TRANSIENT_FAILURE) GOAWAY/연결 끊김 직후일 가능성이 크다. Fail Fast 기본값에서는 이때 즉시 `UNAVAILABLE`이 난다. 일시적 장애를 견디려면 wait-for-ready + 데드라인, 또는 retry 정책([[13 - 안정성 - Retry Health Check Keepalive]])을 쓰라. 단 멱등성을 반드시 확인. 상태 코드의 의미는 [[09 - 에러 모델 - 상태 코드와 Rich Error]].

**Q. 데드라인 없이 wait-for-ready를 켜도 되나요?**
A. 위험하다. 채널이 영원히 READY가 안 되면 RPC도 영원히 매달린다. wait-for-ready는 항상 데드라인([[10 - Deadline 취소 타임아웃]])과 함께 쓰라.

**Q. `Connect()`를 호출하면 연결이 보장되나요?**
A. 보장이 아니라 "시도 트리거"다. IDLE에서 CONNECTING으로 밀어줄 뿐, 성공 여부는 핸드셰이크에 달렸다. 보장하려면 상태가 READY가 될 때까지 관찰하라.

## 핵심 요약

- **채널(channel)은 소켓이 아니라 "타겟으로 가는 가상 연결"이라는 추상화**다. 그 아래 여러 **서브채널(subchannel)**이 있고, 서브채널마다 실제 TCP+TLS+HTTP/2 연결이 하나씩 붙는다. 채널 ⊃ 서브채널 ⊃ 소켓의 3층 구조.
- 채널/서브채널은 **IDLE → CONNECTING → READY → TRANSIENT_FAILURE**(→ 재시도) 의 상태 기계를 돌고, `shutdown()`으로 **SHUTDOWN**에 도달한다. 게으른 IDLE 시작, 유휴 시 READY→IDLE 복귀가 기본.
- **TRANSIENT_FAILURE는 자동 복구한다.** gRPC가 **지수 백오프 + 지터**(기본: INITIAL 1s, MULTIPLIER 1.6, JITTER ±0.2, MAX 120s, MIN_CONNECT_TIMEOUT 20s)로 재연결을 스스로 관리한다. 애플리케이션은 재연결 로직을 짜지 않는다.
- **연결 실패가 곧 RPC 실패는 아니다.** 기본 **Fail Fast**는 즉시 `UNAVAILABLE`, **Wait for Ready**는 (데드라인 안에서) READY까지 대기. RPC의 운명 = 채널 상태 × wait-for-ready × 데드라인.
- **HTTP/2 멀티플렉싱** 덕에 한 연결로 수백 RPC를 보낼 수 있지만, **`MAX_CONCURRENT_STREAMS`(흔히 100)**가 단일 연결의 동시성 천장이다. 고부하에선 round_robin 다중 연결이나 채널 풀링이 필요하다.
- **채널은 thread-safe하고 비싸다. 앱 생애주기 동안 타겟당 1개(또는 정적 소수)를 만들어 공유하라.** "요청마다 채널 생성"은 핸드셰이크·TCP slow start·FD 낭비·서버 부하를 매번 유발하는 치명적 안티패턴.
- **스텁(stub)은 싸고 채널을 공유하는 가벼운 손잡이**다. Java는 blocking/future/async 세 종류, Go는 단일 client + context. 호출별 옵션은 불변 스텁 파생(`withDeadlineAfter` 등)으로.
- **Graceful shutdown**: 클라이언트는 `shutdown()`(진행 중 RPC 보존) vs `shutdownNow()`(취소), 서버는 **GOAWAY**(Last-Stream-ID로 처리 경계 통보)로 무중단 연결 비우기(draining)를 한다.
- **언어 차이**: Go는 `grpc.Dial`(deprecated, eager 옵션) → `grpc.NewClient`(권장, 항상 lazy, `WithBlock` 없음, 기본 dns scheme)로 이동. Java는 `ManagedChannelBuilder`로 만들고 `shutdown/awaitTermination/shutdownNow`로 닫는다.
