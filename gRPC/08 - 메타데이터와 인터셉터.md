이 장이 답하는 질문:

- 메시지 본문(body)과 메타데이터(metadata)는 무엇이 다르고, 왜 따로 존재하는가?
- 메타데이터는 HTTP/2 위에서 물리적으로 어떤 프레임에 어떤 바이트로 실려 가는가? `leading`과 `trailing`은 언제 보내지는가?
- `-bin` 접미사, 소문자 규칙, 예약 헤더(`:method`, `grpc-timeout` 등)는 왜 그렇게 정해졌는가?
- 인터셉터는 어떻게 호출 체인을 감싸며, 등록 순서가 왜 의미를 바꾸는가?
- 인증·요청 ID 주입·트레이싱·패닉 복구를 client/server 양쪽에서 어떻게 구현하는가?

## 1. 본문과 봉투: 메타데이터가 필요한 이유

편지를 부친다고 생각해 보자. 편지지에 적힌 글은 **수신자가 읽을 내용**이다. 그런데 봉투 겉면에는 글의 내용과 무관한 정보가 잔뜩 붙는다. 우표, 우편번호, 등기 여부, 빠른등기 스티커, 발신 우체국 소인. 이것들은 편지의 *의미*가 아니라 편지를 *어떻게 처리할지*를 말해 준다. 우체국 직원은 봉투를 보고 분류·배송하지, 편지를 열어 읽고 처리 방식을 정하지 않는다.

gRPC에서 **메시지 본문(message body)** 은 편지지이고, **메타데이터(metadata)** 는 봉투 겉면이다. 본문은 [[02 - Protocol Buffers 1 - 문법과 타입 시스템]]에서 정의한 protobuf 메시지가 [[03 - Protocol Buffers 2 - 인코딩과 와이어 포맷]]의 규칙대로 직렬화되어 들어간다. 본문은 비즈니스 로직이 읽고 처리하는 "내용"이다. 반면 메타데이터는 그 RPC를 *어떻게* 처리할지에 대한 부가 정보다. 누가 호출했는지(인증 토큰), 이 요청을 추적하는 ID는 무엇인지(trace ID), 클라이언트가 어떤 압축을 쓰는지, 응답을 얼마나 기다릴 것인지 같은 것들이다.

```text
┌─────────────────────────────────────────────┐
│  하나의 RPC 호출                              │
│                                              │
│  ┌────────────────┐   ┌──────────────────┐  │
│  │  메타데이터     │   │  메시지 본문      │  │
│  │  (봉투의 우표)  │   │  (편지지)         │  │
│  │                │   │                  │  │
│  │  authorization │   │  GetOrderRequest │  │
│  │  x-request-id  │   │  { order_id: 42 }│  │
│  │  trace-parent  │   │                  │  │
│  └────────────────┘   └──────────────────┘  │
│   HTTP/2 HEADERS        HTTP/2 DATA 프레임   │
│   /TRAILERS 프레임                           │
└─────────────────────────────────────────────┘
```

이 분리는 단순한 미관 문제가 아니라 **관심사의 분리(separation of concerns)** 라는 설계 결정이다. 만약 인증 토큰이나 trace ID를 메시지 본문(protobuf 필드)에 넣는다면 어떻게 될까? 모든 `proto` 메시지마다 `string auth_token = 99;` 같은 필드를 반복해서 넣어야 하고, 인증 미들웨어가 본문을 역직렬화해야만 토큰을 꺼낼 수 있다. 미들웨어가 비즈니스 타입에 의존하게 된다. 메타데이터로 분리하면, 인증 인터셉터는 본문을 *전혀 건드리지 않고도* 봉투의 우표만 보고 통과/거부를 결정할 수 있다. 우체국 직원이 편지를 뜯지 않고 봉투만 보고 분류하듯이.

> 핵심 직관: **본문은 "무엇을 요청하는가", 메타데이터는 "이 요청을 어떻게 다룰 것인가".** 둘을 섞으면 미들웨어가 비즈니스에 결합된다.

## 2. 메타데이터의 물리적 실체: HTTP/2 HEADERS와 TRAILERS

gRPC는 추상적인 개념이 아니라 [[04 - HTTP2 깊이 보기 - 전송 계층]]에서 다룬 HTTP/2 위에 구체적으로 매핑된다. 메타데이터의 "정체"를 이해하려면 한 번의 unary RPC가 HTTP/2 프레임으로 어떻게 펼쳐지는지를 봐야 한다.

하나의 RPC = 하나의 HTTP/2 **스트림(stream)** 이다. 그 스트림 안에서 프레임이 다음 순서로 흐른다.

```text
시간 →

클라이언트                                서버
   │                                       │
   │── HEADERS (요청 헤더 = 요청 메타데이터)─→│   ┐
   │   :method POST, :path /pkg.Svc/Method  │   │ leading
   │   content-type application/grpc        │   │ (요청)
   │   authorization Bearer ...             │   ┘
   │── DATA (길이-접두 protobuf 메시지)─────→│
   │   END_STREAM (요청 끝)                  │
   │                                       │
   │                                       │ 서버 처리...
   │                                       │
   │←─ HEADERS (응답 초기 헤더 = leading)────│   ┐ leading
   │   :status 200                          │   │ (응답)
   │   content-type application/grpc        │   ┘
   │←─ DATA (길이-접두 protobuf 응답)────────│
   │                                       │
   │←─ HEADERS (응답 종료 = TRAILERS)────────│   ┐ trailing
   │   grpc-status 0                        │   │ (응답)
   │   grpc-message ...                     │   │
   │   END_STREAM (응답 끝)                  │   ┘
   │                                       │
```

여기서 메타데이터가 실리는 자리는 셋이다.

| 자리 | HTTP/2 프레임 | gRPC 용어 | 방향 | 담는 것 |
|------|--------------|-----------|------|---------|
| 요청 헤더 | HEADERS (스트림 시작) | request metadata (leading) | client → server | 인증 토큰, request-id, trace-parent, 압축 알고리즘 |
| 응답 초기 헤더 | HEADERS (응답 DATA 전) | response header / leading | server → client | 서버 측 부가 정보, 일부 트레이싱 응답 |
| 응답 종료 | HEADERS (END_STREAM) | trailer / trailing | server → client | `grpc-status`, `grpc-message`, 처리 통계 |

가장 중요한 통찰 하나: **gRPC는 응답의 최종 상태(status)를 본문이 아니라 trailer에 넣는다.** HTTP/1의 `200 OK`는 응답을 *시작할 때* 정해진다. 그런데 서버가 응답 메시지를 절반쯤 스트리밍한 뒤에 에러가 났다면? HTTP/1에서는 이미 200을 보내버려서 사후에 "사실 실패했다"고 말할 자리가 없다. HTTP/2의 **트레일러(trailing headers)** 는 바로 이 문제를 푼다. 응답 데이터가 다 흐른 *뒤에* `grpc-status: 0`(성공) 혹은 `grpc-status: 13`(내부 오류)을 붙일 수 있다. 그래서 gRPC의 진짜 결과 코드는 HTTP의 `:status`가 아니라 **trailer의 `grpc-status`** 다. 이 메커니즘은 [[09 - 에러 모델 - 상태 코드와 Rich Error]]에서 깊이 다룬다.

이것이 gRPC가 HTTP/1이 아니라 HTTP/2를 요구하는 결정적 이유 중 하나다. HTTP/1.1에도 chunked transfer의 trailer가 명세상 존재하지만 현실의 프록시·브라우저 지원이 형편없다. HTTP/2는 trailer를 1급 시민으로 다룬다.

### 2.1 HPACK: 헤더가 실제로 가는 바이트

HTTP/2의 HEADERS 프레임은 헤더 이름·값을 평문으로 보내지 않는다. **HPACK(RFC 7541)** 이라는 헤더 압축을 거친다. HPACK은 두 가지 무기를 쓴다.

1. **정적 테이블(static table)**: `:method GET`, `:status 200` 같이 흔한 헤더는 미리 번호가 매겨져 있다(RFC 7541 부록 A의 61개 엔트리). `:method GET`은 인덱스 2, `:method POST`는 인덱스 3, `:status 200`은 인덱스 8 같은 식이다. gRPC는 항상 `POST`를 쓰므로 메서드 줄은 인덱스 3 하나(1바이트)로 줄어든다.
2. **동적 테이블(dynamic table)**: 같은 연결에서 한 번 보낸 헤더(예: `authorization: Bearer eyJ...`)는 동적 테이블에 등록되어, 다음 요청부터는 긴 토큰 문자열 대신 인덱스 번호만 보내면 된다.

직접 손으로 디코딩해 보자. 클라이언트가 `te: trailers` 헤더를 보낸다고 하자. HPACK은 "리터럴 헤더 필드 + 증분 인덱싱(literal header field with incremental indexing)" 표현을 쓴다. `te` 헤더의 바이트열을 보면:

```text
바이트:  40 02 74 65 08 74 72 61 69 6c 65 72 73
         │  │  └─┬─┘ │  └────────┬──────────┘
         │  │   "te" │       "trailers"
         │  │        │
         │  │        └ 값 길이 = 0x08 = 8
         │  └ 이름 길이 = 0x02 = 2
         └ 0x40 = 0b0100_0000
           = "Literal Header Field with Incremental Indexing,
              new name" 패턴 (상위 2비트 01, 인덱스 0)
```

- `0x40`: 상위 2비트가 `01`이면 "증분 인덱싱하는 리터럴", 하위 6비트가 `000000`(=0)이면 "이름도 리터럴로 따라온다"는 뜻.
- `0x02`: 이름 길이 2 (HPACK 정수, 최상위 비트 0이면 Huffman 미사용).
- `74 65`: ASCII `t`(0x74) `e`(0x65) = `"te"`.
- `0x08`: 값 길이 8.
- `74 72 61 69 6c 65 72 73`: ASCII `trailers`.

즉 13바이트가 그대로 `te: trailers`다. 그런데 같은 연결에서 두 번째 요청부터는 이 헤더가 동적 테이블에 들어가 있으므로 단 1~2바이트(인덱스 참조)로 줄어든다. **메타데이터를 작고 반복적으로 유지해야 하는 이유**가 여기에 있다(§9.3에서 다시).

### 2.2 ASCII vs 바이너리 값과 `-bin` 접미사

HTTP/2 헤더 값은 원칙적으로 텍스트(특정 바이트 범위의 octet)를 담는 통로다. 그런데 gRPC 메타데이터로는 바이너리 데이터(예: 직렬화된 protobuf, 원본 토큰 바이트, 압축된 trace context)도 싣고 싶다. HPACK과 HTTP/2 헤더는 NUL, 개행 등 임의 바이트를 안전하게 다룬다고 보장하지 않는다.

gRPC의 해법은 명료하다: **키 이름이 `-bin`으로 끝나면 그 값은 바이너리이며, 와이어 위에서는 base64로 인코딩된다.** gRPC 라이브러리가 자동으로 한다.

```text
애플리케이션 코드:    md.set("trace-ctx-bin", []byte{0x01, 0xFF, 0x00, 0x7E})
와이어 위(HEADERS):  trace-ctx-bin: Af8Afg          (base64, 자동)
받는 쪽:            md.get("trace-ctx-bin") → []byte{0x01, 0xFF, 0x00, 0x7E}
```

- `{0x01, 0xFF, 0x00, 0x7E}`을 base64로: 바이트 `01 FF 00 7E` → 비트열 `0000_0001 1111_1111 0000_0000 0111_1110` → 6비트씩 끊으면 `000000 011111 111100 000000 011111 10`(마지막 2비트는 0으로 채워 `100000`) → 인덱스 `0, 31, 60, 0, 31, 32` → base64 알파벳에서 `A f 8 A f g` → `Af8Afg`.
- gRPC는 `-bin` 헤더를 **패딩 없는(unpadded) base64**로 와이어에 싣는다(Go는 `base64.RawStdEncoding`). 표준 패딩을 붙이면 `Af8Afg==`가 되지만, 와이어 스펙상 보내는 쪽은 패딩을 생략하고 받는 쪽은 패딩 유무를 모두 허용해야 한다.
- 받는 쪽 라이브러리는 키가 `-bin`으로 끝나니 base64로 디코딩해 원래 바이트를 복원한다.

규칙을 표로 정리한다.

| 규칙                 | 내용                                | 이유                                                                    |
| ------------------ | --------------------------------- | --------------------------------------------------------------------- |
| 키는 소문자 ASCII       | `Content-Type` 아님, `content-type` | HTTP/2는 헤더 이름이 소문자여야 함(대문자는 프로토콜 오류). 라이브러리가 소문자화하지만 직접 소문자로 다루는 게 안전 |
| 키 문자 집합            | `[a-z0-9-_.]`                     | HTTP 토큰 규칙 + gRPC 관례. 공백·콜론 등 금지                                      |
| `-bin` 접미사         | 값이 바이너리, base64 자동 인·디코딩          | 임의 바이트를 헤더에 안전하게                                                      |
| 같은 키 다중 값          | 허용(콤마 결합 또는 복수 엔트리)               | HTTP 헤더의 다중 값 의미와 동일                                                  |
| 값은 ASCII(비 `-bin`) | 출력 가능 ASCII 권장                    | HPACK 안전성                                                             |

`-bin`이 아닌 일반 키에 멀티바이트(한글 등)나 제어문자를 넣으면 라이브러리·프록시에 따라 깨지거나 거부된다. 사람이 읽을 라벨이라도 ASCII로 두고, 진짜 바이너리는 `-bin`을 써야 한다.

### 2.3 예약된 헤더: 건드리면 안 되는 영역

gRPC 와이어 프로토콜은 일부 헤더를 **자기가 관리**한다. 애플리케이션이 메타데이터로 같은 이름을 설정하려 하면 라이브러리가 무시하거나 에러를 낸다. 이들은 "시스템 봉투" 영역이라 사용자가 손대면 안 된다.

| 예약 헤더 | 누가 정함 | 의미 |
|-----------|----------|------|
| `:method` | HTTP/2 의사 헤더 | 항상 `POST` |
| `:path` | gRPC | `/{package}.{Service}/{Method}` |
| `:authority` | 전송 계층 | 대상 호스트(authority). [[07 - 채널 스텁 커넥션 생명주기]] |
| `:scheme` | 전송 계층 | `http`/`https` |
| `content-type` | gRPC | `application/grpc`(+`+proto`/`+json`) |
| `te` | gRPC | `trailers` (trailer 사용 협상) |
| `grpc-timeout` | gRPC | Deadline 전파. [[10 - Deadline 취소 타임아웃]] |
| `grpc-encoding` | gRPC | 메시지 압축 알고리즘(gzip 등) |
| `grpc-accept-encoding` | gRPC | 받을 수 있는 압축 |
| `grpc-status` | gRPC (trailer) | 최종 상태 코드. [[09 - 에러 모델 - 상태 코드와 Rich Error]] |
| `grpc-message` | gRPC (trailer) | 상태 메시지(퍼센트 인코딩) |
| `user-agent` | gRPC | 클라이언트 식별 |

규칙: **콜론(`:`)으로 시작하는 의사 헤더와 `grpc-`로 시작하는 헤더는 예약어다.** 사용자 정의 메타데이터는 이 접두를 피해야 한다. 관례적으로 `x-` 접두(예: `x-request-id`)나 표준화된 이름(`traceparent`, `authorization`)을 쓴다. `grpc-`로 시작하는 키를 set 하려 하면 대부분의 구현이 조용히 버리거나 거부한다.

`grpc-timeout`이 메타데이터처럼 보여도 사용자가 직접 만지지 않는다는 점이 중요하다. Deadline은 별도의 1급 개념으로, 컨텍스트/CallOption으로 설정하면 라이브러리가 `grpc-timeout: 30S` 같은 형태로 자동 변환한다. 자세한 인코딩(`100m`, `30S`, `1H` 등 단위)은 [[10 - Deadline 취소 타임아웃]]에서 다룬다.

## 3. Leading vs Trailing: 언제 무엇을 보낼 수 있는가

메타데이터의 "타이밍"은 스트리밍을 이해하면 자명해진다. 핵심 제약은 HTTP/2의 물리에서 온다: **HEADERS는 DATA보다 먼저 가야 하고, TRAILERS는 모든 DATA 뒤에 가야 한다.** 한 번 헤더를 흘려보내면(flush) 되돌릴 수 없다.

```text
서버가 응답에서 메타데이터를 보낼 수 있는 두 창(window):

   [응답 시작]                              [응답 끝]
       │                                       │
       ▼                                       ▼
   ┌───────┐  ┌──────┐ ┌──────┐ ┌──────┐  ┌────────┐
   │HEADERS│→ │ DATA │ │ DATA │ │ DATA │ →│TRAILERS│
   │leading│  │msg 1 │ │msg 2 │ │msg 3 │  │trailing│
   └───────┘  └──────┘ └──────┘ └──────┘  └────────┘
       ▲                                       ▲
   여기서 SetHeader/SendHeader               여기서 SetTrailer
   가능. 첫 메시지 전에만!                    응답 종료 직전.
```

규칙을 명시하면:

- **Leading metadata(응답 초기 헤더)**: 서버는 **첫 응답 메시지를 보내기 전**에만 leading 헤더를 보낼 수 있다. 첫 `DATA` 프레임이 나가는 순간 HEADERS는 이미 flush 됐으므로, 그 후에 leading을 추가하려 하면 무시되거나 에러다. Go에서 `grpc.SetHeader(ctx, md)`는 누적이고, `grpc.SendHeader(ctx, md)`는 즉시 flush다. 한 번 `SendHeader`(또는 첫 메시지 전송)가 일어나면 leading은 고정된다.
- **Trailing metadata(트레일러)**: 응답이 끝날 때 항상 보내진다. 서버 핸들러가 정상 반환하든 에러를 반환하든, 라이브러리가 `grpc-status`와 함께 trailer를 flush한다. 사용자가 `grpc.SetTrailer(ctx, md)`로 추가한 항목이 함께 실린다. **에러가 나도 trailer는 간다**는 점이 중요하다 — 실패 사유를 trailer에 실어 보낼 수 있다.

클라이언트 입장에서:

- **Request metadata(leading)**: 클라이언트는 요청을 *시작할 때* 메타데이터를 컨텍스트/CallOption에 실어 보낸다. 이건 요청 HEADERS 프레임에 들어간다. 양방향/클라이언트 스트리밍이라도, 요청 메타데이터는 첫 메시지 전에 고정된다(스트림 헤더는 한 번).
- 클라이언트는 응답의 leading 헤더와 trailer를 따로 받을 수 있다. Go에서는 `grpc.Header(&headerMD)`, `grpc.Trailer(&trailerMD)` CallOption으로 포인터를 넘겨 받는다.

### 3.1 스트리밍에서의 실전 타이밍

server-streaming을 예로 들자. 서버가 헤더에 "총 몇 건이 올지"를 미리 알려주고, 마지막에 "실제로 몇 건 보냈는지"를 trailer로 정정한다고 하자.

```go
func (s *server) ListOrders(req *pb.ListOrdersRequest, stream pb.OrderService_ListOrdersServer) error {
    // leading: 첫 메시지 전에 보내야 한다.
    header := metadata.Pairs("x-estimated-count", "100")
    if err := stream.SendHeader(header); err != nil { // 즉시 flush
        return err
    }

    sent := 0
    for _, o := range s.queryOrders(req) {
        if err := stream.Send(toProto(o)); err != nil { // 이제 DATA가 흐른다
            return err
        }
        sent++
    }

    // trailing: 모든 메시지 후, 반환 직전. SetTrailer는 누적.
    stream.SetTrailer(metadata.Pairs("x-actual-count", fmt.Sprint(sent)))
    return nil
}
```

이 코드의 순서를 어기면 — 예컨대 `stream.Send` 뒤에 `SendHeader`를 부르면 — leading은 이미 나갔으므로 효과가 없다. "헤더는 강이고 한 번 흘러가면 못 거슬러 올라간다"는 직관을 기억하자.

[[05 - 통신의 4가지 방식 - Unary와 Streaming]]에서 본 네 가지 호출 형태별로 메타데이터 창을 정리하면:

| 호출 형태 | 요청 leading | 응답 leading | 응답 trailing |
|-----------|-------------|-------------|--------------|
| Unary | 요청 헤더 1회 | 응답 헤더(본문 전) | 본문 후, 상태와 함께 |
| Server streaming | 요청 헤더 1회 | 첫 응답 메시지 전 | 스트림 끝 |
| Client streaming | 요청 헤더 1회(첫 메시지 전) | 단일 응답 전 | 응답 후 |
| Bidi streaming | 요청 헤더 1회(첫 메시지 전) | 첫 응답 메시지 전 | 스트림 끝 |

공통점: **메타데이터(헤더/트레일러)는 메시지 흐름의 양 끝에만 붙는다.** 메시지 사이사이에 끼워 넣을 수 없다. 메시지 단위로 부가 정보를 보내야 한다면 그건 메타데이터가 아니라 본문 메시지의 필드(예: oneof로 데이터 vs 메타 이벤트)로 모델링해야 한다.

## 4. 컨텍스트 전파: 메타데이터가 나르는 것들

메타데이터의 가장 실용적인 쓰임은 **분산 컨텍스트 전파(distributed context propagation)** 다. 마이크로서비스 환경에서 하나의 사용자 요청은 여러 서비스를 타고 흐른다. 그 흐름을 꿰는 정보 — 누가 호출했나, 이 요청의 추적 ID는 무엇인가, 어느 테넌트인가 — 를 서비스 경계를 넘어 실어 나르는 통로가 메타데이터다.

```text
   사용자
     │  HTTP: Authorization, traceparent
     ▼
┌──────────┐  gRPC metadata:           ┌──────────┐  gRPC metadata:   ┌──────────┐
│ API GW   │─ authorization ──────────→│ Order    │─ traceparent ────→│ Inventory│
│          │  x-request-id: abc123     │ Service  │  x-request-id     │ Service  │
│          │  traceparent: 00-...      │          │  (그대로 전파)     │          │
└──────────┘                           └──────────┘                   └──────────┘
```

대표적으로 메타데이터가 나르는 것들:

1. **인증/인가 토큰**: `authorization: Bearer eyJhbGci...`. OAuth2/JWT 토큰을 표준 `authorization` 헤더에 싣는다. 서버 인터셉터가 검증한다(§7.1).
2. **요청 ID / 상관 ID**: `x-request-id: 7f3a...`. 하나의 사용자 요청에 부여한 고유 ID. 모든 로그에 함께 찍어 여러 서비스의 로그를 한 줄로 꿰맨다.
3. **분산 트레이싱 컨텍스트**: W3C Trace Context의 `traceparent`/`tracestate`, 또는 B3(`x-b3-traceid` 등). OpenTelemetry가 이 헤더들을 메타데이터로 주입·추출해 스팬(span)을 잇는다(§7.4).
4. **테넌트/로캘/기능 플래그**: `x-tenant-id`, `accept-language`, `x-feature-flags`. 멀티테넌시·국제화·실험 컨텍스트.

여기서 자주 헷갈리는 것 하나: **Deadline은 메타데이터로 직접 전파하지 않는다.** Deadline은 1급 개념이라 라이브러리가 남은 시간을 계산해 `grpc-timeout` 헤더로 자동 변환·전파한다. 서비스 A가 30초 deadline으로 호출받고 B를 호출하면, A에 남은 시간(예: 28초)이 B로 전파된다. 이 "deadline 전파"는 메타데이터 메커니즘을 *내부적으로* 쓰지만 사용자는 컨텍스트만 넘기면 된다. 자세한 건 [[10 - Deadline 취소 타임아웃]].

W3C `traceparent`의 형식을 손으로 뜯어보면 메타데이터가 얼마나 압축적인 통로인지 보인다.

```text
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │  └──────────────┬───────────────┘ └───────┬──────┘ │
           버전              trace-id (16바이트)      span-id (8B)  flags
           (00)            전체 요청을 식별          현재 구간     (01=sampled)
```

`Order Service`의 트레이싱 인터셉터는 들어온 요청 메타데이터에서 `traceparent`를 **추출(extract)** 해 부모 스팬을 복원하고 자기 작업을 새 자식 스팬으로 만든 뒤, `Inventory Service`를 호출할 때 새 `span-id`를 담은 `traceparent`를 메타데이터에 **주입(inject)** 한다. 이 추출-주입 사이클을 모든 서비스가 반복하면, 분산된 호출이 하나의 트레이스 트리로 봉합된다. 그 추출·주입을 *자동으로* 해 주는 것이 바로 인터셉터다. 이제 주인공이 등장한다.

## 5. 인터셉터란 무엇인가: RPC를 감싸는 미들웨어

웹 프레임워크의 미들웨어를 써 봤다면 인터셉터는 친숙하다. 요청이 핸들러에 닿기 *전에*, 그리고 응답이 나가기 *후에* 끼어들어 공통 처리를 하는 함수. gRPC 인터셉터(interceptor)는 정확히 그것이다. **RPC 호출 경로를 양파 껍질처럼 감싸는 함수**다.

```text
        바깥                                           안
   ┌─────────────────────────────────────────────────────┐
   │ 인터셉터 A (로깅)                                     │
   │  ┌───────────────────────────────────────────────┐  │
   │  │ 인터셉터 B (인증)                              │  │
   │  │  ┌─────────────────────────────────────────┐  │  │
   │  │  │ 인터셉터 C (메트릭)                     │  │  │
   │  │  │  ┌───────────────────────────────────┐  │  │  │
   │  │  │  │  실제 핸들러 (비즈니스 로직)       │  │  │  │
   │  │  │  │  GetOrder(req) → resp             │  │  │  │
   │  │  │  └───────────────────────────────────┘  │  │  │
   │  │  └─────────────────────────────────────────┘  │  │
   │  └───────────────────────────────────────────────┘  │
   └─────────────────────────────────────────────────────┘

   호출:  A 진입 → B 진입 → C 진입 → 핸들러 → C 복귀 → B 복귀 → A 복귀
```

각 인터셉터는 두 가지 자유를 가진다.

1. **다음을 호출하기 전후에 코드를 끼운다.** 호출 전: 토큰 검증, 타이머 시작, 트레이스 스팬 시작. 호출 후: 응답 로깅, 메트릭 기록, 스팬 종료.
2. **다음을 호출할지 말지, 무엇을 넘길지 결정한다.** 인증 실패면 다음을 *부르지 않고* 즉시 에러 반환(short-circuit). 컨텍스트를 변형해(요청 ID 주입) 다음에 넘기기도 한다.

이 구조의 가치는 **횡단 관심사(cross-cutting concerns)의 분리**다. 인증·로깅·트레이싱은 모든 RPC에 공통으로 필요하지만 어떤 비즈니스 로직에도 속하지 않는다. 이걸 매 핸들러에 복붙하면 재앙이다. 인터셉터로 한 번 작성해 전체 RPC에 일괄 적용하면, 핸들러는 순수한 비즈니스 로직만 남는다.

### 5.1 네 종류의 인터셉터

gRPC는 **양쪽(client/server) × 두 형태(unary/stream) = 4종**의 인터셉터를 정의한다. unary와 stream을 나누는 이유는 둘의 시그니처가 근본적으로 다르기 때문이다. unary는 "요청 1개 → 응답 1개"라 입출력을 직접 잡을 수 있지만 stream은 메시지가 시간에 걸쳐 여러 개 흐르므로 "스트림 객체"를 감싸야 한다.

| 종류 | 위치 | 감싸는 대상 | 메시지 후킹 |
|------|------|------------|------------|
| Unary client | 클라이언트 | 단일 요청/응답 호출 | req/resp 직접 |
| Stream client | 클라이언트 | ClientStream 생성 | 스트림 wrap |
| Unary server | 서버 | 단일 핸들러 호출 | req/resp 직접 |
| Stream server | 서버 | ServerStream 핸들러 | 스트림 wrap |

```text
                  클라이언트 측                    서버 측
              ┌──────────────────┐          ┌──────────────────┐
   Unary      │ UnaryClient      │          │ UnaryServer      │
              │ Interceptor      │  ──RPC──→ │ Interceptor      │
              │ (req,resp 직접)  │          │ (req,resp 직접)  │
              └──────────────────┘          └──────────────────┘
              ┌──────────────────┐          ┌──────────────────┐
   Stream     │ StreamClient     │          │ StreamServer     │
              │ Interceptor      │  ──RPC──→ │ Interceptor      │
              │ (ClientStream    │          │ (ServerStream    │
              │  wrap)           │          │  wrap)           │
              └──────────────────┘          └──────────────────┘
```

## 6. 언어별 인터셉터의 형태

추상적 개념을 구체적 시그니처로 못 박아야 진짜 이해된다. Go, Java, Python 순으로 본다.

### 6.1 Go: 함수형 인터셉터

Go의 인터셉터는 함수다. 핵심은 `handler`(서버) 또는 `invoker`(클라이언트)라는 **"다음 단계"를 받아서 그걸 부르기 전후에 코드를 끼우는** 패턴이다.

```go
// 서버 unary 인터셉터의 시그니처
type UnaryServerInterceptor func(
    ctx context.Context,
    req any,
    info *UnaryServerInfo,    // FullMethod 이름 등 호출 정보
    handler UnaryHandler,     // "다음 단계" = 실제 핸들러(또는 다음 인터셉터)
) (resp any, err error)

// 서버 stream 인터셉터의 시그니처
type StreamServerInterceptor func(
    srv any,
    ss ServerStream,          // 감쌀 스트림
    info *StreamServerInfo,
    handler StreamHandler,    // "다음 단계"
) error
```

가장 단순한 로깅 인터셉터:

```go
func LoggingUnaryServer(
    ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler,
) (any, error) {
    start := time.Now()

    // ── 호출 전 (요청이 핸들러에 닿기 전) ──
    log.Printf("→ %s 시작", info.FullMethod)

    // ── 다음 단계 호출 (핵심!) ──
    resp, err := handler(ctx, req)

    // ── 호출 후 (응답이 나가기 전) ──
    code := status.Code(err)
    log.Printf("← %s 완료 code=%s dur=%s", info.FullMethod, code, time.Since(start))

    return resp, err
}
```

서버 등록:

```go
srv := grpc.NewServer(
    grpc.ChainUnaryInterceptor(   // 여러 개를 바깥→안 순서로
        TracingUnaryServer,       // 가장 바깥
        LoggingUnaryServer,
        AuthUnaryServer,
        RecoveryUnaryServer,      // 가장 안 (핸들러에 가장 가까움)
    ),
    grpc.ChainStreamInterceptor(
        TracingStreamServer,
        LoggingStreamServer,
        AuthStreamServer,
    ),
)
```

클라이언트 측은 대칭이다.

```go
type UnaryClientInterceptor func(
    ctx context.Context,
    method string,
    req, reply any,
    cc *ClientConn,
    invoker UnaryInvoker,         // "다음 단계" = 실제 전송
    opts ...CallOption,
) error

func RequestIDUnaryClient(
    ctx context.Context, method string, req, reply any,
    cc *grpc.ClientConn, invoker grpc.UnaryInvoker, opts ...grpc.CallOption,
) error {
    // 컨텍스트에 메타데이터를 실어 다음으로 넘긴다.
    reqID := uuid.NewString()
    ctx = metadata.AppendToOutgoingContext(ctx, "x-request-id", reqID)
    return invoker(ctx, method, req, reply, cc, opts...) // 다음 단계
}

conn, _ := grpc.NewClient(target,
    grpc.WithChainUnaryInterceptor(RequestIDUnaryClient, RetryUnaryClient),
)
```

### 6.2 Go에서 stream 인터셉터로 메시지 단위 후킹

stream 인터셉터는 메시지가 여러 개 흐르므로 `ServerStream`/`ClientStream`을 **래핑(wrap)**해 `SendMsg`/`RecvMsg`를 가로채야 한다. 이것이 unary와 결정적으로 다른 점이다.

```go
// ServerStream을 감싸 메시지마다 후킹
type wrappedServerStream struct {
    grpc.ServerStream
    onRecv func(any)
    onSend func(any)
}

func (w *wrappedServerStream) RecvMsg(m any) error {
    err := w.ServerStream.RecvMsg(m) // 실제 수신
    if err == nil {
        w.onRecv(m)                  // 메시지 1개마다 후킹
    }
    return err
}

func (w *wrappedServerStream) SendMsg(m any) error {
    w.onSend(m)
    return w.ServerStream.SendMsg(m)
}

func CountingStreamServer(
    srv any, ss grpc.ServerStream,
    info *grpc.StreamServerInfo, handler grpc.StreamHandler,
) error {
    var in, out int
    wrapped := &wrappedServerStream{
        ServerStream: ss,
        onRecv: func(any) { in++ },
        onSend: func(any) { out++ },
    }
    err := handler(srv, wrapped) // 래핑한 스트림을 핸들러에 넘김
    log.Printf("%s in=%d out=%d", info.FullMethod, in, out)
    return err
}
```

여기서 **비용**에 주목하자. stream 인터셉터로 메시지마다 후킹하면, 초당 수천 건의 메시지가 흐르는 스트림에서 매 메시지마다 함수 호출·클로저·로깅이 끼어든다. unary는 호출당 한 번이지만 stream의 메시지 후킹은 메시지당 한 번이다(§9.2에서 재론).

### 6.3 Java: 인터페이스 기반 인터셉터

Java는 함수 대신 인터페이스를 구현한다. 메타데이터를 다루는 `Metadata` 클래스와 컨텍스트를 다루는 `Context` 클래스가 명시적으로 등장한다.

```java
// 서버 인터셉터
public class AuthServerInterceptor implements ServerInterceptor {
    static final Metadata.Key<String> AUTH_KEY =
        Metadata.Key.of("authorization", Metadata.ASCII_STRING_MARSHALLER);
    static final Context.Key<String> USER_CTX = Context.key("userId");

    @Override
    public <ReqT, RespT> ServerCall.Listener<ReqT> interceptCall(
            ServerCall<ReqT, RespT> call,
            Metadata headers,                 // 들어온 요청 메타데이터
            ServerCallHandler<ReqT, RespT> next) {

        String token = headers.get(AUTH_KEY); // 봉투에서 토큰 꺼내기
        String userId = verify(token);
        if (userId == null) {
            call.close(Status.UNAUTHENTICATED.withDescription("토큰 없음/무효"),
                       new Metadata());
            return new ServerCall.Listener<ReqT>() {}; // no-op 리스너, 핸들러 미호출
        }
        // 검증된 userId를 Context에 실어 다음으로
        Context ctx = Context.current().withValue(USER_CTX, userId);
        return Contexts.interceptCall(ctx, call, headers, next);
    }
}
```

바이너리 메타데이터를 다룰 땐 `Metadata.Key.of("trace-bin", Metadata.BINARY_BYTE_MARSHALLER)`처럼 키 마샬러를 바꾼다. 이름이 `-bin`으로 끝나야 한다.

클라이언트 측은 `ClientInterceptor`를 구현하고, `ForwardingClientCall`로 호출을 감싸 메타데이터를 주입한다.

```java
public class RequestIdClientInterceptor implements ClientInterceptor {
    static final Metadata.Key<String> REQ_ID =
        Metadata.Key.of("x-request-id", Metadata.ASCII_STRING_MARSHALLER);

    @Override
    public <ReqT, RespT> ClientCall<ReqT, RespT> interceptCall(
            MethodDescriptor<ReqT, RespT> method, CallOptions opts, Channel next) {
        return new SimpleForwardingClientCall<ReqT, RespT>(
                next.newCall(method, opts)) {
            @Override
            public void start(Listener<RespT> responseListener, Metadata headers) {
                headers.put(REQ_ID, UUID.randomUUID().toString()); // 봉투에 우표 붙이기
                super.start(responseListener, headers);
            }
        };
    }
}
```

Java의 stream 후킹은 `ForwardingServerCallListener`/`ForwardingClientCall`로 `onMessage`/`sendMessage`를 오버라이드해 한다. 개념은 Go의 wrappedStream과 같다.

### 6.4 Python: 추상 클래스 인터셉터

Python(`grpcio`)도 인터셉터를 제공한다. 서버는 `grpc.ServerInterceptor`, 클라이언트는 `grpc.UnaryUnaryClientInterceptor` 등 호출 형태별 클래스를 구현한다.

```python
import grpc, uuid

class RequestIdClientInterceptor(grpc.UnaryUnaryClientInterceptor):
    def intercept_unary_unary(self, continuation, client_call_details, request):
        # 기존 메타데이터에 우표를 덧붙인다 (튜플 리스트)
        md = list(client_call_details.metadata or [])
        md.append(("x-request-id", str(uuid.uuid4())))
        new_details = client_call_details._replace(metadata=md)
        return continuation(new_details, request)  # 다음 단계

channel = grpc.intercept_channel(
    grpc.insecure_channel("localhost:50051"),
    RequestIdClientInterceptor(),
)
```

서버 측:

```python
class AuthServerInterceptor(grpc.ServerInterceptor):
    def intercept_service(self, continuation, handler_call_details):
        md = dict(handler_call_details.invocation_metadata)
        token = md.get("authorization", "")
        if not verify(token):
            def deny(request, context):
                context.abort(grpc.StatusCode.UNAUTHENTICATED, "토큰 무효")
            return grpc.unary_unary_rpc_method_handler(deny)
        return continuation(handler_call_details)  # 통과
```

세 언어를 관통하는 공통 골격:

```text
1. "다음 단계"를 인자로 받는다 (handler / invoker / continuation / next)
2. 다음 단계 호출 전: 메타데이터/컨텍스트를 읽거나 쓴다
3. 다음 단계를 호출한다 (또는 호출하지 않고 short-circuit)
4. 다음 단계 호출 후: 결과를 가공/관찰한다
```

## 7. 대표 활용: 인터셉터로 무엇을 하는가

### 7.1 인증/인가 (서버 인터셉터)

가장 흔한 용도. 모든 RPC가 핸들러에 닿기 전에 토큰을 검증한다. 실패하면 핸들러를 *부르지 않고* `UNAUTHENTICATED`/`PERMISSION_DENIED`로 차단한다.

```go
func AuthUnaryServer(
    ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler,
) (any, error) {
    // 공개 메서드는 통과 (예: 헬스체크)
    if info.FullMethod == "/grpc.health.v1.Health/Check" {
        return handler(ctx, req)
    }
    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "메타데이터 없음")
    }
    auth := md.Get("authorization") // []string
    if len(auth) == 0 || !strings.HasPrefix(auth[0], "Bearer ") {
        return nil, status.Error(codes.Unauthenticated, "Bearer 토큰 필요")
    }
    claims, err := verifyJWT(strings.TrimPrefix(auth[0], "Bearer "))
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "토큰 검증 실패")
    }
    // 검증된 신원을 컨텍스트에 실어 핸들러로
    ctx = context.WithValue(ctx, userKey{}, claims.Subject)
    return handler(ctx, req)
}
```

`UNAUTHENTICATED`(인증 안 됨, 코드 16)와 `PERMISSION_DENIED`(인증은 됐으나 권한 없음, 코드 7)의 구분은 [[09 - 에러 모델 - 상태 코드와 Rich Error]]를 따른다. 실제 토큰 전송은 TLS 위에서 해야 안전하다 — 평문 연결에 Bearer 토큰을 실으면 도청된다([[11 - 보안 - TLS mTLS 인증]]).

### 7.2 로깅 (서버/클라이언트)

호출의 메서드·코드·소요시간·요청 ID를 구조화 로그로 남긴다. §6.1의 `LoggingUnaryServer`가 기본형이다. 핵심은 **메타데이터의 `x-request-id`를 로그에 함께 찍어** 여러 서비스의 로그를 상관(correlate)시키는 일이다.

```go
func LoggingUnaryServer(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    reqID := firstOrEmpty(metadataGet(ctx, "x-request-id"))
    start := time.Now()
    resp, err := handler(ctx, req)
    log.Printf(`{"method":%q,"code":%q,"dur_ms":%d,"req_id":%q}`,
        info.FullMethod, status.Code(err), time.Since(start).Milliseconds(), reqID)
    return resp, err
}
```

### 7.3 메트릭 (서버/클라이언트)

호출 수·지연·에러율을 Prometheus 등으로 집계. 보통 메서드명과 상태 코드를 라벨로 단다.

```go
var rpcDuration = prometheus.NewHistogramVec(
    prometheus.HistogramOpts{Name: "grpc_server_handling_seconds"},
    []string{"method", "code"},
)

func MetricsUnaryServer(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    start := time.Now()
    resp, err := handler(ctx, req)
    rpcDuration.WithLabelValues(info.FullMethod, status.Code(err).String()).
        Observe(time.Since(start).Seconds())
    return resp, err
}
```

라벨 카디널리티 주의: `FullMethod`는 유한하니 안전하지만 `x-request-id` 같은 무한 값은 절대 라벨로 쓰면 안 된다(메트릭 폭발). 관찰성 전반은 [[14 - 관찰성과 디버깅 - Reflection grpcurl]].

### 7.4 분산 트레이싱 컨텍스트 주입·추출

§4에서 본 `traceparent` 추출-주입을 인터셉터가 자동화한다. 서버 인터셉터는 들어온 메타데이터에서 부모 컨텍스트를 추출하고 클라이언트 인터셉터는 나가는 메타데이터에 현재 컨텍스트를 주입한다.

```go
// 서버: 들어온 메타데이터에서 trace context 추출 → 자식 스팬 시작
func TracingUnaryServer(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    md, _ := metadata.FromIncomingContext(ctx)
    parent := propagator.Extract(ctx, MetadataCarrier(md)) // traceparent 읽기
    ctx, span := tracer.Start(parent, info.FullMethod)
    defer span.End()
    resp, err := handler(ctx, req)
    if err != nil {
        span.RecordError(err)
    }
    return resp, err
}

// 클라이언트: 현재 스팬을 나가는 메타데이터에 주입
func TracingUnaryClient(ctx context.Context, method string, req, reply any,
    cc *grpc.ClientConn, invoker grpc.UnaryInvoker, opts ...grpc.CallOption) error {
    md, _ := metadata.FromOutgoingContext(ctx)
    md = md.Copy()
    propagator.Inject(ctx, MetadataCarrier(md)) // traceparent 쓰기
    ctx = metadata.NewOutgoingContext(ctx, md)
    return invoker(ctx, method, req, reply, cc, opts...)
}
```

실무에서는 이걸 직접 쓰기보다 OpenTelemetry의 `otelgrpc` 같은 표준 인터셉터를 등록한다. 직접 만들 때의 교훈은 **추출은 서버 인터셉터, 주입은 클라이언트 인터셉터**라는 대칭성이다.

### 7.5 패닉 복구 (서버 인터셉터)

핸들러가 패닉(Go), 예외(Java/Python)를 던지면 기본적으로 연결이나 서버가 위태로워질 수 있다. 복구 인터셉터는 패닉을 잡아 `INTERNAL` 에러로 변환해 클라이언트에 깔끔히 돌려준다. **가장 안쪽(핸들러에 가장 가까운)에 두어야** 핸들러 패닉을 바로 잡는다.

```go
func RecoveryUnaryServer(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (resp any, err error) {
    defer func() {
        if r := recover(); r != nil {
            stack := debug.Stack()
            log.Printf("PANIC %s: %v\n%s", info.FullMethod, r, stack)
            err = status.Errorf(codes.Internal, "내부 오류") // 스택은 로그에만!
        }
    }()
    return handler(ctx, req)
}
```

보안상 중요: 패닉 메시지·스택트레이스를 클라이언트 에러에 노출하지 말 것. 내부 구현을 누설한다. 로그에만 남기고 클라이언트엔 일반적 `INTERNAL`을 준다.

### 7.6 요청 검증 (서버 인터셉터)

protobuf 메시지가 `protoc-gen-validate` 같은 도구로 `Validate()` 메서드가 있으면 인터셉터에서 일괄 검증해 핸들러 진입 전에 `INVALID_ARGUMENT`로 거른다.

```go
type validator interface{ Validate() error }

func ValidateUnaryServer(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    if v, ok := req.(validator); ok {
        if err := v.Validate(); err != nil {
            return nil, status.Errorf(codes.InvalidArgument, "검증 실패: %v", err)
        }
    }
    return handler(ctx, req)
}
```

### 7.7 재시도·서킷브레이커 (클라이언트 인터셉터)

클라이언트 인터셉터에서 `invoker`를 여러 번 부르면 재시도가 된다. 다만 단순 재시도는 멱등성(idempotency)을 깨뜨릴 수 있고 gRPC는 내장 retry 정책(서비스 설정 기반)을 별도로 제공한다. 인터셉터 레벨 재시도는 세밀한 제어가 필요할 때만. 이 주제는 [[13 - 안정성 - Retry Health Check Keepalive]]에서 본격적으로 다루므로 여기서는 "클라이언트 인터셉터가 invoker를 감싸 재시도/차단 로직을 넣을 수 있다"는 점만 짚는다.

```go
func RetryUnaryClient(ctx context.Context, method string, req, reply any,
    cc *grpc.ClientConn, invoker grpc.UnaryInvoker, opts ...grpc.CallOption) error {
    var err error
    for attempt := 0; attempt < 3; attempt++ {
        err = invoker(ctx, method, req, reply, cc, opts...)
        if status.Code(err) != codes.Unavailable { // 재시도 가능 코드만
            return err
        }
        time.Sleep(backoff(attempt))
    }
    return err
}
```

## 8. 체이닝과 순서: 양파의 결을 다듬기

### 8.1 체이닝 라이브러리

표준 gRPC는 `ChainUnaryInterceptor`/`ChainStreamInterceptor`(Go), 여러 ServerInterceptor 등록(Java)으로 체이닝을 지원한다. 생태계에는 `go-grpc-middleware`처럼 자주 쓰는 인터셉터 모음(로깅, 인증, 복구, 트레이싱, 메트릭, 검증)과 깔끔한 체이닝 API를 제공하는 라이브러리가 있다.

```go
import (
    "github.com/grpc-ecosystem/go-grpc-middleware/v2/interceptors/recovery"
    "github.com/grpc-ecosystem/go-grpc-middleware/v2/interceptors/logging"
)

srv := grpc.NewServer(
    grpc.ChainUnaryInterceptor(
        TracingUnaryServer,                 // 1. 가장 바깥
        logging.UnaryServerInterceptor(lg), // 2. 로깅
        AuthUnaryServer,                    // 3. 인증
        ValidateUnaryServer,                // 4. 검증
        recovery.UnaryServerInterceptor(),  // 5. 가장 안 (핸들러 직전)
    ),
)
```

### 8.2 등록 순서가 의미를 바꾼다

체인의 순서는 단순한 취향이 아니라 **정확성에 영향을 준다.** 양파의 바깥에 둔 인터셉터가 먼저 진입하고 나중에 복귀한다.

```text
등록 순서:  [Tracing, Logging, Auth, Validate, Recovery]

진입(요청):  Tracing → Logging → Auth → Validate → Recovery → 핸들러
복귀(응답):  Tracing ← Logging ← Auth ← Validate ← Recovery ← 핸들러
```

왜 이 순서인가? 각 위치의 논리를 따져보자.

| 순서 | 인터셉터 | 왜 이 자리인가 |
|------|---------|--------------|
| 1 (최외곽) | Tracing | 모든 처리(인증 실패 포함)를 스팬에 담으려면 가장 바깥. 인증 실패도 트레이스에 보여야 함 |
| 2 | Logging | 트레이스 ID가 이미 컨텍스트에 있어야 로그에 같이 찍힌다. 그래서 Tracing보다 안쪽 |
| 3 | Auth | 검증·핸들러보다 먼저 차단해 불필요한 작업 방지 |
| 4 | Validate | 인증된 요청만 검증. 검증 비용을 인증 통과자에게만 |
| 5 (최내곽) | Recovery | 핸들러 패닉을 즉시 잡으려면 핸들러에 가장 가깝게 |

**대표적 함정**: 트레이싱을 로깅보다 *안쪽*에 두면, 로깅 인터셉터가 실행될 때 아직 trace 컨텍스트가 없어서 로그에 trace ID가 안 찍힌다. 그러면 로그와 트레이스를 상관시킬 수 없다. 그래서 "트레이싱이 로깅보다 바깥"이 정석이다.

또 다른 함정: Recovery를 *바깥쪽*에 두면, 안쪽 인터셉터(예: 메트릭)가 핸들러 패닉의 영향을 받은 뒤에야 복구가 일어난다. 메트릭이 패닉으로 기록 안 될 수 있다. Recovery는 가장 안쪽이 안전하다(단, 인터셉터 자체의 버그까지 잡으려면 별도 고려가 필요).

클라이언트 측도 동일 원리다. 예컨대 재시도 인터셉터를 트레이싱보다 바깥에 두면 재시도 전체가 하나의 스팬에 묶이고 안쪽에 두면 각 시도가 별도 스팬이 된다 — 어느 쪽을 원하는지에 따라 순서를 정한다.

## 9. 함정과 주의점

### 9.1 메타데이터·컨텍스트의 불변성

인터셉터에서 메타데이터를 변형할 때 **공유 상태를 직접 수정하면 안 된다.** Go의 `metadata.MD`는 맵이라 여러 고루틴이 공유할 수 있다. 변형 전 `md.Copy()`로 복사해야 안전하다.

```go
// 위험: 원본 메타데이터를 직접 수정 (다른 RPC와 공유될 수 있음)
md, _ := metadata.FromOutgoingContext(ctx)
md.Set("x-tenant", tenant)   // ✗ 공유 맵 변형

// 안전: 복사 후 수정
md, _ := metadata.FromOutgoingContext(ctx)
md = md.Copy()               // ✓
md.Set("x-tenant", tenant)
ctx = metadata.NewOutgoingContext(ctx, md)
```

Java의 `Context`는 불변(immutable)이라 `withValue`가 항상 새 컨텍스트를 만든다 — 언어가 강제로 안전을 보장하는 좋은 예. Go의 `context.Context`도 불변이지만 `metadata.MD`(맵)는 가변이라 손이 가는 부분이다. 더 깊은 컨텍스트 전파·취소 의미는 [[10 - Deadline 취소 타임아웃]].

또한 incoming과 outgoing 메타데이터를 헷갈리지 말 것. Go에서 서버가 받은 건 `FromIncomingContext`, 클라이언트가 보낼 건 `FromOutgoingContext`/`AppendToOutgoingContext`다. 서버에서 받은 incoming 메타데이터를 **자동으로** 다음 호출의 outgoing으로 전파해 주지 않는다 — 컨텍스트 전파를 원하면 인터셉터에서 명시적으로 복사해 넘겨야 한다.

```go
// 서버 인터셉터에서 incoming의 request-id를 outgoing으로 잇기
func PropagateUnaryServer(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    if md, ok := metadata.FromIncomingContext(ctx); ok {
        if ids := md.Get("x-request-id"); len(ids) > 0 {
            ctx = metadata.AppendToOutgoingContext(ctx, "x-request-id", ids[0])
        }
    }
    return handler(ctx, req)
}
```

### 9.2 스트리밍 메시지 가로채기 비용

§6.2에서 짚었듯, stream 인터셉터로 매 메시지를 후킹하면 비용이 메시지 수에 비례한다. 양방향 스트림에서 초당 1만 메시지를 후킹하면 매 메시지마다 클로저 호출·로깅·복사가 일어난다. 권장: 메시지 단위 후킹은 정말 필요할 때만, 그리고 후킹 함수를 극도로 가볍게(카운터 증가 정도) 유지하라. 메시지 내용 전체를 매번 로깅하면 처리량이 폭락한다. 흐름 제어·백프레셔 관점의 추가 논의는 [[15 - 성능과 Flow Control 백프레셔]].

### 9.3 메타데이터 크기 폭증과 HPACK 동적 테이블 압박

메타데이터는 공짜가 아니다. §2.1에서 봤듯 HPACK 동적 테이블에 들어가 압축되지만 **테이블 크기는 유한**(기본 4KB 권장)하다. 큰 메타데이터를 매 요청 다르게 보내면(예: 매번 다른 거대 JWT, 매번 다른 base64 페이로드) 동적 테이블이 계속 갱신·축출(eviction)되어 압축 효율이 떨어지고 매 요청 헤더 바이트가 커진다.

```text
좋은 패턴: 작고 반복적인 메타데이터
  x-request-id: 16바이트 UUID   → 매번 다르지만 짧음 (OK)
  authorization: 같은 토큰 반복  → 동적 테이블 1회 등록 후 인덱스 참조 (최적)

나쁜 패턴: 크고 매번 다른 메타데이터
  x-debug-context-bin: 8KB base64 → 동적 테이블 초과, 매번 풀 전송 (HPACK 무력화)
  x-trace-blob: 매 요청 다른 4KB  → 테이블 축출 유발, 압축 효율 붕괴
```

또한 일부 게이트웨이·프록시는 헤더 총량에 상한(예: 8KB, 16KB)을 둔다. 메타데이터가 이를 넘으면 RPC가 거부(`RESOURCE_EXHAUSTED`나 연결 거부)된다. 큰 데이터는 메타데이터가 아니라 **본문 메시지**로 보내야 한다. "봉투에 책 한 권을 넣지 마라" — 봉투는 우표를 붙이는 곳이지 내용을 담는 곳이 아니다.

### 9.4 예약 영역 침범과 대소문자

§2.3에서 봤듯 `grpc-`/`:` 접두 키는 라이브러리가 관리한다. 사용자 키를 `grpc-my-thing`으로 지으면 일부 구현이 버린다. 또 HTTP/2는 헤더 이름이 반드시 소문자여야 하므로, 대문자 키를 set하면 라이브러리가 소문자화하거나(대부분) 오류를 낸다(일부). 항상 소문자로 다루는 습관을 들이자.

### 9.5 trailer-only 응답과 에러 시 메타데이터

서버가 즉시 에러로 거부하면(예: 인증 실패) 응답 DATA가 없고 trailer만 가는 **trailer-only 응답**이 된다. 이때 클라이언트의 `Header()` CallOption은 빈 헤더를 받고 에러 상세는 trailer(및 status)에서 온다. 인터셉터로 추가한 trailing 메타데이터는 이 경우에도 전달되므로, 실패 컨텍스트(예: `x-rate-limit-remaining`)를 trailer에 실어 보낼 수 있다.

## 10. 종합 예제: 요청 ID 부여 + Bearer 토큰 검증

이제 배운 것을 묶어 client/server 양쪽 인터셉터를 일반적인 "주문 서비스"로 구현한다. 시나리오: 클라이언트는 모든 요청에 고유 `x-request-id`를 부여하고 `authorization: Bearer <token>`을 싣는다. 서버는 토큰을 검증하고 요청 ID를 로그에 찍으며 응답 trailer에 처리 시간을 실어 보낸다.

### 10.1 proto 정의

```proto
syntax = "proto3";
package order.v1;
option go_package = "example.com/order/v1;orderv1";

service OrderService {
  rpc GetOrder(GetOrderRequest) returns (GetOrderResponse);
}

message GetOrderRequest  { int64 order_id = 1; }
message GetOrderResponse {
  int64  order_id = 1;
  string status   = 2;
  int64  amount   = 3;
}
```

[[06 - 코드 생성과 protoc 툴체인]]의 절차로 `protoc`/`buf`를 돌려 `orderv1` 패키지의 스텁을 생성했다고 가정한다.

### 10.2 클라이언트 인터셉터 (요청 ID + 토큰 주입)

```go
package main

import (
    "context"

    "github.com/google/uuid"
    "google.golang.org/grpc"
    "google.golang.org/grpc/metadata"
)

// 모든 호출에 x-request-id와 authorization을 주입하는 클라이언트 인터셉터
func clientAuthAndReqID(token string) grpc.UnaryClientInterceptor {
    return func(ctx context.Context, method string, req, reply any,
        cc *grpc.ClientConn, invoker grpc.UnaryInvoker, opts ...grpc.CallOption) error {

        reqID := uuid.NewString()
        // outgoing 컨텍스트에 메타데이터 덧붙이기 (봉투에 우표)
        ctx = metadata.AppendToOutgoingContext(ctx,
            "x-request-id", reqID,
            "authorization", "Bearer "+token,
        )

        // 응답 헤더/트레일러를 받아볼 포인터
        var header, trailer metadata.MD
        opts = append(opts, grpc.Header(&header), grpc.Trailer(&trailer))

        err := invoker(ctx, method, req, reply, cc, opts...) // 다음 단계

        // 서버가 trailer에 실어 보낸 처리 시간 읽기
        if dur := trailer.Get("x-processing-ms"); len(dur) > 0 {
            // 관찰: 서버 처리 시간 = dur[0] ms, req_id = reqID
        }
        return err
    }
}

func dial(target, token string) (*grpc.ClientConn, error) {
    return grpc.NewClient(target,
        grpc.WithTransportCredentials(loadTLS()), // 토큰은 TLS 위에서만!
        grpc.WithChainUnaryInterceptor(clientAuthAndReqID(token)),
    )
}
```

`metadata.AppendToOutgoingContext`는 기존 메타데이터를 보존하면서 덧붙이므로, 여러 인터셉터가 각자 키를 더해도 충돌하지 않는다. 토큰을 실어 보내므로 반드시 `WithTransportCredentials`(TLS)를 쓴다 — [[11 - 보안 - TLS mTLS 인증]].

### 10.3 서버 인터셉터 (토큰 검증 + 로깅 + trailer)

```go
package main

import (
    "context"
    "fmt"
    "log"
    "strings"
    "time"

    "google.golang.org/grpc"
    "google.golang.org/grpc/codes"
    "google.golang.org/grpc/metadata"
    "google.golang.org/grpc/status"
)

type ctxKey string
const userIDKey ctxKey = "userID"

// 1) 인증 인터셉터 (안쪽: 검증 통과한 요청만 다음으로)
func serverAuth(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {

    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "메타데이터 없음")
    }
    auths := md.Get("authorization")
    if len(auths) == 0 || !strings.HasPrefix(auths[0], "Bearer ") {
        return nil, status.Error(codes.Unauthenticated, "Bearer 토큰 필요")
    }
    userID, err := verifyToken(strings.TrimPrefix(auths[0], "Bearer "))
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "토큰 검증 실패")
    }
    ctx = context.WithValue(ctx, userIDKey, userID) // 신원을 핸들러로
    return handler(ctx, req)
}

// 2) 로깅 + trailer 인터셉터 (바깥: 모든 결과를 관찰)
func serverLogging(ctx context.Context, req any,
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {

    start := time.Now()
    reqID := first(metadata.ValueFromIncomingContext(ctx, "x-request-id"))

    resp, err := handler(ctx, req)

    dur := time.Since(start)
    // 응답 trailer에 처리 시간을 실어 보낸다
    _ = grpc.SetTrailer(ctx, metadata.Pairs(
        "x-processing-ms", fmt.Sprint(dur.Milliseconds())))

    log.Printf(`{"method":%q,"code":%q,"dur_ms":%d,"req_id":%q}`,
        info.FullMethod, status.Code(err), dur.Milliseconds(), reqID)
    return resp, err
}

func first(ss []string) string { if len(ss) > 0 { return ss[0] }; return "" }

func newServer() *grpc.Server {
    return grpc.NewServer(
        grpc.ChainUnaryInterceptor(
            serverLogging, // 바깥: 인증 실패도 로깅하고 trailer를 단다
            serverAuth,    // 안쪽: 통과한 요청만 핸들러로
        ),
    )
}
```

### 10.4 흐름 추적: 봉투 한 장의 일생

이 예제에서 한 번의 `GetOrder` 호출이 메타데이터 관점에서 어떻게 흐르는지 ASCII로 추적한다.

```text
클라이언트                                                     서버
   │                                                            │
 [clientAuthAndReqID 진입]                                      │
   │ ctx += x-request-id: 7f3a..., authorization: Bearer eyJ... │
   │                                                            │
   │── HEADERS ───────────────────────────────────────────────→│
   │   :path /order.v1.OrderService/GetOrder                    │
   │   content-type application/grpc                            │
   │   x-request-id 7f3a...                                     │
   │   authorization Bearer eyJ...                              │
   │── DATA (GetOrderRequest{order_id:42}) END_STREAM ─────────→│
   │                                                          [serverLogging 진입] start=t0
   │                                                          [serverAuth 진입]
   │                                                            │ md.Get("authorization") OK
   │                                                            │ verifyToken → userID=u_99
   │                                                          [핸들러 GetOrder 실행]
   │                                                            │ resp = {42, "PAID", 12000}
   │                                                          [serverAuth 복귀]
   │                                                          [serverLogging 복귀]
   │                                                            │ SetTrailer x-processing-ms: 7
   │                                                            │ log{method,code=OK,req_id=7f3a}
   │←─ HEADERS (응답 leading) :status 200 ─────────────────────│
   │←─ DATA (GetOrderResponse{42,"PAID",12000}) ───────────────│
   │←─ HEADERS (trailing) grpc-status 0 ───────────────────────│
   │   x-processing-ms 7  END_STREAM                            │
 [clientAuthAndReqID 복귀]                                      │
   │ trailer.Get("x-processing-ms") = ["7"]                     │
   ▼                                                            ▼
```

이 다이어그램이 이 장 전체를 압축한다. 봉투(메타데이터)는 HEADERS와 TRAILERS에 실리고 인터셉터는 그 봉투를 양쪽 끝에서 읽고 쓰며 본문(DATA)은 그 사이를 흐른다. 인증은 본문을 건드리지 않고 봉투만으로 통과/거부를 결정하며 로깅은 봉투의 request-id로 분산 로그를 꿰고 trailer는 응답이 끝난 뒤에 처리 통계를 덧붙인다.

### 10.5 grpcurl로 메타데이터 직접 보내기

서버를 띄워두고 [[14 - 관찰성과 디버깅 - Reflection grpcurl]]의 도구로 메타데이터를 직접 실어 호출해 보면 개념이 손에 잡힌다.

```bash
# -H 로 메타데이터(헤더)를 직접 주입한다
grpcurl -plaintext \
  -H 'x-request-id: 7f3a-test-0001' \
  -H 'authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1Xzk5In0.sig' \
  -d '{"order_id": 42}' \
  localhost:50051 order.v1.OrderService/GetOrder

# 응답 헤더/트레일러까지 보려면 -v
grpcurl -plaintext -v \
  -H 'authorization: Bearer <token>' \
  -d '{"order_id": 42}' \
  localhost:50051 order.v1.OrderService/GetOrder
# Response trailers received:
#   x-processing-ms: 7
```

`-v`로 보면 `Response headers received`(leading)와 `Response trailers received`(trailing)가 분리되어 나온다 — §3에서 본 두 창이 실제로 눈에 보인다. 토큰 없이 호출하면 `Code: Unauthenticated`로 거부되는 것도 확인할 수 있다.

## 핵심 요약

- **메타데이터 = 봉투의 우표.** RPC 메시지 본문(protobuf)과 별개의 key-value 목록으로, 인증·추적·요청 ID 같은 "어떻게 처리할지"를 나른다. 본문과 분리해 미들웨어가 비즈니스 타입에 결합하지 않게 한다.
- **물리적 실체는 HTTP/2 HEADERS·TRAILERS.** 요청 헤더(leading)·응답 초기 헤더(leading)·응답 종료(trailing) 세 자리에 실린다. gRPC의 진짜 결과 코드 `grpc-status`는 본문이 아니라 **trailer**에 온다 — 이것이 gRPC가 HTTP/2를 요구하는 핵심 이유 중 하나.
- **키 규칙**: 소문자 ASCII, `-bin` 접미사면 바이너리(base64 자동), `grpc-`/`:` 접두는 예약(`:path`, `content-type`, `grpc-timeout`, `grpc-status` 등). 사용자 키는 `x-`나 표준 이름을 쓴다. Deadline은 메타데이터가 아니라 별도 1급 개념으로 `grpc-timeout`에 자동 변환된다.
- **타이밍**: leading은 첫 메시지 전에만, trailing은 응답 끝에 항상(에러 시에도). 메시지 사이엔 못 낀다 — "헤더는 한 번 흐르면 못 거슬러 올라간다".
- **인터셉터 = RPC를 감싸는 양파 껍질 미들웨어.** client/server × unary/stream 4종. "다음 단계(handler/invoker/continuation)"를 받아 전후에 코드를 끼우거나 short-circuit 한다. unary는 req/resp 직접, stream은 스트림을 래핑해 메시지 단위 후킹.
- **등록 순서가 정확성을 좌우한다.** 바깥이 먼저 진입·나중에 복귀. 트레이싱은 로깅보다 바깥(로그에 trace ID를 담기 위해), 복구는 핸들러에 가장 가깝게(패닉 즉시 포착).
- **함정**: 공유 메타데이터 직접 변형 금지(복사 후 수정), 스트림 메시지 후킹 비용은 메시지 수에 비례, 큰/매번 다른 메타데이터는 HPACK 동적 테이블을 무력화하고 프록시 헤더 상한을 초과시킨다. 큰 데이터는 본문으로.
- **대표 활용**: 인증/인가, 로깅(+request-id 상관), 메트릭, 트레이싱(서버 추출·클라이언트 주입), 패닉 복구, 요청 검증, 재시도/서킷브레이커. 모두 핸들러를 순수하게 유지하기 위한 횡단 관심사 분리.
