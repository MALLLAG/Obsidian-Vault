**이 장이 답하는 질문:**

- 코틀린은 정말 "Java에 설탕을 뿌린 언어"에 불과한가, 아니면 별개의 언어인가?
- "코틀린 = JVM 언어"라는 등식은 어디서부터 틀렸는가? Native·JS·Wasm은 곁다리가 아니라 무엇인가?
- JetBrains는 왜 굳이 새 언어를 만들었나 — Scala도 Groovy도 있었는데?
- "간결·안전·상호운용·도구"라는 네 기둥은 서로 어떻게 얽혀 있고, 어디서 충돌하는가?
- 정적 타입·널 안전·함수형 지향이라는 세 축은 각각 언어의 무엇을 결정하는가?
- 코틀린에는 "언어 명세(specification)"가 존재하는가? 표준 라이브러리는 언어의 일부인가?
- 코틀린은 Java를 대체하려는가, 공존하려는가 — 그 답이 언어 설계를 어떻게 규정했는가?
- 하나의 소스가 네 개의 완전히 다른 런타임으로 컴파일된다는 것은 정확히 무슨 뜻이며, 무엇이 공통이고 무엇이 타깃마다 갈라지는가?

---

이 노트는 "학습노트/Kotlin" 시리즈의 문을 여는 장이다. 앞선 장이 없으니 이어받을 흐름은 없지만, 대신 이 장은 뒤따르는 45개 장 전체가 딛고 설 지반을 놓는다. 세미콜론 추론이든([[03 - 렉시컬 구조와 세미콜론 추론]]) 널 안전성이든([[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]]) 코루틴이든([[42 - suspend와 코루틴 - 언어 수준의 중단]]), 그 모든 세부는 "코틀린이 어떤 언어이기를 의도했는가"라는 하나의 원점에서 뻗어 나온 가지다. 이 장의 목적은 그 원점을 정확히 세우는 것이다.

우리가 여기서 정면으로 부수려는 두 개의 통념이 있다. 하나는 **"코틀린은 Java의 문법 설탕일 뿐"** 이라는 오해(M1)다. 이 통념은 코틀린을 Java 위에 얹은 편의 레이어로 격하시키고, 그 결과 코틀린이 가진 별도의 타입 시스템 — 널 가능성이 타입에 새겨지고, 변성이 선언 지점에서 표현되며, `Nothing`이라는 바닥 타입이 존재하고, 스마트 캐스트가 제어 흐름을 읽는 — 을 통째로 보이지 않게 만든다. 다른 하나는 **"코틀린 = JVM 언어"** 라는 오해(M2)다. 코틀린이 태어난 곳이 JVM 생태계인 것은 맞지만, JVM은 코틀린의 네 정식 타깃 중 하나일 뿐이며, 언어 명세 자체는 특정 런타임에 매이지 않도록 설계됐다.

이 장이 **다루는** 것은 이 두 교정을 축으로 한 큰 그림이다: 언어의 역사와 철학, 세 개의 설계 축, 네 개의 컴파일 타깃과 멀티플랫폼, 언어 명세와 표준 라이브러리의 위상, 그리고 Java와의 관계. 이 장이 **다루지 않는** 것은 그 각각의 기계적 세부다. 컴파일러가 실제로 어떻게 K2 프론트엔드와 IR 백엔드를 거치는지는 [[02 - 컴파일러의 해부 - K2와 IR 백엔드와 바이트코드]]가 소유하고, 타입 계층의 정확한 형태는 [[05 - 타입 시스템의 지형 - Any와 Unit과 Nothing]]이, 원시 타입 컴파일의 진실은 [[07 - 수 1 - Int와 Long과 부동소수점]]과 [[08 - 수 2 - 박싱과 오버플로와 비트와 부호 없는 정수]]가 맡는다. 여기서는 그 세부들이 "왜 그렇게 설계됐는가"를 관통하는 실을 먼저 쥔다.

## 1. 코틀린은 어디서 왔는가 — JetBrains와 실용주의의 기원

### 1.1 도구 회사가 언어를 만든다는 것

코틀린을 이해하는 가장 빠른 길은 "누가, 왜 만들었나"를 보는 것이다. 코틀린은 대학 연구실이나 표준화 위원회가 아니라, **IntelliJ IDEA를 만드는 IDE 회사 JetBrains**가 만들었다. 이 출신은 언어의 성격을 뿌리부터 규정한다. JetBrains는 언어를 "논문으로 증명할 아름다운 계산 모델"로 본 것이 아니라, "매일 수백만 줄의 코드를 다루는 도구 회사가 실제로 쓸 무기"로 봤다.

> **역사 메모**: 코틀린은 2011년 7월 JetBrains가 공개했고, 2012년 Apache 2.0 라이선스로 오픈소스가 됐으며, 첫 안정 버전 1.0은 2016년 2월에 나왔다. 이름은 상트페테르부르크 근처의 **코틀린 섬(Kotlin Island)** 에서 따왔는데, 이는 Java가 인도네시아 자바 섬에서 이름을 딴 것을 의식한 명명이다. JetBrains의 주요 개발 거점 중 하나가 상트페테르부르크였다.

JetBrains의 동기는 자기들의 필요에서 나왔다. 그들의 주력 제품 IntelliJ IDEA는 방대한 Java 코드베이스였다. Java는 안정적이었지만 장황했고, 당시 Java(6~7 시절)는 언어 진화가 느렸다. 그렇다고 Scala로 갈아탈 수도 없었다 — Scala는 표현력은 높았지만 **컴파일이 느렸고**, 이는 IDE를 만드는 회사에게 치명적이었다. IDE의 심장은 빠른 증분 컴파일과 정확한 코드 분석인데, 언어 자체가 그 발목을 잡으면 안 됐다.

그래서 요구사항이 아주 구체적이었다: **(1) Java와 100% 상호운용될 것**(기존 IntelliJ 코드와 섞여야 하니까), **(2) Java만큼 빠르게 컴파일될 것**, **(3) Java보다 훨씬 안전하고 간결할 것**. 이 세 조건이 동시에 성립하는 언어는 당시 시장에 없었다. Scala는 (1)(3)은 되지만 (2)가 안 됐고, Groovy는 동적 타입이라 (3)의 정적 안전성이 약했다. 코틀린은 이 빈틈을 겨냥해 태어난 "필요의 산물"이다.

### 1.2 실용주의라는 설계 원칙

이 출신에서 나오는 코틀린의 첫 번째이자 가장 중요한 철학이 **실용주의(pragmatism)** 다. 코틀린은 "이론적으로 완벽한 언어"를 지향하지 않는다. 대신 "현업 개발자가 오늘 당장 쓰기에 가장 덜 아픈 언어"를 지향한다. 이 태도는 언어 곳곳에 각인돼 있다.

예를 들어 코틀린은 순수 함수형 언어가 아니다. Haskell처럼 부수효과를 타입으로 격리하지도, 불변성을 강제하지도 않는다. `var`도 있고, 예외도 던지며, `null`도 (통제된 형태로) 존재한다. 왜? 현실의 코드는 상태를 가지고, 실패하며, 값이 없는 경우가 있기 때문이다. 코틀린은 이 현실을 부정하는 대신, 그것을 **더 안전하게 다루는 도구**를 준다. 불변을 강제하는 대신 `val`을 권하고, 검사 예외를 없애되 예외 자체는 남기며, `null`을 금지하는 대신 타입에 새겨 컴파일러가 감시하게 한다.

```kotlin
// 순수주의라면 금지했을 것들을, 코틀린은 "통제된 형태로" 허용한다
var counter = 0                    // 가변 변수 — 필요하면 쓴다
counter += 1

val list = mutableListOf(1, 2, 3)  // 가변 컬렉션 — 실용적으로 흔하다
list.add(4)

fun risky(): Int = throw RuntimeException("실패")  // 예외 — 없애지 않았다

val name: String? = null           // null — 금지가 아니라 타입에 새김
```

이 실용주의는 언어 진화 방식에도 나타난다. 코틀린은 매년 새 버전을 내되, **기존 코드를 깨뜨리지 않는 것**을 최우선으로 둔다. 새 기능은 대개 실험적(experimental) 딱지를 달고 나와 옵트인(opt-in)으로 검증받은 뒤에야 안정화된다. 이 신중한 진화 모델은 [[46 - 멀티플랫폼과 진화와 어두운 구석]]에서 자세히 다룬다.

### 1.3 Android가 준 도약, 그러나 Android 언어는 아니다

코틀린의 대중화에서 가장 큰 사건은 **2017년 Google I/O에서의 Android 공식 지원 선언**이다. 2019년에는 Google이 아예 "Android는 이제 Kotlin-first"라고 선언하며 신규 Android 개발의 기본 언어로 밀었다. 많은 개발자가 여기서 처음 코틀린을 접했고, 그래서 "코틀린 = Android 언어"라는 인상도 널리 퍼졌다.

하지만 이는 **성장의 계기이지 정체성의 정의가 아니다**. 코틀린은 Android보다 6년 먼저 나왔고, Android는 코틀린의 여러 무대 중 큰 무대일 뿐이다. 코틀린 언어 자체에는 Android에 대한 어떤 특수 문법도, 특수 타입도 없다. Android SDK는 그냥 Java로 쓰인 거대한 라이브러리이고, 코틀린은 그것을 상호운용으로 호출할 뿐이다. 서버 백엔드, 데스크톱, 데이터 과학, 빌드 스크립트(Gradle Kotlin DSL), 그리고 뒤에 볼 멀티플랫폼까지, 코틀린의 무대는 훨씬 넓다.

이 구분은 뒤에서 M2를 교정할 때 다시 중요해진다. Android가 JVM(정확히는 ART) 위에서 돈다는 사실이 "코틀린은 JVM 언어"라는 오해를 강화하기 때문이다. 정체성과 인기 있는 사용처를 혼동하지 말아야 한다.

## 2. 설계 철학의 네 기둥 — 간결·안전·상호운용·도구

JetBrains가 코틀린을 소개할 때 반복하는 네 단어가 있다: **간결(concise), 안전(safe), 상호운용(interoperable), 도구 친화(tooling-friendly)**. 이 넷은 슬로건이 아니라 실제 설계 결정마다 작동하는 판단 기준이다. 하나씩, 그리고 그것들이 어떻게 서로 밀고 당기는지 보자.

### 2.1 간결 — 보일러플레이트의 제거, 그러나 마법은 아니다

간결함은 코틀린을 처음 만난 사람이 가장 먼저 느끼는 매력이다. 같은 일을 Java보다 훨씬 적은 코드로 표현한다.

```java
// Java — 값 객체 하나에 이만큼
public final class Point {
    private final int x;
    private final int y;
    public Point(int x, int y) { this.x = x; this.y = y; }
    public int getX() { return x; }
    public int getY() { return y; }
    @Override public boolean equals(Object o) { /* ... 장황 ... */ }
    @Override public int hashCode() { /* ... */ }
    @Override public String toString() { return "Point(x=" + x + ", y=" + y + ")"; }
}
```

```kotlin
// Kotlin — 한 줄
data class Point(val x: Int, val y: Int)
// => equals/hashCode/toString/copy/component1/component2 자동 생성
```

하지만 간결함에 대한 코틀린의 태도는 미묘하다. 코틀린은 **"덜 쓰게" 하되 "덜 명확하게" 하지는 않으려** 한다. 이는 코틀린이 의도적으로 거부한 것들을 보면 드러난다. 코틀린에는 Scala 같은 자유로운 기호 연산자 정의가 없고([[19 - 연산자 오버로딩과 관례]]에서 보듯 정해진 관례 이름만 오버로드 가능), 암시적 변환(implicit conversion)도 없다. `Int`를 `Long`이 기대되는 자리에 그냥 넣을 수 없다 — `.toLong()`을 명시해야 한다([[07 - 수 1 - Int와 Long과 부동소수점]]). 왜? 암시적 변환은 코드를 짧게 만들지만 **읽는 사람이 무슨 일이 일어나는지 알 수 없게** 만들기 때문이다. 코틀린의 간결함은 "타이핑을 줄이는 것"이지 "일어나는 일을 숨기는 것"이 아니다.

> **흔한 오해**: "간결한 언어 = 마법 같은 언어"라는 등식은 코틀린엔 맞지 않는다. `data class`가 자동 생성하는 `equals`는 완전히 예측 가능한 규칙(주 생성자 프로퍼티 비교)을 따르며([[27 - 데이터 클래스]]), 그 결과물은 디컴파일하면 그대로 Java 코드로 보인다. 간결함의 원천은 "컴파일러가 뻔한 코드를 대신 써주는 것"이지 "런타임 마법"이 아니다.

### 2.2 안전 — 실패를 컴파일 타임으로 당긴다

두 번째 기둥 안전은 코틀린의 가장 깊은 정체성이다. 핵심 발상은 **"런타임에 터질 오류를 최대한 컴파일 타임으로 당긴다"** 이다. 그 대표가 널 안전성이다. Java에서 `NullPointerException`은 런타임에, 보통 프로덕션에서 터진다. 코틀린은 `null`이 될 수 있는 값과 없는 값을 **타입 수준에서 구분**하여, 널 가능한 값을 널 검사 없이 역참조하려 하면 **컴파일러가 거부**한다.

```kotlin
fun length(s: String): Int = s.length      // s는 절대 null 아님 — 보장됨
fun lengthN(s: String?): Int = s.length    // 컴파일 에러: s가 null일 수 있음
//                              ^ Only safe (?.) or non-null asserted (!!) calls allowed
```

이것이 왜 "안전"인가 하면, 이 검사가 **런타임 비용 없이** 이뤄지기 때문이다. 널 안전성은 대부분 컴파일 타임 분석이며(M7 — [[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]]이 소유), 컴파일된 바이트코드에는 별도의 감시 코드가 거의 들어가지 않는다. 실패의 가능성을 **타입에 인코딩**하고 컴파일러가 그 타입을 추적하게 하는 것 — 이것이 코틀린식 안전의 핵심 기법이다. 같은 발상이 언어 곳곳에 반복된다: 봉인 클래스로 상태 공간을 닫고([[30 - sealed 클래스와 대수적 데이터 타입]]), `when`의 완전성 검사로 빠진 경우를 잡고([[17 - 표현식으로서의 제어 흐름 - if와 when]]), 불변 `val`을 기본으로 권한다.

안전의 또 다른 면은 **의도적 불편함**이다. 코틀린 클래스는 기본이 `final`이라 상속하려면 `open`을 명시해야 하고([[24 - 상속과 오버라이딩과 초기화 순서]]), 오버라이드도 `override` 키워드가 필수다. 이 "명시 강제"는 타이핑을 늘리지만, "실수로 상속되거나 실수로 오버라이드되는" 종류의 버그를 원천 차단한다. 안전과 간결이 여기서 충돌하는데, 코틀린은 안전을 택한다.

### 2.3 상호운용 — 절대 타협하지 않은 제약

세 번째 기둥 상호운용은 앞서 봤듯 코틀린 탄생의 전제 조건이었다. 코틀린은 **기존 Java 코드를 한 줄도 안 고치고 그대로 호출**할 수 있어야 했고, 반대로 Java에서 코틀린 코드를 부를 수도 있어야 했다. 이 요구가 얼마나 강력한 제약이었는지는 아무리 강조해도 지나치지 않다. 상호운용 때문에 코틀린은 수많은 설계에서 "이상적인 선택"을 포기하고 "Java와 맞물리는 선택"을 해야 했다.

```kotlin
// Java 표준 라이브러리를 그냥 쓴다 — 래퍼도, 변환도 없다
import java.util.ArrayList
import java.time.LocalDate

val list = ArrayList<String>()   // Java 클래스를 직접 인스턴스화
list.add("hello")
val today = LocalDate.now()      // Java 8 API 그대로
```

이 상호운용성이 만든 흔적은 언어 도처에 있다. 컴파일 시 코틀린 함수는 JVM 백엔드에서 Java가 부를 수 있는 메서드가 되고, 코틀린 프로퍼티는 `getX()/setX()` 형태로 노출되며([[22 - 프로퍼티와 백킹 필드]]), 코틀린의 `Unit`은 Java에선 사실상 `void`처럼 보이도록 매핑된다. 심지어 코틀린이 새로 도입한 개념들(널 가능성 등)도 Java 쪽에서 **플랫폼 타입**으로 매끄럽게 다뤄지게 설계됐다(M8 — [[45 - Java 상호운용]]). 이 모든 세부의 총람은 45장이 소유한다.

> **명세 정밀**: 상호운용은 **JVM 타깃에서 가장 깊다**. Kotlin/JS는 JavaScript 생태계와, Kotlin/Native는 C/Objective-C/Swift와 상호운용한다. 즉 "상호운용"이라는 기둥은 하나의 대상(Java)이 아니라 **각 타깃의 호스트 생태계와의 맞물림**을 뜻한다. Java 상호운용은 그중 가장 성숙한 사례일 뿐이다.

### 2.4 도구 — 언어가 IDE를 배려한다

네 번째 기둥 도구는 자주 간과되지만, JetBrains가 만든 언어답게 코틀린의 뼈대에 새겨져 있다. 코틀린은 **컴파일러가 빠르고, 오류 메시지가 정확하며, IDE가 코드를 잘 이해할 수 있도록** 설계됐다. 언어 문법조차 도구를 배려한다. 예컨대 타입이 항상 이름 뒤에 오는 `val x: Int` 형식(Pascal 계열 순서)은 파서가 왼쪽에서 오른쪽으로 읽으며 "이건 선언이다"를 빨리 확정하게 해준다.

이 기둥이 가장 극적으로 드러난 사건이 **K2 컴파일러**다. 코틀린 2.0에서 기본이 된 K2는 프론트엔드를 완전히 새로 쓴 것으로, 컴파일 속도와 타입 추론 정확도를 크게 끌어올렸다. 이는 "언어가 완성됐으니 이제 손 뗀다"가 아니라 "도구로서의 언어를 끝없이 벼린다"는 태도의 산물이다. K2의 내부는 [[02 - 컴파일러의 해부 - K2와 IR 백엔드와 바이트코드]]가 다룬다.

네 기둥은 서로 완벽히 조화롭지 않다. 간결은 때로 명시성(안전)과, 안전은 때로 상호운용(Java의 느슨함을 받아들여야 함)과 부딪친다. 코틀린 설계의 묘미는 이 긴장을 어떻게 균형 잡았는가에 있고, 앞으로의 모든 장은 사실상 그 균형점들을 하나씩 확대해 보는 것이다.

## 3. "Java의 문법 설탕"이라는 착각 — 별도의 타입 시스템 (M1 교정)

이제 이 장이 소유한 첫 번째 오개념을 정면으로 부순다.

### 3.1 오해의 정체

**M1: "코틀린은 Java의 문법 설탕(syntactic sugar)일 뿐이다."** 이 오해는 그럴듯한 관찰에서 출발한다. 코틀린은 JVM 바이트코드로 컴파일되고, Java 라이브러리를 쓰며, `data class`를 디컴파일하면 평범한 Java 클래스가 나온다. 그러니 "결국 Java로 번역되는 편의 문법 아니냐"고 생각하기 쉽다.

이것이 틀린 이유는, **"무엇으로 컴파일되는가"와 "어떤 언어인가"는 다른 질문**이기 때문이다. C도 어셈블리로 컴파일되지만 C를 "어셈블리의 문법 설탕"이라 하지 않는다. TypeScript는 JavaScript로 컴파일되지만 별개의 타입 시스템을 가진 별개의 언어다. 컴파일 타깃이 같다는 것은 **런타임을 공유한다**는 뜻이지, **언어의 의미론(semantics)이 같다**는 뜻이 아니다.

문법 설탕이란 엄밀히는 "같은 언어 안에서 더 짧게 쓰는 표기"를 말한다(예: Java의 향상된 for문은 이터레이터 루프의 설탕이다). 코틀린이 문법 설탕이려면 코틀린의 모든 구성물이 "Java에 이미 있는 개념의 짧은 표기"여야 한다. 그런데 코틀린에는 **Java에 대응 개념 자체가 없는 것들**이 수두룩하다. 그것이 별개 언어라는 증거다.

### 3.2 Java에 대응물이 없는 것들 — 타입 시스템 차원

코틀린이 별개의 타입 시스템을 가졌다는 것을, Java에 없는 개념들로 열거해 보자.

**(1) 널 가능성이 타입의 일부다.** Java에서 `String`은 언제나 `null`일 수 있고, 그 사실은 타입에 나타나지 않는다. 코틀린에서 `String`과 `String?`은 **서로 다른 타입**이다. 이건 편의 표기가 아니라 타입 시스템의 근본 확장이다.

```kotlin
val a: String = "x"    // null 불가 타입
val b: String? = null  // null 가능 타입 — a와 b는 다른 타입
// b를 String이 요구되는 곳에 넣으면 컴파일 에러 (b는 String의 하위타입 아님... 정확히는 String이 String?의 하위타입)
```

Java에는 이런 구분이 없다. 따라서 이건 "Java 개념의 짧은 표기"가 될 수 없다 — 대응 개념 자체가 없으니까.

**(2) `Nothing` 바닥 타입.** 코틀린에는 값이 하나도 없는 타입 `Nothing`이 있고, 이것은 **모든 타입의 하위타입**이다(M11 — [[05 - 타입 시스템의 지형 - Any와 Unit과 Nothing]]). `throw`의 타입이 `Nothing`이고, 그래서 아래 코드가 타입 검사를 통과한다.

```kotlin
val name: String = person.name ?: throw IllegalStateException("이름 없음")
// throw의 타입이 Nothing이고 Nothing은 String의 하위타입이므로
// 엘비스 ?:의 양변 타입이 String으로 합쳐진다
```

Java에는 바닥 타입이 없다. `throw`는 문(statement)이지 값이 있는 표현식이 아니다. 이건 타입 이론 차원의 차이지 표기의 차이가 아니다.

**(3) 선언 지점 변성(declaration-site variance).** 코틀린은 `class Box<out T>`처럼 클래스를 선언하는 곳에서 변성을 지정한다(M20 — [[33 - 제네릭 1 - 타입 파라미터와 변성]]). Java는 사용하는 곳에서 `List<? extends T>` 같은 와일드카드로만 변성을 표현한다. 이는 제네릭 타입 시스템의 구조 자체가 다르다는 뜻이다.

**(4) 스마트 캐스트.** 코틀린 컴파일러는 제어 흐름을 분석해 타입을 자동으로 좁힌다(M13 — [[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]]).

```kotlin
fun describe(x: Any): String {
    if (x is String) {
        return x.length.toString()  // x가 여기선 String으로 자동 인식 — 캐스트 안 씀
    }
    return "other"
}
```

Java(구버전)는 `if (x instanceof String)` 뒤에도 `((String) x).length()`처럼 명시적 캐스트가 필요했다. 스마트 캐스트는 **컴파일러의 데이터 흐름 분석**이라는 별도 메커니즘이다.

### 3.3 Java에 대응물이 없는 것들 — 언어 구조 차원

타입 시스템만이 아니다. 언어 구조 차원에서도 Java에 없는 것들이 즐비하다.

| 코틀린 개념 | Java 대응 | 설명 |
|-------------|-----------|------|
| 확장 함수 | 없음 | 클래스를 안 고치고 함수를 "붙여" 쓴다(정적 디스패치, M14). [[35 - 확장 함수와 프로퍼티]] |
| 프로퍼티 | 필드+게터/세터의 관례일 뿐 | 코틀린 프로퍼티는 접근자 추상, 백킹 필드는 필요할 때만(M35). [[22 - 프로퍼티와 백킹 필드]] |
| `when` 표현식 | switch 문 | 완전성 검사·임의 조건·값 반환(M12). [[17 - 표현식으로서의 제어 흐름 - if와 when]] |
| 봉인 클래스 | 없음(Java 17에 뒤늦게 도입) | 제한된 계층 + 완전성. [[30 - sealed 클래스와 대수적 데이터 타입]] |
| 최상위 함수 | 없음(모든 것이 클래스 안) | 파일에 함수를 바로 선언. [[04 - 파일과 패키지와 선언 - 프로그램의 골격]] |
| `object` 선언 | 없음 | 언어 차원의 싱글턴(M24). [[28 - object와 companion - 싱글턴과 동반 객체]] |
| 연산자 관례 | 없음 | `plus`, `get`, `invoke` 등 관례 이름 오버로드(M40). [[19 - 연산자 오버로딩과 관례]] |
| 인라인 함수 | 없음 | 람다를 호출부에 펼쳐 넣음, `reified`(M17,M18). [[15 - 인라인 함수와 reified]] |
| `suspend` | 없음 | 언어 수준의 중단(M33,M34). [[42 - suspend와 코루틴 - 언어 수준의 중단]] |

이 표의 어느 항목도 "Java 개념의 짧은 표기"가 아니다. 각각은 코틀린이 새로 정의한 의미론이다. `if`가 표현식이라는 것부터가 이미 Java와 다르다 — 코틀린엔 삼항 연산자 `? :`가 없는데, 필요가 없기 때문이다. `val max = if (a > b) a else b`처럼 `if` 자체가 값을 낸다.

```kotlin
// if가 표현식 — Java의 삼항 연산자를 대체하는 게 아니라, 개념이 다르다
val max = if (a > b) a else b
val category = when {
    score >= 90 -> "A"
    score >= 80 -> "B"
    else -> "C"
}  // when도 값을 내는 표현식
```

### 3.4 그러면 왜 Java로 번역되는 것처럼 보이나

M1이 끈질긴 이유는 관찰이 부분적으로 맞기 때문이다. JVM 백엔드에서 코틀린은 결국 JVM 바이트코드가 되고, 그 바이트코드는 Java 컴파일러가 내는 것과 같은 형식이다. 즉 **런타임 표현은 공유**한다. 확장 함수는 정적 메서드로, 프로퍼티는 게터/세터로, `object`는 정적 필드를 가진 클래스로 컴파일된다.

핵심은 이것이 **"컴파일 결과"이지 "언어의 정체성"이 아니라는 것**이다. 코틀린 컴파일러는 코틀린의 풍부한 의미론을 JVM이라는 상대적으로 단순한 실행 모델로 **내려 번역(lowering)** 한다. 이 번역 과정에서 널 가능성 정보 같은 것들은 (일부는 애노테이션으로 남지만) 상당 부분 소거된다. 소거된다는 것은 그 정보가 **컴파일 타임에만 존재했다**는 뜻 — 즉 Java에 없는 별도의 정적 검사 계층이 있었다는 결정적 증거다.

그리고 결정적으로, **JVM 백엔드는 코틀린의 여러 백엔드 중 하나일 뿐이다**. Kotlin/Native로 컴파일하면 애초에 JVM 바이트코드가 나오지 않는다. LLVM을 거쳐 네이티브 기계어가 나온다. "Java로 번역된다"는 관찰은 JVM 타깃에서만 부분적으로 성립하는 것이고, 이는 곧 다음 절의 M2로 이어진다.

> **명세 정밀**: 코틀린에는 **공식 언어 명세**(Kotlin language specification)가 존재하며, 이 명세는 특정 백엔드(JVM)에 매이지 않고 언어의 의미론을 정의한다. Java와 별개의 문법(grammar), 별개의 타입 시스템, 별개의 이름 해소(name resolution) 규칙을 가진다. 명세가 별도로 있다는 사실 자체가 "설탕이 아니라 언어"임을 증명한다. 명세의 위상은 이 장 7절에서 다룬다.

## 4. 세 개의 축 — 정적 타입·널 안전·함수형 지향

코틀린이 별개의 언어라면, 그 언어를 세우는 축은 무엇인가? 세 개의 축으로 요약할 수 있다. 이 셋은 서로 독립적이지 않고, 하나가 다른 둘을 강화한다.

### 4.1 첫째 축 — 정적·강타입

코틀린은 **정적 타입(statically typed)** 언어다. 모든 표현식의 타입이 컴파일 타임에 결정되고 검사된다. 그러나 코틀린의 정적 타입은 **타입 추론(type inference)** 과 결합돼 있어서, 동적 언어처럼 가벼워 보인다.

```kotlin
val count = 42          // 타입 명시 없지만 Int로 추론됨 — 여전히 정적 타입
val name = "Alice"      // String으로 추론
val pi = 3.14           // Double로 추론

// 하지만 타입은 확정돼 있다
count = "hello"         // 컴파일 에러: Int에 String 대입 불가
```

여기서 중요한 구분: **타입 추론 ≠ 동적 타입**. `var x = 5` 다음에 `x = "hello"`가 안 되는 것은, `x`의 타입이 첫 대입 순간 `Int`로 **영구히 확정**됐기 때문이다. 이것이 JavaScript 같은 동적 언어와의 결정적 차이다(자매 시리즈 JavaScript의 동적 타입과 대조하면 선명하다). 코틀린은 "타이핑은 덜 하되 안전은 다 챙긴다"는 실용주의를, 타입 추론이라는 도구로 실현한다.

코틀린은 또한 **강타입(strongly typed)** 이다. 앞서 봤듯 암시적 변환이 없다. `Int`와 `Long` 사이조차 자동 변환이 없어 `.toLong()`을 써야 한다. 이 엄격함은 "42 + '2' = ?" 같은 언어별로 결과가 제각각인 함정을 원천 차단한다.

정적·강타입은 나머지 두 축의 **전제**다. 널 안전성이 컴파일 타임에 작동하려면 타입이 정적이어야 하고, 함수형 도구들(고차 함수, 제네릭)이 안전하려면 타입이 강해야 한다.

### 4.2 둘째 축 — 널 안전

두 번째 축 널 안전은 앞서 안전 기둥에서 소개했지만, 여기서는 "언어를 세우는 축"으로서의 위상을 강조한다. 널 안전은 코틀린 타입 시스템의 **일부이지 부가 기능이 아니다**. 모든 타입은 널 가능 버전(`T?`)과 널 불가 버전(`T`)의 짝을 이루며, 이 구분이 타입 계층 전체에 스며 있다.

```text
타입 계층에서 nullable의 위상 (개념도)

        Any?                    ← 모든 nullable 타입의 최상위
       /    \
     Any    (null 만 담는 영역)
    /   \
 String  Int ...
    |
 String?  ← String ∪ {null}, String의 상위타입
```

이 축이 언어에 미치는 파장은 광범위하다. 안전 호출 `?.`, 엘비스 `?:`, 널 단언 `!!`, 안전 캐스트 `as?` 같은 연산자들이 모두 이 축에서 나온다. 스마트 캐스트도 상당 부분 널 검사와 얽혀 있다. `lateinit`, `by lazy` 같은 초기화 도구들도 "널 불가인데 나중에 채운다"는 문제를 풀기 위해 존재한다([[11 - 변수와 초기화 - val과 const와 lateinit]]). 널 안전이라는 축 하나가 언어 표면의 상당 부분을 결정한다. 그 전모는 [[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]]이 소유한다.

### 4.3 셋째 축 — 함수형 지향

세 번째 축은 **함수형 지향(functional-oriented)** 이다. 주의할 단어는 "지향"이다. 코틀린은 순수 함수형 언어가 아니라(4.1절 실용주의 참조), **객체지향과 함수형을 융합한 다중 패러다임 언어**다. 함수형 "지향"이란, 함수를 일급 시민으로 다루고 함수형 스타일을 편하게 쓸 수 있게 만들었다는 뜻이다.

```kotlin
// 함수가 일급 시민 — 변수에 담고, 인자로 넘기고, 반환한다
val square: (Int) -> Int = { x -> x * x }
val nums = listOf(1, 2, 3, 4, 5)

val result = nums
    .filter { it % 2 == 1 }   // 고차 함수에 람다 전달
    .map { it * it }          // 함수형 파이프라인
    .sum()
// => 1 + 9 + 25 = 35
```

이 축에서 나오는 것들: 함수 타입([[13 - 함수 타입과 람다와 함수 참조]]), 람다와 클로저([[14 - 클로저와 캡처]]), 고차 함수와 인라인([[15 - 인라인 함수와 reified]]), 스코프 함수([[16 - 스코프 함수와 수신 객체 관용구]]), 컬렉션 함수형 연산([[39 - 컬렉션 연산과 함수형 파이프라인]]), 시퀀스의 지연 평가([[40 - 시퀀스와 지연 평가]]). 그리고 `Nothing`을 활용한 표현식 지향 설계 — `if`, `when`, `try`가 모두 값을 내는 표현식이라는 것 — 도 함수형 지향의 산물이다. 문(statement)보다 표현식(expression)을 선호하는 것은 함수형 언어의 특징이다.

세 축을 겹쳐 보면 코틀린의 성격이 선명해진다. **정적·강타입이라는 견고한 바닥 위에, 널 안전이라는 안전벽을 세우고, 함수형 지향이라는 표현력을 얹은 것.** 그리고 이 모든 것을 실용주의가 조율한다 — 어느 축도 극단으로 밀지 않는다. 정적 타입이되 추론으로 부드럽게, 널 안전이되 `!!`라는 탈출구를 남기고, 함수형이되 `var`와 부수효과를 허용한다.

## 5. 하나의 언어, 네 개의 타깃 — JVM·Native·JS·Wasm (M2 교정)

이제 이 장이 소유한 두 번째 오개념을 부순다.

### 5.1 오해의 정체

**M2: "코틀린은 JVM 언어다."** 이 오해는 M1보다도 널리 퍼져 있고, 여러 이유로 강화된다. 코틀린은 JVM 생태계에서 태어났고, 가장 성숙한 타깃이 JVM이며, 가장 유명한 사용처인 Android도 (ART라는) JVM 계열 위에서 돈다. 많은 튜토리얼이 JVM을 전제로 쓰이고, `System.out.println` 같은 Java API가 예제에 흔히 등장한다. 그래서 "코틀린 = JVM 위에서 도는 Java의 대안" 이미지가 굳어졌다.

이것이 틀린 이유는, **JVM은 코틀린의 네 정식 컴파일 타깃 중 하나일 뿐**이기 때문이다. 코틀린 컴파일러는 프론트엔드(언어 분석)와 백엔드(코드 생성)로 나뉘는데, 프론트엔드는 타깃과 무관하게 공통이고, 백엔드가 여럿이다. 같은 코틀린 소스가 JVM 바이트코드로도, 네이티브 기계어로도, JavaScript로도, WebAssembly로도 컴파일된다. 언어 명세는 특정 런타임을 전제하지 않도록 설계됐다.

> **명세 정밀**: 코틀린의 타입 계층 최상단은 `java.lang.Object`가 **아니라** `kotlin.Any`다. 이 사실 하나가 M2를 무너뜨린다. 만약 코틀린이 JVM 언어라면 최상위 타입이 `Object`여야 할 텐데, 코틀린은 자체의 `Any`를 정의하고, JVM 백엔드에서만 그것을 `Object`로 매핑한다. Kotlin/Native나 Kotlin/JS에서 `Any`는 `Object`와 아무 관계가 없다. 언어의 타입 시스템이 특정 런타임에 독립적으로 정의돼 있다는 뜻이다.

### 5.2 네 타깃의 지형

네 타깃을 하나씩 보자. 각 타깃은 "같은 언어, 다른 런타임"이라는 구조를 이룬다.

```text
                    Kotlin 소스 (.kt)
                          |
                 프론트엔드 (K2/FIR)   ← 파싱·타입추론·진단, 타깃 무관
                          |
                    공통 IR (중간표현)
              /        |        |        \
         JVM 백엔드  Native   JS 백엔드  Wasm 백엔드
             |      백엔드       |          |
        .class      LLVM      .js        .wasm
        바이트코드   → 기계어   JavaScript  WebAssembly
             |         |         |          |
          JVM/ART   OS 직접   브라우저/    Wasm 런타임
                    실행      Node.js    (브라우저 등)
```

**(1) Kotlin/JVM** — 가장 성숙한 타깃. 코틀린 소스를 JVM 바이트코드(`.class`)로 컴파일하여 JVM(또는 Android의 ART)에서 실행한다. Java 생태계 전체와 상호운용한다. 서버 백엔드, Android, 데스크톱의 주력.

**(2) Kotlin/Native** — 코틀린을 **LLVM을 거쳐 네이티브 기계어**로 컴파일한다. JVM 없이 OS에서 직접 도는 실행 파일이 나온다. iOS, macOS, Linux, Windows, 임베디드 등을 겨냥하며, C/Objective-C/Swift와 상호운용한다. 가비지 컬렉션은 자체 런타임이 담당한다. iOS 앱에서 코틀린 로직을 공유하는 KMP의 핵심 타깃이다.

**(3) Kotlin/JS** — 코틀린을 **JavaScript로 컴파일**한다. 브라우저나 Node.js에서 실행되며, npm 생태계·DOM API와 상호운용한다. 웹 프런트엔드나 풀스택 코틀린에 쓰인다.

**(4) Kotlin/Wasm** — 코틀린을 **WebAssembly**로 컴파일한다. 상대적으로 최신 타깃으로, 브라우저 등 Wasm 런타임에서 돈다.

> **역사 메모**: 타깃들의 성숙도는 다르다. Kotlin/JVM과 Kotlin/JS는 오래 안정화됐고, Kotlin/Native도 안정 타깃이다. **Kotlin/Wasm은 상대적으로 나중에 나온 타깃으로, 2.x 시점 기준 아직 발전 중(Alpha 수준의 부분들이 있음)** 이다. 따라서 "네 정식 타깃"이라 할 때 Wasm의 성숙도는 나머지 셋과 같지 않음을 정확히 인지해야 한다. (제목의 "세 개의 타깃"은 성숙한 JVM/Native/JS를 강조한 표현이고, Wasm까지 세면 넷이다.)

### 5.3 "같은 언어"란 정확히 무엇이 같은가

여기서 정밀함이 필요하다. "하나의 소스가 네 타깃으로 컴파일된다"는 말이, "아무 코틀린 코드나 네 타깃에서 다 돈다"는 뜻은 **아니다**. 무엇이 공통이고 무엇이 갈라지는지 명확히 하자.

**공통인 것 (언어 코어):**
- 문법과 의미론 — `val`/`var`, 클래스, 함수, 널 안전, 제네릭, 람다, `when`, 확장 함수 등 언어 자체의 규칙.
- 타입 시스템 — `Any`/`Unit`/`Nothing`, 널 가능성, 변성.
- **공통 표준 라이브러리(kotlin-stdlib)의 공통 부분** — `List`, `Map`, `String` 연산, 컬렉션 함수형 API, 코루틴 기초 등 타깃 무관하게 정의된 부분.

**타깃마다 갈라지는 것:**
- **호스트 플랫폼 API** — `java.util.ArrayList`는 JVM에서만, DOM API는 JS에서만, POSIX는 Native에서만 존재한다. Java 표준 라이브러리를 쓴 코드는 JVM에서만 돈다.
- **일부 저수준 동작** — 원시 타입 표현, 박싱 규칙, 메모리 모델([[43 - 동시성과 메모리 모델]])은 백엔드마다 다르다. 예컨대 JVM 백엔드는 `Int`를 원시 `int`로 컴파일하지만(M4), 다른 백엔드는 다르게 표현할 수 있다.
- **상호운용 대상** — JVM은 Java, JS는 JavaScript, Native는 C/Swift.

```kotlin
// 이 코드는 네 타깃 모두에서 돈다 — 순수 코틀린 + 공통 stdlib만 사용
fun average(nums: List<Int>): Double {
    if (nums.isEmpty()) return 0.0
    return nums.sum().toDouble() / nums.size
}

// 이 코드는 JVM 타깃에서만 돈다 — Java API 사용
import java.time.LocalDate
fun daysUntilNewYear(): Long { /* java.time 의존 */ ... }
```

이 구분이 바로 다음 절의 멀티플랫폼(KMP) 이야기의 전제다. 공통 부분에 로직을 쓰고, 갈라지는 부분만 타깃별로 채우는 것이 KMP의 발상이다.

### 5.4 왜 이 오해가 위험한가

M2가 단지 부정확한 표현에 그치지 않고 **실질적 해를 끼치는** 지점이 있다. "코틀린 = JVM"이라 믿으면, 코틀린 코드를 짤 때 무심코 JVM 전용 API를 섞어 쓰게 되고, 그러면 그 코드는 멀티플랫폼으로 못 옮긴다. 또 "널 가능 `Int?`는 항상 박싱된다"(M5) 같은 JVM 전용 사실을 언어 보편 사실로 착각하게 된다. 실제로 박싱은 JVM 백엔드의 구현 세부이지 언어 명세가 강제하는 것이 아니다([[08 - 수 2 - 박싱과 오버플로와 비트와 부호 없는 정수]]).

그래서 이 시리즈 전체는 규율을 하나 둔다: **JVM에서만 참인 사실은 반드시 "JVM 백엔드에서"라고 한정한다.** "코틀린은 `Int`를 원시 타입으로 컴파일한다"가 아니라 "JVM 백엔드에서 코틀린은 `Int`를 원시 `int`로 컴파일한다"고 쓴다. 이 한정 습관이 M2를 실무 차원에서 예방한다.

## 6. 코틀린 멀티플랫폼(KMP)의 구조

네 타깃이 있다는 사실이 진짜 힘을 발휘하는 곳이 **코틀린 멀티플랫폼(Kotlin Multiplatform, KMP)** 이다. KMP는 "하나의 코드베이스에서 로직을 공유하고, 플랫폼별 부분만 따로 구현"하게 해준다.

### 6.1 공통 소스셋과 플랫폼 소스셋

KMP 프로젝트는 소스를 **소스셋(source set)** 으로 나눈다. `commonMain`은 모든 타깃이 공유하는 공통 코드이고, `jvmMain`/`iosMain`/`jsMain` 등은 각 타깃 전용 코드다. 공통 코드는 오직 공통 표준 라이브러리와 다른 공통 코드에만 의존할 수 있다.

```text
프로젝트 소스셋 구조 (개념)

  commonMain/     ← 모든 타깃 공유, 순수 코틀린 + 공통 stdlib만
     └ 비즈니스 로직, 도메인 모델, 검증 규칙 ...
        │
        │ (아래는 타깃별로 갈라짐)
        │
  jvmMain/    androidMain/    iosMain/    jsMain/
     └ Java     └ Android       └ Cocoa    └ 브라우저
       API        API             API        API
```

### 6.2 expect / actual — 공통 선언, 플랫폼 구현

공통 코드가 플랫폼별 기능을 필요로 할 때 쓰는 장치가 `expect`/`actual`이다. 공통 소스셋에서 `expect`로 "이런 선언이 있다"고 약속하고, 각 플랫폼 소스셋에서 `actual`로 실제 구현을 준다.

```kotlin
// commonMain — "이런 게 있을 것이다"라는 기대 선언
expect fun currentPlatformName(): String

// jvmMain — JVM용 실제 구현
actual fun currentPlatformName(): String = "JVM"

// iosMain — iOS(Native)용 실제 구현
actual fun currentPlatformName(): String = "iOS"
```

이 구조 덕분에 공통 코드는 `currentPlatformName()`을 자유롭게 부르되, 실제 무엇이 실행될지는 컴파일되는 타깃이 결정한다. 이는 3.3절에서 본 "함수형 파이프라인" 같은 로직을 한 번만 쓰고 여러 플랫폼에서 재사용하게 만든다. KMP의 정확한 문법과 소스셋 위계, 계층적 공통화 등은 [[46 - 멀티플랫폼과 진화와 어두운 구석]]가 소유한다.

### 6.3 KMP가 "모든 것을 공유"하지 않는다는 점

여기서 흔한 과장을 경계하자. KMP는 "한 번 짜서 어디서나 돈다(write once, run anywhere)"의 마케팅 재판이 아니다. KMP의 실용주의적 입장은 **"로직은 공유하되, UI 같은 플랫폼 고유 부분은 각자"** 다. 실제로 널리 쓰이는 패턴은 도메인 로직·네트워크·검증 같은 공유 가능한 부분을 `commonMain`에 두고, UI는 각 플랫폼의 네이티브 방식(iOS는 SwiftUI, Android는 Compose 등)으로 짜는 것이다. (물론 Compose Multiplatform처럼 UI까지 공유하는 접근도 있지만, 그건 선택이지 강제가 아니다.)

이 절제된 야심이야말로 코틀린 실용주의의 표현이다. "이론적으로 모든 것을 공유할 수 있다"가 아니라 "현실적으로 공유해서 이득인 것만 공유한다"는 것이다.

## 7. 언어 명세의 위상 — 스펙과 표준 라이브러리

"코틀린은 어떤 언어인가"를 끝까지 밀면 "코틀린을 정의하는 것은 무엇인가"에 이른다. 여기엔 세 층위가 있다: 언어 명세, 컴파일러, 표준 라이브러리.

### 7.1 언어 명세 — 무엇이 코틀린을 코틀린이게 하는가

코틀린에는 **공식 언어 명세(Kotlin language specification)** 가 있다. 이는 언어의 문법, 타입 시스템, 이름 해소, 제어 흐름, 오버로드 해소 등의 규칙을 형식적으로 기술한 문서다. 명세의 존재는 두 가지를 함의한다.

첫째, 3.4절에서 봤듯 **코틀린이 별개의 언어**라는 증거다. 명세는 Java나 JVM을 전제하지 않고 코틀린 자체의 의미론을 정의한다.

둘째, **명세와 구현(컴파일러)이 원리상 분리**된다는 것이다. 명세는 "무엇이 올바른 코틀린 프로그램이고 그것이 무슨 의미인가"를 정하고, 컴파일러는 "그 명세를 어떻게 실현하는가"를 담당한다. 실제로는 JetBrains의 참조 컴파일러가 사실상의 표준 역할을 하지만, 개념적으로 이 둘은 다른 층이다. 이 구분은 "언어의 보장"과 "구현의 재량"을 나누는 이 시리즈의 서술 원칙과 직결된다 — 명세가 보장하는 것은 단정하고, 컴파일러 구현에 달린 것은 "구현/백엔드에 따라"로 한정한다.

> **명세 정밀**: 코틀린 명세는 여전히 진화 중인 문서이며, 언어의 모든 최신 기능이 즉시 반영되는 것은 아니다. 실무에서 "코틀린의 정본 동작"을 확인하는 가장 확실한 방법은 컴파일러의 실제 동작(과 공식 문서)이다. 그럼에도 명세가 별도로 존재한다는 사실 자체가, 코틀린이 "구현이 곧 정의"인 임시방편 언어가 아니라 **정의된 언어**임을 말해준다.

### 7.2 컴파일러 — 명세의 실현자

두 번째 층은 컴파일러다. 앞서 여러 번 언급한 K2 컴파일러가 현재의 참조 구현이다. 컴파일러는 프론트엔드(언어를 이해)와 백엔드(코드를 생성)로 나뉘고, 백엔드가 네 타깃으로 갈라진다는 것은 5절에서 봤다. 이 장에서 중요한 것은 위상 하나다: **컴파일러는 언어가 아니다.** 코틀린이라는 언어는 명세로 정의되고, 컴파일러는 그것을 특정 런타임 위에 구현한다. 그래서 "JVM 백엔드가 `Int`를 원시형으로 만든다"는 컴파일러(그것도 특정 백엔드)의 사실이지 언어의 사실이 아니다. 컴파일러의 실제 해부는 [[02 - 컴파일러의 해부 - K2와 IR 백엔드와 바이트코드]]가 담당한다.

### 7.3 표준 라이브러리(kotlin-stdlib) — 언어인가 라이브러리인가

세 번째 층이 미묘하다. **표준 라이브러리 `kotlin-stdlib`** 는 `List`, `Map`, `String`의 확장 함수들, `let`/`run` 같은 스코프 함수, 컬렉션 연산, 범위(range), `Pair`/`Triple`, `Result` 등 우리가 "코틀린 코드"라 여기는 것 대부분을 제공한다. 이건 언어의 일부인가, 아니면 그냥 딸려 오는 라이브러리인가?

답은 **"둘 사이"** 다. 정확히는, **언어 문법 자체는 stdlib를 필요로 하지 않지만, 몇몇 언어 기능은 stdlib의 특정 타입과 맞물려 있다.** 예를 들어:

- `for (x in collection)`이 도는 것은 언어의 `for` 문법이지만, `collection`이 `Iterator`를 제공해야 하고 그 `Iterator`/`Iterable`은 stdlib에 정의돼 있다.
- 문자열 템플릿 `"$name"`은 언어 문법이지만, 결과 타입 `String`은 stdlib(정확히는 코어 타입)다.
- 위임 프로퍼티 `by lazy`에서 `lazy`는 stdlib 함수다([[23 - 위임 프로퍼티]]).

그래서 코틀린을 배운다는 것은 사실 **언어 코어 + stdlib**를 함께 배우는 것이다. 이 시리즈도 그렇게 다룬다 — 예컨대 컬렉션([[38 - 컬렉션 - 읽기 전용과 가변]])이나 스코프 함수([[16 - 스코프 함수와 수신 객체 관용구]])는 엄밀히는 라이브러리지만, 언어 경험의 필수 부분이라 언어처럼 다룬다.

> **흔한 오해**: "`listOf`, `let`, `map`은 언어 키워드다"라고 생각하기 쉽지만, 이들은 전부 **stdlib의 함수**다. `let`은 `public inline fun <T, R> T.let(block: (T) -> R): R`라는 확장 함수이고([[35 - 확장 함수와 프로퍼티]]), `map`도 컬렉션 확장 함수다. 언어 키워드(`val`, `fun`, `when`, `is`, `in`)와 라이브러리 함수를 구분하는 것은 언어를 정확히 이해하는 첫걸음이다. 이 경계는 [[03 - 렉시컬 구조와 세미콜론 추론]]에서 키워드 분류로 정밀화한다.

한편 stdlib에는 타깃별로 갈라지는 부분도 있다(5.3절). 공통 부분은 모든 타깃에서 쓸 수 있고, `kotlin.io`의 파일 API처럼 특정 타깃에만 의미 있는 부분은 그 타깃 stdlib에만 있다.

## 8. Java와의 관계 — 대체가 아니라 공존

코틀린을 이해하는 마지막 큰 축은 Java와의 관계다. 이 관계는 "코틀린이 어떤 언어인가"를 규정한 결정적 힘이었다.

### 8.1 코틀린은 Java를 죽이려 하지 않았다

흔한 서사는 "코틀린이 Java를 대체한다"이지만, 이는 코틀린의 실제 설계 철학을 오해한 것이다. 코틀린의 목표는 **Java를 죽이는 것이 아니라 Java와 공존하는 것**이었다. 이는 이상이 아니라 필연이었다 — 앞서 봤듯 JetBrains는 자기들의 거대한 Java 코드베이스와 섞어 쓰려고 코틀린을 만들었다. Java 코드를 다 버리고 코틀린으로 재작성하는 것은 애초에 선택지가 아니었다.

이 "공존" 전제가 코틀린 설계를 얼마나 강하게 규정했는지 보자. 만약 코틀린이 Java를 완전히 대체할 작정이었다면, 훨씬 급진적인 설계를 할 수 있었다 — Java와 다른 컬렉션 프레임워크, Java와 다른 예외 모델, Java 타입과 매핑되지 않는 자체 타입 시스템. 하지만 코틀린은 그러지 않았다. 코틀린의 `List`는 Java의 `List`와 상호운용되고, 코틀린의 예외는 Java의 `Throwable` 계층을 그대로 쓰며([[41 - 예외와 Nothing]]), 코틀린의 `Int`는 JVM에서 Java의 `int`/`Integer`와 매끄럽게 오간다. 상호운용이라는 제약이 자유로운 설계를 막았지만, 동시에 그것이 코틀린을 "당장 쓸 수 있는 언어"로 만들었다.

### 8.2 공존이 남긴 흉터 — 플랫폼 타입

공존에는 대가가 있었다. 가장 아픈 흉터가 **플랫폼 타입(platform type)** 이다. Java 코드에는 널 가능성 정보가 없다(널 애노테이션이 없는 한). Java 메서드가 `String`을 반환할 때, 그것이 `null`일 수 있는지 코틀린은 알 수 없다. 코틀린이 이걸 어떻게 처리할까? 세 선택지가 있었다.

1. **모두 널 불가로 취급** → Java가 실제로 `null`을 주면 예상치 못한 NPE. 게다가 안전하다는 착각을 줌.
2. **모두 널 가능으로 취급** → 모든 Java 호출에 `?.`나 `!!`를 붙여야 함. 상호운용이 지옥이 됨.
3. **"모르는 타입"으로 취급하고 개발자에게 맡김** → 플랫폼 타입 `T!`.

코틀린은 3번을 택했다. 플랫폼 타입 `T!`는 "널 가능성 검사를 유예"하는 타입으로, 널 불가처럼도 널 가능처럼도 쓸 수 있되, 실제로 `null`이면 사용처에서 NPE가 난다(M8 — [[45 - Java 상호운용]]). 이것은 코틀린의 널 안전성에 뚫린 구멍이지만, **의도된 실용적 타협**이다. 완벽한 안전을 고집했다면(2번) Java 상호운용이 불가능했을 것이고, 그러면 코틀린은 태어나지도 못했을 것이다.

```kotlin
// Java의 System.getProperty(String)은 String을 반환하지만 null일 수 있다
// 코틀린은 이를 String! (플랫폼 타입)으로 본다
val v = System.getProperty("user.name")  // 타입: String! (널 가능성 미정)
val len = v.length   // 컴파일러가 막지 않음 — 하지만 v가 null이면 여기서 NPE
```

플랫폼 타입은 M7("널 안전성은 런타임 검사다")과 M8("플랫폼 타입은 안전하다")이 만나는 지점이다. 코틀린 코드 내부에서 널 안전은 컴파일 타임 보장이지만, Java 경계를 넘는 순간 그 보장이 유예되고 런타임 NPE의 문이 열린다. 이 상세는 [[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]]과 [[45 - Java 상호운용]]이 나눠 소유한다.

### 8.3 공존의 다른 흔적들

플랫폼 타입 외에도 공존의 흔적은 많다. 코틀린의 **검사 예외(checked exception) 부재**([[41 - 예외와 Nothing]], M30)도 공존과 관련 있다 — 코틀린은 검사 예외를 강제하지 않지만, Java와 상호운용하기 위해 `@Throws` 애노테이션으로 Java 쪽에 예외 시그니처를 노출할 수단을 남겼다. 코틀린의 프로퍼티가 JVM에서 `getX()/setX()`로 컴파일되는 것도([[22 - 프로퍼티와 백킹 필드]]), Java에서 코틀린 프로퍼티를 자연스럽게 쓰게 하려는 배려다. `@JvmStatic`, `@JvmField`, `@JvmName`, `@JvmOverloads` 같은 애노테이션 무리는 전부 "코틀린을 Java에서 편하게 쓰게 하는" 상호운용 도구다([[45 - Java 상호운용]]).

이 모든 것을 관통하는 통찰: **코틀린의 많은 설계는 "이상적 선택"과 "Java 공존" 사이의 협상 결과다.** 코틀린을 깊이 이해한다는 것은, 어떤 설계가 순수한 언어 철학에서 나왔고 어떤 설계가 Java와의 타협에서 나왔는지를 구분할 수 있다는 뜻이다. 이 구분은 앞으로 여러 장에서 반복해 마주친다.

## 9. 왜 코틀린은 이렇게 생겼는가 — 트레이드오프와 경계

마지막으로, 지금까지의 실을 하나로 묶고, 이 장이 **다루지 않는** 경계를 명확히 못박자.

### 9.1 세 힘의 합성으로서의 코틀린

코틀린의 모든 설계는 세 힘의 합성이다.

첫째, **실용주의**. "현업이 오늘 쓸 언어"라는 목표가 이론적 순수함보다 앞선다. 그래서 `var`도, 예외도, `null`도, `!!`도 남았다. 순수 함수형이 아니라 다중 패러다임인 이유다.

둘째, **안전에의 집착**. "런타임 오류를 컴파일 타임으로 당긴다"는 원칙이 언어의 뼈대를 이룬다. 널 안전, 봉인 클래스, `when` 완전성, 기본 `final`, 명시적 `override` — 모두 이 원칙의 표현이다. 실용주의와 안전은 자주 부딪치고, 코틀린은 대개 안전 쪽으로 살짝 기운다(그러나 극단으로 가진 않는다 — `!!`라는 탈출구를 남긴 것처럼).

셋째, **상호운용/이식성**. Java와 공존해야 한다는 제약이 초기 설계를 규정했고, 나중엔 네 타깃으로의 이식성이 언어를 특정 런타임에 매이지 않게 만들었다. `Any`가 `Object`가 아닌 것, stdlib에 공통 부분이 있는 것, JVM 전용 사실을 언어 사실과 구분해야 하는 것 — 모두 이 힘의 산물이다.

```text
코틀린 설계를 빚은 세 힘 (요약)

  실용주의 ──────┐
  "당장 쓸 언어"  │
                ├──→  다중 패러다임, 통제된 가변성/예외/null,
  안전에의 집착 ──┤       타입에 새긴 안전, 명시 강제,
  "오류를 컴파일   │       Java 공존 + 4개 타깃 이식성
   타임으로"      │
                │
  상호운용/이식성 ┘
  "Java 공존, 멀티타깃"
```

### 9.2 이 장이 다루지 않는 것 — 경계 설정

이 장은 코틀린의 큰 지형도를 그렸을 뿐, 그 어느 지형의 세부도 파고들지 않았다. 명확히 경계를 긋자.

- **컴파일러가 실제로 어떻게 동작하는가** — K2 프론트엔드, FIR, IR, 백엔드별 코드 생성, 증분 컴파일, 컴파일러 플러그인 → [[02 - 컴파일러의 해부 - K2와 IR 백엔드와 바이트코드]].
- **문법의 최소 단위** — 토큰, 키워드 분류, 세미콜론 추론, 리터럴 → [[03 - 렉시컬 구조와 세미콜론 추론]].
- **타입 계층의 정확한 형태** — `Any`/`Unit`/`Nothing`의 위상, nullable의 타입 이론적 자리 → [[05 - 타입 시스템의 지형 - Any와 Unit과 Nothing]].
- **널 안전의 기계적 세부** — `?.`/`?:`/`!!`/`as?`, 스마트 캐스트 알고리즘, 플랫폼 타입 → [[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]].
- **원시 타입과 박싱의 진실** — "코틀린엔 원시 타입이 없다"(M4)의 정확한 의미 → [[07 - 수 1 - Int와 Long과 부동소수점]], [[08 - 수 2 - 박싱과 오버플로와 비트와 부호 없는 정수]].
- **Java 상호운용의 총람** — 플랫폼 타입, 애노테이션, 이름 맹글링 → [[45 - Java 상호운용]].
- **멀티플랫폼의 구체적 문법과 언어 진화** — `expect`/`actual`, 소스셋 위계, 옵트인, 안정성 등급 → [[46 - 멀티플랫폼과 진화와 어두운 구석]].

이 장에서 딱 하나를 가져가야 한다면 이것이다: **코틀린은 JVM 위에 얹은 Java의 설탕이 아니라(M1), 자체 명세와 타입 시스템을 가진 별개의 언어이며, JVM은 그 언어가 딛는 네 땅 중 하나(M2)다.** 이 두 교정을 지반 삼아, 다음 장부터 우리는 그 별개의 언어가 기계 수준에서 어떻게 작동하는지를 하나씩 해부한다.

지금 우리는 지도의 축척과 범례를 읽은 셈이다. "코틀린 = Java 설탕"이라는 낡은 등식과 "코틀린 = JVM"이라는 좁은 등식을 둘 다 내려놓고 나면, 비로소 이 언어의 실제 지형 — 널이 타입에 새겨지고, 함수가 일급 시민이며, 하나의 소스가 네 개의 런타임으로 갈라지는 그 지형 — 이 눈에 들어온다. 이 장이 답하려 했던 질문들, "설탕인가 언어인가", "JVM 언어인가 멀티타깃 언어인가", "왜 이렇게 생겼는가"는 결국 하나의 답으로 수렴한다: 코틀린은 실용주의와 안전과 이식성이라는 세 힘이 협상한 결과로서, 그 협상의 흔적을 언어 곳곳에 새긴 독립된 언어다. 그 흔적을 하나하나 읽어내는 것이 앞으로 45개 장이 할 일이다.

## 핵심 요약

- **코틀린은 Java의 문법 설탕이 아니라 별개의 언어다 (M1).** 널 가능성이 타입에 새겨지고, `Nothing` 바닥 타입·선언 지점 변성·스마트 캐스트·확장 함수·프로퍼티·`when` 표현식 등 Java에 대응 개념 자체가 없는 것들을 가진다. "무엇으로 컴파일되는가"와 "어떤 언어인가"는 다른 질문이다.
- **코틀린은 JVM 언어가 아니라 멀티타깃 언어다 (M2).** JVM/Native(LLVM)/JS/Wasm이 정식 타깃이며(성숙도는 다름 — Wasm은 상대적으로 발전 중), 타입 계층 최상단이 `java.lang.Object`가 아니라 `kotlin.Any`라는 사실이 이를 증명한다.
- **코틀린은 도구 회사 JetBrains의 실용적 필요에서 태어났다.** Java와의 100% 상호운용 + 빠른 컴파일 + 더 나은 안전이라는 세 조건을 동시에 만족하는 언어가 없어서 만들었고, 이 출신이 언어의 실용주의를 뿌리부터 규정했다.
- **네 개의 설계 기둥은 간결·안전·상호운용·도구다.** 이 넷은 조화롭지만은 않으며(간결 vs 안전, 안전 vs 상호운용), 코틀린 설계의 묘미는 그 긴장을 균형 잡은 방식에 있다.
- **언어를 세우는 세 축은 정적·강타입, 널 안전, 함수형 지향이다.** 정적·강타입이 바닥이고, 그 위에 널 안전이라는 안전벽을, 다시 그 위에 함수형 지향이라는 표현력을 얹었으며, 실용주의가 셋 모두를 극단에서 절제시킨다.
- **널 안전은 대부분 컴파일 타임 보장이다 (M7 참조).** 실패의 가능성을 타입에 인코딩하고 컴파일러가 추적하게 하는 것이 코틀린식 안전의 핵심 기법이며, 런타임 비용이 거의 없다.
- **하나의 소스가 네 타깃으로 컴파일되지만, 모든 코드가 네 타깃에서 도는 것은 아니다.** 언어 코어·타입 시스템·공통 stdlib는 공통이고, 호스트 API(Java·DOM·POSIX)와 저수준 표현(박싱·메모리 모델)은 타깃마다 갈라진다.
- **JVM 전용 사실은 반드시 "JVM 백엔드에서"라고 한정해야 한다.** "코틀린은 `Int`를 원시형으로 컴파일한다"는 JVM 백엔드의 사실이지 언어의 사실이 아니다(M4, M5 예고). 이 한정 습관이 M2를 실무에서 예방한다.
- **코틀린 멀티플랫폼(KMP)은 "로직 공유, 플랫폼별 부분 분리"의 실용적 접근이다.** `commonMain`에 공통 로직을, `expect`/`actual`로 플랫폼별 구현을 두며, "모든 것을 공유"가 아니라 "공유해서 이득인 것만 공유"한다.
- **코틀린에는 언어 명세가 존재하며, 명세·컴파일러·표준 라이브러리는 다른 층이다.** 명세는 언어의 의미론을 정의(별개 언어의 증거), 컴파일러는 그것을 특정 런타임에 구현, stdlib는 `let`·`map`·`listOf` 같은 "코틀린처럼 보이는 라이브러리"를 제공한다 — 이들은 키워드가 아니다.
- **코틀린은 Java를 대체하려는 게 아니라 공존하려 했고, 그 제약이 설계를 규정했다.** 플랫폼 타입 `T!`(M8 예고), 검사 예외 부재의 `@Throws` 배려, 프로퍼티의 게터/세터 컴파일 등은 모두 Java 공존의 흔적이자 실용적 타협이다.
- **코틀린의 모든 설계는 실용주의·안전에의 집착·상호운용/이식성이라는 세 힘의 합성이다.** 어떤 설계가 순수 철학에서 왔고 어떤 것이 Java 타협에서 왔는지 구분하는 것이 언어를 깊이 이해하는 열쇠다.
