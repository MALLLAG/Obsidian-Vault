**이 장이 답하는 질문:**

- "Kotlin 컴파일러"는 하나의 프로그램인가, 아니면 여러 개의 부품이 이어 붙은 파이프라인인가?
- 2.0에서 기본이 된 K2/FIR은 옛날 컴파일러(K1)와 정확히 무엇이 다른가? 왜 JetBrains는 프론트엔드를 통째로 다시 썼는가?
- 하나의 소스가 어떻게 JVM 바이트코드·Native 기계어·JavaScript·Wasm이라는 네 갈래로 갈라지는가? 그 갈림길에 있는 IR(중간 표현)은 무엇인가?
- `inline`, `reified`, `const`는 런타임 마법인가, 아니면 컴파일 타임에 이미 흔적도 없이 사라지는가?
- `Foo.kt` 파일 하나는 몇 개의 `.class` 파일로 변하는가? 아무 데도 안 만든 `FooKt`라는 클래스는 어디서 왔는가?
- 타입 추론은 파이프라인의 어느 단계에서, 무엇을 보고 일어나는가?
- kapt와 KSP는 왜 성능이 다르며, all-open 같은 플러그인은 컴파일러의 어느 지점에 끼어드는가?
- 증분 컴파일은 무엇을 재사용하고 무엇을 다시 계산하는가?
- 내가 쓴 코드가 실제로 어떤 바이트코드가 됐는지 어떻게 눈으로 확인하는가?

[[01 - 코틀린이란 무엇인가 - 설계 철학과 세 개의 타깃]]에서 우리는 Kotlin이 "JVM 언어"가 아니라 JVM·Native·JS·Wasm이라는 네 타깃을 겨냥하는 멀티플랫폼 언어이며(M2), Java의 문법 설탕이 아니라 널 가능성·변성·`Nothing`을 가진 독립된 타입 시스템을 갖춘 언어임을(M1) 확인했다. 그 주장들은 하나의 사실을 전제로 한다. **언어의 의미는 컴파일러가 결정한다.** `val`이 재대입 불가라는 것, `Int?`가 널을 표현할 수 있다는 것, 확장 함수가 정적으로 디스패치된다는 것 — 이 모든 규칙은 명세에 적혀 있고, 컴파일러가 그 명세를 코드로 강제한다. 그래서 언어를 기계 수준에서 이해하려면, 그 규칙을 집행하는 기계 자체를 열어봐야 한다.

이 장은 그 기계를 해부한다. 소스 텍스트 한 조각이 렉서(lexer)를 지나 구문 트리가 되고, 프론트엔드가 이름을 해소(resolution)하고 타입을 추론하고 진단(diagnostic)을 뱉고, 공통 중간 표현(IR)으로 낮춰진 뒤, 네 개의 백엔드 중 하나가 그것을 실제 타깃 코드로 바꾸는 여정을 따라간다. 특히 2.0에서 기본이 된 **K2/FIR** 프론트엔드가 무엇을 다시 설계했는지, 그리고 왜 하나의 프론트엔드에서 네 개의 백엔드가 뻗어 나오는 IR 구조가 Kotlin 진화의 핵심이었는지를 본다.

이 장이 *다루는 것*은 파이프라인의 뼈대와, 그 위에서 벌어지는 "컴파일 타임에 무엇이 확정·소거·인라이닝되는가"이다. 이 장이 *다루지 않는 것*: `inline`/`reified`의 의미론적 세부는 [[15 - 인라인 함수와 reified]]가 소유하고, 타입 소거의 전모는 [[34 - 제네릭 2 - 타입 소거와 reified와 바운드]]가, `suspend`의 CPS 변환은 [[42 - suspend와 코루틴 - 언어 수준의 중단]]가, Java 상호운용의 바이트코드 계약은 [[45 - Java 상호운용]]가 소유한다. 여기서는 그것들을 "컴파일러의 어느 단계에서 처리되는가"라는 파이프라인의 좌표로만 자리매김한다.

논증의 궤적은 이렇다. 먼저 컴파일이 왜 단일 변환이 아니라 여러 단계의 파이프라인인지를 세우고(1절), 프론트엔드가 텍스트를 의미로 바꾸는 과정을 렉서부터 진단까지 따라간 뒤(2·3·4절), 그 의미가 IR이라는 공통 나무로 응결되어 네 백엔드로 갈라지는 지점(5절)과 JVM 백엔드가 그 나무를 `.class`로 낮추는 구체(6절)를 본다. 그다음 컴파일 타임에 흔적을 지우는 변환들(7절), 컴파일러에 외부인이 끼어드는 통로인 플러그인(8절), 이 모든 것을 빠르게 반복하는 증분 컴파일(9절), 마지막으로 이 모든 주장을 직접 눈으로 검증하는 디컴파일 실습(10절)으로 닫는다.

## 1. 컴파일러라는 파이프라인 — 왜 한 덩어리가 아닌가

### 1.1 "번역"은 하나의 도약이 아니다

순진한 그림에서 컴파일러는 텍스트를 받아 기계어를 뱉는 블랙박스다. 실제로는 그렇지 않다. `kotlinc`가 하는 일은 소스 텍스트라는 *구조 없는 문자열*을, 의미가 완전히 확정된 *구조화된 표현*으로 여러 번에 걸쳐 조금씩 끌어올리는 것이다. 각 단계는 앞 단계의 산출물을 입력으로 받아, 정보가 더 풍부하고 더 낮은(기계에 가까운) 표현을 내놓는다.

이렇게 단계를 나누는 이유는 세 가지다. 첫째, **분리된 관심사**. "이 텍스트가 문법적으로 옳은가"와 "이 이름이 무엇을 가리키는가"와 "이 표현식의 타입은 무엇인가"는 서로 다른 질문이고, 다른 알고리즘을 요구한다. 둘째, **재사용**. 프론트엔드(의미 분석)가 만든 결과를 여러 백엔드가 공유할 수 있다면, JVM용·JS용·Native용 컴파일러가 타입 추론 로직을 각자 세 번 구현하지 않아도 된다. 셋째, **도구화**. IntelliJ IDEA의 코드 완성·인라인 에러 표시·리팩터링은 컴파일러의 프론트엔드와 같은 코드를 공유한다. 편집기가 매 키 입력마다 전체 컴파일을 돌릴 수는 없으므로, 프론트엔드는 "빠르고 부분적으로 실행 가능"하도록 설계돼야 한다.

```text
소스 텍스트 (.kt)
      │  렉서(lexer): 문자열 → 토큰 스트림
      ▼
   토큰 스트림
      │  파서(parser): 토큰 → 구문 트리(PSI/AST)
      ▼
   구문 트리                     ┐
      │  이름 해소·타입 추론·진단  │  프론트엔드
      ▼                          │  (K1: descriptors / K2: FIR)
   해소된 의미 트리               ┘
      │  (K1: PSI2IR / K2: FIR2IR)
      ▼
   백엔드 IR (공통 중간 표현)     ┐
      │  로워링(lowering) 패스들   │  백엔드
      ▼                          │  (JVM / JS / Native / Wasm)
   타깃 코드 (.class / .js / 기계어 / .wasm)  ┘
```

### 1.2 프론트엔드와 백엔드라는 두 세계

이 파이프라인에는 뚜렷한 이음매가 하나 있다. **프론트엔드**는 "이 프로그램은 무슨 뜻인가"에 답한다 — 파싱, 이름 해소, 타입 추론, 그리고 명세 위반에 대한 진단(에러·경고). 프론트엔드는 타깃을 모른다. `val x: Int = 3`이 JVM에서 `int`가 될지 JS에서 `number`가 될지는 프론트엔드의 관심사가 아니다. 프론트엔드의 산출물은 "타입이 다 붙고 이름이 다 해소된, 그러나 아직 어떤 기계와도 무관한 의미 트리"다.

**백엔드**는 "이 뜻을 이 기계에서 어떻게 실현하는가"에 답한다. 여기서 비로소 타깃이 등장한다. JVM 백엔드는 `when`을 `tableswitch`/`lookupswitch` 바이트코드로, 데이터 클래스의 `equals`를 실제 메서드로, 인라인 함수를 호출처에 펼친 코드로 낮춘다. JS 백엔드는 같은 의미 트리를 JavaScript 함수와 프로토타입으로 낮춘다.

이 분리가 Kotlin에 준 실질적 이득은 [[01 - 코틀린이란 무엇인가 - 설계 철학과 세 개의 타깃]]에서 말한 멀티타깃 전략 그 자체다. 하나의 언어 의미론(프론트엔드)에서 네 개의 실행 환경(백엔드)으로 뻗어 나갈 수 있는 것은, 이음매가 명확한 파이프라인 덕분이다. 그 이음매의 이름이 바로 5절에서 볼 **IR**이다.

> **명세 정밀**: "컴파일러"는 단수 명사처럼 쓰이지만 실제로는 다수의 협력 컴포넌트다. `kotlinc-jvm`, `kotlinc-js`, Native 컴파일러(`konanc` 계열)는 프론트엔드를 공유하면서 백엔드가 다른, 사실상 서로 다른 실행 파일이다. "Kotlin 컴파일러가 X를 한다"는 서술을 볼 때는 항상 "프론트엔드가 하는가, 특정 백엔드가 하는가"를 되물어야 한다. 예컨대 원시 타입으로의 컴파일(M4)은 **JVM 백엔드**의 결정이지, 언어나 프론트엔드의 결정이 아니다.

### 1.3 한 번 훑기 vs 여러 번 훑기

옛 언어(초기 C, Pascal)의 컴파일러는 성능·메모리 제약 때문에 소스를 앞에서 뒤로 한 번만 훑는 단일 패스(single-pass)를 지향했고, 그래서 "사용 전에 선언"을 문법으로 강제했다. Kotlin은 다르다. 최상위 선언의 순서가 자유롭고([[04 - 파일과 패키지와 선언 - 프로그램의 골격]]에서 다룰 전방 참조), 타입 추론이 함수 경계를 넘어 협력하며, 완전성(exhaustiveness) 검사 같은 전역적 판단이 필요하다. 이는 컴파일러가 소스 전체를 **여러 번** 훑을 수 있어야 함을 뜻한다.

```kotlin
fun describe(n: Int): String = classify(n)     // 아래에 선언된 함수를 위에서 참조
fun classify(n: Int): String =
    if (n > 0) "양수" else if (n < 0) "음수" else "영"
// => 컴파일 성공. 선언 순서 무관.
```

이 코드가 컴파일되려면, `describe`를 분석하는 시점에 이미 `classify`의 시그니처가 알려져 있어야 한다. 그래서 프론트엔드는 최소 두 겹으로 일한다. 먼저 모든 선언의 *헤더*(시그니처)를 수집하고, 그다음에 각 함수의 *본문*을 분석한다. 이 "헤더 먼저, 본문 나중" 구조는 K2/FIR의 다단계 해소에서 훨씬 정교해지는데, 그 이야기는 3절에서 한다.

## 2. 프론트엔드 1 — 렉서와 파서, 텍스트가 트리가 되기까지

### 2.1 렉서 — 문자에서 토큰으로

파이프라인의 첫 관문은 렉서(어휘 분석기)다. 렉서는 소스를 문자 단위로 읽어 **토큰(token)** 이라는 의미 단위로 자른다. `val x = 42`는 `[val][x][=][42]`라는 네 토큰(키워드, 식별자, 연산자, 정수 리터럴)이 된다. 공백과 줄바꿈, 주석은 대부분 여기서 걸러지지만, Kotlin에서는 줄바꿈이 완전히 버려지지 않는다 — 세미콜론 추론([[03 - 렉시컬 구조와 세미콜론 추론]] 소유)이 줄바꿈 정보를 필요로 하기 때문이다.

렉서 수준에서 이미 언어의 성격이 드러난다. Kotlin의 식별자는 유니코드 문자를 허용하고, 백틱으로 감싸면 키워드조차 이름으로 쓸 수 있다(``` `fun` ```, ``` `is` ```). 문자열 리터럴은 세 종류(`"..."`, `"""..."""`, 템플릿 `$x`)를 구분해야 하고, 숫자 리터럴은 밑줄 구분자·진법 접두사·타입 접미사(`1_000`, `0xFF`, `3.14f`, `100L`)를 인식해야 한다. 이 어휘 규칙들은 [[03 - 렉시컬 구조와 세미콜론 추론]]와 [[07 - 수 1 - Int와 Long과 부동소수점]]가 자세히 다룬다.

> **흔한 오해**: "렉서가 타입을 안다"는 착각. 렉서는 `42`가 정수 리터럴이라는 *어휘적* 사실만 안다. 그 값이 `Int`인지 `Long`인지, 나아가 `Byte`로 좁혀지는지는 훨씬 뒤인 타입 추론 단계(4절)에서 문맥을 보고 결정된다. 어휘 분류와 타입 결정은 파이프라인상 멀리 떨어진 별개의 사건이다.

### 2.2 파서 — 토큰에서 구문 트리로

파서는 토큰 스트림을 받아 언어 문법에 따라 **구문 트리**를 만든다. 역사적으로 Kotlin 컴파일러와 IntelliJ는 이 트리를 PSI(Program Structure Interface)라는 형태로 표현해 왔다. PSI는 편집기 친화적이다 — 원본 텍스트의 모든 문자(공백·주석 포함)를 보존하고, 노드가 원본의 어느 위치에서 왔는지 추적한다. 그래서 같은 트리로 컴파일도 하고, "이 심볼의 정의로 이동" 같은 IDE 기능도 구현한다.

```kotlin
val total = price * quantity + tax
```

이 한 줄이 만드는 트리의 골격은 이렇다.

```text
프로퍼티 선언 (val total = ...)
└─ 대입식 초기화
   └─ 이항식 (+)
      ├─ 이항식 (*)
      │  ├─ 참조 (price)
      │  └─ 참조 (quantity)
      └─ 참조 (tax)
```

파서가 `*`를 `+`보다 트리의 더 깊은 곳에 놓는 것에 주목하자. 연산자 우선순위는 문법에 고정돼 있고([[19 - 연산자 오버로딩과 관례]]에서 볼 M40의 근거), 사용자가 `operator fun times`를 정의하더라도 우선순위는 바꾸지 못한다 — 이는 언어 문법이 파서 단계에서 이미 못 박은 사실이기 때문이다. 파싱 단계는 아직 `price`가 무엇인지, `*`가 어떤 함수를 부르는지 전혀 모른다. 그것은 다음 단계의 일이다.

### 2.3 구문 오류 대 의미 오류

파서 단계에서 나오는 진단은 **구문 오류**다: 괄호 짝이 안 맞음, `val` 뒤에 이름이 없음, `when`의 분기 문법이 깨짐 같은, "문장 구조 자체가 문법에 어긋남"이다. 파서는 관대하다 — 하나의 오류에서 멈추지 않고 최대한 복구(error recovery)해 트리를 마저 만들려 한다. 그래야 편집기가 한 번에 여러 오류를 표시하고, 오류 지점 이후의 코드에도 완성 제안을 줄 수 있다.

```kotlin
val x = (1 + 2          // 컴파일 에러: 닫는 괄호 ')' 가 없음 (구문 오류)
fun f(): Int = "hello"  // 컴파일 에러: String을 Int로 반환할 수 없음 (의미/타입 오류)
```

두 오류는 성격이 다르다. 첫 줄은 파서가 잡는다("구조가 안 됨"). 둘째 줄은 파서를 통과한다 — 문법적으로는 완벽히 옳은 함수 선언이다. 그 오류는 **의미 분석**, 즉 타입 검사에서야 드러난다. 이 구분은 실무에서 중요하다. "빨간 줄"이 떴을 때 그것이 구조의 문제인지 의미의 문제인지를 알면, 어느 단계를 의심할지가 정해진다.

## 3. 프론트엔드 2 — K1에서 K2로, FIR이라는 재설계

### 3.1 K1의 한계 — 디스크립터와 BindingContext

파싱 이후, 프론트엔드는 트리의 각 이름과 표현식에 "의미"를 붙여야 한다. `price`는 어느 프로퍼티를 가리키는가? `price * quantity`의 `*`는 어느 `times` 함수를 호출하는가? 그 결과 타입은 무엇인가? 2.0 이전의 컴파일러(오늘날 K1이라 부른다)는 이 정보를 **디스크립터(descriptor)** 라는 객체와 **BindingContext**라는 거대한 조회 테이블에 담았다.

문제는 이 설계가 두 가지 표현을 병행 관리했다는 점이다. 한쪽에는 구문을 담은 PSI 트리가 있고, 다른 쪽에는 의미를 담은 디스크립터들이 있으며, BindingContext가 "이 PSI 노드 → 이 디스크립터"라는 방대한 매핑으로 둘을 이어 붙였다. 이 구조는 오래 작동했지만 무거웠다. 새 언어 기능(예: 더 똑똑한 스마트 캐스트, 더 정교한 타입 추론)을 넣을 때마다 두 표현과 그 사이 매핑을 함께 손봐야 했고, 이는 컴파일 속도와 유지보수성 모두에 짐이었다.

### 3.2 K2와 FIR — 하나의 진화하는 트리

K2 프론트엔드의 핵심은 **FIR(Front-end Intermediate Representation, 프론트엔드 중간 표현)** 이다. 아이디어는 이렇다. 별도의 디스크립터 세계와 그 매핑을 두는 대신, **단일한 트리를 여러 번의 변환 패스로 점진적으로 풍부하게 만든다.** 처음에는 이름도 안 해소되고 타입도 안 붙은 "날 것의 FIR(raw FIR)"이 PSI에서 생성되고, 그다음 일련의 해소 단계(resolution transformer)가 이 트리를 제자리에서 조금씩 채워 나가, 마지막에는 모든 이름이 해소되고 모든 표현식에 타입이 붙은 "완전히 해소된 FIR"이 된다.

```text
raw FIR
  │  1) 임포트/패키지 해소
  │  2) 슈퍼타입 해소   (클래스 계층 확정)
  │  3) 타입 해소       (선언에 쓰인 타입 참조 확정)
  │  4) 시그니처(헤더) 해소  (함수/프로퍼티 시그니처 확정)
  │  5) 본문 해소       (표현식·타입 추론·스마트캐스트·진단)
  ▼
resolved FIR  ── (FIR2IR) ──▶  백엔드 IR
```

이 다단계 구조가 1.3절에서 말한 "헤더 먼저, 본문 나중"을 명시적 파이프라인으로 승격시킨 것이다. 슈퍼타입과 시그니처가 본문보다 먼저 확정되기 때문에, 선언 순서와 무관하게 서로를 참조할 수 있고, 순환 참조(A가 B를 상속하고 B가 A를 참조)도 단계별로 안전하게 다뤄진다.

> **역사 메모**: K2는 2.0(2024년 5월)에서 기본 컴파일러가 됐다. "K2"는 컴파일러 세대를, "FIR"은 그 프론트엔드가 쓰는 자료 표현을 가리키는 말이라 종종 같은 의미로 통용되지만 엄밀히는 다르다 — FIR은 K2 프론트엔드의 *구현 표현*이고, K2는 그 프론트엔드를 포함한 컴파일러 전체의 이름이다. 백엔드 IR(5절)과 혼동하지 말 것: 이름이 둘 다 "IR"로 끝나지만, FIR은 프론트엔드의 의미 트리, (백엔드) IR은 프론트엔드와 백엔드의 이음매에 있는 별개의 표현이다.

### 3.3 K2가 바꾼 사용자 체감

K2는 대부분 "더 빠르고 더 정확"이라는 형태로 조용히 체감된다. 컴파일 속도가 개선됐고, 특히 스마트 캐스트([[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]] 소유)와 타입 추론이 이전보다 더 많은 경우를 이해한다. 예를 들어 제어 흐름을 통과한 널 가능성 판단이 더 정밀해졌다.

```kotlin
fun printLength(s: String?) {
    if (s == null) return
    // 여기서 s는 스마트 캐스트로 String (Nothing 반환 return이 흐름을 끊었으므로)
    println(s.length)   // => 안전. ?. 불필요.
}
```

이런 흐름 분석은 K1도 상당 부분 했지만, K2는 흐름 분석과 타입 추론을 더 통합된 방식으로 처리해 경계 사례에서 덜 놀란다. 다만 이 장에서 강조할 것은 결과가 아니라 *위치*다. **스마트 캐스트도, 널 안전성 검사도, 완전성 검사도 모두 이 프론트엔드 단계에서 벌어진다.** 즉 대부분 컴파일 타임 사건이다(M7과 연결 — 널 안전성은 런타임 검사가 아니라 주로 컴파일 타임 분석이다). 런타임에 남는 검사는 `!!`, 플랫폼 타입, 리플렉션, Java 상호운용처럼 컴파일러가 확신할 수 없는 지점뿐이다.

## 4. 프론트엔드 3 — 타입 추론, 컴파일러가 타입을 만들어내는 곳

### 4.1 추론은 "생략"이 아니라 "계산"이다

Kotlin은 정적·강타입 언어지만 타입을 거의 안 적는다. `val x = 42`에서 `x`가 `Int`인 것을 우리가 안 적어도 컴파일러가 안다. 이 "안다"는 것은 단순히 오른쪽을 왼쪽에 복사하는 일이 아니다. 타입 추론(type inference)은 표현식들 사이의 제약(constraint)을 수집하고 그것을 만족하는 타입을 *푸는* 과정이다.

```kotlin
val a = 42            // Int (정수 리터럴의 기본 타입)
val b = 42L           // Long (L 접미사)
val c = 3.14          // Double (부동소수 리터럴의 기본 타입)
val d = listOf(1, 2, 3)          // List<Int>
val e = listOf(1, 2L)            // List<Long>  ← 공통 상위 타입으로 통일
val f = listOf(1, "x")           // List<Any>   ← 더 위로 올라감
val g = if (cond) 1 else "x"     // Any (K2 기준. 참고로 K1은 Comparable<*> & Serializable 교집합을 줬다)
```

`listOf(1, 2L)`에서 컴파일러는 두 인자의 타입(`Int`, `Long`)을 보고 원소 타입 `T`에 대한 제약을 세운 뒤, 둘을 모두 담을 수 있는 최소 공통 상위 타입을 계산해 `Long`으로 정한다. `listOf(1, "x")`는 `Int`와 `String`의 공통 상위 타입인 `Any`로 올라간다(Kotlin의 타입 계층은 [[05 - 타입 시스템의 지형 - Any와 Unit과 Nothing]]가 소유한다). 추론은 이렇게 "가장 구체적이면서 모두를 만족하는 타입"을 찾는 최적화 문제에 가깝다.

### 4.2 리터럴의 타입은 문맥이 정한다

숫자 리터럴은 타입 추론이 문맥과 협력하는 좋은 예다. `42`는 그 자체로는 "정수 리터럴"일 뿐, 어떤 구체 타입도 확정되지 않은 상태다. 기대 타입(expected type)이 있으면 그쪽으로 맞춰지고, 없으면 기본형 `Int`가 된다.

```kotlin
val x: Long = 42       // 기대 타입이 Long → 42가 Long 리터럴로 해석됨. toLong() 불필요.
val y: Byte = 42       // 기대 타입 Byte, 42가 Byte 범위 안 → OK
val z: Byte = 200      // 컴파일 에러: 200은 Byte(-128..127) 범위를 벗어남
val w = 42             // 기대 타입 없음 → 기본형 Int
```

`val x: Long = 42`가 `42.toLong()` 없이도 되는 것은, 기대 타입 `Long`이 리터럴 해석에 되먹임되기 때문이다. 반면 변수끼리는 이런 자동 확대가 없다 — Kotlin에는 암시적 확대 변환이 없다(M4의 이웃 주제, [[07 - 수 1 - Int와 Long과 부동소수점]] 소유).

```kotlin
val i: Int = 42
val l: Long = i        // 컴파일 에러: Int는 Long이 아님. i.toLong() 필요.
```

리터럴 `42`(문맥이 타입을 정할 수 있는 상수식)와 변수 `i`(이미 타입이 `Int`로 확정된 값)의 차이가 여기서 갈린다. 전자는 추론이 기대 타입을 향해 굽힐 수 있는 대상이고, 후자는 이미 굳은 값이다.

### 4.3 추론이 함수 경계를 넘을 때 — 그리고 그 한계

람다와 제네릭이 얽히면 추론은 인자와 반환을 넘나든다.

```kotlin
val lengths = listOf("a", "bb", "ccc").map { it.length }
// map의 원소 타입 String → it: String → it.length: Int → 결과 List<Int>
```

`map`의 수신 리스트가 `List<String>`이므로 람다 파라미터 `it`이 `String`으로 추론되고([[13 - 함수 타입과 람다와 함수 참조]]에서 볼 M42의 `it`), 그 결과 `it.length`가 `Int`, 최종 결과가 `List<Int>`로 흐른다. 추론은 이렇게 "밖에서 안으로"(리스트 타입 → 람다 파라미터) 그리고 "안에서 밖으로"(람다 반환 → 결과 타입) 정보를 양방향으로 흘린다.

그러나 추론에는 원칙적 한계가 있다. **함수의 공개 시그니처(파라미터·반환 타입)는 명시적이어야 하는 경우가 많다.** 재귀 함수의 반환 타입, 공개 API의 반환 타입 등은 추론에만 맡기면 순환이 생기거나 의도가 흐려진다.

```kotlin
fun factorial(n: Int) = if (n <= 1) 1 else n * factorial(n - 1)
// 컴파일 에러: 재귀 함수는 반환 타입을 추론할 수 없음 → 명시 필요
fun factorial2(n: Int): Int = if (n <= 1) 1 else n * factorial2(n - 1)  // OK
```

`factorial`을 추론하려면 `factorial`의 반환 타입을 이미 알아야 하는 순환이 생긴다. 그래서 컴파일러는 재귀 함수에 반환 타입 명시를 요구한다. 이는 추론이 만능이 아니라, "지역적으로 결정 가능한 것"과 "전역적 계약으로 못 박아야 하는 것"을 구분하는 언어 설계의 반영이다.

> **성능 주의**: 타입 추론, 특히 복잡한 제네릭·람다 체인의 추론은 컴파일러가 가장 무겁게 일하는 지점 중 하나다. 아주 긴 빌더 체인이나 깊이 중첩된 람다에서 컴파일이 느려진다면, 중간 지점에 명시적 타입을 달아 제약 해결의 범위를 끊어 주는 것이 실질적 처방이 된다. 추론은 편의지 공짜가 아니다.

## 5. IR — 하나의 나무, 네 개의 백엔드

### 5.1 왜 공통 중간 표현이 필요했나

프론트엔드가 완전히 해소된 FIR을 만들었다. 이제 이것을 실제 코드로 바꿔야 하는데, 타깃이 넷이다. 순진하게 하면 "FIR → JVM 바이트코드", "FIR → JS", "FIR → 기계어", "FIR → Wasm"이라는 네 개의 독립 변환기를 만들게 되고, 각 변환기는 데이터 클래스 생성·인라인 함수 펼치기·`when` 완전성 등 수많은 공통 로직을 제각각 다시 구현해야 한다. 유지보수 악몽이다.

해법이 **IR(Intermediate Representation, 중간 표현)** 이다. 프론트엔드의 결과를 먼저 타깃 독립적인 공통 IR로 낮추고(FIR2IR), 그다음 그 IR에 대해 **로워링(lowering)** 이라 부르는 변환 패스들을 적용해 점점 저수준으로 만든 뒤, 최종 단계에서만 타깃별 코드로 방출한다. 많은 로워링(예: 데이터 클래스의 멤버 합성, 지역 함수 처리, `when`의 정규화)은 타깃과 무관하므로 IR 수준에서 한 번만 구현해 모든 백엔드가 공유한다.

```text
                     resolved FIR
                          │  FIR2IR
                          ▼
                    공통 백엔드 IR
                          │  타깃 공통 로워링 패스들
                          ▼
             ┌────────────┼────────────┬────────────┐
             ▼            ▼            ▼            ▼
        JVM 백엔드    JS 백엔드    Native 백엔드   Wasm 백엔드
             │            │            │            │
             ▼            ▼            ▼            ▼
        .class 파일    .js 파일   LLVM→기계어     .wasm
```

이 IR 기반 구조로의 통합은 Kotlin 백엔드 진화의 큰 줄기였다. JVM 백엔드가 오랫동안 IR 없이 PSI에서 곧장 바이트코드를 뽑던 시절이 있었지만, 지금은 JVM을 포함한 모든 타깃이 공통 IR을 거친다. 그 결과 언어 기능 하나를 추가할 때 백엔드 로직을 네 번 고치는 대신, IR 로워링을 한 번 고쳐 네 타깃에 반영하는 것이 가능해졌다.

### 5.2 로워링 — 고수준 개념을 저수준으로 "낮추기"

로워링은 IR 트리를 입력받아, 더 원시적인 IR 트리를 출력하는 변환이다. 이름 그대로, 언어의 고수준 개념을 기계가 이해할 수 있는 저수준 구성으로 "낮춘다". 예를 들어 다음 개념들이 IR 로워링에서 풀린다(각각의 상세 의미론은 소유 장에 있다).

| 고수준 개념    | 로워링이 하는 일                                             | 소유 장                                   |
| --------- | ----------------------------------------------------- | -------------------------------------- |
| 데이터 클래스   | `equals`/`hashCode`/`toString`/`componentN`/`copy` 합성 | [[27 - 데이터 클래스]]                       |
| 기본 인자     | 인자 채움 로직·합성 오버로드/`$default` 생성                        | [[12 - 함수 - 인자와 vararg와 지역 함수와 꼬리 재귀]] |
| 인라인 함수    | 호출처에 본문 펼치기, 람다 인라이닝                                  | [[15 - 인라인 함수와 reified]]               |
| `when`    | 조건 사슬/`switch`류로 정규화                                  | [[17 - 표현식으로서의 제어 흐름 - if와 when]]      |
| 지역/익명 함수  | 별도 함수/클래스로 추출                                         | [[14 - 클로저와 캡처]]                       |
| `suspend` | CPS·상태 기계로 변환                                         | [[42 - suspend와 코루틴 - 언어 수준의 중단]]      |

여기서 핵심 통찰: **우리가 "언어 기능"이라 부르는 많은 것이 실은 컴파일러가 IR 단계에서 자동 생성하는 코드다.** 데이터 클래스의 `copy`는 런타임 라이브러리가 주는 마법이 아니라, 컴파일러가 우리 대신 써 준 평범한 메서드다. 10절에서 디컴파일로 이 "써 준 코드"를 직접 보게 될 것이다.

### 5.3 IR가 있어도 백엔드마다 다른 것

IR이 공통이라고 해서 네 타깃의 결과가 같다는 뜻은 아니다. IR 이후, 각 백엔드는 자기 타깃의 현실에 맞춰 고유한 결정을 내린다. 대표적으로 **원시 타입**이 그렇다.

```kotlin
fun add(a: Int, b: Int): Int = a + b
```

JVM 백엔드에서 이 함수의 `Int`는 가능한 곳에서 원시 `int`로 컴파일된다(M4 — Kotlin 소스에 원시 타입이 없다고 해서 실행 코드에도 없는 게 아니다. 소스엔 없지만 JVM 백엔드가 원시형으로 낮춘다). JS 백엔드에서는 `int`라는 개념 자체가 없으니 JavaScript의 `number`로 표현된다. 같은 IR, 다른 실현.

널 가능성의 표현도 백엔드마다 다르다. `Int?`는 널을 표현해야 하므로 JVM 백엔드에서는 대개 박싱된 `java.lang.Integer`가 되지만(M5 — "항상 박싱"은 오해다. 널을 표현할 필요가 있을 때 박싱되며, 백엔드가 최적화할 여지도 있다), 그 정확한 조건은 [[08 - 수 2 - 박싱과 오버플로와 비트와 부호 없는 정수]]가 소유한다. 요점은 **"어떤 타입이 어떻게 표현되는가"는 프론트엔드가 아니라 백엔드의 관할**이라는 것이다. 그래서 이 장은 그런 질문을 "JVM 백엔드에서"라는 한정어와 함께만 다룬다.

> **명세 정밀**: "Kotlin은 원시 타입으로 컴파일한다"는 문장은 반쪽만 맞다. 정확히는 "JVM 백엔드가, 가능한 문맥에서, `Int` 같은 타입을 원시 `int`로 컴파일한다"이다. Native·JS·Wasm 백엔드에는 JVM의 `int`/`Integer` 이분법이 존재하지 않는다. 멀티타깃 언어를 논할 때 백엔드를 명시하지 않은 "컴파일된다" 서술은 항상 의심하라(M2·M4).

## 6. JVM 백엔드 — `.kt`가 `.class`가 되는 여정

### 6.1 파일 하나가 클래스 하나가 아니다

JVM은 "모든 코드는 클래스 안에 있다"는 세계다. 하지만 Kotlin은 파일 최상위에 함수와 프로퍼티를 둘 수 있다([[04 - 파일과 패키지와 선언 - 프로그램의 골격]] 소유). 이 둘을 화해시키기 위해, JVM 백엔드는 **파일당 하나의 합성 클래스**를 만들어 그 파일의 최상위 선언들을 담는다. 이름 규칙은 "파일명 + `Kt`"다.

```kotlin
// 파일: MathUtils.kt
package geometry

const val PI = 3.14159
fun areaOfCircle(r: Double): Double = PI * r * r
```

이 파일은 JVM 백엔드에서 `geometry.MathUtilsKt`라는 클래스로 컴파일되고, `areaOfCircle`은 그 클래스의 `public static` 메서드가 된다. 그래서 [[45 - Java 상호운용]]에서 Java 코드가 이 함수를 부를 때는 `MathUtilsKt.areaOfCircle(...)`처럼 클래스명을 거친다.

```java
// Java에서 본 모습
double a = geometry.MathUtilsKt.areaOfCircle(2.0);
```

이 클래스 이름은 `@file:JvmName`으로 바꿀 수 있고, 여러 파일의 최상위 선언을 한 클래스로 합칠 수도 있다(`@JvmMultifileClass`) — 그 상세는 [[45 - Java 상호운용]]가 소유한다. 여기서 배울 것은 **`FooKt`는 마법 클래스가 아니라 JVM의 "클래스만 있다" 제약을 만족시키려 컴파일러가 만든 컨테이너**라는 사실 하나다.

### 6.2 클래스 하나가 파일 하나가 아니다 — 반대 방향

역방향도 성립한다. 소스 파일 하나가 여러 `.class`를 만들 수 있다. 클래스마다 `.class`가 하나씩 나오는 것은 물론이고, 다음도 각각 별개 `.class`를 만든다.

```kotlin
// 파일: Shapes.kt
class Circle(val r: Double)                 // Circle.class
class Rectangle(val w: Double, val h: Double) // Rectangle.class

data class Point(val x: Int, val y: Int)    // Point.class (+ 합성 멤버 내장)

fun makeGreeter(name: String): () -> String {
    return { "Hi, $name" }   // 람다 → 별도 합성 클래스 (또는 인라인이면 사라짐)
}

object Registry                              // Registry.class (싱글턴, INSTANCE 필드 포함)
enum class Dir { N, S, E, W }                // Dir.class
```

람다·익명 객체·`object`·중첩 클래스([[31 - 중첩 클래스와 이너 클래스와 익명 객체]])는 각기 클래스 파일을 낳는다. 그래서 한 `.kt`를 컴파일하면 여러 개의 `.class`가 우수수 나오는 것이 정상이다. "파일 : 클래스 = 1 : 1"이라는 Java식 직관은 Kotlin에서 양방향으로 깨진다.

### 6.3 프로퍼티는 필드 하나가 아니다

Kotlin 프로퍼티는 필드가 아니라 접근자(getter/setter)의 추상이다(M35, [[22 - 프로퍼티와 백킹 필드]] 소유). JVM 백엔드는 이것을 다음으로 낮춘다: 필요하면 백킹 필드(private 필드) 하나, 그리고 `getX()`(그리고 `var`면 `setX(...)`) 메서드들.

```kotlin
class Account(var balance: Long)
```

JVM 백엔드에서 이 클래스는 대략 이렇게 낮춰진다(개념적 Java 등가물).

```java
public final class Account {
    private long balance;               // 백킹 필드
    public Account(long balance) { this.balance = balance; }
    public final long getBalance() { return this.balance; }        // getter
    public final void setBalance(long value) { this.balance = value; } // setter
}
```

`account.balance = 100`이라는 프로퍼티 대입은 실제로는 `account.setBalance(100)` 메서드 호출로 컴파일된다. 반대로 계산 프로퍼티(백킹 필드 없이 게터만 있는 것)는 필드 없이 메서드만 생성된다. `@JvmField`를 붙이면 접근자 없이 public 필드로 노출되는데, 이 트레이드오프의 상세는 [[22 - 프로퍼티와 백킹 필드]]와 [[45 - Java 상호운용]]가 다룬다.

### 6.4 클래스는 기본이 final이고, 함수는 기본이 static이 아니다

JVM 백엔드가 방출하는 접근 제어자에도 언어 철학이 새겨진다. Kotlin 클래스는 기본이 `final`이므로([[24 - 상속과 오버라이딩과 초기화 순서]] 소유), 위 `Account`의 바이트코드에서 클래스와 메서드에 `final`이 붙는다. `open`을 명시해야만 상속·오버라이드 가능한 형태로 방출된다. 이는 "상속은 명시적 설계 결정"이라는 Kotlin의 입장을 바이트코드 수준에서 강제한 것이다(8절의 all-open 플러그인이 왜 필요한지의 배경이기도 하다).

가시성도 매핑된다. `public`은 Java `public`으로, `private`은 `private`으로 간다. 하지만 `internal`(모듈 가시성, M44, [[26 - 가시성 한정자]] 소유)은 JVM에 대응 개념이 없어, `public`으로 방출하되 이름을 맹글링(mangling)해 Java에서 실수로 부르기 어렵게 만든다. 언어의 가시성 모델이 타깃의 가시성 모델과 정확히 겹치지 않을 때 백엔드가 어떻게 타협하는지를 보여주는 예다.

## 7. 컴파일 타임에 사라지는 것들 — inline, reified, const

### 7.1 `const val` — 상수는 호출처로 복사된다

`const val`은 컴파일 상수다(M36, [[11 - 변수와 초기화 - val과 const와 lateinit]] 소유). 이 사실의 백엔드적 의미는 강력하다. **`const val`을 참조하는 곳에는 그 이름이 아니라 값 자체가 바이트코드에 박힌다.**

```kotlin
const val MAX_RETRIES = 3
fun retryPolicy(): Int = MAX_RETRIES
```

`retryPolicy`의 바이트코드는 `MAX_RETRIES`라는 필드를 읽지 않는다. 그냥 상수 `3`을 상수 풀에서 로드한다(`iconst_3` / `ldc`류). 즉 컴파일 시점에 값이 호출처로 인라인되는 것이다. 이는 성능상 이점이지만 함정도 낳는다: 라이브러리의 `const val`을 바꾸고 그 라이브러리만 다시 빌드해도, 그 값을 이미 자기 바이트코드에 박아 넣은 소비자를 다시 컴파일하지 않으면 옛 값이 남는다. 이 "상수 인라이닝으로 인한 재컴파일 의존성"은 9절의 증분 컴파일이 신경 써야 하는 문제다.

```kotlin
val runtimeMax = 3            // 일반 val: 필드 read (인라인 아님)
const val compileMax = 3      // const val: 호출처에 값 3이 박힘
```

일반 `val`과의 차이는 여기서 갈린다. `const`는 원시 타입/String이며 최상위 또는 `object`/companion에 있고 리터럴로 초기화되는, 컴파일 타임에 값이 확정되는 상수만 될 수 있다 — 그 제약이 바로 "호출처에 값을 박아 넣기" 위한 전제 조건이다.

### 7.2 `inline` — 함수가 호출이 아니라 펼침이 된다

`inline` 함수는 호출되는 대신 호출처에 본문이 펼쳐진다(M17, [[15 - 인라인 함수와 reified]] 소유). 이것은 성능 힌트가 아니라 **의미론적 변환**이며, IR 로워링(5.2절) 단계에서 벌어진다.

```kotlin
inline fun measure(block: () -> Unit) {
    val start = System.nanoTime()
    block()
    println("elapsed: ${System.nanoTime() - start}")
}

fun run() {
    measure { println("일") }
}
```

`run`을 컴파일하면 `measure`를 호출하는 코드가 남지 않는다. 대신 `measure`의 본문과 람다 `{ println("일") }`이 `run` 안에 직접 펼쳐진다 — 개념적으로:

```kotlin
fun run() {              // 인라이닝 후의 개념적 모습
    val start = System.nanoTime()
    println("일")         // 람다도 인라인 → 함수 객체 할당 없음
    println("elapsed: ${System.nanoTime() - start}")
}
```

여기서 두 가지가 사라진다. 함수 호출의 프레임 비용과, **람다를 담을 함수 객체의 할당**이다. 인라인이 아닌 고차 함수라면 `{ println("일") }`이 `Function0` 인터페이스를 구현하는 객체로 만들어져 힙에 할당되지만([[13 - 함수 타입과 람다와 함수 참조]]의 FunctionN), 인라인이면 그 객체 자체가 존재하지 않게 된다. 이것이 인라인의 진짜 값어치이자, 동시에 코드가 여러 호출처에 복제되어 커지는 대가의 원천이다.

### 7.3 `reified` — 소거된 타입을 되살리는 컴파일 타임 트릭

제네릭 타입은 JVM에서 소거된다(M19, [[34 - 제네릭 2 - 타입 소거와 reified와 바운드]] 소유). 즉 런타임에 `List<String>`과 `List<Int>`는 구분되지 않는다. 그런데 `inline` 함수의 타입 파라미터에 `reified`를 붙이면, 마치 런타임에 타입을 아는 것처럼 `is T`나 `T::class`를 쓸 수 있다. 이것이 가능한 비결은 순전히 컴파일 타임에 있다.

```kotlin
inline fun <reified T> firstOfType(items: List<Any>): T? {
    for (item in items) if (item is T) return item   // is T 사용 가능
    return null
}

val nums = listOf("a", 1, "b", 2, 3.0)
val n = firstOfType<Int>(nums)   // => 1
```

`firstOfType<Int>`을 호출하면, 인라이닝될 때 `T`가 구체 타입 `Int`로 **치환**된다. 즉 호출처에는 `if (item is Int)`라는, 소거와 무관한 구체적 검사가 펼쳐진다. 런타임에 타입 정보를 실어 나른 게 아니라, 컴파일 타임에 각 호출처마다 타입을 박아 넣은 것이다. 그래서 `reified`는 반드시 `inline`과 함께여야 하고(치환할 "펼쳐질 자리"가 있어야 하므로), 최상위 타입만 되살릴 수 있다(M18 — `List<String>`의 `String`은 여전히 소거된다. `reified`가 아는 것은 `T` 자체이지 `T`의 타입 인자가 아니다).

```kotlin
inline fun <reified T> typeName(): String = T::class.simpleName ?: "?"
typeName<String>()          // => "String"
typeName<List<Int>>()       // => "List"  ← 안쪽 Int는 알 수 없음(소거)
```

이 세 예(`const`, `inline`, `reified`)의 공통 교훈: **런타임 마법처럼 보이는 많은 기능이 실은 컴파일 타임에 코드 변형으로 구현된다.** 성능·타입 정보가 "실행 중에 생긴다"고 오해하기 쉽지만, 컴파일러가 미리 코드를 써 두었을 뿐이다. 이 시각은 성능을 추론할 때(무엇이 할당되고 무엇이 안 되는가)와 한계를 이해할 때(왜 `reified`가 인라인 없이는 안 되는가) 모두 필요하다.

> **성능 주의**: 인라인이 좋다고 아무 함수에나 `inline`을 붙이면 안 된다. 본문이 큰 함수를 여러 곳에서 부르면, 그 본문이 모든 호출처에 복제되어 바이트코드가 팽창하고(코드 캐시 압박) 오히려 느려질 수 있다. `inline`의 정당한 이유는 대개 (1) 람다 인자의 객체 할당 제거, (2) `reified`, (3) 비지역 반환이다. 그 외의 "그냥 빠를 것 같아서"는 근거가 약하다. 자세한 판단 기준은 [[15 - 인라인 함수와 reified]]에.

## 8. 컴파일러 플러그인 — 외부인이 컴파일러에 끼어드는 통로

### 8.1 왜 플러그인이 필요한가 — final의 벽

6.4절에서 Kotlin 클래스는 기본 `final`이라고 했다. 그런데 일부 JVM 프레임워크(대표적으로 프록시 기반 라이브러리들)는 클래스를 런타임에 상속해 프록시를 만들어야 작동한다. `final` 클래스는 상속이 불가능하니 곤란하다. 매번 `open`을 손으로 붙이는 것은 번거롭고 실수가 나기 쉽다.

이 문제를 컴파일러 수준에서 푸는 것이 **컴파일러 플러그인**이다. `all-open` 플러그인은 지정한 애노테이션이 붙은 클래스들을 컴파일 시점에 자동으로 `open`으로 만든다. 소스에는 `final`이지만 바이트코드에는 `open`으로 방출되는 것이다.

```kotlin
// all-open 플러그인이 (가령) @Entity를 대상으로 설정돼 있다면
@Entity
class Product(val name: String)   // 소스상 final처럼 보이지만, 컴파일 결과는 open
```

비슷한 플러그인으로 `no-arg`(지정 애노테이션 클래스에 합성 무인자 생성자 추가), `kotlinx.serialization`(직렬화 코드 생성), `Compose` 관련 플러그인 등이 있다. 이들의 공통점은 **언어를 바꾸지 않고 컴파일러의 동작을 확장**한다는 것이다. 플러그인은 IR 로워링 단계 등에 훅(hook)을 걸어 IR을 추가·변형한다.

> **흔한 오해**: "이건 Kotlin 언어 기능이다"와 "이건 컴파일러 플러그인이 해 주는 것이다"를 혼동하기 쉽다. `data class`의 멤버 합성은 언어 명세의 일부(컴파일러 코어)지만, `@Serializable`의 직렬화 코드 생성은 별도 플러그인이다. 후자는 그 플러그인이 빌드에 설정돼 있을 때만 작동한다. "왜 이 애노테이션이 아무 효과가 없지?"의 흔한 답은 "해당 컴파일러 플러그인이 안 켜져 있다"이다.

### 8.2 kapt — Java 스텁을 거치는 옛길

애노테이션 프로세싱은 "애노테이션이 붙은 선언을 읽어 새 코드를 생성"하는, 오래된 JVM 생태계 기법이다. Java에는 `javac`의 애노테이션 처리 API(APT)가 있고, 수많은 라이브러리가 이를 기반으로 코드를 생성한다. 문제는 이 API가 **Java 소스 모델**을 전제한다는 것이다. Kotlin 코드는 Java 소스가 아니다.

`kapt`(Kotlin Annotation Processing Tool)의 해법은 우회다. Kotlin 코드로부터 **Java 스텁(stub)**, 즉 시그니처만 있고 본문은 빈 가짜 Java 소스를 생성한 뒤, 기존 Java 애노테이션 프로세서를 그 스텁 위에서 돌린다.

```text
Kotlin 소스 ──▶ (kapt) Java 스텁 생성 ──▶ javac APT ──▶ 생성 코드
                     ▲ 이 스텁 생성 단계가 느림
```

이 스텁 생성이 kapt의 병목이다. 실제로는 필요 없는 정보까지 담은 Java 소스를 만들고, `javac`를 한 번 더 돌리는 비용이 든다. 그래도 kapt는 Java 프로세서 생태계 전체와 호환된다는 큰 장점이 있어 오래 표준으로 쓰였다. 현재 kapt는 유지보수 모드에 가깝고, 새 코드에는 아래의 KSP가 권장된다.

### 8.3 KSP — Kotlin을 직접 이해하는 새길

**KSP(Kotlin Symbol Processing)** 는 스텁 우회를 없앤다. KSP는 Java 소스 모델이 아니라 **Kotlin의 해소된 심볼(symbol)을 직접** 프로세서에 노출한다. 프로세서는 Kotlin 함수·프로퍼티·타입을 있는 그대로 순회하며 코드를 생성한다.

| | kapt | KSP |
|---|---|---|
| 동작 방식 | Java 스텁 생성 → javac APT | Kotlin 심볼 API 직접 제공 |
| 속도 | 느림(스텁+javac 왕복) | 빠름(스텁 없음) |
| Kotlin 이해도 | Java로 투영된 근사 | 널 가능성·`suspend` 등 Kotlin 개념 그대로 |
| 생태계 | 기존 Java 프로세서 호환 | KSP용으로 작성/이식된 프로세서 |
| 권장도(현재) | 유지보수 모드 | 신규 권장 |

KSP가 빠른 근본 이유는 파이프라인상 위치다. KSP는 프론트엔드가 이미 해소해 둔 심볼 정보를 재사용하므로, Java 스텁을 만들고 `javac`를 다시 돌리는 중복 작업이 없다. 또 KSP 프로세서는 `String?`과 `String`을 구분하고 `suspend` 함수를 인식하는 등, Java로 투영하면 사라졌을 Kotlin 고유 정보를 그대로 본다. 다만 KSP는 프로세서가 KSP API로 (재)작성돼야 하므로, 생태계 전환이 필요했다.

리플렉션([[44 - 애노테이션과 리플렉션]] 소유)과의 관계도 짚어 둘 만하다. KSP·kapt는 **컴파일 타임 코드 생성**이고, 리플렉션은 **런타임 타입 조사**다. 같은 목적(애노테이션 기반 동작)을 컴파일 타임에 정적으로 풀면 KSP, 런타임에 동적으로 풀면 리플렉션이다. 전자는 시작이 빠르고 안전하지만 유연성이 덜하고, 후자는 그 반대다. 이 트레이드오프는 [[44 - 애노테이션과 리플렉션]]가 정본으로 다룬다.

## 9. 증분 컴파일과 빌드 — 왜 두 번째 빌드는 빠른가

### 9.1 전부 다시 컴파일하지 않기

작은 프로젝트도 파일이 수백 개다. 한 파일을 고칠 때마다 전부 다시 컴파일하면 개발 반복이 느려진다. **증분 컴파일(incremental compilation)** 은 "바뀐 것과, 그 변경에 영향받는 것만" 다시 컴파일한다. 핵심 질문은 "무엇이 영향받는가"를 어떻게 아느냐다.

컴파일러(+빌드 도구)는 이전 컴파일의 산출물과 함께, 파일들 사이의 **의존 관계**를 기록해 둔다. 어떤 파일이 바뀌면, 그 파일이 노출하는 API(공개 시그니처)가 바뀌었는지 본다. 만약 함수 *본문*만 바뀌고 공개 시그니처는 그대로라면, 그 파일을 참조하는 다른 파일들은 다시 컴파일할 필요가 없다 — 그들이 의존하는 계약(시그니처)이 안 변했으니까. 반대로 공개 시그니처가 바뀌면(파라미터 추가, 반환 타입 변경 등), 그 선언을 쓰는 모든 파일이 다시 컴파일 대상(dirty)이 된다.

```text
Foo.kt의 private 헬퍼 본문만 수정
  → Foo.kt만 재컴파일, 소비자는 그대로

Foo.kt의 public 함수 시그니처 변경
  → Foo.kt + 그 함수를 부르는 모든 파일 재컴파일
```

### 9.2 ABI — 다시 컴파일을 촉발하는 경계

이 판단의 기준이 **ABI(Application Binary Interface)**, 즉 "다른 모듈/파일이 의존할 수 있는 공개 표면"이다. 공개 함수 시그니처, 공개 프로퍼티 타입, 클래스 계층 등이 ABI에 속한다. private 구현 세부는 ABI가 아니다. 증분 컴파일은 "ABI가 바뀌었는가"를 기준으로 파급 범위를 정한다.

7.1절의 `const val` 인라이닝이 여기서 다시 문제가 된다. `const val`의 값은 소비자의 바이트코드에 박히므로, 그 값을 바꾸면 시그니처는 그대로여도 소비자를 다시 컴파일해야 옳은 값이 반영된다. 증분 컴파일은 이런 "값 수준 의존성"도 추적해야 한다. 이는 증분 컴파일이 단순히 "시그니처만 보면 되는" 문제가 아님을 보여준다.

> **성능 주의**: 증분 컴파일의 이점을 극대화하려면 공개 API를 자주 흔들지 않는 게 좋다. 널리 쓰이는 유틸 파일의 public 시그니처를 자주 바꾸면 매번 대규모 재컴파일이 유발된다. 반대로 구현을 `private`/`internal`로 잘 감싸 두면, 그 내부를 아무리 고쳐도 파급이 그 파일(또는 모듈)에 머문다. 캡슐화([[26 - 가시성 한정자]])는 설계 미덕일 뿐 아니라 빌드 성능 전략이기도 하다.

### 9.3 컴파일 산출물과 메타데이터

JVM 백엔드의 산출물은 `.class` 파일들이지만, 그 안에는 Java에는 없는 정보도 실린다. Kotlin 컴파일러는 각 클래스 파일에 `@Metadata` 애노테이션을 심어, 널 가능성·프로퍼티·기본 인자·`internal` 원래 이름 등 **Java 바이트코드로는 표현되지 않는 Kotlin 고유 정보**를 보존한다. 다른 Kotlin 모듈이 이 클래스를 라이브러리로 참조할 때, 컴파일러는 이 메타데이터를 읽어 원래의 Kotlin 시그니처를 복원한다.

이 메타데이터가 있기에, 컴파일된 라이브러리를 쓰는 쪽에서도 널 안전성·기본 인자 같은 기능이 그대로 작동한다. 만약 메타데이터가 없다면(순수 Java 라이브러리를 쓸 때가 그렇다), 컴파일러는 널 가능성을 알 수 없어 **플랫폼 타입 `T!`** 로 취급하게 된다(M8, [[06 - 널 안전성 - nullable와 스마트 캐스트와 플랫폼 타입]]·[[45 - Java 상호운용]] 소유). 즉 "Kotlin 라이브러리는 널 안전한데 Java 라이브러리는 안 그런" 이유가 바로 이 메타데이터의 유무다.

## 10. 디컴파일 — 진실을 직접 확인하는 법

### 10.1 왜 눈으로 봐야 하는가

이 장의 주장은 대부분 "컴파일러가 X를 한다"는 형태다. 데이터 클래스가 `equals`를 생성한다, `const`가 값을 박는다, 인라인이 람다 객체를 없앤다. 이런 주장을 믿는 가장 확실한 방법은 컴파일 결과를 직접 보는 것이다. Kotlin의 실용주의 중 하나는 이 검증을 쉽게 만들어 뒀다는 점이다.

가장 흔한 경로는 IntelliJ IDEA의 **Tools → Kotlin → Show Kotlin Bytecode**다. 이것은 현재 파일의 JVM 바이트코드를 보여주고, 그 창의 **Decompile** 버튼을 누르면 그 바이트코드를 다시 *Java 소스로* 역번역해 보여준다. 즉 "내 Kotlin 코드가 어떤 Java에 해당하는가"를 근사적으로 볼 수 있다. 명령줄에서는 컴파일한 `.class`에 `javap -c`를 돌려 원시 바이트코드 명령을 볼 수도 있다.

> **역사 메모**: 이 "디컴파일해서 확인" 문화는 Kotlin 학습의 오랜 관습이다. 초창기부터 "이 문법 설탕이 실제로 뭐가 되는지"를 Java로 역번역해 이해하는 방식이 널리 쓰였다. 다만 주의: 디컴파일 결과는 **JVM 백엔드**의 산출물을 근사한 것이며, 컴파일러 버전에 따라 세부(합성 클래스 이름, 브릿지 메서드 유무 등)가 달라질 수 있다. 개념 확인용이지 명세가 아니다(M2 — 이 결과는 다른 백엔드에는 해당하지 않는다).

### 10.2 데이터 클래스를 디컴파일하면

```kotlin
data class Point(val x: Int, val y: Int)
```

이 한 줄을 디컴파일하면(개념적 Java 등가물), 컴파일러가 우리 대신 써 준 코드가 드러난다.

```java
public final class Point {
    private final int x;
    private final int y;
    public Point(int x, int y) { this.x = x; this.y = y; }
    public final int getX() { return this.x; }
    public final int getY() { return this.y; }
    public final int component1() { return this.x; }   // 구조 분해용
    public final int component2() { return this.y; }
    public final Point copy(int x, int y) { return new Point(x, y); }
    public String toString() { return "Point(x=" + x + ", y=" + y + ")"; }
    public int hashCode() { return this.x * 31 + this.y; }   // 개념적
    public boolean equals(Object other) { /* x,y 비교 */ }
}
```

여기서 5.2절과 [[27 - 데이터 클래스]]의 주장들이 눈으로 확인된다. `component1`/`component2`가 구조 분해([[20 - 구조 분해]])를 떠받치고, `copy`가 새 인스턴스를 만드는 평범한 메서드일 뿐임(그래서 얕은 복사, M22)이 보이고, `equals`/`hashCode`가 **주 생성자 프로퍼티만** 쓴다는 것(M21 — body에 선언한 프로퍼티는 여기 안 들어온다)이 드러난다. "마법"이라 부르던 것이 실은 생성된 평범한 코드다.

### 10.3 인라인과 const를 디컴파일하면

7절의 주장도 확인할 수 있다. 다음을 컴파일해 소비 함수를 디컴파일하면:

```kotlin
const val LIMIT = 10
inline fun repeatTwice(action: () -> Unit) { action(); action() }

fun demo() {
    val x = LIMIT              // 디컴파일: int x = 10;  ← 상수가 박힘
    repeatTwice { println("!") }  // 디컴파일: println("!"); println("!");  ← 펼쳐짐, 람다 객체 없음
}
```

`LIMIT`이 있던 자리에 `10`이 박히고, `repeatTwice` 호출이 사라지고 본문이 두 번 펼쳐지며, 람다를 담을 함수 객체가 아예 안 만들어진 것을 볼 수 있다. 반대로 `inline`을 떼고 같은 코드를 디컴파일하면, `repeatTwice`가 호출로 남고 람다가 `Function0` 구현 객체로 만들어지는 것이 보인다. 이 대조가 "인라인이 무엇을 없애는가"를 가장 선명하게 가르쳐 준다.

### 10.4 검증하는 습관이 곧 이해다

디컴파일은 단순한 요령이 아니라 인식론적 태도다. "컴파일러가 이렇게 할 것이다"라는 추측을 "이렇게 한다"는 관찰로 바꾼다. 성능이 궁금할 때(무엇이 할당되는가), 상호운용이 궁금할 때(Java에서 이 함수 이름이 뭔가), 언어 기능의 실체가 궁금할 때(이 애노테이션이 정말 코드를 생성했는가) — 답은 대개 컴파일 결과에 있다. 이 습관은 이 시리즈 전체에서 되풀이 쓰일 것이다. 앞으로 어떤 장에서 "컴파일러가 X를 한다"고 말할 때, 독자는 그것을 직접 열어 볼 수 있어야 한다.

---

우리는 소스 텍스트 한 조각이 렉서에서 토큰이 되고, 파서에서 구문 트리가 되며, K2/FIR 프론트엔드가 이름을 해소하고 타입을 추론하고 진단을 뱉어 "완전히 해소된 의미 트리"를 만들고, 그것이 공통 IR로 낮춰져 네 백엔드로 갈라지고, JVM 백엔드에서 마침내 `FooKt`와 `.class`들로 응결되는 전 과정을 따라왔다. 그 여정에서 도입부의 질문들이 하나씩 회수됐다. "Kotlin 컴파일러"는 프론트엔드와 여러 백엔드가 IR을 이음매로 협력하는 파이프라인이었고, K2는 디스크립터+BindingContext의 옛 세계를 하나의 진화하는 FIR 트리로 대체한 재설계였다. `inline`·`reified`·`const`는 런타임 마법이 아니라 컴파일 타임에 코드로 풀려 흔적을 지우는 변환이었고, `FooKt`는 "클래스만 있다"는 JVM의 제약을 만족시키려 컴파일러가 만든 컨테이너였다. 그리고 이 모든 주장은 디컴파일이라는 단순한 도구로 직접 검증 가능하다는 것 — 이것이 이 장이 다음 장들에 물려주는 가장 실용적인 습관이다. 이제 우리는 이 기계가 가장 먼저 읽는 것, 즉 텍스트의 어휘 규칙 자체로 내려간다. [[03 - 렉시컬 구조와 세미콜론 추론]]에서 렉서가 실제로 어떤 규칙으로 토큰을 자르고, 왜 Kotlin이 세미콜론 없이도 문장의 끝을 아는지를 본다.

## 핵심 요약

- **컴파일은 단일 도약이 아니라 파이프라인이다.** 렉서(문자→토큰) → 파서(토큰→구문 트리) → 프론트엔드(이름 해소·타입 추론·진단) → IR(공통 중간 표현) → 백엔드(타깃 코드)의 여러 단계로, 각 단계는 앞의 산출물을 더 낮은 표현으로 끌어내린다.
- **"Kotlin 컴파일러"는 단수가 아니다.** 프론트엔드는 하나(타깃 독립적)지만 백엔드는 JVM·JS·Native·Wasm으로 넷이다. "컴파일된다"는 서술은 항상 어느 백엔드인지 한정해야 하며, 원시 타입 컴파일은 JVM 백엔드의 결정이다(M2, M4).
- **K2/FIR은 프론트엔드의 재설계다.** 옛 K1의 디스크립터+BindingContext 이중 구조 대신, 단일 FIR 트리를 다단계 해소 패스로 점진적으로 채워 완전히 해소된 의미 트리를 만든다. 2.0에서 기본이 됐다.
- **널 안전성·스마트 캐스트·완전성 검사는 프론트엔드의 컴파일 타임 사건이다.** 런타임에 남는 널 검사는 `!!`·플랫폼 타입·리플렉션·Java 상호운용처럼 컴파일러가 확신할 수 없는 지점뿐이다(M7).
- **타입 추론은 생략이 아니라 제약 해결이다.** 리터럴의 타입은 기대 타입이라는 문맥이 되먹여 정하고(`val x: Long = 42`), 변수 사이의 암시적 확대는 없으며, 재귀·공개 시그니처는 명시를 요구한다.
- **IR은 프론트엔드와 백엔드의 이음매다.** 하나의 IR을 로워링 패스로 낮추면 데이터 클래스 합성·기본 인자·`when` 정규화 같은 공통 변환을 한 번만 구현해 네 타깃이 공유한다. 우리가 "언어 기능"이라 부르는 많은 것이 실은 컴파일러가 IR 단계에서 생성한 코드다.
- **`.kt`와 `.class`는 1:1이 아니다.** 파일 최상위 선언은 `파일명+Kt` 클래스에 담기고(JVM 백엔드), 한 파일의 여러 클래스·`object`·람다·enum은 각기 별개 `.class`를 낳는다. `FooKt`는 마법이 아니라 "클래스만 있다"는 JVM 제약의 산물이다.
- **컴파일 타임에 사라지는 것들이 있다.** `const val`은 호출처에 값이 박히고(M36), `inline`은 함수 본문과 람다를 호출처에 펼쳐 함수 객체 할당을 없애며(M17), `reified`는 인라인 치환으로 소거된 타입을 호출처마다 되살린다 — 단 최상위 타입만(M18, M19).
- **컴파일러 플러그인은 언어를 안 바꾸고 컴파일러를 확장한다.** all-open은 `final`을 `open`으로 여닫고, kapt는 Java 스텁을 거쳐 느리며, KSP는 Kotlin 심볼을 직접 노출해 빠르고 Kotlin 고유 정보(널 가능성·`suspend`)를 보존한다.
- **증분 컴파일은 ABI 변경을 기준으로 파급을 정한다.** 본문만 바뀌고 공개 시그니처가 그대로면 소비자를 재컴파일하지 않지만, `const val` 값처럼 소비자 바이트코드에 박히는 것은 값 수준 의존성으로 추적해야 한다. 캡슐화는 빌드 성능 전략이기도 하다.
- **`.class`에는 `@Metadata`로 Kotlin 고유 정보가 실린다.** 이 덕에 컴파일된 Kotlin 라이브러리는 널 안전성·기본 인자를 보존하지만, 메타데이터 없는 순수 Java 라이브러리는 플랫폼 타입 `T!`로 취급된다(M8).
- **디컴파일은 추측을 관찰로 바꾸는 검증 습관이다.** IDEA의 Show Kotlin Bytecode → Decompile 또는 `javap -c`로, 데이터 클래스 합성·상수 인라이닝·람다 소거를 직접 눈으로 확인할 수 있다. 이 습관이 이 시리즈 전체의 방법론이다.
